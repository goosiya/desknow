# Story 1.1: 외부 서비스 가입 & 사전 설정 (선행 — 사람이 직접 수행)

Status: done

## Story

As a 프로젝트 소유자(KTH),
I want 구현 착수 전에 필요한 모든 외부 서비스에 가입하고 API 키·과금·플랫폼 등록을 직접 완료한 뒤, 그 값을 백엔드가 안전하게 로드·검증하도록 정리하길,
so that 이후 모든 개발 스토리가 외부 의존성 차단 없이 진행되고, 키는 프론트에 노출되지 않고 백엔드에서만 관리된다(NFR-6).

> ⚠️ **이 스토리는 "사람만" 수행 가능한 선행 작업이 핵심이다.** dev 에이전트는 가입·결제·플랫폼 등록을 **대신 수행하지 않는다.** dev 에이전트의 역할은 정확히 다음 둘이다:
> 1. **준비물 생성** — 소유자용 체크리스트 문서, `apps/api/.env.example`(키 이름·용도·획득 출처 문서화), 루트 `.gitignore`, 키 로딩 코드(`apps/api/app/core/config.py` — pydantic-settings `Settings`), 그리고 키 누락 시 fail-fast 하는 진단/검증 수단.
> 2. **로드/검증 확인** — 소유자가 채운 `.env` 값으로 백엔드(또는 최소 진단 스크립트)가 필수 키를 정상 로드하고, 누락 시 명확한 오류로 기동 실패함을 확인.
>
> **이 스토리가 완료되기 전 Phase 0 스파이크(1.3)·챗봇(E7)·배포는 시작할 수 없다.**

## Acceptance Criteria

1. **(인간)** 아래 "필요 서비스 체크리스트"의 각 서비스에 대해 소유자가 가입·키 발급·과금 설정을 완료하면, 발급된 모든 키·시크릿이 `apps/api/.env`에 채워진다.
2. **(dev)** `apps/api/.env.example`에 모든 필수/선택 키가 **키 이름·용도·획득 출처(어느 서비스 콘솔에서 받는지)** 주석과 함께 문서화된다. 실제 비밀 값은 절대 포함하지 않는다(플레이스홀더만).
3. **(dev)** 어떤 키도 프론트 코드(`apps/web`·`apps/admin`·`apps/mobile`)나 저장소에 커밋되지 않는다 — 루트 `.gitignore`가 `.env`(및 변형)·시크릿 파일을 제외하고, `.env.example`만 추적된다.
4. **(dev)** `core/config`(pydantic-settings)로 백엔드(또는 최소 진단 스크립트)를 기동했을 때, **필수 키**(카카오 REST·OpenAI·임베딩 모델명 등)가 누락되면 **어떤 키가 누락됐는지 명시하는 명확한 오류 메시지와 함께 기동이 실패**한다(fail-fast).
5. **(dev)** 모든 필수 키가 존재하면 정상 기동되고, **키 존재 여부를 확인하는 진단/헬스 출력**(값 자체는 마스킹)이 제공된다.
6. **(인간+문서)** 카카오맵 약관의 서비스 유형 적합성 1차 확인 결과가 기록된다. **부적합으로 판명되면** 그 사실과 대안(지도 제공자 대체 검토) 필요성이 문서에 남아 후속 스토리(E3) 진행 판단에 반영된다.
7. **(인간)** 지도 JS SDK 도메인 키는 허용 도메인 화이트리스트로 제한되어, 프론트에 노출되는 유일한 키(도메인 키)도 타 도메인에서 무단 사용되지 않는다(NFR-6).

## Dev Notes

### 이 스토리의 본질과 경계

- **인간 게이트 + dev 산출물 혼합 스토리.** AC 1·6·7은 소유자가 수행/확인하는 항목이고, dev 에이전트는 **AC 2·3·4·5의 산출물(문서·`.env.example`·`.gitignore`·`config.py`·진단)** 을 만들고 검증한다. 가입/결제/콘솔 등록을 코드로 자동화하려 하지 말 것.
- **키 격리는 이 스토리의 존재 이유다(NFR-6).** 프론트에 나가는 키는 **카카오 JS 도메인 키 단 하나**이며, 이마저도 도메인 화이트리스트로 보호된다. 나머지(카카오 REST·OpenAI·Anthropic·Google·Railway DB)는 **백엔드 환경변수 전용**. `.env.example`/문서/코드 어디에도 실제 비밀을 남기지 말 것.
  [Source: prd.md#보안(L623), architecture.md#Authentication & Security(L164)]

### ⚠️ 결정적 시퀀싱 변수 — 스캐폴드는 아직 없다

- 현재 리포에는 **`apps/` 디렉터리가 존재하지 않는다**(확인 완료). 모노레포 + FastAPI 골격(`uv init`, FastAPI/uvicorn 등 풀 의존성)은 **Story 1.2에서 생성**된다. 그러나 sprint 순서상 **1.1이 1.2보다 먼저** 실행된다.
- **권고 접근(채택):** dev 에이전트는 산출물을 **최종 정규 경로**(`apps/api/.env.example`, `apps/api/app/core/config.py`, 루트 `.gitignore`, `docs/external-services-setup.md`)에 생성한다. 부모 디렉터리는 필요 시 생성. 키 로딩 검증을 위해 `apps/api`에 **최소 파이썬 환경**만 부트스트랩한다(예: `uv init` + `uv add pydantic-settings` 만, 또는 단일 검증 스크립트). **FastAPI/uvicorn/SQLModel 등 풀 의존성 설치는 1.2의 책임**이며 여기서 중복 설치하지 말 것.
- Story 1.2가 동일 `apps/api/app/core/config.py`를 그대로 재사용·확장하도록 설계할 것(1.2의 AC가 `core/`에 `config(pydantic-settings)`를 포함 — L326). **이 스토리에서 만든 `Settings`가 1.2 골격의 config 원천이 된다.**
- AC 4·5의 "백엔드를 기동하면" 요구는 풀 FastAPI 기동이 아니라 **`Settings` 인스턴스화 시점의 fail-fast**로 충족 가능하다. pydantic-settings는 인스턴스 생성 시 필수 필드 누락이면 즉시 `ValidationError`를 던지므로, 최소 진단 스크립트로도 동일 보증을 실증할 수 있다. 1.2에서 `main.py`가 `get_settings()`를 호출하면 동일 fail-fast가 앱 기동에 자연 연결된다.

### Testing standards

- 백엔드 테스트는 `apps/api/tests/` **미러 구조 + pytest**. 단, 이 스토리는 풀 스캐폴드 전이라 정식 pytest 스위트는 1.2/1.4 이후가 자연스럽다.
- **최소 검증 의무:** Task 6의 두 시나리오(누락 시 fail-fast / 충족 시 정상+마스킹 출력)를 실제 실행하고 **출력을 Completion Notes에 캡처**한다. 가능하면 `tests/core/test_config.py`로 (a) 필수 키 누락 시 `ValidationError`, (b) 모든 필수 키 존재 시 정상 인스턴스화, (c) 진단 출력이 비밀 값을 평문 노출하지 않음(마스킹) 을 검증하는 단위 테스트를 추가(pytest 환경이 부트스트랩 가능할 때).
  [Source: architecture.md#Structure Patterns(L253)]

### 흔한 실수 방지 (anti-patterns)

- ❌ 가입/결제/콘솔 등록을 스크립트로 자동화 시도 → **하지 말 것.** 인간 작업이다.
- ❌ `.env.example`이나 문서·코드에 실제 키 값 기입 → 절대 금지(플레이스홀더만).
- ❌ 프론트(`apps/web`/`admin`/`mobile`)에 도메인 키 외 키 노출 → 금지. 도메인 키도 화이트리스트 전제.
- ❌ 1.2의 풀 스캐폴드(FastAPI/Next/Expo 등)를 여기서 미리 생성 → 범위 밖. config 검증에 필요한 최소만.
- ❌ 진단 출력에 비밀 값 평문 노출 → 반드시 마스킹.
- ❌ 필수 키에 기본값 부여(누락을 숨김) → fail-fast가 깨진다. 필수는 기본값 없는 필드로.

### Project Structure Notes

- **정합:** 산출물 경로(`apps/api/.env.example`, `apps/api/app/core/config.py`)는 아키텍처 디렉터리 구조와 정확히 일치한다.
- **의도된 변이(rationale):** 본래 `apps/api` 풀 스캐폴드는 Story 1.2 소관이나, sprint 순서가 1.1을 먼저 두므로 **1.1은 키 검증에 필요한 최소 파이썬 부트스트랩만** 수행하고 나머지 의존성·앱 구조는 1.2가 채운다. 1.1이 만든 `config.py`/`.env.example`은 1.2가 재사용한다. (1.2 AC가 `core/`에 config 포함을 명시 — L326.)

## Latest Tech Information

- **pydantic-settings (v2.x, 2026-06 기준 안정):** `BaseSettings`는 `pydantic`에서 분리되어 `pydantic-settings` 패키지로 제공. v2 패턴은 `class Settings(BaseSettings): model_config = SettingsConfigDict(env_file=".env", env_file_encoding="utf-8", extra="ignore")`. 필드에 기본값을 주지 않으면 환경/`.env`에 없을 때 인스턴스 생성 시 `ValidationError` → **자연스러운 fail-fast**. 여러 필수 키 누락을 한 번에 모아 보고하려면 `Settings()` 호출을 try/except로 감싸 `ValidationError.errors()`를 사람이 읽기 좋게 출력.
- **uv (Astral):** `uv init`로 `pyproject.toml` 생성, `uv add pydantic-settings`로 최소 의존성. 풀 백엔드 의존성(fastapi, uvicorn[standard], sqlmodel, psycopg[binary], pyjwt, pwdlib[argon2], pgvector)은 1.2에서 추가 — 여기서는 중복 추가 금지.
- 정확한 최신 버전 핀은 1.2 스캐폴드 시점에 재확인(아키텍처: "버전은 init 시점 최신으로 재확인", L118). 이 스토리는 버전 고정 책임이 아니므로 핀 강제 불필요.
