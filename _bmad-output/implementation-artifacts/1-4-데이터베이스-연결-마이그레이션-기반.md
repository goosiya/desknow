## Story

As a 개발자,
I want PostgreSQL + pgvector 연결과 Alembic 마이그레이션 기반, 핵심 설정 모듈(`core/db`)을 갖추길,
so that 이후 스토리(1.7 users, 2.1 rooms, 4.1 reservations, 7.2 pgvector 등)가 자신에게 필요한 테이블만 마이그레이션으로 점진 추가할 수 있다.

## Acceptance Criteria

1. **(AC1 — 연결 & pgvector & fail-fast)** `core/config`와 `core/db`가 갖춰진 상태에서 백엔드를 기동하면, SQLModel 엔진이 PostgreSQL에 연결되고 pgvector 확장이 활성화(존재 보장)되며, **연결 실패 시 명확한 오류로 fail-fast** 한다.
2. **(AC2 — Alembic 베이스라인 & 점진 확장)** `alembic upgrade head`를 실행하면 마이그레이션이 적용되고, **빈 베이스라인(도메인 테이블 0개)에서 시작**해 이후 스토리가 점진적으로 테이블을 추가할 수 있다. **이 스토리는 모든 테이블을 미리 만들지 않는다.**
3. **(AC3 — 네이밍 규약)** DB/컬럼/제약 네이밍은 아키텍처 규약을 따른다 — 복수 snake_case 테이블, `*_at`는 UTC `timestamptz`, 제약은 `uq_{table}_{cols}`/`idx_{table}_{cols}`/`fk_…`/`pk_…`/`ck_…`. 규약이 메타데이터에 등록되어 이후 모든 마이그레이션이 자동으로 따른다.

> **AC1 "pgvector 활성화"의 책임 소재(이 스토리의 명시적 결정):** 확장 활성화(`CREATE EXTENSION IF NOT EXISTS vector`)는 **Alembic 베이스라인 마이그레이션이 단일 출처로 소유**한다(스키마 관리의 정공법, 최소권한 — 런타임 롤이 `CREATE EXTENSION` 권한을 가질 필요 없음). 백엔드 **기동 시 `core/db`는 연결 확인(`SELECT 1`) + `vector` 확장 존재 검증**을 수행하고, 미존재 시 "`alembic upgrade head`를 먼저 실행하라"는 **명확한 actionable 오류로 fail-fast** 한다. → AC1의 "활성화"는 베이스라인 마이그레이션이, "연결+fail-fast"는 startup 검증이 충족한다.

### 정확한 네이밍 규약 dict (AC3 — 그대로 사용)

`app/core/db.py`에서 모델 정의 이전에 설정. 아키텍처 §Naming Patterns(`uq_{table}_{cols}`, `idx_{table}_{cols}`)에 정렬:

```python
NAMING_CONVENTION = {
    "ix": "idx_%(table_name)s_%(column_0_N_name)s",
    "uq": "uq_%(table_name)s_%(column_0_N_name)s",
    "ck": "ck_%(table_name)s_%(constraint_name)s",
    "fk": "fk_%(table_name)s_%(column_0_name)s_%(referred_table_name)s",
    "pk": "pk_%(table_name)s",
}
SQLModel.metadata.naming_convention = NAMING_CONVENTION
```

- 아키텍처가 인덱스를 `idx_`, UNIQUE를 `uq_`로 명시 → SQLAlchemy 기본 `ix_`가 아닌 `idx_`로 오버라이드. 예: `uq_reservations_room_slot`(architecture.md L233).
- `%(column_0_N_name)s`는 다중 컬럼 제약(예 `UNIQUE(room_id, slot_start)`)을 `uq_reservations_room_id_slot_start`로 펼침. 단일 출처 규약이라 후속 스토리가 제약명을 손으로 짓지 않아도 된다.
- 시각 컬럼: `*_at`은 UTC `timestamptz`(architecture.md L234) — 후속 스토리 모델이 `Field(sa_column=Column(TIMESTAMP(timezone=True)))` 또는 SQLModel 기본 `datetime`+timezone로 강제. 이 스토리는 테이블이 없어 강제 대상 없음(규약 문서화만).

### `DATABASE_URL` — 개발 DB는 Supabase (provider-neutral, 이관은 배포 시점)

> **이 스토리는 DB provider에 무관하다.** core/db 엔진·Alembic 마이그레이션은 PostgreSQL+pgvector이기만 하면 Supabase든 Railway든 동일하게 돈다. **개발 단계 `DATABASE_URL`은 Supabase를 가리킨다**(소유자 선택). **Supabase→Railway 실데이터 이관은 배포 시점**의 별도 작업이며 1.4 범위 밖(1.3에서 경로만 실증, 충실도 검증은 실 이관 스토리로 이월).

- **스킴**: SQLAlchemy 기본 `postgresql://`는 psycopg2를 찾는다. 우리는 `psycopg[binary]`(v3, pyproject L13)이므로 **`postgresql+psycopg://`** 가 정답. Supabase/Railway 콘솔 문자열은 보통 `postgresql://…`(일부 `postgres://`) → config 검증기에서 자동 정규화(소유자 편의 + 드라이버 정합).
- **⚠️ Supabase 연결 모드 함정**: Supabase는 ① **직접 연결**(`db.<ref>.supabase.co:5432`) ② **Session pooler**(`…pooler.supabase.com:5432`) ③ **Transaction pooler**(`…pooler.supabase.com:6543`)를 제공한다. **Alembic 마이그레이션·psycopg3는 직접 연결 또는 Session pooler(5432) 사용** — Transaction pooler(6543)는 prepared statement/세션 상태를 끊어 마이그레이션·일부 psycopg3 기능이 깨진다. README/`.env.example` 주석에 "6543(transaction pooler) 금지, 5432 사용" 명시. (IPv4 환경이면 Session pooler가 안전.)
- **1.3 실증(이관 경로, 참고용)**: 소스 **Supabase(PG17.6)** → 타깃 **Railway(PG18.4, pgvector 0.8.2)**, `CREATE EXTENSION vector` 선실행 후 복원·HNSW 생존(deferred-work.md L13-16). PG14+ 필요 충족 — Supabase·Railway 둘 다 만족.

### pgvector — 이 스토리에서 / 후속 스토리에서

- **이 스토리**: 확장 활성화만(테이블·컬럼 없음). 베이스라인 마이그레이션 `op.execute("CREATE EXTENSION IF NOT EXISTS vector")`.
- **후속(7.2)**: 벡터 컬럼은 `from pgvector.sqlalchemy import Vector` → `Field(sa_type=Vector(1536))`(차원 1536 = `text-embedding-3-small`, 1.3에서 고정). 그 마이그레이션 파일은 `from pgvector.sqlalchemy import Vector` import 필요(SQLAlchemy 미네이티브 타입 — 검색결과 GitHub alembic#1324·pgvector-python 확인).

### 흔한 실수 방지 (anti-patterns)

- ❌ `SQLModel.metadata.create_all()` 사용 → AC2 위반(스키마는 Alembic 단독). 마이그레이션만.
- ❌ 이 스토리에서 도메인 테이블(users/rooms/…) 생성 → 범위 밖. 빈 베이스라인만.
- ❌ `postgresql://`(psycopg2 가정) 그대로 엔진에 전달 → 드라이버 불일치. `+psycopg`로 정규화.
- ❌ `alembic.ini`에 `sqlalchemy.url` 하드코딩(비밀 노출) → env.py가 settings에서 주입.
- ❌ `with TestClient(app)`로 기존 test_main 변경 → lifespan DB 검증 트리거, CI 회귀.
- ❌ 연결 실패 시 raw `OperationalError` 그대로 노출(스택만, 안내 없음) → 명확 한국어 오류로 래핑(단 비밀번호 마스킹).
