# Story 2.2: 공간 등록 & 주소 검색

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 제공자,
I want 공간 정보를 입력하고 주소 검색으로 위치를 확정해 저장하길,
So that 등록 한 번으로 내 공간이 지도/목록에 노출된다 (FR-22, UJ-3, SM-5).

## Acceptance Criteria

**AC1 — 공간 등록(provider 인증 쓰기)**
**Given** 제공자 계정으로 로그인한 상태
**When** 이름·시간당 금액·영업시간·주소·수용 인원·룸 형태(개방형/독립룸)·부대시설(주차·빔프로젝터/TV·커피머신·화이트보드·wifi·기타 다중 선택)을 입력하고 저장하면
**Then** 룸이 생성되어(`is_active=true`로 시작) 지도/목록 조회의 데이터로 노출 가능해진다.

**AC2 — 주소 검색(카카오 지오코딩) → 좌표 + 행정동 코드 저장**
**Given** 주소 검색(카카오 지오코딩)
**When** 주소를 선택하면
**Then** 좌표(lat/lng)와 행정동 코드가 함께 저장되어 반경 검색(FR-9)과 행정동 검색(FR-8)을 모두 지원한다. **REST 키는 백엔드 전용**이라 지오코딩은 반드시 백엔드 프록시를 경유한다(NFR-6 — 프론트 직접 호출 금지).

**AC3 — 영업시간 같은 날 내(자정 넘김 거부)**
**Given** 영업시간 입력
**When** 자정을 넘기는 시간(예: 22:00~02:00)을 입력하면
**Then** 같은 날 내(0~24시) 제약으로 거부된다(친절한 422 + DB CHECK 이중 방어).

**AC4 — 제공자당 1개 제약**
**Given** 이미 룸을 1개 보유한 제공자
**When** 추가 등록을 시도하면
**Then** MVP 제약(제공자당 1개)으로 차단된다(409 — 친절한 선검사 + DB UNIQUE 이중 방어).

**AC5 — 5분 이내 등록 흐름 + 권한 강제**
**And** 등록 흐름은 제공자가 5분 이내 마칠 수 있도록 백엔드 계약이 단순하고(단일 `POST` + 주소검색), `provider` 역할만 등록 가능하다(booker/미인증 차단). *지도에서 자기 핀 확인은 E3 지도 화면 소유 — 본 스토리는 그 데이터(좌표/가용성 입력)를 확립한다.*

**AC6 — Epic 1 회고 deferred 회수(P2·P3) — 출처: deferred-work.md 회수 라우팅 결정**
**Given** rooms insert 쓰기 경로(본 스토리가 처음 도입)와 provider 권한으로의 쓰기
**When** 쓰기 서비스/마이그레이션을 구현하면
**Then** **(P2)** `IntegrityError` 포괄 캐치를 **제약명 식별 후 선별 변환**으로 교체하고(무관 위반은 re-raise), **rooms 신규 코드와 1.7 `auth.register_user`를 동일 패턴으로 정합화**한다. **(P3)** `rooms.room_type`·`users.role` 두 enum 컬럼에 **DB CHECK 제약**을 한데 묶어 부여하고(자유 문자열 → defense-in-depth), 검증은 Pydantic `Literal` 스키마(1.7 `role` 패턴)로 1차 차단한다. 또한 2.1 code-review에서 defer된 **rooms 값 범위 CHECK**(price/capacity 음수·lat/lng 범위 밖)를 본 쓰기 경로 도입과 함께 회수한다.

### 권장 구현 관용구 (실측 대조 — 프리플라이트 ④)

- **provider RBAC 라우트**(1.8 `require_role` 첫 실소비):
  ```python
  @router.post("/", response_model=RoomPublic, status_code=201,
      responses={401: {"model": ErrorResponse}, 403: {"model": ErrorResponse},
                 409: {"model": ErrorResponse}, 422: {"model": ErrorResponse}})
  def create_room(data: RoomCreateRequest,
      principal: AuthPrincipal = Depends(require_role("provider")),
      session: Session = Depends(get_session)) -> Room:

  ```

- **제약명 추출(P2, ④a 실측)** — `app/core/db.py`:
  ```python
  def violated_constraint(exc: IntegrityError) -> str | None:
      diag = getattr(getattr(exc, "orig", None), "diag", None)
      return getattr(diag, "constraint_name", None) if diag is not None else None
  ```
  실 위반(중복 provider_id)을 일으켜 `diag.constraint_name == "uq_rooms_provider_id"`를 실측 확인.

- **단일 트랜잭션 다중 insert**(`create_room`): `session.add(room)` + 각 `session.add(BusinessHours(room_id=room.id, ...))` 후 단일 `commit`. `room.id`는 `default_factory=uuid.uuid4`라 commit 전 이미 존재(FK 연결 가능).
