# Story 2.3: 공간 수정 & 영업시간 변경 독립성

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 제공자(provider),
I want 내가 등록한 공간 정보와 요일별 영업시간을 수정하길,
so that 운영 상황에 맞춰 정보를 갱신하되 **기존에 확정된 예약은 영향받지 않는다** (FR-22).

## Acceptance Criteria

**AC1 — 수정이 이후 도출 슬롯에 반영**
**Given** 등록된 공간(본인 소유)
**When** 정보(이름·금액·인원·룸형태·부대시설·좌표/행정동)·영업시간을 수정하면
**Then** 변경이 저장되고, **이후 도출되는 슬롯**(`derive_slots`)이 변경된 영업시간을 사용한다.

**AC2 — 영업시간 축소 시 기존 확정 예약 독립**
**Given** 영업시간을 축소해 기존 확정 예약의 슬롯이 도출 집합 밖으로 나가는 경우
**When** 변경을 저장하면
**Then** 이미 확정된 예약은 자신의 점유 슬롯(`slot_start`)을 **자체 보유**하므로 정상 표시·취소·후기 대상이 되고 영향받지 않는다 — 즉, **수정 경로는 reservations 데이터를 절대 건드리지/재계산하지 않는다**(슬롯은 도출값이지 저장 테이블이 아니며, 예약은 자기 `slot_start`를 행에 보유한다). reservations 테이블은 E4에서 생기므로 본 스토리에서는 이 **구조적 불변식을 코드·테스트·문서로 보증**한다(차감 연결은 Story 4.9).

**AC3 — 삭제 미제공**
**And** 스터디룸 삭제는 제공하지 않으며(DELETE 엔드포인트 없음), 운영 중단은 계정 비활성(FR-31, E8)으로 갈음한다. `is_active` 토글도 본 스토리 범위 밖(E8).

**AC4 — RBAC + 소유권 백엔드 최종 강제**
**Given** provider 인증 토큰
**When** 수정 엔드포인트를 호출하면
**Then** provider 역할만 통과(booker/admin 403, 미인증 401)하고, **본인 소유 공간만** 수정 가능하다(타인 소유/미존재 room_id는 거부 — 소유권은 백엔드 최종, architecture.md L277).

**AC5 — 검증 & 422 단일화**
**Given** 수정 입력
**When** 자정 넘김 영업시간(close ≤ open)·범위 밖 값(음수 금액 등)·미지정 enum·요일 중복을 보내면
**Then** Pydantic 검증 → 1.5 `validation_exception_handler`가 **422 + `{detail:{code:"VALIDATION_ERROR"}}`** 로 단일화한다(2.2 등록 검증 규약과 동일).

### 핵심 함정 (LLM 개발자 실수 차단 — 프리플라이트)

1. **⚠️ 라우팅 순서(최우선):** 신규 `/{room_id}`는 정적 `/geocode` **뒤에** 선언. 2.2 `router.py` docstring(L17-19)이 정확히 이 시나리오를 경고했다 — "추가 시 이 순서를 반드시 보존한다." 위반 시 `GET /rooms/geocode`가 `room_id="geocode"`로 잡혀 UUID 422. **회귀 테스트로 고정**(Task 5).
2. **PATCH 부분 수정 시맨틱:** "미제공"과 "명시 null"을 구분. `model_dump(exclude_unset=True)`로 **온 필드만** 적용. `business_hours` 키 부재=영업시간 불변, 키 존재=전체 교체. 전체 객체 덮어쓰기(PUT 시맨틱)로 착각해 미제공 필드를 `None`으로 밀어버리면 데이터 손실.
3. **영업시간 교체 원자성:** 삭제+삽입+룸 필드 변경을 **단일 트랜잭션**. 중간 commit 금지(부분 저장=영업시간 누락 룸). 2.2 `create_room` 단일 트랜잭션 미러.
4. **소유권 = 백엔드 최종(architecture.md L277):** 라우터 RBAC(`require_role("provider")`)는 "provider인가"만 보고 "**이 room의 주인인가**"는 안 본다. 소유권 검사는 **서비스에서** `room.provider_id == principal.user_id`로. 누락 시 임의 provider가 타인 공간 수정(권한 상승).
5. **`IntegrityError` 선별 변환(회고 P2):** 포괄 `except IntegrityError → 단일 도메인 에러` 금지. `violated_constraint`로 제약명 식별, 무관 위반 re-raise. 본 경로엔 변환 대상이 없어 **rollback 후 re-raise가 기본**.
6. **도메인 404 vs 프레임워크 404:** `DomainError(ROOM_NOT_FOUND, status=404)`는 표준 스키마(`{detail:{code,message}}`)로 응답된다 — Starlette 미존재-경로 404(미표준화)와 다르다. `DEFAULT_STATUS`에 매핑 추가 필수.
7. **마이그레이션 만들지 말 것:** 스키마 무변경. Alembic autogenerate를 돌려도 빈 diff여야 정상. 빈 리비전 생성 금지.
8. **SDK 재생성 의무:** 신규 엔드포인트 → openapi.json + SDK 재생성 안 하면 Layer A/B 레드. 2.2 dev에서 검증된 절차 그대로(export_openapi → generate → pytest Layer A → check:drift Layer B).
9. **Fake 충실도(회고 A5):** `FakeRoomSession`이 `get`/`delete`를 안 다뤄 자기충족 통과 위험. `get`은 room_id별 분기, `delete`는 호출 기록, 소유권/삭제건수를 실제로 단언(1.8 FakeSession introspection 확장 선례).

### AC2 "독립성"의 정확한 의미 (가장 중요 — 오해 차단)

reservations 테이블은 **아직 없다**(E4). 따라서 AC2는 "기존 예약을 보호하는 추가 로직을 짠다"가 **아니다**. 정반대로 — **아키텍처가 이미 독립성을 구조로 보장**한다(architecture.md L149-150 "확정 예약은 slot_start 자체 보유 → 영업시간 변경에 독립"):

- 슬롯은 **물리 테이블이 아니라 도출**(`derive_slots`)된다. 영업시간을 바꾸면 *다음 도출*이 달라질 뿐, 저장된 무엇도 소급 변하지 않는다.
- 예약(E4)은 행에 자기 `slot_start`(UTC)를 **보유**한다. 영업시간이 그 슬롯을 도출집합 밖으로 밀어도 예약 행은 그대로 존재 → 표시·취소·후기 대상.

본 스토리 dev의 책임은 **이 불변식을 깨지 않는 것**: `update_room`이 reservations를 조회/삭제/재계산하는 코드를 **넣지 않는다**(넣을 테이블도 없다). 테스트는 (a) 수정 후 `derive_slots`가 신규 시간을 쓰고 (b) 수정 서비스가 reservation류 데이터를 안 건드림을 단언한다. E4가 reservations를 추가하면 그 행이 `slot_start`를 보유하므로 자동으로 독립이 유지된다.
