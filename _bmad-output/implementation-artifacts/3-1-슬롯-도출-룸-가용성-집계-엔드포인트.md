# Story 3.1: 슬롯 도출 & 룸 가용성 집계 엔드포인트

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 개발자,
I want 룸별 "오늘 현재시각 이후 남은 빈 슬롯 수"를 서버에서 1회 집계해 반환하는 엔드포인트를 두길,
so that 핀 색(FR-5)이 **서버 집계값**으로 결정되고 클라이언트가 핀마다 슬롯을 N회 계산하지 않는다 (NFR-2 성능, architecture.md L362 금지 안티패턴).

## Acceptance Criteria

**AC1 — 룸별 집계가 서버에서 1회 계산되어 반환된다**
**Given** 룸 집합(활성 룸)
**When** 가용성 집계를 요청하면
**Then** 각 룸의 "**오늘(룸 타임존=KST) 현재시각 이후** 남은 빈 슬롯 수"가 **서버측에서 1회** 계산되어 반환된다(클라이언트가 핀마다 슬롯을 재계산하지 않는다).

**AC2 — 슬롯 도출 규칙 = (영업시간 − 휴무 − 예약), 예약은 현재 공집합**
**Given** 슬롯 도출
**When** 집계를 계산하면
**Then** `derive_slots`(Story 2.1)의 `(요일별 영업시간 − 휴무일 − 예약)` 규칙을 사용하며, **reservations가 아직 존재하지 않으므로 예약 집합을 공집합(`frozenset()`)으로 취급**한다. 예약 차감 연결은 **Story 4.9**에서 수행한다(본 스토리는 그 연결 지점을 남겨둔다).

**AC3 — 응답은 핀 색 결정에 충분한 집계값만 제공한다**
**And** 응답은 핀 색 결정에 충분한 **집계값(룸별 잔여 빈 슬롯 수)만** 제공하고, 클라이언트 N회 계산을 유발하지 않는다. 좌표·가격 등 룸 메타데이터는 본 응답에 싣지 않는다(그건 탐색/목록 엔드포인트의 책임 — Story 3.2·3.4·3.5).

**AC4 — derive_slots 호출부 room 격리(회고 deferred 회수)**
**Given** 여러 룸의 `business_hours` 행
**When** 룸별로 집계하면
**Then** 각 룸의 슬롯은 **그 룸의 영업시간 행으로만** 도출된다(룸 간 영업시간이 섞이지 않는다). 이는 Story 2.1 code review에서 `derive_slots`가 `weekday`만 필터하고 `room_id`는 보지 않는다고 식별돼 **본 스토리(3.1)를 트리거로 명시 deferred된 항목의 회수**다(deferred-work.md "derive_slots room_id 미필터").

## Dev Notes

### 이 스토리의 본질 — 백엔드 집계 1개 추가, 신규 모듈 0

Story 2.1이 만든 순수 함수 `derive_slots`의 **첫 실 소비처**다. 신규 파일 0 — 기존 `rooms/{schemas,service,router}.py`에 각각 작은 추가만 한다. 본 스토리는 **백엔드 집계 엔드포인트까지**이며, 핀 렌더·색·아이콘(FR-5 시각화)은 **Story 3.2**(프론트)가 이 엔드포인트를 소비해 구현한다. [Source: epics.md#Story-3.1 / architecture.md L362]

### derive_slots 재사용 — 슬롯 도출 단일 출처 (중복 금지)

슬롯은 **물리 테이블이 아니라 도출**된다: `슬롯 = (요일별 영업시간 − 휴무일 − 예약)`, 고정 1시간 단위, 같은 날 내. 이 로직은 `app/rooms/service.py:derive_slots`(2.1)에 이미 있고 **순수 함수**(DB 미접근, 입력 주입)다. 본 집계 서비스는 ORM 조회 결과를 그 함수에 주입할 뿐, 슬롯 계산을 재구현하지 않는다. [Source: apps/api/app/rooms/service.py:52-130 / architecture.md L148-150·L360-362]

`derive_slots` 계약 요약(소비 시 반드시 지킬 것):
- 입력 `business_hours`는 **그 룸의 행만**(AC4 — room_id 그룹핑). `holiday_dates`는 `date`(datetime 금지), `target_date`는 `date`, `reserved_starts`는 tz-aware UTC(naive 금지) — 위반 시 `ValueError` fail-fast.
- 반환은 **오름차순 tz-aware UTC `slot_start` 리스트**. 휴무·미영업·전부예약 시 `[]`.
- `open_time`/`close_time`은 KST **벽시계**, `slot_start`는 **UTC 인스턴트**(09:00 KST = 00:00 UTC). 변환은 `derive_slots`가 담당. [Source: apps/api/app/rooms/service.py:59-86]

### "오늘 현재시각 이후" — 시간 규약 (core/time)

- **"오늘"=룸 타임존 기준 날짜.** `today_in_tz(ROOM_TZ, now=now)`로 구한다 — **절대 프로세스/UTC 로컬 날짜로 판정하지 않는다**(architecture.md L296 금지, NFR-1 결정성). KST는 UTC+9라 UTC 15:00 이후는 KST 익일이다. [Source: apps/api/app/core/time.py:60-67]
- **"현재시각 이후"=`slot_start >= now`.** `derive_slots` 반환값(aware UTC)과 `now`(aware UTC)를 **직접 비교**한다. `is_within_hours`/`hours_until`을 쓰지 말 것 — 그건 6h/24h 윈도우용이고 1.5에서 float 경계 정밀도가 deferred다. 직접 인스턴트 비교가 그 함정을 우회한다. 경계는 `>=`(now에 시작하는 슬롯 포함)로 고정. [Source: deferred-work.md "hours_until 부동소수 경계 정밀도" — 트리거는 E4/4.7이며 본 스토리 아님]
- **`now` 주입:** core/time 함수가 `now`를 주입받는 패턴 그대로 서비스도 `now: datetime | None = None`을 받아 테스트 결정성을 확보한다. 라우터는 미주입(=`now_utc()`).

### ★회고 deferred 회수 — derive_slots room_id 미필터 (AC4)

deferred-work.md에 **본 스토리(3.1)를 트리거로 명시**된 항목이 있다:

### ★신선도(freshness) 계약 — 단계별로 다르다 (KTH 확정 2026-06-15)

MVP는 **지도 핀의 실시간(WebSocket/SSE) 갱신을 하지 않는다.** 대신 화면 단계별로 신선도를 차등한다:

| 화면 | 신선도 | 메커니즘 |
|------|--------|----------|
| **지도 핀**(본 스토리 3.1·3.2) | 느슨(snapshot) | 이 집계 응답은 **요청 시점 스냅샷**. 다른 유저 예약이 즉시 반영되지 않아도 됨(색 힌트). 서버 캐싱 없이 매 요청 계산하되, 타 유저에게 push는 없음 |
| **바텀시트**(3.3) | **열 때 최신** | 핀의 캐시된 집계를 **재사용하지 않고** 여는 순간 그 룸을 새로 조회 → "예약 가능 여부"가 현재 상태 반영 |
| **상세·슬롯**(4.2·4.3) | **진입 시 최신** | 이미 예약된 슬롯은 **선택 불가**로 표시(`derive_slots`에 `reserved_starts` 채워짐 — 4.9 차감 연결) |
| **예약 확정**(4.5·4.6) | 엄격 | fetch~tap 사이 누가 채갔어도 `UNIQUE(room_id,slot_start)` + 단일 트랜잭션 → **409 SLOT_CONFLICT**로 차단(FR-15) |

**복귀 시 핀 갱신(KTH 확정 2026-06-15):** 유저 A가 핀 탭 → 바텀시트/상세에서 "이미 찼음"을 본 뒤 **바텀시트를 내리거나 지도로 돌아오면 그 핀은 회색(마감)으로 바뀌어야 한다.** 메커니즘 = **실시간 푸시가 아니라 "복귀 시 재조회"** — 지도 화면 복귀 시 가용성 집계를 refetch(TanStack Query `refetchOnMount`/복귀 invalidate)해 현재값을 다시 그린다(`['rooms',filters]` 무효화는 3.2 소유). 단 **예약 때문에** 회색이 되려면 4.9(예약 차감 연결)가 있어야 한다 — 그 전엔 예약이 수치를 줄이지 않는다(메커니즘은 3.2에서 확립, 예약-구동 변화는 4.9 후 완성).

**3.1에 주는 함의(중요):**
- 이 엔드포인트(핀 집계)는 의도적으로 **느슨한 스냅샷**이다 — 실시간성 요구 없음. **서버측 캐시를 넣지 않는다**(매 요청 계산이 곧 그 시점 최신이므로 충분; 캐시는 오히려 stale 키 무효화 부담만 늘림). **`GET`은 멱등**이라 지도 복귀 때마다 자유롭게 재조회해도 안전·저비용 — 위 "복귀 시 핀 갱신"이 백엔드에서 막히지 않는다.
- 바텀시트/상세의 "현재 예약 가능"은 **이 엔드포인트가 아니라** 단일 룸·슬롯 단위 신선 조회(3.3·4.3)의 책임이다 — 그래서 3.1은 **전체 룸 집계만** 하고 단일 룸 조회 파라미터를 추가하지 않는다(스코프 경계). architecture.md의 TanStack Query 키 분리(`['rooms',filters]` 핀집계 ↔ `['room',roomId,'slots',date]` 상세)와 정합.
- "이미 예약된 슬롯=불가"의 실제 동작은 **4.9 차감 연결** 후에 성립한다(3.1·4.3 모두 `derive_slots`를 쓰므로 4.9가 `reserved_starts`를 채우면 두 경로 다 자동 반영).
