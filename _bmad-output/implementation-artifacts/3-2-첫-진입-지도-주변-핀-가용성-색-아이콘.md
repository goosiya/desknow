# Story 3.2: 첫 진입 지도 + 주변 핀 + 가용성 색/아이콘

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 예약자,
I want 첫 화면(웹)에서 추가 조작 없이 주변 룸을 카카오맵 핀으로 보고, 예약 가능 여부를 **색 + 아이콘**으로 즉시 인지하길,
so that 지금 내 근처 어디가 비는지 한눈에 안다 (FR-4, FR-5, UJ-2, UX-DR2, NFR-2·5).

> **이 스토리는 프로젝트 최초의 실 프론트엔드 기능 화면이다.** 지금까지 1.6은 디자인 토큰·UI 셸(골격)만 만들었고, 1.7~2.3의 프론트 폼은 전부 전용 후속 스토리로 이관됐다. 본 스토리가 그 첫 소비처다.

## 범위 결정 (KTH 확정 2026-06-15 — create-story)

1. **표면 = 웹 전용(`apps/web`).** Phase 0 스파이크(1.3)가 **웹 카카오맵 e2e(지도+핀+지오코딩)를 이미 증명**했고 약관 적합 판정(E3 go)을 받았다. 반면 **모바일(RN) 카카오맵은 미검증**(스파이크는 SSE만 검증)이고 RN 기기검증은 Expo SDK56↔Expo Go 비호환으로 **E7에 보류 중**이다. 따라서 본 스토리는 **`apps/mobile` 지도를 건드리지 않는다** — 모바일 지도는 별도 스토리(E7 RN 기기검증과 동반)로 분리한다. 첫 방문 온보딩 오버레이(3.9)도 웹부터다.
2. **좌표 공급 백엔드 = 본 스토리에 `GET /api/v1/rooms` 포함(수직 슬라이스).** 3.1 가용성 엔드포인트는 의도적으로 `{room_id, remaining_slots}`만 반환하고 **좌표는 "탐색/목록 엔드포인트(3.2·3.4·3.5)의 책임"으로 명시 이관**했다(3.1 AC3). 핀을 찍으려면 룸 좌표·이름이 필요하므로 **공개 `GET /rooms`(활성 룸의 핀 메타)** 백엔드 엔드포인트를 본 스토리에서 추가하고, 프론트가 `room_id`로 가용성과 인메모리 조인한다. 백엔드 부분은 availability처럼 작다.
3. **프론트 테스트 인프라 = `apps/web`에 vitest + jsdom + React Testing Library 도입.** 첫 프론트 기능이라 러너가 없다(현재 `packages/ui` parity vitest만 존재). 순수 로직(가용성 조인·핀 색/아이콘 도출·aria-label 빌더) 단위테스트 + 카카오 SDK를 모킹한 컴포넌트 상태 테스트를 도입하고 turbo `test` 게이트에 편입한다.

## Acceptance Criteria

**AC1 — 첫 진입 시 추가 조작 없이 주변 룸 핀이 카카오맵에 뜬다 (≤2초 p90)**
**Given** 위치 권한 허용(웹 `navigator.geolocation`)
**When** 웹 첫 화면(`/` 스터디룸 찾기)을 열면
**Then** 현재 위치를 중심으로 활성 룸이 **카카오맵 JS SDK 핀**으로 표시되고, **첫 지도 표시는 양호 네트워크에서 ≤2초(p90)에 시작**된다(그 전까지 지도 스켈레톤 + 핀 placeholder). 핀 좌표는 본 스토리가 추가하는 `GET /api/v1/rooms`(활성 룸 좌표·이름)에서 온다.

**AC2 — 핀 색 + 아이콘이 서버 집계 가용성으로 결정된다(색 단독 신호 금지)**
**Given** 3.1 가용성 집계(`GET /api/v1/rooms/availability`)와 본 스토리의 룸 목록(`GET /api/v1/rooms`)
**When** 핀을 렌더하면
**Then** `room_id`로 둘을 조인해 각 룸의 `remaining_slots`를 구하고 — `>= 1`이면 **예약 가능(`pin-available` `#157F45` 초록 + 체크(✓) 아이콘)**, `0`(또는 가용성 미존재)이면 **마감(`pin-full` `#7E7466` 회색 + X(✕) 아이콘)**으로 표시한다. **색은 절대 단독 신호가 아니며 항상 아이콘을 병행**한다(architecture.md L295·DESIGN.md 핀 규칙). **클라이언트가 핀마다 슬롯을 재계산하지 않는다**(서버 집계값 사용 — architecture.md L293 금지 안티패턴). `remaining_slots >= 1` 분기만 한다(자명한 분기, 재계산 아님).

**AC3 — 핀을 탭하면 해당 룸 바텀시트가 열린다(콘텐츠는 3.3 소유)**
**Given** 렌더된 핀
**When** 핀을 탭/클릭(또는 키보드 Enter)하면
**Then** 해당 룸의 바텀시트가 열린다. **본 스토리는 열림 동작(핀→시트)과 최소 시트 셸**(룸 이름 + 예약 가능/마감 상태 + "상세 정보는 곧" 자리)까지 구현하고, **1차/2차 정보 위계·즐겨찾기·드래그·포커스 트랩 등 풍부한 내용은 Story 3.3가 채운다**(스코프 경계). 시트는 떠오르는 유일 면(`elevation.sheet`·상단만 `rounded.xl`)이다.

**AC4 — 스크린리더가 핀을 "○○ 스터디룸, 예약 가능/마감"으로 읽는다 (NFR-5)**
**Given** 스크린리더(웹)
**When** 핀에 도달하면
**Then** `"{룸 이름} 스터디룸, 예약 가능"`(가용) 또는 `"{룸 이름} 스터디룸, 마감"`(마감)으로 읽힌다. 핀은 `role="button"`(또는 버튼 요소) + `aria-label`을 갖고, **키보드로 도달·실행 가능**하며 터치 타겟 ≥44×44px이다. **색맹 대비: 색 외에 아이콘(✓/✕)이 독립 신호**로 존재한다.

**AC5 — 막다른 화면 금지: 로딩/빈/에러 상태가 다음 행동을 안내한다**
**Given** 찾기(지도) 화면
**When** ① 로딩 중 / ② 주변 활성 룸 0개 / ③ 지도 로드 실패(카카오 SDK·키) / ④ 위치 권한 거부 상황이면
**Then** 각각 — ① **지도 스켈레톤 + 핀 placeholder**(≤2초), ② **"이 근처엔 아직 없어요"**(반경 확대/행정동 전환 유도 자리), ③ **"지도를 못 불러왔어요"** + 재시도, ④ **위치 거부 안내**("위치를 못 받았어요. 동네를 골라서 찾아볼게요.") + 기본 지역(서울 중심)으로 폴백해 핀 표시 — 가 표시된다. **막다른 빈 화면이 없다.** 마이크로카피는 친근한 해요체. *(완전한 상태 매트릭스는 3.8, 행정동 콤보 우회 전체 구현은 3.6 소유 — 본 스토리는 막힘 방지 최소 안내까지.)*

## Dev Notes

### 이 스토리의 본질 — 최초 프론트 기능(웹 지도) + 작은 백엔드 좌표 엔드포인트

수직 슬라이스: **백엔드** `GET /rooms`(활성 룸 핀 메타, availability처럼 작음) + **프론트** 카카오맵·핀·상태·최소 바텀시트 셸 + **테스트 인프라**(vitest/jsdom/RTL 최초 도입). 모바일·풍부한 바텀시트(3.3)·행정동 콤보(3.4·3.6)·반경(3.5)·완전 상태 매트릭스(3.8)·온보딩(3.9)은 **범위 밖**. [Source: 범위 결정 섹션 / epics.md#Story-3.2]

### ★핵심 데이터 흐름 — 두 공개 GET을 `room_id`로 조인 (좌표·가용성 책임 분리)

3.1이 가용성 응답에서 좌표를 **의도적으로 뺐다**(AC3: "좌표·가격 등은 탐색/목록 엔드포인트 3.2·3.4·3.5 책임"). 본 스토리가 그 좌표 엔드포인트를 만든다:

- `GET /api/v1/rooms` → `list[RoomMapItem]{room_id,name,lat,lng}` (본 스토리 신규·공개)
- `GET /api/v1/rooms/availability` → `list[RoomAvailability]{room_id,remaining_slots}` (3.1·공개)
- **프론트가 `room_id`로 인메모리 조인** → `RoomPin{room_id,name,lat,lng,remaining_slots}` → 좌표에 색+아이콘 핀.

TanStack 키: `['rooms','map']`(좌표) + `['rooms','availability']`(색). architecture.md L270 `['rooms', filters]`(핀집계)의 구체화다. 둘 다 지도 복귀 시 재조회(신선도). [Source: 3.1 AC3·L147·L154 / architecture.md L270-272]

### ★신선도(freshness) 계약 — 핀=느슨한 스냅샷, 복귀 시 재조회 (KTH 확정, 3.1 계승)

- **핀 집계/목록은 느슨한 스냅샷**(실시간 푸시 없음·서버 캐시 없음). `GET`은 멱등이라 지도 복귀마다 재조회 안전·저비용.
- **복귀 시 핀 갱신:** 유저가 핀→바텀시트/상세에서 "이미 찼음"을 본 뒤 지도로 돌아오면 그 핀이 회색으로 바뀌어야 한다 → **실시간 푸시 아님, "복귀 시 재조회"**(TanStack `refetchOnMount:'always'` 또는 복귀 invalidate로 `['rooms',*]` 재조회). **`['rooms',*]` 무효화는 본 스토리(3.2) 소유.**
- 단 **예약 때문에** 회색이 되려면 **4.9(예약 차감 연결)**가 있어야 한다. 본 스토리는 **메커니즘만 확립**(재조회 배선)하고, 예약-구동 변화는 4.9 후 완성된다. 현재 `remaining_slots`는 예약 미반영(3.1 AC2 공집합).
- 바텀시트/상세의 "현재 예약 가능"은 이 핀 엔드포인트가 아니라 단일 룸 신선 조회(3.3·4.3) 책임 — 그래서 `GET /rooms`는 **목록 메타만** 싣는다(과조회 금지). [Source: availability-freshness-policy 메모 / 3.1 Dev Notes 신선도 표 / architecture.md L271]

### 금지 안티패턴 — 클라 슬롯 재계산·색 단독 (architecture.md L292-296)

- **클라가 핀마다 슬롯 N회 계산 금지** → 서버 집계값(`remaining_slots`)만 쓴다. `pinStatus`의 `>= 1` 분기는 자명한 분기일 뿐 재계산이 아니다(슬롯 도출은 서버 `derive_slots`가 1회 끝냄). `business_hours`/예약을 클라에서 슬롯으로 환산하는 코드를 **절대 작성하지 않는다**.
- **색만으로 상태 표현 금지** → 핀은 색 + 아이콘(✓/✕) + (스크린리더) 텍스트를 **항상 병행**. `pinVisual`이 색·아이콘을 함께 반환하고, `pinAriaLabel`이 텍스트를 보장한다. 테스트가 아이콘·라벨의 색-독립 존재를 단언. [Source: architecture.md L293·L295 / DESIGN.md 핀 규칙 L256-257]

### 카카오맵 — 웹 JS SDK(스파이크 검증 패턴 이식)

- Phase 0 스파이크가 `apps/web`(Next 16)에서 **지도+핀+지오코딩 e2e**를 증명했고 **약관 적합(E3 go)** 판정했다. 스파이크 코드는 폐기됐으므로 **로더 패턴만 재작성**: `//dapi.kakao.com/v2/maps/sdk.js?appkey={JS_KEY}&autoload=false` 동적 `<script>` → `kakao.maps.load(cb)`. [Source: 1-3 스파이크 Task 2 / docs/phase0-spike-results.md ①]

### SDK 소비 — `@desknow/api-client`만, 직접 fetch 금지 (1.9 가드 발효)
