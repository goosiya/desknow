# Story 3.8: 탐색 상태 매트릭스 마감

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 예약자,
I want 탐색 화면의 빈/에러/네트워크 단절/로딩 상태가 모두 설계되어 있길,
so that 어떤 상황에서도 막히지 않고 다음 행동을 안내받는다 (UX-DR9, NFR-5).

> **이 스토리의 본질 — "마감(seal)" 스토리.** 새 기능을 만드는 게 아니라, 3.2~3.7이 부분적으로 만들어 둔
> 탐색 표면의 **상태 매트릭스(주요 화면 × {빈 / 에러 / 네트워크 단절 / 로딩})를 빈 셀 없이 닫는다**.
> 코드를 실제로 뜯어본 결과: 로딩·빈·에러는 대부분 이미 있고, **네트워크 단절(network disconnect) 셀이
> 세 화면 모두에서 완전히 비어 있다** — 이것이 가장 큰 신규 작업이다. 그 외 지도 화면의 빈/에러 셀에
> AC가 요구하는 **다음-행동 액션 버튼**이 빠져 있어 채운다.

## Acceptance Criteria

epics.md(L672-688) 원문 BDD:

1. **(AC1 — 찾기 지도)** **Given** 찾기(지도) 화면 **When** 주변 룸 없음/지도 로드 실패/네트워크 단절/로딩 상황이면 **Then** 각각
   - **빈:** "이 근처엔 아직 없어요" + **반경 확대·행정동 전환 액션**
   - **에러:** "지도를 못 불러왔어요" + **재시도·목록 우회 액션**
   - **네트워크 단절:** "네트워크 연결이 끊겼어요. 연결되면 다시 보여드릴게요." + **마지막 핀 캐시 유지**
   - **로딩:** 지도 스켈레톤(≤2초)

   상태가 표시된다.

2. **(AC2 — 찾기 목록)** **Given** 찾기(목록) 화면 **When** 결과 없음/조회 실패/네트워크 단절/로딩 상황이면 **Then** 각각
   - **빈:** 빈 상태 안내(검색방식별 다음-행동 제안)
   - **에러:** 재시도 액션
   - **네트워크 단절:** **캐시된 목록 표시** + **네트워크 단절 표시(토스트)**
   - **로딩:** Skeleton 행

   이 표시된다.

3. **(AC3 — 마이크로카피 규칙)** **And** 모든 마이크로카피는 친근한 해요체이며 막힘 뒤에는 항상 다음 행동을 제시한다(UX-DR10, EXPERIENCE.md:106).

### KTH 확정 사항 (2026-06-16 — create-story 시점)

> AC 자체에 없거나 해석이 갈리는 지점을 사전 확정한다. 이 절은 dev agent에게 **AC와 동등한 구속력**을 갖는다.

- **확정① — 용어/카피: "오프라인" 금지, "네트워크 단절"로 명명.** [[terminology-network-disconnect-not-offline]]
  "오프라인"은 *로그인 상태인지 네트워크 상태인지 모호*하다.
  - **내부(스토리·코드·주석·식별자):** "네트워크 단절"로 표기. 식별자는 `useOnlineStatus`/`isOnline`(navigator API 표준어라 허용)이되, 사용자 의미는 네트워크.
  - **화면 카피(사용자 노출):** **"네트워크 연결이 끊겼어요. 연결되면 다시 보여드릴게요."** 로 고정.
    EXPERIENCE.md 매트릭스 원안 "오프라인이에요…"(L133-134)를 **이 문구로 덮는다**.
  - 여기서 네트워크 단절 = `navigator.onLine === false`(인터넷 연결 끊김). **로그인/세션과 무관**하고, **위치 권한 거부(3.6)와도 별개 축**이다. 탐색(지도·목록)은 애초에 공개·무인증 화면이라 로그인은 상태 변수가 아니다.

- **확정② — 네트워크 단절 표시 = 경량 커스텀(신규 의존성 0).**
  토스트 라이브러리(sonner 등) **도입하지 않는다**. Epic 3 전체의 "신규 의존성 0" 기조와 일관.
  `useSyncExternalStore` 기반 `useOnlineStatus` 훅 + `role="status"`/`aria-live` 경량 `NetworkNotice`
  컴포넌트로 구현한다. 매트릭스의 "토스트"는 자동 소멸 토스트 위젯이 아니라 **단절 동안 떠 있다가
  연결되면 사라지는 인라인 배너**로 충족한다(자동 소멸 큐 불필요 — 상태가 곧 표시).

- **확정③ — 마감 범위 = 현존 3개 탐색 표면(지도 + 목록 + 즐겨찾기).**
  AC1/AC2는 지도·목록만 명시하나, 3.7이 이미 만든 `/favorites` 페이지도 매트릭스에 행이 있고(EXPERIENCE.md:138
  "캐시 표시") 네트워크 단절 셀이 비어 있다. **같은 `useOnlineStatus`/`NetworkNotice`를 만드는 김에
  즐겨찾기 페이지의 단절 셀까지 닫는다.** 이는 AC 문구를 넘는 확장이며 KTH가 명시 승인.
  - **범위 밖(각 스토리 소유):** 상세(4.2)·슬롯피커(4.3)·예약현황(E4/4.8)·챗봇(E7)·제공자 등록(후속)
    — 화면 자체가 아직 스텁/미구현이라 각자 만들 때 상태를 짠다. **모바일(apps/mobile) 전체 제외**(3.2~3.7 일관, E7).
  - **바텀시트(RoomSheet) 자체의 단절 셀은 범위 밖** — 시트는 3.3이 신선 단일 조회(로딩/에러)를 이미 소유하며,
    시트는 화면이 아니라 떠오르는 면이다. 단절 시 시트를 새로 여는 동작은 드물고, 3.3/4.x 영역. 본 스토리는 **화면 레벨**만 마감.

### 네트워크 단절 처리 — 설계 핵심(공통 패턴)

```
isOnline = useOnlineStatus()           // navigator.onLine + online/offline 이벤트
showError = isOnline && (서버/SDK 실패)  // 단절을 일반 에러로 오인 금지(최우선 게이팅)
!isOnline →
  - NetworkNotice 표시("네트워크 연결이 끊겼어요…")
  - 마지막 성공 데이터(TanStack 캐시 잔존)를 그대로 렌더 → "캐시 표시/마지막 핀 유지"
  - 재연결 → query-client 기본 refetchOnReconnect(true·미오버라이드)가 자동 재조회
```

- **왜 영속 캐시가 필요 없나:** TanStack Query는 마지막 성공 응답을 메모리 캐시에 보유한다. 단절은 *다음 refetch를 막을 뿐* 이미 렌더된 데이터를 지우지 않는다. "마지막 핀 캐시 유지"·"캐시된 목록 표시"는 이 메모리 캐시 잔존으로 충족된다(세션 내). IndexedDB persist는 본 스토리 범위 밖(과설계).
- **재연결 자동 복구:** `query-client.ts`는 `staleTime`/`refetchOnMount`/`retry`만 설정하고 `refetchOnReconnect`는 미오버라이드 → TanStack 기본 `true`. `online` 이벤트 시 stale 쿼리 자동 재조회. **이 기본을 끄지 말 것.**
- **단절 ≠ 로그인 ≠ 위치거부(3축 분리):** 셋을 한 분기로 합치지 말 것(확정①). 즐겨찾기에서 특히 — 미로그인(401)을 단절로, 단절을 미로그인으로 덮으면 카피가 거짓이 된다.
