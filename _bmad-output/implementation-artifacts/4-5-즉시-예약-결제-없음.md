# Story 4.5: 즉시 예약 (결제 없음)

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 예약자,
I want 선택한 연속 슬롯을 결제 없이 즉시 확정하길,
so that 최소 단계로 예약을 끝낸다 (FR-14, UJ-1·2, SM-1·2).

> **이 스토리는 Story 4.4가 `ReservationPanel`에 남긴 "4.5 seam"(선택 상태 `selection` 노출 → 실제 예약 확정)을 채운다.** 4.4는 연속 슬롯 **선택**과 하단 **선택 요약(범위·날짜·시간·금액) readout까지**만 했다(`예약하기` CTA·POST·성공 피드백·캐시 무효화 0 — dead 버튼도 만들지 않음). 4.5는 그 `selection`을 읽어 **① 백엔드 POST 예약 엔드포인트 신설**(예약 단위 + 슬롯 점유 행 all-or-nothing 생성 — 4.1 `create_reservation` 서비스 프리미티브 소비) **② SDK 재생성 ③ `예약하기` CTA·POST 제출·"예약이 완료됐어요!" 인라인 성공 피드백·선택 초기화·슬롯/핀 invalidate**를 배선한다. **동시성 충돌 특화 안내("앗, 방금 다른 분이…")·인접 빈 슬롯 즉시 재표시는 Story 4.6**(본 스토리는 제출 실패 시 generic 우아 안내+재시도까지 — 막다른 화면 금지).

## 범위 결정 (KTH 확정 2026-06-16 — create-story)

이 스토리가 닿는 네 경계를 다음과 같이 확정한다:

1. **인증/권한 = `require_role("booker")`.** 예약 생성은 booker(예약자) 행위 — provider/admin은 403 `FORBIDDEN_ROLE`로 차단한다(favorites의 역할무관 `get_current_principal`과 달리 **역할 제약**). 미인증은 401 `UNAUTHENTICATED`. [Source: epics.md#Story 4.5 / core/security.py `require_role`]

2. **확정 시 신선 재검증(server-side).** 요청 슬롯이 **확정 시점에 실제로 빈 슬롯인지** 서버가 재확인한다 — `derive_slots`(4.3 도출 로직) 재사용으로 요청 `slot_start[]`이 ① 룸 영업시간 내 ② 미과거(ROOM_TZ 현재시각 이후) ③ 휴무일 아님 ④ (이미 예약된 슬롯은 `uq_reservation_slots_room_slot` UNIQUE가 최종 차단) 인지 검증하고, 위반 시 거부한다. **룸 존재/활성(`is_active`) 404 `ROOM_NOT_FOUND` 포함.** stale 선택(클라가 오래된 슬롯을 들고 확정)을 차단해 [[availability-freshness-policy]](예약확정=409 차단·FR-15)와 정합. **신규 ErrorCode 0** — "요청 슬롯이 현재 예약 불가" = 기존 `SLOT_CONFLICT`(409)로 통일(과거/영업시간외/타예약 점유 모두 "지금 잡을 수 없는 슬롯"), 입력 형식 위반(빈 배열·naive·교차일)은 `VALIDATION_ERROR`(422·Pydantic).

3. **실패 UX 경계 = 4.5는 generic 우아 실패까지.** 4.5는 제출 실패 시 **generic 안내 + 재시도**(막다른 화면 금지)까지만 구현한다. **SLOT_CONFLICT 특화 카피("앗, 방금 다른 분이 먼저 잡았어요. 가까운 빈 시간을 다시 보여드릴게요.")·인접 빈 슬롯 즉시 재표시는 Story 4.6** AC 소관으로 유지(EXPERIENCE L136·L205). 백엔드 409 SLOT_CONFLICT는 4.5에 이미 존재(4.1 프리미티브)하나, 그 **특화 프론트 UX**만 4.6으로 미룬다.

4. **성공 후 = 상세 유지 + 인라인 성공 배너.** "예약이 완료됐어요!" `role="status"` 인라인 배너(자동소멸 toast 아님 — sonner 미도입 일관)를 표시하고, **`selection` 초기화 + `['room', roomId, 'slots', date]` 슬롯 쿼리 / 핀 집계(`['rooms', ...]`) invalidate + 페이지 유지**. 예약현황 `/reservations`로 이동하지 **않는다**(4.8 미구축 → dead link 회피). 3스텝 이내 보존(NFR-3).

## Acceptance Criteria

**AC1 — POST 예약 엔드포인트: booker 인증 + all-or-nothing 점유 (FR-14, SM-1·2, 범위결정 #1)**
**Given** booker로 인증된 요청과 빈 연속 슬롯 `slot_start[]`
**When** `POST /api/v1/rooms/{room_id}/reservations`로 예약을 확정하면
**Then** 결제 단계 없이 예약 단위 1건(`status='confirmed'`) + 그 슬롯 점유 행들이 **단일 트랜잭션 all-or-nothing**으로 생성되고(부분 점유 0건 — 일부만 잡힌 채 확정되는 상태 없음), `ReservationPublic`(id·room_id·booker_id·status·created_at`…Z`·slot_starts)이 201로 반환된다.
- **서비스 소비:** 4.1 `reservations.service.create_reservation(session, booker_id=, room_id=, slot_starts=)`를 그대로 호출(재구현 금지). `booker_id`는 인증 principal에서·`room_id`는 경로에서·`slot_start[]`는 본문에서.
- **인증/권한(범위결정 #1):** `Depends(require_role("booker"))` — provider/admin 403 `FORBIDDEN_ROLE`, 미인증 401 `UNAUTHENTICATED`. responses 계약에 401·403·404·409 노출(OpenAPI/SDK).
- **결제 없음(FR-14):** 결제·금액 차감 단계 없음. 금액은 표시 전용(4.4). 확정 = 점유 행 생성이 전부.

**AC2 — 확정 시 신선 재검증 + 룸 404 (범위결정 #2, [[availability-freshness-policy]] / FR-15)**
**Given** 예약 확정 요청
**When** 요청 슬롯을 처리하면
**Then** 서버가 요청 `slot_start[]`이 **확정 시점에 실제 빈 슬롯**인지 재검증한다 — `derive_slots`(4.3 도출 로직) 재사용으로 ① 룸 영업시간 내 ② 미과거(ROOM_TZ) ③ 휴무 아님을 확인하고, 하나라도 도출 가능한 가용 슬롯 집합에 없으면 `SLOT_CONFLICT`(409)로 거부한다(stale 선택 차단).
- **룸 가드:** 미존재/비활성(`is_active=false`) 룸 → 404 `ROOM_NOT_FOUND`(rooms 도메인 헬퍼 재사용).
- **이미 예약된 슬롯:** `uq_reservation_slots_room_slot` UNIQUE가 INSERT 시 최종 차단 → 전체 ROLLBACK·409 `SLOT_CONFLICT`(4.1 변환 그대로). 신선 재검증과 UNIQUE가 이중 방어(검증은 과거/영업시간외/휴무, UNIQUE는 타 예약 점유).
- **입력 형식:** 빈 `slot_start[]`·naive datetime·중복·교차일(서로 다른 ROOM_TZ 날짜) = 422 `VALIDATION_ERROR`(Pydantic/명시 검증 — service `ValueError`가 500으로 새지 않게 라우터/스키마에서 선차단).
- **신규 ErrorCode 0**(SLOT_CONFLICT·ROOM_NOT_FOUND·FORBIDDEN_ROLE·UNAUTHENTICATED·VALIDATION_ERROR 전부 기존).

**AC3 — SDK 재생성 (계약 노출, 1.9 파이프라인)**
**Given** 신규 POST 엔드포인트의 OpenAPI 스키마
**When** OpenAPI export + SDK 생성을 실행하면
**Then** `@desknow/api-client`에 `reservationsCreateReservation`(operationId `reservations_create_reservation`)과 `ReservationCreateRequest`/`ReservationPublic` 타입이 **snake_case 보존**으로 생성되고, 드리프트 게이트(Layer A·B)가 그린이다.

**AC4 — `예약하기` CTA + POST 제출 + 인라인 성공 피드백 + 선택 초기화/무효화 (FR-14, 범위결정 #3·#4)**
**Given** 빈 연속 슬롯이 선택된 상태(`selection` 보유 — 4.4)
**When** `예약하기`를 누르면
**Then** 결제 단계 없이 즉시 확정 요청이 가고, 성공 시 **"예약이 완료됐어요!"** `role="status"` 인라인 배너가 표시되며, `selection`이 초기화되고 `['room', roomId, 'slots', selectedDate]` 슬롯 쿼리 + 핀 집계 쿼리가 invalidate된다(점유 즉시 반영).
- **CTA 위치:** 선택 요약 바(4.4) 하단/내부에 `예약하기` 버튼 — **선택이 있을 때만 활성**(선택 없으면 버튼 미표시 또는 disabled+안내 "시간을 선택해 주세요"는 4.4가 이미 처리 → 버튼은 선택 있을 때만 렌더). dead 버튼 금지.
- **제출:** `selection`(인덱스 구간) → `slots[startIndex..endIndex].slot_start[]`(서버 UTC ISO 그대로) 추출 → `reservationsCreateReservation`. 제출 중 버튼 disabled + "예약 중…"(이중 제출 방지·`isPending`).
- **mutation 패턴:** `useToggleFavorite`(useFavorites.ts) canonical 미러 — `useMutation` + `onSuccess`에서 정확 키 invalidate(광역 금지). **옵티미스틱 불필요**(예약은 서버 확정이 진실 — 응답 후 성공 표시·낙관적 추가 금지: 충돌 가능성). `pricePerHour`는 결제 의미 아님(표시 전용 유지).
- **성공 피드백(범위결정 #4):** 인라인 `role="status"` 배너(NetworkNotice/FavoriteButton hint 패턴) — "예약이 완료됐어요!". 페이지 유지(상세). `/reservations` 이동 없음(4.8 미구축).

**AC5 — 제출 실패 generic 우아 안내 + 재시도 (막다른 화면 금지, 범위결정 #3)**
**Given** 예약 제출이 실패(409 SLOT_CONFLICT·404·5xx·네트워크 단절)
**When** 응답이 오면
**Then** **generic 우아 안내 + 재시도** 경로가 제공된다(예: "예약을 완료하지 못했어요. 다시 시도해 주세요." + 다시 시도) — 화면이 막히지 않고 `selection`은 유지되어 재시도 가능.
- **SLOT_CONFLICT 특화 카피·인접 빈 슬롯 재표시는 4.6**(본 스토리는 모든 실패를 generic으로 우아 처리). 에러코드 노출 금지(표준 에러 스키마의 `message`만 사용하거나 고정 카피).
- **네트워크 단절:** `useOnlineStatus`/`NetworkNotice` 기존 게이팅 일관(단절을 서버 에러로 오인 금지).

**AC6 — 3스텝 이내 + 무회귀 (NFR-3, 4.2/4.3/4.4 보존)**
**Given** 단독 예약 흐름
**When** 상세 진입부터 예약 완료까지 진행하면
**Then** 3스텝 이내(스텝=별개 탭/입력, 스크롤·드래그는 1스텝)로 완료된다 — 상세 진입(전개) → 날짜·슬롯 선택 → `예약하기`.
- 4.2(상세 전개)·4.3(달력/슬롯/빈날 분기)·4.4(연속 선택·요약·날짜변경 리셋·키보드/SR) 전부 **무회귀**. CTA·성공 배너를 그 위에 덧대되 기존 분기(로딩/실패/빈날/단절)는 그대로.
- `prefers-reduced-motion` 존중·모든 마이크로카피 해요체.

### 엔드포인트 형태 — 중첩 `/rooms/{room_id}/reservations` (architecture L243)

### 신선 재검증 설계 — derive_slots 단일 출처 재사용 (범위결정 #2, [[availability-freshness-policy]])

- 확정 시점에 요청 슬롯이 **실제 빈 슬롯인지** 서버가 재확인한다(클라가 들고 있던 stale 선택 차단). `rooms.service`의 슬롯 도출(`get_room_slots`/`derive_slots`)을 **재사용**해 요청 날짜의 가용 `slot_start` 집합을 구하고 `요청 ⊆ 가용` 검증. **슬롯 도출 로직 재구현 금지**(architecture L354·L362·회고 회수 — reservations가 rooms 슬롯 SQL 직접 접근 금지, rooms.service 경유).
- **이중 방어:** 신선 재검증은 **과거·영업시간외·휴무**를 거른다. **이미 예약된 슬롯**은 `get_room_slots`가 4.9 배선 전이라 안 빠질 수 있으나 `uq_reservation_slots_room_slot` UNIQUE가 INSERT 시 최종 차단(→ 전체 ROLLBACK·409). 따라서 4.5는 4.9 없이도 안전.
- **에러 매핑(신규 코드 0):** "요청 슬롯이 현재 예약 불가"(과거·영업시간외·휴무·타예약 점유) = **`SLOT_CONFLICT`(409)** 단일화([[availability-freshness-policy]] 확정 시 409 차단). 입력 형식 위반(빈·naive·중복·교차일) = **`VALIDATION_ERROR`(422)**. service `ValueError`(빈/naive/중복)가 500으로 새지 않게 **스키마/라우터에서 선차단**.

### 4.5/4.6 경계 — happy-path 확정은 4.5, 동시성 특화 UX는 4.6 (범위결정 #3 · 에픽 경계)

- **4.5 소유:** POST 엔드포인트(all-or-nothing·신선 재검증)·SDK·`예약하기` CTA·POST 제출·"예약이 완료됐어요!" 인라인 성공·selection 초기화·슬롯/핀 invalidate·**제출 실패 generic 우아 안내+재시도**(막다른 화면 금지).
- **4.6 소유:** 동시성 불변식(`ON CONFLICT` 후 affected=N 검증·겹치는 연속 구간 동시 확정 테스트·SM-4)·**SLOT_CONFLICT 특화 프론트 UX**("앗, 방금 다른 분이 먼저 잡았어요. 가까운 빈 시간을 다시 보여드릴게요." + 인접 빈 슬롯 즉시 재표시·EXPERIENCE L136·L205). 백엔드 409 SLOT_CONFLICT 자체는 4.5에 이미 존재(4.1 변환) — 4.6은 그 **동시성 보강 + 특화 UX**.
- **즉, 4.5 프론트는 모든 실패를 generic으로 우아 처리**하고, 4.6이 409만 특화한다. 4.5에 SLOT_CONFLICT 전용 카피·인접 재표시 만들지 않는다(dead UX·중복 금지).

### 성공 후 동작 — 상세 유지 + 인라인 배너 (범위결정 #4)

- "예약이 완료됐어요!" `role="status"` 인라인 배너(자동소멸 toast 아님 — sonner 미도입 일관·NetworkNotice 패턴). 페이지 유지(상세). `selection` 초기화 + 정확 키 invalidate(`["room",roomId,"slots",date]`·핀 집계). **`/reservations` 이동 없음**(4.8 미구축 → dead link 회피). 예약현황으로의 이동/링크는 4.8이 추가.
- **옵티미스틱 금지:** 예약은 서버 확정이 진실(충돌 가능) → 응답 성공 후에만 성공 표시. favorites 옵티미스틱 패턴을 **그대로 베끼지 말 것**(토글과 달리 예약은 실패 시 잘못 잡힌 것처럼 보이면 안 됨).
- **4.9 상호작용:** invalidate 후 슬롯 재조회 시 방금 잡은 슬롯이 (4.9 차감 배선 전엔) 여전히 `available`로 보일 수 있음 — 그건 4.9(`confirmed_slot_starts` → `derive_slots` 배선) 소관. 4.5는 invalidate까지·실제 차감 표시는 4.9. (혼동 방지: 4.5에서 임시 차감 로직 만들지 말 것 — 4.9 단일 지점.)
