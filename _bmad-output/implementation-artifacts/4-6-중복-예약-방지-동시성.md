# Story 4.6: 중복 예약 방지 (동시성)

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 예약자,
I want 같은 시간에 다른 사람과 동시에 눌러도 한쪽만 성공하고 실패해도 우아하게 안내받길,
so that 잘못 잡힐 걱정 없이 안심하고 예약한다 (FR-15, UJ-1 edge, NFR-7, SM-4).

> **이 스토리는 "검증 + 특화 UX" 스토리다 — 새 데이터 계층/엔드포인트를 만들지 않는다.** 동시성 불변식(정확히 하나 성공·부분 점유 0·all-or-nothing·실패 시 409)은 **이미 4.1 + 4.5에 구현 완료**다: ① 4.1 `create_reservation`이 `uq_reservation_slots_room_slot` UNIQUE + 단일 트랜잭션 다중행 INSERT → 어느 슬롯이라도 충돌 시 **전체 ROLLBACK(0건) → `SLOT_CONFLICT`(409)** ② 4.5 POST 엔드포인트가 이를 소비하고 모든 실패를 **generic**으로 우아 처리. 4.6은 그 위에 **(A) 동시성 불변식을 자동화 테스트로 증명(SM-4)** 하고 **(B) `SLOT_CONFLICT`(409)에만 특화 프론트 UX**(EXPERIENCE L136 카피 + 인접 빈 슬롯 즉시 재표시)를 얹는다. 4.5가 명시적으로 4.6에 남긴 seam(`ReservationPanel.tsx` L14·L176 "SLOT_CONFLICT 특화/인접 재표시는 4.6")을 채운다.

## 범위 결정 (KTH 확정 2026-06-17 — create-story)

이 스토리가 닿는 세 경계를 다음과 같이 확정한다:

1. **백엔드 동시성 메커니즘 = 기존 유지·검증 전담(BE 프로덕션 코드 변경 0).** 4.1 `create_reservation`의 'plain 다중행 INSERT + UNIQUE 위반 → 전체 ROLLBACK → `SLOT_CONFLICT`'를 **그대로 둔다**. 에픽 AC1 문구의 `ON CONFLICT 후 affected=N 검증, 아니면 ROLLBACK`은 **전체 ROLLBACK 방식으로 등가 충족**된다 — 전체 ROLLBACK은 affected=N 카운팅 없이도 all-or-nothing(부분 점유 0건)을 **구조적으로 더 강하게** 보장한다(부분 커밋 자체가 불가능). 따라서 4.6은 증명된 서비스를 재작성하지 않고(회귀 위험 0), 동시성 불변식을 **자동화 테스트로 증명**하며 이 등가성을 Dev Notes에 문서화한다. [Source: epics.md#Story 4.1 L720-736 / 4.6 L818-830 / architecture.md L118 / reservations/service.py:71-136]

2. **동시성 테스트 = 둘 다(결정적 직렬 + 진짜 race).** ① **결정적 직렬 불변식 테스트(SQLite·always-on, 라이브 DB 불요):** 겹치는 연속 구간(14~17시 vs 16~18시)을 **순차** 확정 시 둘째 예약은 단 하나의 슬롯도 점유하지 않고(부분 점유 0건) 전체 0건이며 `SLOT_CONFLICT`, 동일 슬롯 순차 확정 시 정확히 하나만 성립함을 단언한다(메커니즘 증명·매 게이트 실행). ② **진짜 멀티스레드 race 테스트(skipif `TEST_DATABASE_URL` Postgres):** 동일 슬롯에 N개 스레드가 **동시** 확정 → 정확히 1건 201·나머지 409 `SLOT_CONFLICT`·슬롯 점유 중복 0건(진짜 동시성 = SM-4 본격 검증). 라이브 DB 정책([[dev-workflow-policy-live-db-migration]])·기존 `test_reservations_migration.py` skipif 선례와 정합. [Source: epics.md#Story 4.6 L830(SM-4) / tests/integration/test_reservations_migration.py]

3. **프론트 = 재조회 + 특화 카피 + selection 초기화(신규 슬롯 로직 0).** `SLOT_CONFLICT`(409) 감지 시 ① 그날 슬롯 쿼리(`["room", roomId, "slots", selectedDate]`)를 invalidate → 기존 `SlotGrid` 재조회(그리드 자체가 "인접 빈 슬롯 재표시") ② "앗, 방금 다른 분이 먼저 잡았어요. 가까운 빈 시간을 다시 보여드릴게요." 특화 카피 ③ stale `selection` 초기화(사용자가 새로고침된 그리드에서 재선택). **신규 슬롯 계산 로직 0**(nearest-available 하이라이트 미도입). **데이터 정확도 한계는 4.9 소관 명시:** 4.9(예약 차감) 배선 전이라 `get_room_slots`가 `reserved_starts=frozenset()`이므로, 재조회해도 **방금 남이 잡은 슬롯이 여전히 `available`로 보일 수 있다** — 인접 빈 슬롯 재표시의 *데이터 정확성*은 4.9가 완성한다(4.6은 재조회·특화 UX·재선택 경로까지). [Source: ux-designs/…/EXPERIENCE.md L136·L205 / epics.md#Story 4.6 L826-828 / rooms/service.py get_room_slots reserved_starts seam]

## Acceptance Criteria

**AC1 — 동시 확정 시 정확히 하나만 성공 (동일 슬롯) — 자동화 테스트로 증명 (FR-15, SM-4, 범위결정 #1·#2)**
**Given** 동일 슬롯에 대한 동시 확정 시도
**When** 두(이상) 예약이 경합하면
**Then** 정확히 하나만 성공한다(슬롯 점유 중복 0건). 예약은 선택 슬롯 전체에 대해 단일 트랜잭션 다중행 INSERT로 원자적으로 성립/실패하며(`uq_reservation_slots_room_slot` UNIQUE 위반 시 **전체 ROLLBACK → 0건**), 이는 에픽 AC1의 `ON CONFLICT 후 affected=N 검증, 아니면 ROLLBACK`을 **등가(전체 ROLLBACK이 부분 점유를 구조적으로 불가능하게 함)로 충족**한다.
- **검증:** ① 결정적 직렬 테스트(SQLite·always-on) — 동일 슬롯 두 번째 확정 시 `SLOT_CONFLICT`·점유 행 1개만 잔존. ② 진짜 멀티스레드 race 테스트(skipif `TEST_DATABASE_URL` Postgres) — N스레드 동시 확정 시 201 정확히 1건·409 N-1건·`reservation_slots` 해당 슬롯 행 정확히 1개.
- **BE 프로덕션 코드 변경 0**(4.1 `create_reservation` 무변경 — 범위결정 #1).

**AC2 — 겹치는 연속 예약의 동시 확정: 한쪽 전체 성립·다른 쪽 부분 점유 0 (FR-15, SM-4)**
**Given** 겹치되 시작 슬롯이 다른 두 연속 예약(14~17시 vs 16~18시)의 동시 확정
**When** 경합하면
**Then** 한쪽 예약 **전체**가 성립하고 다른 쪽은 **단 하나의 슬롯도 점유하지 않는다**(부분 점유 0건 — 16·17시가 한쪽에만 잡히고 다른 쪽이 일부만 성립하는 상태 없음).
- **검증:** 결정적 직렬 테스트(14~17 확정 성공 후 16~18 확정 → 16·17 겹침으로 `SLOT_CONFLICT`·16~18은 18시 빈 슬롯조차 점유 0건, 전체 ROLLBACK) + (가능 시) 진짜 race 변형. all-or-nothing의 **부분 점유 0**이 핵심 단언.

**AC3 — 실패 측: SLOT_CONFLICT 특화 안내 + 인접 빈 슬롯 즉시 재표시 (FR-15, UX-DR10, EXPERIENCE L136·L205, 범위결정 #3)**
**Given** 실패한 쪽(409 `SLOT_CONFLICT` 반환)
**When** 충돌이 반환되면
**Then** **"앗, 방금 다른 분이 먼저 잡았어요. 가까운 빈 시간을 다시 보여드릴게요."** 특화 카피가 표시되고(에러코드 노출 금지), 그날 슬롯이 즉시 재조회되어(`["room", roomId, "slots", selectedDate]` invalidate → `SlotGrid` 재표시 = 인접 빈 슬롯 재노출) stale `selection`이 초기화되어 사용자가 재선택할 수 있다(막다른 화면 금지).
- **409 판별:** SDK throw 에러 본문 `{detail:{code,message}}`의 `detail.code === "SLOT_CONFLICT"`로 식별한다(`RoomDetail.tsx`의 `isRoomNotFound` 패턴 미러 → `isSlotConflict` 헬퍼). HTTP 상태는 에러 객체에 직접 안 실리므로 **코드로 분기**(1.5 에러 계약).
- **generic 실패와 분기:** `SLOT_CONFLICT`만 특화. 그 외 실패(404·5xx·기타)는 4.5의 generic 안내("예약을 완료하지 못했어요. 다시 시도해 주세요." + selection 유지·"다시 시도")를 **그대로 유지**(무회귀).
- **데이터 정확도(4.9 경계):** 4.9 차감 배선 전이라 재조회 시 방금 점유된 슬롯이 여전히 `available`로 보일 수 있다 — 재조회·특화 UX·재선택 경로는 4.6, **실제 차감 표시는 4.9**(임시 차감 로직 만들지 말 것 — 4.9 단일 지점).

**AC4 — generic 실패 UX 무회귀 + 4.5 동작 보존 (4.5 보존, 막다른 화면 금지)**
**Given** `SLOT_CONFLICT`가 아닌 실패(404·5xx·네트워크 단절)
**When** 응답이 오면
**Then** 4.5의 generic 안내 + 재시도(selection 유지)가 그대로 동작하고, 단절은 `showSubmitError = isOnline && isError` 게이팅으로 `NetworkNotice`에 일원화된다(단절을 서버 충돌로 오인 금지).
- 4.5 happy-path(성공 배너·selection 초기화·정확 키 invalidate)·4.3/4.4 분기(로딩/실패/빈날/단절·선택 상호작용·요약·키보드/SR) **전부 무회귀**.

**AC5 — 동시성 불변식 자동화 테스트 통과 (SM-4, 범위결정 #2)**
**Given** 동시성 불변식(동일 슬롯 + 겹치는 연속 구간)
**When** 테스트 스위트를 실행하면
**Then** 결정적 직렬 불변식 테스트(SQLite·always-on)가 매 게이트에서 통과하고, 진짜 멀티스레드 race 테스트(skipif `TEST_DATABASE_URL`)가 라이브 DB 환경에서 통과한다(CI에 Postgres 없으면 자동 skip → 회귀 0).

### 동시성 메커니즘 — 왜 전체 ROLLBACK이 에픽 AC를 등가 충족하는가 (범위결정 #1)

- **에픽 AC1 문구:** "단일 트랜잭션 multi-row INSERT(`ON CONFLICT` 후 affected=N 검증, 아니면 ROLLBACK)". 이는 "넣은 N개가 다 들어갔는지 affected 수로 확인하고, 아니면 되돌린다"는 한 가지 구현 제안이다.
- **4.1 구현(유지):** `ON CONFLICT` 절 없이 plain INSERT → 어느 슬롯이라도 UNIQUE 위반 시 **`IntegrityError` → 전체 ROLLBACK(0건) → `SLOT_CONFLICT`**. 이는 **affected=N 카운팅 없이도 all-or-nothing을 더 강하게** 보장한다: `ON CONFLICT DO NOTHING`은 충돌 행을 조용히 건너뛰어 부분 커밋 가능성을 만들고 affected 비교가 *추가로* 필요한 반면, 전체 ROLLBACK은 **부분 커밋 자체가 구조적으로 불가능**하다(트랜잭션 전부 또는 전무). 결과 불변식(정확히 하나·부분 점유 0·409)은 동일하므로 4.1을 재작성하지 않는다(증명된 코드 변경·회귀 위험 회피).
- **따라서 4.6 BE = 테스트만.** `create_reservation`·router 무변경. 테스트가 이 등가성을 명시 단언한다.
[Source: architecture.md L118 / reservations/service.py:78-136 docstring / epics.md#Story 4.6 L820]

### 동시성 테스트 — 직렬(메커니즘) + 진짜 race(SM-4) (범위결정 #2)

- **왜 둘 다:** SQLite는 단일 라이터라 진짜 race를 못 만든다(직렬화됨) → 메커니즘은 증명하나 SM-4의 "동시"는 미검증. 진짜 멀티스레드 race는 Postgres가 UNIQUE 위반을 커밋 시점에 직렬화하는 실제 동작을 본다. 결정적 직렬은 **매 게이트** 돌아 회귀를 잡고, race는 **라이브 DB에서만**(skipif) SM-4를 본격 검증한다.
- **race 테스트 실측:** 스레드별 **독립 Session/connection**(공유 세션이면 진짜 경합 아님). `threading.Barrier(N)`로 동시 출발 정렬 후 동시 `create_reservation`. 결과 = 201 정확히 1·`SLOT_CONFLICT` N-1·점유 행 1개. `ThreadPoolExecutor` + `future.result()`로 예외(DomainError) 집계.
- **기존 골격 재사용:** `test_reservations_migration.py`가 이미 skipif·alembic·monkeypatch env·`get_engine.cache_clear()`·전용 테스트 DB 경고를 갖춤. race 테스트를 같은 파일에 추가하거나 동형 신규 파일로. **라이브 Supabase 금지**(`upgrade head`/`downgrade base` 파괴적 — 전용 테스트 DB만).
[Source: tests/integration/test_reservations_migration.py / epics.md#Story 4.6 L830(SM-4)]

### 프론트 SLOT_CONFLICT UX — 재조회 + 특화 카피 + selection 초기화 (범위결정 #3)

- **409 판별:** SDK는 `throwOnError` 시 파싱된 본문 `{detail:{code,message}}`를 throw(HTTP 상태 미동봉) → `detail.code`로 분기. `isRoomNotFound`(RoomDetail.tsx:44-54) 그대로 미러해 `isSlotConflict`.
- **분기 동작:**
  - `SLOT_CONFLICT` → 특화 카피("앗, 방금 다른 분이 먼저 잡았어요. 가까운 빈 시간을 다시 보여드릴게요.") + `["room",id,"slots",date]` invalidate(훅 onError) + `selection` 초기화 → 새로고침된 `SlotGrid`에서 재선택.
  - 그 외 실패 → 4.5 generic(selection 유지·"다시 시도") **무변경**.
  - 단절 → `NetworkNotice` 우선(`showSubmitError = isOnline && isError`로 제외) — 단절을 충돌로 오인 금지.
- **렌더 배치 주의:** 4.5의 generic 실패 카피는 `safeSelection` 블록 **안**에 있다. SLOT_CONFLICT는 selection을 초기화하므로(요약 바·CTA 사라짐), **특화 카피는 그 블록 밖(슬롯 영역 레벨)에서 selection 무관하게** 렌더해야 selection 초기화 후에도 보인다 + 새로고침된 그리드로 재선택 유도. 막다른 화면 금지.
- **set-state-in-effect 회피(3.5/3.6 실발생):** selection 초기화는 mutation 콜백(`mutate(..., { onError })`)/이벤트 핸들러에서. effect setState 0. 카피 표시는 `isError && isSlotConflict(error)` 파생(렌더 중 계산·상태 추가 불요).
