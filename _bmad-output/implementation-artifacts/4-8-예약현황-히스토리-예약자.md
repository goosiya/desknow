## Story

As a 예약자,
I want 다가오는 예약과 지난 예약을 한 곳에서 보길,
so that 내 예약을 관리하고 취소·상세로 이어간다 (FR-17, UJ-1·2).

> **이 스토리는 예약자 본인의 예약 목록 FE 표면을 처음 만드는 "수직 슬라이스" 스토리다.** 지금까지 예약자가 자기 예약을 보는 화면이 **없다**(`/reservations`는 placeholder — `apps/web/src/app/reservations/page.tsx:1-11`, Story 1.6 셸 자리). 4.5(즉시 예약)·4.7(취소 엔드포인트)이 데이터·엔드포인트를 깔았고, 4.7이 **"취소 버튼·6h 비활성 UX·cancellable 노출"을 명시적으로 4.8에 위임**했다(4-7 범위 결정 #1·#3). 따라서 4.8 = **(A) 예약자 본인 예약 목록 조회 엔드포인트**(`GET /reservations`) + **(B) 예약 시간 스냅샷**(취소/거절해도 히스토리에 시간 잔존 — 아래 범위 결정 #1) + **(C) `/reservations` 페이지 FE**(다가오는/지난 구분·룸 상세 이동·취소 버튼[6h 비활성+안내]·상태 매트릭스) + **(D) SDK 재생성**. 취소 호출은 4.7 `reservationsCancelReservation`을 **소비**하고, 슬롯/핀 가용성 invalidate는 **4.9**에 위임한다(범위 결정 #2).

## 범위 결정 (KTH 확정 2026-06-17 — create-story)

이 스토리가 닿는 두 경계를 다음과 같이 확정한다:

1. **취소/거절 예약의 시간 히스토리 = reservations 행에 슬롯 스냅샷 저장(마이그레이션 1 + 4.1 `create_reservation` 확장).** 4.1 설계상 `reservation_slots` 행은 취소/거절 시 **DELETE**된다(슬롯 재활성 = FULL `uq_reservation_slots_room_slot` UNIQUE 불변식의 근거 — `models.py:128-176`). 그 결과 취소/거절 예약은 점유 행이 0건이 되어 **원래 예약 시간 정보가 사라진다**. 에픽 AC는 "지난(이용 완료/취소/거절)" 예약을 모두 히스토리로 요구하므로, **`Reservation`에 슬롯 시작시각 스냅샷을 생성 시 1회 저장**(immutable — 취소/거절해도 잔존)해 히스토리에서 정확한 날짜·시간을 표시한다. **이 스냅샷은 표시 전용**이며 점유의 진실의 원천(`reservation_slots`)·4.9 차감 불변식은 **그대로 유지**한다(스냅샷은 `derive_slots`/가용성 집계에 절대 쓰이지 않음 — 활성 점유만이 차감 대상). [Source: epics.md#Story 4.8 L852-868 / reservations/models.py:74-176 / KTH 확정]

2. **취소 배선 경계 = 4.8은 취소 호출 + `['reservations']` 자기 list 갱신만, 슬롯/핀 가용성 invalidate는 4.9.** 4.8 list 행의 취소 버튼은 4.7 `reservationsCancelReservation`(SDK)을 소비하고, 취소 성공 시 **`['reservations']` 자기 목록만 invalidate**(취소된 예약이 다가오는→지난[취소됨] 섹션으로 즉시 이동 반영)한다. **슬롯/핀 가용성 invalidate**(`['room', roomId, 'slots', date]`·`['rooms', 'availability']`)는 **Story 4.9**(가용성 차감·갱신 단일 지점)가 소유한다 — 4.9 차감이 아직 배선되지 않아(`reserved_starts=frozenset()`) 취소 후 슬롯/핀을 invalidate해도 실제 UI 변화가 없으므로, 차감 배선과 함께 4.9에서 일괄 처리한다(4.5/4.7이 이미 동일 경계로 위임 — 4-7 Dev Notes "4.7/4.8/4.9 경계"). 4.8에 임시 슬롯/핀 invalidate를 만들지 말 것. [Source: 4-7 L164-168 / epics.md#Story 4.9 L882-884 / KTH 확정]

## Acceptance Criteria

**AC1 — 다가오는/지난 예약 구분 표시 + 룸 상세 이동 (FR-17, UJ-1·2)**
**Given** 로그인한 예약자가 예약현황 화면(`/reservations`)에 진입하면
**When** 본인 예약이 있으면
**Then** **다가오는 예약**(`confirmed` + 마지막 슬롯 종료가 현재 이후)과 **지난 예약**(이용 완료=`confirmed`이고 모든 슬롯이 과거 / 취소됨=`cancelled` / 거절됨=`rejected`)이 **시각적으로 구분된 섹션**으로 표시되고, 각 예약 행에서 **룸 상세(`/rooms/{room_id}`)로 이동**할 수 있다.
- 각 행은 **룸 이름 · 예약 날짜·시간 범위(예: "6월 20일 14:00–17:00") · 상태 배지**(확정/이용 완료/취소됨/거절됨 — 색 단독 금지, 텍스트 동반)를 표시한다.
- 취소/거절 예약도 **스냅샷 시간**(범위 결정 #1)으로 날짜·시간을 표시한다(시간 정보 누락 없음 — 단, 스냅샷 도입 *이전*에 이미 취소된 레거시 행은 시간 미표시 허용 — 마이그레이션 백필 한계).

**AC2 — 취소 가능 예약의 취소 동작 연결 + 6h 비활성 (FR-16·17, 4.7 위임)**
**Given** 다가오는(`confirmed`) 예약 행
**When** 예약 시작(스냅샷 슬롯 중 **가장 이른** 시작시각)까지 **6시간 이상** 남았으면
**Then** **취소 버튼이 활성**되고, 누르면 4.7 취소 엔드포인트(`reservationsCancelReservation`)를 호출해 취소하며, 성공 시 해당 예약이 **지난(취소됨) 섹션으로 즉시 이동**한다(`['reservations']` 재조회).
**And Given** 시작까지 **6시간 미만** 남은 `confirmed` 예약
**Then** 취소 버튼이 **비활성**되고 **"이제 6시간이 안 남아서 취소가 어려워요."** 안내가 표시된다(에픽 4.7 AC1의 UI 책임 — FE 계산).
**And** 취소/거절/이용 완료 예약에는 취소 버튼을 **노출하지 않는다**.
- **서버 권위 최종 강제(클럭 스큐 graceful):** FE가 6h 계산으로 활성 표시했더라도 클럭 스큐로 서버가 **409 `CANCEL_WINDOW_PASSED`**를 반환하면, "방금 취소 가능 시간이 지났어요." 등 **친절한 안내 + 목록 재조회**(버튼 상태 갱신)로 우아하게 처리한다(막다른 화면·에러코드 노출 금지). 4.7 엔드포인트가 6h 경계를 timedelta로 최종 강제한다(`service.py:217-269`).
- **6h 계산:** `earliest_slot_start - now >= 6시간`이면 취소 가능(순수 duration — ROOM_TZ 불요, 4.7 범위 결정 #3). `earliest`는 스냅샷 슬롯 중 최소값.

**AC3 — 빈/에러/네트워크 단절/로딩 상태 (UX-DR9, NFR-5)**
**Given** 예약현황 화면
**When** 각 상태에 진입하면
**Then** 다음이 각각 설계된다(막다른 화면 금지 — FavoriteList 선례 `FavoriteList.tsx` 미러):
- **세션 판별 중** → 스켈레톤. **세션 판별 실패**(네트워크/5xx) → "로그인 상태를 확인하지 못했어요." + 다시 시도(로그아웃 UI로 오인 금지).
- **미로그인**(401→`user=null`) → "로그인하면 예약 내역을 볼 수 있어요." + 로그인 유도 링크(전용 로그인 화면은 별도 스토리).
- **네트워크 단절**([[terminology-network-disconnect-not-offline]]) → 캐시된 목록 있으면 행 + `NetworkNotice`, 없으면 `NetworkNotice`만(에러보다 우선·재연결 시 자동 재조회).
- **로딩** → Skeleton 행(전역 스피너 금지). **에러**(온라인 진짜 실패) → "예약 내역을 못 불러왔어요." + 다시 시도.
- **빈** → "아직 예약이 없어요. 마음에 드는 곳을 찾아볼까요?" + 찾기(`/`) 링크(EXPERIENCE.md 빈 상태 카피 정신).

**AC4 — 본인 예약만 조회 (RBAC·소유권, FR-17·architecture.md §Boundaries)**
**Given** `GET /reservations` 요청
**When** 인증된 booker가 호출하면
**Then** **본인(`booker_id == principal.user_id`) 예약만** 반환한다(타인 예약 노출 0). 미인증=401 `UNAUTHENTICATED`, provider/admin=403 `FORBIDDEN_ROLE`(`require_role("booker")` — 4.5/4.7 `_require_booker` 싱글톤 재사용). 예약이 없으면 빈 리스트(정상 200).
- 각 행에 **룸 이름**을 싣되, reservations.service는 **rooms를 import하지 않는다**(4.9 역방향 의존 → 순환 import 방지 — Dev Notes "도메인 경계"). 이름 합성은 **라우터(조합 계층)**가 `session.get(Room, …)`으로 수행한다(favorites 선례). 비활성 룸의 예약도 이름·히스토리를 표시한다(상세 진입은 룸 상태에 따름).

### 슬롯 시간 스냅샷 설계 (범위 결정 #1 — KTH 확정)

- **문제:** `reservation_slots`는 취소/거절 시 DELETE되어(`_release_slots` — `service.py:145-156`) 종료 상태 예약은 점유 행 0건 → 시간 정보 소실. 그러나 4.1 설계상 이 DELETE가 **FULL UNIQUE(room_id, slot_start)의 근거**(활성 점유만 잔존)라 바꿀 수 없다.
- **해법(KTH 확정):** `Reservation`에 **`slot_starts` 스냅샷 컬럼**(JSON, UTC ISO `...Z` 문자열 오름차순)을 추가하고 `create_reservation`이 **생성 시 1회 기록**. 취소/거절 전이는 이 컬럼을 **건드리지 않는다**(immutable 히스토리). 표시는 이 스냅샷에서, 점유·차감·UNIQUE는 `reservation_slots`에서 — **두 출처의 역할이 다름**(스냅샷=표시 전용·점유행=불변식). **4.9 차감 불변식 그대로 유지**(스냅샷은 `derive_slots`/가용성에 절대 미사용).
- **왜 ISO 문자열 + `sa.JSON`:** ① `sa.JSON`이 datetime 직렬화 불가 → ISO `...Z` 문자열. ② UTC `...Z`는 사전식=시간순이라 `min()`=earliest(파싱·정렬 불요). ③ `sa.JSON`은 SQLite(단위 테스트)+Postgres 양쪽 동작(`rooms.amenities`의 postgresql `JSONB`는 SQLite 불가라 미러 금지). ④ 응답이 이미 `...Z`라 직렬화도 그대로.
- **마이그레이션 백필(중요):** 신규 컬럼 NOT NULL이므로 기존 행 처리 필요. **confirmed 예약** = `reservation_slots`에서 `slot_start` 모아 백필. **이미 취소/거절된 레거시 행** = 슬롯 0건 → `[]`(시간 정보 이미 소실 — AC1 "스냅샷 도입 이전 레거시는 시간 미표시 허용"). 마이그레이션: 컬럼 nullable 추가 → 데이터 백필 → NOT NULL·default `[]`(또는 `server_default`). 라이브 Supabase엔 4.5/4.7 dev 예약이 있을 수 있으니 백필 실측.
- **비-연속 슬롯 주의:** `create_reservation`은 임의 슬롯을 받지만(4.4 UI만 연속 강제) 표시 범위는 min..max+1h로 요약한다(연속 전제). 스냅샷은 **전체 리스트**라 비연속도 보존(향후 정밀 표시 가능).

### 본인 예약 목록 엔드포인트 — `GET /reservations`(top-level) (AC4)

- **왜 top-level인가:** "내 예약"은 **룸 비결합**(여러 룸에 걸친 본인 예약)이라 기존 중첩 `/rooms/{room_id}/reservations`(생성·취소)와 별개다. `favorites`(`/favorites`)와 동형으로 top-level `GET /reservations`를 둔다 → 새 `APIRouter(prefix="/reservations")` 인스턴스(같은 `tags=["reservations"]`).
- **도메인 경계(architecture.md L354 — 핵심):** `reservations.service`는 **rooms를 import하지 않는다**. 이유: **4.9가 `rooms.service → reservations.service`(confirmed_slot_starts) 역방향 의존을 추가**하므로, reservations.service가 rooms를 import하면 **순환 import**가 된다(service docstring `service.py:29-31`가 명시: *"rooms를 import하지 않는다 — 4.9가 역방향으로 호출"*). 따라서 **룸 이름 합성은 라우터(조합 계층)**가 `session.get(Room, …)`로 한다(favorites가 service에서 하는 것과 달리, reservations는 4.9 제약 때문에 라우터에서). 라우터→`rooms.models` import는 service↔service 순환이 아니라 안전.
- **flat 목록 + FE 분류:** 서버는 본인 예약을 `created_at desc` flat 반환(favorites 선례). 다가오는/지난 분류·정렬은 **FE**가 `now` 기준 계산(서버 시계 의존·캐시 신선도 문제 회피). cancellable도 FE 계산(4.7 범위 결정 #3) — 서버는 cancellable 필드 미노출, 취소 엔드포인트가 6h를 최종 강제.

### 6h 취소 가능 계산 (AC2 — 4.7 위임)

- **FE 계산:** `cancellable = status === "confirmed" && (earliest_slot_start - now) >= 6h`. `earliest = slot_starts[0]`(스냅샷 오름차순 보장). 순수 duration(두 UTC 인스턴트 차)이라 ROOM_TZ 불요(4.7 범위 결정 #3). <6h면 버튼 비활성 + "이제 6시간이 안 남아서 취소가 어려워요."(에픽 4.7 AC1 카피와 동일).
- **서버 최종 강제:** FE가 활성 표시했어도 클럭 스큐로 6h 경과 시 서버가 409 `CANCEL_WINDOW_PASSED`. FE는 `isCancelWindowPassed`로 잡아 친절 안내 + 목록 재조회(버튼 갱신). 4.7 service가 timedelta로 경계 강제(`service.py:262-266`).
