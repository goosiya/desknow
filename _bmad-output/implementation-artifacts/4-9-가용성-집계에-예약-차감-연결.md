## Story

As a 개발자,
I want Story 3.1의 가용성 집계와 Story 4.3의 슬롯 표시가 실제 예약을 차감하도록 연결하길,
so that 핀 색·슬롯 가용성이 실제 점유를 정확히 반영하고 예약 확정·취소 직후 즉시 갱신된다 (FR-5·FR-12 완성, NFR-2).

## Acceptance Criteria

1. **(AC1 — 차감 반영)** `reservation_slots`에 활성 점유 행이 존재할 때, 가용성 집계(3.1 `aggregate_availability`)와 슬롯 도출(4.3 `get_room_slots`) 및 단일 룸 신선 카운트(`get_room_summary`·`room_remaining_slots`)·목록 카운트(`search_rooms`)가 (영업시간 − 휴무일 − **예약**) 규칙으로 계산되어, 예약된 슬롯이 차감/비활성으로 정확히 반영된다.
2. **(AC2 — 슬롯 그리드 표시)** `get_room_slots`에서 예약된 슬롯은 응답에서 **사라지지 않고** `status="reserved"`로 표시되어 SlotGrid가 비활성(취소선)으로 렌더한다. 단, `next_available_date`(다음 빈 날) 계산은 예약을 차감해 만석인 날을 건너뛴다.
3. **(AC3 — 즉시 갱신: 확정)** 예약 확정 뮤테이션 성공 시 관련 TanStack Query 키(`['room', roomId, 'slots', date]`·`['rooms','availability']`)가 invalidate되어 가용성이 즉시 갱신된다. (4.5에서 이미 배선됨 — 4.9 차감 배선으로 *실제 데이터 변화*가 표면화된다.)
4. **(AC4 — 즉시 갱신: 취소)** 예약 취소 뮤테이션 성공 시 슬롯/핀 가용성 키(`['room', roomId, 'slots']` prefix 전 날짜 + `['rooms','availability']`)가 추가로 invalidate되어, 재활성된 슬롯이 즉시 다시 가용으로 보인다. (4.8은 `['reservations']`만 invalidate했고 슬롯/핀은 4.9 위임.)
5. **(AC5 — stale 선택 무효화)** 슬롯 그리드를 보던 사용자의 선택 슬롯 중 하나라도 재조회로 `available`이 아니게 되면, 선택을 **전부 비우고** 중립 안내("선택한 시간 중 일부가 방금 예약됐어요. 다시 선택해 주세요.")를 표시한 뒤 재선택을 유도한다(막다른 화면 금지). render-time 파생으로 set-state-in-effect 회피(반복함정 #7).

### ⚠️⚠️ 최대 함정 — `get_room_slots`는 "차감(소멸)"이 아니라 "reserved 표시(비활성)"

`derive_slots`는 마지막에 `sorted(starts - reserved)`로 **예약 슬롯을 결과에서 제거**한다(service.py:151). 따라서:

- **카운트 4곳**(aggregate/summary/remaining/search): "남은 빈 슬롯 수"이므로 차감(제거)이 **정답** — `derive_slots(..., reserved_starts=reserved)`로 빼고 `>= current` 카운트.
- **`get_room_slots` 표시 슬롯**: 예약 슬롯이 **사라지면 안 된다** — AC는 "슬롯 **비활성**에 반영"(취소선·범례), 슬롯 그리드는 예약된 시간도 보여주되 선택 불가여야 한다. 따라서 derive_slots엔 `frozenset()`(전 슬롯 유지)를 주고, **`_slot_status`에 실제 reserved set**을 줘서 `status="reserved"`로 표기한다. `_slot_status`는 `reserved → past → available` 우선순위로 이미 구현됨(service.py:453, 4.3에서 미리 설계).
- **`get_room_slots`의 `next_available_date`**: 만석인 날은 "가용"이 아니므로 차감 적용 — `_next_available_date`가 후보 날짜마다 `derive_slots(..., reserved_starts=reserved)`로 빼야 정확. reserved set은 전 날짜를 포함하지만 instant 매칭이라 각 후보 날짜의 슬롯에만 적용됨(타 날짜 슬롯과 우연 일치 불가 — UTC 정시 인스턴트).

이 비대칭(표시=유지+표기 / 카운트·next=차감)을 놓치면 슬롯 그리드에서 예약 시간이 통째로 사라져 "원래 영업 안 하는 시간"처럼 보이는 회귀가 난다.

### 도메인 경계 & 순환 import (검증 완료 — 안전)

`reservations.service`의 차감 reader를 `rooms.service`가 호출하는 **단방향** 의존이다(architecture.md L354 도메인 경계 준수 — rooms가 `reservation_slots` SQL을 직접 만지지 않고 reader 경유).

- 방향: `rooms.service → reservations.service → reservations.models → core`. **순환 없음**(실측 확인: `reservations.models`는 `app.core.time`만, `reservations.service`는 rooms를 import하지 않음).
- 4.1/4.8 docstring이 이 방향을 명시 설계: "4.9가 역방향으로 `rooms.service`에서 `confirmed_slot_starts`를 호출"(reservations/service.py:30-31), "service가 rooms를 import하면 4.9 역방향과 순환"(reservations/router.py:215-218 — 그래서 4.8 룸 합성을 라우터에 둠).
- **모듈 상단 import 사용**(함수-로컬 import 불요 — 진짜 순환이 없음). dev-story 중 만에 하나 ImportError 사이클이 뜨면 함수-로컬 지연 import로 폴백하고 사유 기록. import 안전 회귀 테스트(두 모듈 동시 import) 1건 권장.

### `on_or_after` 최적화 (권장)

reader에 `on_or_after=current`를 넘기면 과거 예약 슬롯이 집합에서 빠진다. 카운트/그리드 모두 `>= current`만 의미가 있어 **동작 보존**이면서 집합 크기를 미래로 제한한다(공개 무인증 엔드포인트 — 데이터 증가 시 set 비대 방지). 4.1 `confirmed_slot_starts`는 이미 `on_or_after` 파라미터를 지원(service.py:310). 벌크판도 동일 시그니처로 추가.

### FE: 취소 invalidation (AC4)

`useCreateReservation`(4.5)은 이미 `['room',roomId,'slots',date]`+`['rooms','availability']`를 invalidate한다(useCreateReservation.ts:43-46) — **변경 불요**. 4.9 차감 배선으로 그 invalidate가 비로소 *실제 데이터 변화*를 표면화한다.

`useCancelReservation`(4.8)은 `['reservations']`만 invalidate(useCancelReservation.ts:43). 취소는 슬롯을 재활성(DELETE)하므로 슬롯/핀이 갱신돼야 한다 → `onSuccess`에 `['room', roomId, 'slots']` **prefix**(취소 응답이 날짜를 안 주므로 — `slot_starts=[]` 반환 — 룸 단위 prefix가 전 날짜 슬롯 쿼리를 덮음) + `['rooms','availability']` 추가. TanStack `invalidateQueries`는 기본 prefix 매칭이라 `['room',roomId,'slots']`가 모든 `['room',roomId,'slots',<date>]`를 무효화한다. **광역 `['rooms']` 금지**(KTH 확정 — 룸 범위 prefix는 deferred L153 "지도/시트 캐시 휩쓸기" 위반 아님).

### FE: stale 선택 무효화 (AC5 — 의무회수)

[[dev-workflow-policy-deferred-and-repeat-mistakes]] — 트리거(4.9 차감 배선)가 **지금 발화**하므로 의무회수(재-defer 금지). 회수 대상:
- **deferred L214**(4.5): 백그라운드 refetch로 슬롯 내용 변경 시 stale selection 제출 — "실제 차감 반영·선택 무효화는 Story 4.9 seam".
- **deferred L224**(4.6): "동일 길이·다른 내용 배열이면 의도 외 슬롯 POST 가능. 4.9 차감 배선 시 selection 무효화 로직과 함께 해소".

핵심: 4.9 차감 후 예약 슬롯은 **인덱스가 안 밀린다**(사라지지 않고 `reserved`로 표시 — 같은 인덱스 status만 변경). 그래서 기존 `safeSelection`의 **bounds 가드(L75)는 통과**하지만 내용이 stale인 정확히 그 케이스(L224 "동일 길이·다른 내용")가 발생. 해결:
- `safeSelection` 조건에 **content 검사 추가**: 선택 구간 모든 슬롯 `status === "available"`(순수 헬퍼 `isSelectionStillAvailable`). 하나라도 아니면 `null`.
- **KTH 확정 = 전체 초기화 + 안내**: `selectionInvalidated`를 렌더-중 파생(`selection !== null && safeSelection === null && data`)해 중립 안내 표시. **setState-in-effect 금지**(반복함정 #7) — `selection` state를 effect/렌더-중 setState로 비우지 않고, 파생값 `safeSelection=null`로 CTA·요약을 숨기고 안내를 띄운다(사용자 관점=선택 비워짐, 재선택 시 `handleSelect`가 교체, 날짜 변경 시 `prevDate` 패턴이 리셋). `SlotGrid`엔 `safeSelection` 전달(stale 강조 차단).
- 우선순위: 제출 `showSlotConflict`(409 후) 카피와 동시 표출 금지 — conflict 우선.

기존 패턴 재사용: `prevDate` 렌더-중-조정(ReservationPanel.tsx:50-54)이 set-state-in-effect 회피의 canonical 선례. `safeSelection`(L75)이 이미 파생 가드의 자리 — content 검사만 얹는다.
