## Story

As a 예약자,
I want 예약이 임박하면(시작 24시간 이내) 접속 시 배너로 안내받고 원하면 '다시 보지 않기'로 숨기길,
so that 약속을 잊지 않으면서도 불필요한 반복 알림에 시달리지 않는다 (FR-18, UJ-1, UX-DR5).

이 스토리는 **Epic 5의 두 번째 스토리이자 5.1 인프라의 첫 실트리거(도출)**다. 5.1이 세운
영속 통지 기계(`notifications` 테이블·서비스 프리미티브·전역 배너·`useNotifications` 훅)를
**소비**해 도래 리마인드를 **읽기 시점에 도출**하고, '다시 보지 않기'로 사용자별 영속 억제한다.

**KTH 확정(2026-06-17, 본 스토리 — 5.1 확정 #1·#3 충돌 해소):**
> **도출-only + 리마인드 전용 dismiss (확정 #1 우선).** `reservation_reminder`는 **GET 시점에
> 행 없이 도출**(읽기전용·부작용 0)한다. '다시 보지 않기' 시에만 `reservation_id` 키 **억제행을
> born-dismissed로 생성**하는 **신규 엔드포인트** `POST /notifications/reminders/{reservation_id}/dismiss`
> 를 둔다. 5.1 generic `POST /notifications/{id}/dismiss`(status_change '확인')와 **dismiss
> 메커니즘(`dismissed_at`)은 공유**하되, 도출 리마인드는 행 id가 없어 **reservation_id 키 엔드포인트**가
> 별도다(5.1 확정 #3의 "동일 엔드포인트"는 "동일 dismiss 메커니즘"으로 완화 — #3 본문이 이미
> "리마인드=억제행 생성"으로 생성차이를 인정). 5.1 확정 #1 "평소 행 없음(도출)·'다시 보지 않기'
> 시 억제행 생성"을 글자 그대로 충족.

**왜 GET 도출(materialize-on-GET 아님):** GET을 읽기전용으로 유지(HTTP safe — 프리페치/캐시가
행을 만들지 않음)하고, 리마인드 노출마다 행이 쌓이지 않으며(억제건만 1행), `create_notification`
동시삽입 경합(deferred-work.md L246)을 읽기 경로에 노출하지 않는다.

## Acceptance Criteria

(epics.md Story 5.2 BDD + UX-DR5 + KTH 본 스토리 확정 기반)

**AC1 — 도래 리마인드 도출 (읽기 시점·부작용 0)**
**Given** 로그인 사용자의 **확정(`confirmed`)** 예약 중 **가장 이른 점유 슬롯이 지금부터 24시간 이내**(정확히 24시간, 달력일 아님)인 예약
**When** `GET /api/v1/notifications`에 접속하면
**Then** 해당 예약마다 `reservation_reminder` 통지 항목이 **행 생성 없이 도출**되어 응답에 포함된다(이미 '다시 보지 않기'한 예약·종료 상태[`cancelled`/`rejected`] 예약·24h 밖 예약은 제외). GET은 DB 쓰기를 하지 않는다(`commit`/`add` 0 — 억제건 조회만). 24h 경계 판정은 **`timedelta` 정수 비교**로 하며(부동소수 `hours_until` 미사용) **UTC 절대시각** 기준이다. [Source: deferred-work.md L103 의무 회수]

**AC2 — '다시 보지 않기' 억제 (사용자별 영속)**
**Given** 도래 리마인드 배너
**When** '다시 보지 않기'를 선택하면
**Then** `POST /api/v1/notifications/reminders/{reservation_id}/dismiss`(204·멱등)가 `(user_id, reservation_id, reservation_reminder)` **억제행을 born-dismissed로 생성**(`dismissed_at` 설정)해 **사용자별로 영속**되고, 같은 사용자 재접속 시 그 예약 리마인드는 더는 도출되지 않는다. 타 사용자/미존재/비소유 예약은 404 `NOTIFICATION_NOT_FOUND`(소유권 누설 금지). 멱등(재클릭·재시도 안전).

**AC3 — 배너 표시·카피·라벨 (UX-DR5)**
**Given** 도출된 리마인드 항목
**When** 전역 배너 슬롯이 렌더하면
**Then** 면 위 플랫(`bg-secondary`/`bg-muted` + 아이콘 + 텍스트, 색 단독 금지) 배너에 **룸 이름·시작 시각을 포함한 리마인드 카피**("○○ 예약이 곧 다가와요. {KST 날짜·시각}…")가 render-time 파생으로 표시되고, **닫기 버튼 라벨은 '다시 보지 않기'**(status_change='확인'과 분기)이며 클릭 시 AC2 억제를 호출한다(≥44px·`aria-label`). 출현은 슬롯 `aria-live`가 안내(푸시 아님 — 접속 시점에만).

**AC4 — 두 종류 공존·독립 트리거**
**Given** 같은 사용자에게 도래 리마인드(도출)와 상태변경 통지(`status_change` stored)가 동시에 있으면
**When** GET이 둘을 머지하면
**Then** 두 배너가 **독립 트리거**로 함께 표시되고(리마인드='다시 보지 않기'→억제행 / status_change='확인'→기존행 dismiss), 각자의 dismiss가 **상대를 건드리지 않는다**(키·엔드포인트 분리). SDK가 재생성되어 OpenAPI 드리프트 게이트(Layer A·B) 그린.

**AC5 — 결정성·경계·게이트**
**Given** 24h 경계 정밀도(deferred L103)·도메인 경계·반복 함정
**When** 구현하면
**Then** 24h 경계는 `timedelta` 비교 + **경계 테스트**(정확히 24h·24h 직전/직후·지금·이미 시작)로 회귀 고정되고, 도출 조합은 **라우터/조합 헬퍼**가 수행해 `notifications.service`는 자기 테이블만 만지며(rooms/reservations import 0·순환 회피), 백엔드 `pytest`/`ruff`/`mypy` + 웹 `vitest`/`lint`/`check-types`/`build` + `turbo` 전체 그린이다. **본 스토리는 새 마이그레이션이 없다**(5.1 `notifications` 스키마 재사용 — `type` CHECK에 `reservation_reminder` 이미 포함). 라이브 Supabase head 불변 확인(드리프트 0). ## Dev Notes

### 🎯 스토리 본질 (반드시 먼저 읽을 것)

이건 **5.1 인프라의 첫 실트리거(도출 소비)**다. 5.1이 "동작하는 통지 기계"를 세웠고, 5.2는

- **이 스토리가 만드는 것:** ① GET 도출(24h내 확정예약 → 행 없이 reminder 항목 머지) ② 리마인드 전용 dismiss 엔드포인트(`/reminders/{reservation_id}/dismiss` → born-dismissed 억제행) ③ 조합 헬퍼(`reminders.py`)·service 프리미티브 2종(`dismissed_reminder_reservation_ids`·`suppress_reminder`) ④ 24h 경계 정책(timedelta)·경계 테스트(L103 회수) ⑤ 배너 카피·라벨 분기 + 리마인드 dismiss 훅.
- **이 스토리가 만들지 않는 것:** 상태변경 통지 표시 정밀 카피(=5.3) · 거절/취소 시 status_change **생성** 배선(=6.2/8.3) · 모바일 배너(=E7) · N+1 배치 조회(=favorites 계열 deferred).
- **마이그레이션 없음:** 5.1 `notifications` 테이블이 `reservation_reminder` 행(type CHECK 포함·reason nullable)을 이미 수용. 5.2는 **행을 쓰는 코드만 추가**(억제행 생성)하고 스키마는 불변.

### 🔑 KTH 본 스토리 확정 (2026-06-17) — 5.1 #1·#3 충돌 해소

5.1의 KTH 확정 #1("평소 행 없음·다시보지않기 시 억제행 생성")과 #3("동일 `{id}` 엔드포인트
소비")은 **도출 리마인드엔 행 id가 없다**는 사실에서 충돌한다. 본 스토리에서 KTH는:

- **확정 #1 우선 = 도출-only + 리마인드 전용 dismiss.** GET은 **행 없이 도출**(읽기전용). '다시
  보지 않기'에서만 `reservation_id` 키 억제행을 **born-dismissed 생성**. dismiss **메커니즘**
  (`dismissed_at` 설정)은 5.1 프리미티브(`create_notification`+`dismiss_notification`)를 **그대로
  재사용**하되, 진입 **엔드포인트**는 도출 리마인드용(`/reminders/{reservation_id}/dismiss`)이
  별도다. #3의 "동일 엔드포인트"는 "동일 dismiss 메커니즘"으로 완화(#3 본문이 이미
  "리마인드=억제행 생성"으로 생성 경로 차이를 인정).
- **기각된 대안(materialize-on-GET):** GET이 reminder 행을 born-pending 생성해 generic
  `{id}` dismiss를 쓰는 안. GET 쓰기 부작용(HTTP safe 위반)·노출마다 행 누적·읽기 경로 create
  경합(deferred L246) 때문에 기각. #1 "평소 행 없음"과도 모순.

### 🧱 도출 로직 상세 (AC1)

```
GET /api/v1/notifications  (읽기전용 — DB 쓰기 0)
  now = now_utc()                                   # 1회 캡처
  pending      = service.list_pending(user_id)      # status_change(억제행은 born-dismissed라 미반환)
  suppressed   = service.dismissed_reminder_reservation_ids(user_id)   # {reservation_id, ...}
  reservations = reservations.service.list_booker_reservations(user_id)
  reminders = [ r for r in reservations
               if r.status == 'confirmed'
               and r.id not in suppressed
               and r.slot_starts                     # 방어(confirmed는 ≥1)
               and is_within_reminder_window(fromisoformat(r.slot_starts[0]), now) ]
  → items = [reminder→NotificationItem(id=None, slot_start=earliest, reason=None, created_at=None, room_name 합성)]  (slot_start asc)
          + [pending →NotificationItem(id=행id, slot_start=None, created_at=행값, room_name 합성)]  (created_at desc)
```

- **억제건 = `notifications` 행(type=reservation_reminder, dismissed_at NOT NULL)** — 도출 시 제외.
  born-dismissed라 `list_pending`(dismissed_at IS NULL)엔 절대 안 나옴 → 머지 중복 0.
- **confirmed만:** cancelled/rejected는 점유 슬롯 DELETE(4.7)라 도래 의미 없음 + status_change가
  별도로 다룸. 자연 상호배타(같은 예약이 confirmed↔terminal 동시 불가).
- **earliest = `slot_starts[0]`:** create 시 `sorted(isoformat_utc(...))` 오름차순 스냅샷이라 [0]이
  최早. `reservation_slots` 추가 쿼리 불요(N+1 회피). 단일/연속/비연속 슬롯 무관.

### ⚠️ 반복 함정 프리플라이트 (dev-workflow-policy 누적 교훈 — 위반 시 리뷰 차단)
