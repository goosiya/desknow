# Story 5.4: 카카오톡 예약 공유

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 예약자,
I want 확정 예약을 **한 번의 동작**으로 카카오톡에 공유(룸 이름·날짜·시간 범위 + 룸 상세 링크 포함)하길,
so that 일행에게 장소·시간을 자신 있게 전달한다 (FR-19, UJ-1 climax).

이 스토리는 **Epic 5의 마지막 "공유" 책임**이다. 5.1~5.3(알림 인프라·도래 리마인드·상태변경 통지)이
끝난 뒤, UJ-1(총무 "지우")의 **climax beat** — "확정 화면에서 카카오톡 공유 한 번"(EXPERIENCE.md
L233) — 을 실제로 구현한다. **순수 프론트(웹) 스토리**다: 카카오 JS SDK(Share)를 동적 로드해
`Kakao.Share.sendDefault()`로 공유하며, **백엔드·DB·OpenAPI·SDK 재생성 변경이 전혀 없다**(공유 데이터는
이미 클라가 가진 예약 응답 필드에서 합성).

**KTH 확정 대기 — Dev 진입 전 확인 권장(스토리 끝 "열린 질문" 참조):** 공유 템플릿 타입(text vs feed),
진입점 범위(예약현황+확정 climax 둘 다 vs 예약현황만), 모바일 처리. 본 스토리는 **권장안을 적용해
작성**되었고(아래), 다른 선택을 원하면 해당 AC/Task만 좁히면 된다. [[comms-discuss-dont-collapse-decisions]]

## Acceptance Criteria

(epics.md Story 5.4 BDD[L942-958] + FR-19[L60] + UX EXPERIENCE.md[L233,L51,L57] 기반)

**AC1 — 한 번의 동작으로 카카오톡 공유 (룸·일시 핵심 정보 포함)**
**Given** 확정(`confirmed`) 예약
**When** 공유 버튼을 누르면
**Then** **한 번의 클릭**으로 카카오톡 공유 시트가 열리고, 공유 내용에 **룸 이름 + 예약 날짜 + 시간
범위(KST 벽시계) + 룸 상세 링크**(`{origin}/rooms/{room_id}`)가 포함된다. 추가 확인 단계·중간 모달 없음
("한 번에" — UJ-1 climax). 공유 텍스트는 기존 KST 포맷터(`formatDateKorean`·`formatSlotLabel` —
slots.ts/reservations.ts)를 **재사용**해 생성한다(상대일 클라 재판정 금지·절대 표기 — deferred L207 정책,
[[terminology-network-disconnect-not-offline]]와 무관한 시간 규약).

**AC2 — 플랫폼 분기 (웹=Kakao JS SDK / 모바일=후속 dev-build)**
**Given** 웹 표면(apps/web)
**When** 공유를 구현하면
**Then** **카카오 JS SDK(`Kakao.Share.sendDefault()`)** 로 동작한다. SDK는 신규 `apps/web/src/lib/kakao-share.ts`
로더가 `<script>` 동적 주입 → `Kakao.init(NEXT_PUBLIC_KAKAO_JS_KEY)`(멱등 가드 `Kakao.isInitialized()`)로
1회 초기화한다(maps 로더 `kakao-map.ts` 패턴 재사용 — **신규 env 변수 0**, 같은 JS 키). RN(apps/mobile)은
`@react-native-kakao/share`(EAS dev build)가 필요하나 **모바일 예약 화면 자체가 아직 placeholder**(Epic 4
미구현)이므로 본 스토리 범위 밖 — **"모바일 dev-build 푸시" 버킷으로 명시 라우팅**(E7=챗봇 에픽 ≠ 모바일,
혼동 금지). [[web-mobile-parity-on-changes]]

**AC3 — 진입점: 예약현황 확정 예약 행 + 즉시예약 확정 직후(climax)**
**Given** 확정 예약의 표면
**When** 공유 버튼을 배치하면
**Then** ① **예약현황(`ReservationRow`)** — `status === "confirmed"`인 행(다가오는·이용 완료 모두 — 확정
예약은 공유 가능)에 공유 버튼 노출(취소 버튼과 **형제** — 중첩 인터랙티브 금지·`ReservationRow` a11y 규칙
준수). ② **즉시예약 성공 직후(`ReservationPanel` "예약이 완료됐어요!" 배너)** — UJ-1 climax. 두 곳 모두
공유 데이터는 이미 보유한 필드에서 합성(추가 API 조회 0): 예약현황=`ReservationListItem.{room_name,slot_starts}`,
즉시예약=`createReservation.data.slot_starts`(`ReservationPublic`) + RoomDetail이 패널에 전달하는 룸 이름
prop(신규 `roomName` prop — 추가 조회 0, RoomDetail이 `useRoomSummary.data.name`을 이미 보유). 취소·거절
예약에는 미노출(확정만 — AC "확정 예약").

**AC4 — 카톡 외 SNS 미제공 + graceful degrade (막다른 화면 금지)**
**Given** 공유 기능
**When** 노출/실패하면
**Then** ① 카카오톡 **외 SNS(페북·트위터·일반 Web Share API 등) 진입점은 제공하지 않는다**(epics L958 —
카톡 단일). ② SDK 로드 실패·키 미설정·`window.Kakao` 미초기화는 **graceful degrade**: 공유 시도 시
조용히 실패하지 않고 친근한 안내(`role="status"`, 예: "지금은 공유를 할 수 없어요. 잠시 후 다시 해주세요.")
+ 재시도 가능. 버튼이 화면/예약 행을 **크래시시키지 않는다**(maps 로더처럼 reject→안내, throw 전파 금지).
③ 색 단독 금지 — 버튼=아이콘(`Share2`/lucide)+텍스트("공유" 또는 aria-label) 병행. ④ 터치 타겟 ≥44px
(`tap-target`). ⑤ 카카오 도메인 미등록(키 화이트리스트 밖)도 같은 안내로 degrade(프로드 도메인 등록은 배포 시).

**AC5 — deferred 의무 회수: injectScript 실패 시 `<script>` 정리 (L60)**
**Given** 신규 `kakao-share.ts`가 maps 로더의 `injectScript` 패턴(동적 `<script>` 주입)을 **그대로 확장**함
([[dev-workflow-policy-deferred-and-repeat-mistakes]] 트리거 — 같은 함정 재발)
**When** 스크립트 load/error 후
**Then** **신규 share 로더와 기존 `kakao-map.ts` 양쪽** 모두 load 성공/error 후 `<script>` 태그를 정리
(`script.remove()`)하거나 재사용을 보장해 dead `<script>` 누적을 막는다(deferred-work.md L60 의무 회수 —
재-defer 금지, E7/E8/배포 예외 아님). 회수 후 deferred-work.md L60에 ✅ 회수 마킹. **단 maps 로더의 기존
동작(Promise 캐시·중복 주입 가드·도메인 거부 reject·재시도)은 불변** — 정리 로직만 추가.

**AC6 — 결정성·경계·게이트**
**Given** 도메인 경계·반복 함정·게이트
**When** 구현하면
**Then** ① **백엔드 변경 0**(엔드포인트·스키마·service 무변경)·**마이그레이션 0**·**OpenAPI/SDK 재생성 0**·
**신규 의존성 0**(lucide `Share2`는 기존·카카오 JS SDK는 `<script>` 주입이라 npm 의존성 아님)·**신규 env
변수 0**(`NEXT_PUBLIC_KAKAO_JS_KEY` 재사용)·**신규 ErrorCode 0**. ② 직접 fetch 0(공유는 SDK `<script>`
주입 + `window.Kakao` — eslint no-direct-fetch 가드와 무관, maps와 동일). ③ 공유 텍스트 생성은 **순수
함수**(테스트 가능·KST 결정성)·표시는 **render-time 파생**(set-state-in-effect 금지). ④ 웹 `vitest`/`lint`
(직접 fetch 0·set-state-in-effect 0)/`check-types`/`build` + `turbo` 전체 그린. ⑤ 백엔드·DB 무변경이므로
라이브 Supabase head 불변(alembic 변경 없음 — `upgrade head` 불필요/no-op). [[dev-workflow-policy-live-db-migration]]

### 🔑 권장 기술 결정 (KTH 확정 권장 — 다르면 좁히면 됨)

[[comms-discuss-dont-collapse-decisions]] — 권장안과 트레이드오프를 풀어서 제시(스토리 끝 "열린 질문"에서
KTH 확정 권장):

1. **공유 템플릿 = `text`(권장) vs `feed`.** `Kakao.Share.sendDefault`의 `feed` 템플릿은 `content.imageUrl`이
   **필수**라 호스팅된 이미지 URL이 필요한데 현재 이미지 인프라가 없다. `text` 템플릿은 텍스트(≤200자)+링크만
   필요해 **룸·일시 핵심 정보(FR-19 AC)를 이미지 자산 없이 충족**한다. → **권장: text**. (리치 카드를 원하면
   feed + 정적 브랜드 이미지 호스팅이 후속 필요 — 본 스토리 비포함.)
2. **진입점 = 예약현황 행 + 즉시예약 climax 둘 다(권장) vs 예약현황만.** UX는 UJ-1 climax를 "확정 화면에서
   공유 한 번"(L233)으로 명시하고 IA는 예약현황을 "공유로 연결"(L51)로 명시한다 — **둘 다 스펙 근거 있음**.
   두 곳이 `KakaoShareButton` 하나를 공유하므로 한계비용이 낮다. → **권장: 둘 다**. (climax만 부담되면
   예약현황 행만으로 좁힐 수 있음 — 그 경우 ReservationPanel/RoomDetail prop 추가 불요.)
3. **링크 타깃 = 룸 상세 딥링크 `{origin}/rooms/{room_id}`(권장).** 수신자가 같은 룸을 바로 본다(예약은
   본인만 보므로 예약 상세 링크는 부적절). `window.location.origin`은 등록 도메인(localhost:3000 등록 완료
   — 1-1 AC7). 프로드 도메인 카카오 콘솔 등록은 **배포 스토리**가 처리(deferred).
4. **모바일 = "모바일 dev-build 푸시" 버킷.** [[web-mobile-parity-on-changes]] 점검 결과: 모바일 카카오
   공유는 `@react-native-kakao/share`+EAS dev build가 필요하고, **모바일 예약 화면 자체가 placeholder**
   (apps/mobile/src/app/reservations.tsx — "Epic 4" 주석). 공유의 전제(예약 화면)가 없어 본 스토리에서
   동등 적용 불가 → 명시 라우팅. **E7=챗봇 에픽이지 모바일 버킷이 아님**(혼동 주의 — 메모리).

### 🌐 카카오 JS SDK(Share) 기술 세부 (web research — 최신 정확성)

- **maps SDK ≠ share SDK(중요):** maps=`dapi.kakao.com/v2/maps/sdk.js`는 `window.kakao`(소문자·maps
  네임스페이스)만 노출 — `Kakao.Share`가 없다. 공유는 **카카오 JS SDK**(`https://t1.kakaocdn.net/kakao_js_sdk/2.7.6/kakao.min.js`)를
  별도 주입해야 `window.Kakao`(대문자·`.init`/`.Share`/`.isInitialized`)가 생긴다. → **별도 로더 필요**(maps
  로더 확장 아님). SRI `integrity` + `crossorigin="anonymous"` 권장(카카오 문서 명시 — 정확한 해시는 dev가
  공식 문서/CDN에서 확인).
- **초기화:** load 후 `Kakao.init(NEXT_PUBLIC_KAKAO_JS_KEY)` 1회 — `Kakao.isInitialized()` 멱등 가드(중복
  init 시 경고). JavaScript 앱 키 사용(REST 키 아님).
- **공유 호출:** `Kakao.Share.sendDefault({ objectType: "text", text, link: { webUrl, mobileWebUrl } })`.
  (v1의 `Kakao.Link`는 deprecated — v2 `Kakao.Share` 사용.) `text` 템플릿은 이미지 불요. 한 번 호출=공유
  시트 1회(추가 단계 없음 — AC1 "한 번에").
- **도메인 등록:** 공유 링크(`webUrl`)·`Kakao.init`은 카카오 콘솔 [플랫폼 → Web 도메인]에 등록된 도메인에서만
  동작(localhost:3000 등록 완료 — 1-1 AC7). 미등록 도메인은 로드/공유 실패 → AC4 graceful degrade.

### ⚠️ 반복 함정 프리플라이트 (dev-workflow-policy 누적 교훈 — 위반 시 리뷰 차단)

[[dev-workflow-policy-deferred-and-repeat-mistakes]]
1. **injectScript `<script>` 미정리(L60 — 본 스토리 의무 회수):** share 로더가 maps `injectScript`를 미러하므로
   같은 함정 재발 → **양쪽** load/error 후 `script.remove()`. 재-defer 금지.
2. **set-state-in-effect 금지:** SDK 로드는 **클릭 이벤트**에서 lazy(effect 아님). 공유 텍스트는 순수 함수/
   render-time. (maps는 mount-effect 로드지만 share는 사용자 동작 시점이 자연 — 불필요한 mount-effect 금지.)
3. **직접 fetch 금지:** 공유는 `<script>` 주입 + `window.Kakao`(백엔드 호출 0) — maps와 동일하게 no-direct-fetch
   가드 무관. 신규 SDK 엔드포인트 호출 없음.
4. **색 단독 금지:** 공유 버튼=아이콘(Share2)+텍스트.
5. **dead 버튼/막다른 화면 금지:** SDK 실패·키 미설정·도메인 거부 → 친근한 안내(`role="status"`)+재시도,
   throw 전파로 화면/예약 행 크래시 금지(maps 로더 reject 패턴 — AC4).
6. **중첩 인터랙티브 a11y 금지:** 예약현황 공유 버튼은 상세 Link/취소 버튼과 **형제**(ReservationRow L3-4
   규칙) — Link 안 button 중첩 금지.
7. **신규 env/의존성/스키마 0:** JS 키 재사용·카카오 SDK는 `<script>`(npm 아님)·백엔드 무변경. 잘못
   `package.json`에 카카오 패키지 추가하거나 새 env 도입 금지.
8. **시간 규약:** 공유 텍스트 KST는 **기존 Asia/Seoul 포맷터 재사용**(상대일 "내일" 클라 재판정 금지 —
   deferred L207 정책·절대 날짜 표기).
