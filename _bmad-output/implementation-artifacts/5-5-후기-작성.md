# Story 5.5: 후기 작성

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 예약자(booker),
I want 이용 완료한 예약에 별점과 후기를 남기길,
so that 다른 사람에게 도움을 주고 내 경험을 기록한다 (FR-20, UJ-1, D4).

이 스토리는 **Epic 5의 첫 풀스택 도메인 스토리**다. 5.1~5.4(알림 인프라·리마인드·상태변경 통지·카카오 공유)가
끝난 뒤, **reviews 도메인을 0에서 생성**한다(현재 `apps/api/app/reviews/`는 빈 골격). **5.4와 정반대 성격**:
5.4는 순수 웹 프론트(백엔드·DB·SDK 0)였으나 **5.5는 신규 테이블·마이그레이션·엔드포인트·SDK 재생성·웹 UI를
모두 포함하는 풀스택 스토리**다. reservations 도메인(4.x)이 깐 패턴(상태머신·소유권 가드·me_router·라우터 룸 메타
합성·신선 재검증)을 **참조 선례로 미러**한다.

**KTH 확정(create-story 결정 — 반영 완료):**
1. **작성자 표시 = 익명(미표시)** — `users` 테이블엔 email만 있고 이름 필드가 없다. 공개 룸 상세에 email 노출은
   프라이버시 누수 → **작성자 식별 미노출**(별점+텍스트+작성일만). users 변경/마이그레이션 0. FR-20 BDD는 작성자명
   미요구. [[region-code-legal-not-admin-dong]]와 무관한 별개 프라이버시 결정.
2. **텍스트 = 필수(1~500자)** — 별점만 남기는 빈 후기 금지("별점+텍스트" FR-20 문구 정합).
3. **작성 게이팅 = `ReservationListItem.has_review` 필드 추가** — 기존 `GET /reservations` 응답에 예약별 후기 작성
   여부를 함께 실어 예약현황에서 "미작성 이용완료 → 작성 폼 / 작성됨 → 후기 완료 표시"로 죽은 버튼을 막는다(SDK 재생성).

## Acceptance Criteria

(epics.md Story 5.5 BDD[L960-976] + FR-20[L63] + UX EXPERIENCE.md[L51,L78,L103] + review-accessibility.md[L61] 기반)

**AC1 — 이용 완료 예약에 별점+텍스트 작성 → 저장 + 룸 상세 노출**
**Given** 본인(booker)의 **이용 완료**(`confirmed` 상태를 유지한 채 **마지막 슬롯 종료 시각이 지난**) 예약
**When** 별점(1~5 정수) + 텍스트(1~500자)로 후기를 작성하면(`POST /api/v1/reservations/{reservation_id}/reviews`)
**Then** 201 + `ReviewPublic`로 저장되고, 그 룸 상세의 후기 목록(`GET /api/v1/rooms/{room_id}/reviews`)에 노출된다.
**이용 완료 판정은 백엔드가 진실 경계**: `reservation.status == "confirmed"` **그리고**
`max(slot_starts) + 1시간(슬롯 길이) < now_utc()`. 슬롯 길이 1h는 reservations 도메인 내재 규약(별도 duration 필드
없음 — `derive_slots` 1h 격자). [[availability-freshness-policy]] 정신: 작성 확정 시점에 서버가 자격을 재판정한다.

**AC2 — 작성 거부: 취소·거절·미완료·중복(이용 완료만, 예약당 1회)**
**Given** ① 취소(`cancelled`)/거절(`rejected`)된 예약, 또는 ② 아직 이용 완료 전(마지막 슬롯 미종료) 확정 예약, 또는
③ 이미 후기를 작성한 예약
**When** 작성을 시도하면
**Then** ①② → **409 `RESERVATION_NOT_COMPLETED`**("이용 완료된 예약만 후기를 쓸 수 있어요."), ③ → **409
`REVIEW_ALREADY_EXISTS`**("이미 후기를 작성한 예약이에요."). 타인 예약·미존재 예약 = **404 `RESERVATION_NOT_FOUND`**
(존재 누설 금지 — reservations cancel 가드 미러). 중복 차단의 **진실의 원천 = `reviews.reservation_id` UNIQUE
제약**(서비스 동시성 경합도 IntegrityError→409로 최종 차단). 입력 형식 위반(별점 범위 밖·텍스트 0자/501자+·naive 등)
= **422 `VALIDATION_ERROR`**(Pydantic 선차단 → service `ValueError` 500 누출 금지 — 반복 함정).

**AC3 — RBAC·소유권(booker 본인만)**
**Given** 작성 엔드포인트
**When** 호출되면
**Then** `Depends(require_role("booker"))`로 **booker만** 통과(provider/admin=403 `FORBIDDEN_ROLE`, 미인증=401
`UNAUTHENTICATED`). 소유권: `reservation.booker_id != principal.user_id`면 404 `RESERVATION_NOT_FOUND`(미존재와 동일
404로 합쳐 타인 예약 존재 누설 금지 — reservations cancel/4.7 선례). 후기 **조회**(`GET /rooms/{room_id}/reviews`)는
**공개·무인증**(룸 상세가 공개 표면 — `GET /rooms/{room_id}` 동일 근거, PRD §FR-2).

**AC4 — 룸 상세 후기 노출(익명·별점 시각화 a11y)**
**Given** 룸 상세 화면(`apps/web/src/features/detail/RoomDetail.tsx` 3차 섹션)
**When** 후기 목록을 노출하면
**Then** 각 후기는 **별점(1~5)·텍스트·작성일**만 표시하고 **작성자 식별 정보는 노출하지 않는다**(KTH 결정 1 — 익명).
별점은 **색 단독 금지**: 별 아이콘(채움/빈 별) + **숫자 텍스트("4/5" 또는 aria-label "별점 5점 만점에 4점")** 병행
(review-accessibility.md L61 — "후기 별점은 키보드/SR 입력·읽기 가능해야"). 후기 0건이면 **막다른 화면 금지**: "아직
후기가 없어요" 빈 상태 카피. 최신순(`created_at` 내림차순) 정렬.

**AC5 — 예약현황 작성 진입점(미작성 이용완료에만 폼·작성 후 완료 표시)**
**Given** 예약현황(`apps/web/src/features/reservation/`)의 예약 행
**When** 후기 작성 UI를 배치하면
**Then** **이용 완료(`confirmed` + 마지막 슬롯 종료 경과) 그리고 `has_review === false`** 행에만 후기 작성 폼/버튼을
노출한다. `has_review === true`면 "후기 완료"(또는 작성한 별점) 표시·작성 폼 미노출(**죽은 버튼 0**). 취소/거절/다가오는
확정 행엔 미노출. 후기 작성 폼 = **별점 입력(키보드·SR 접근 가능 — 라디오/버튼 그룹, `aria-label`)** + **텍스트
입력(가시 라벨 `for/id` 연결·글자수 카운터 "n/500"·필수 별표+'필수' 텍스트)** + **에러는 색+아이콘+텍스트**
(review-accessibility.md L61). 작성 성공 시 그 행은 즉시 "후기 완료"로 전환(옵티미스틱 또는 invalidate)되고, 해당 룸
상세 후기 캐시(`['rooms', roomId, 'reviews']`)를 무효화해 새 후기가 상세에 반영되게 한다.

**AC6 — 비포함(명시 제외)**
**Given** FR-20 스코프 경계
**When** 구현하면
**Then** **후기 수정/삭제 미제공**(작성 후 불변 — edit/delete 엔드포인트·UI 0), **"내가 쓴 후기 모아보기" 미제공**
(별도 목록 화면 0), **제공자 답글 미제공**(FR-21 = Story 5.6 — 본 스토리 비포함), **카톡 외 공유·이미지 첨부 0**.

**AC7 — 데이터 모델·마이그레이션·도메인 경계·게이트**
**Given** reviews 도메인 신규 + 라이브 DB
**When** 구현하면
**Then** ① 신규 `reviews` 테이블 + alembic 마이그레이션(down_revision=`a7c9e1b3d5f2` 현 head). 제약명 ≤63자
(reservations `test_models.py` 패턴) — `uq_reviews_reservation`(예약당 1회)·`ck_reviews_rating`(1~5)·FK
`fk_reviews_*`. ② 신규 ErrorCode 2개(`RESERVATION_NOT_COMPLETED`·`REVIEW_ALREADY_EXISTS`) — core/errors에 등록 +
`DEFAULT_STATUS` 매핑(둘 다 409). ③ **도메인 경계:** reviews.service는 `reviews` 테이블만 만진다. 예약 자격 판정에
필요한 `Reservation`(status·slot_starts·booker_id·room_id) 조회는 **라우터/조합 계층**이 `session.get`으로 하거나
reviews.service가 `reservations.models`만 import(service↔service 순환 금지 — `list_reservations`의 룸 메타 합성 선례
참조). 시간 판정은 `app.core.time`(`now_utc` 단일 진입점) 재사용 — 슬롯 종료시각 직접 datetime 산술 시 tz-aware 보장.
④ **SDK 재생성:** 신규 엔드포인트 + `ReservationListItem.has_review` 추가 → openapi.json export → `packages/api-client`
`npm run generate` → 드리프트 0. snake_case 보존(camelCase 변환 금지). ⑤ **라이브 마이그레이션 의무:** dev 완료 시
라이브 Supabase에 `alembic upgrade head` **실제 실행**([[dev-workflow-policy-live-db-migration]]). ⑥ 백엔드 `pytest`·
웹 `vitest`/`lint`(직접 fetch 0·set-state-in-effect 0)/`check-types`/`build` + `turbo` 전체 그린.

- 룸 소재지 타임존(Asia/Seoul=`ROOM_TZ`) 기준 판정이 필요한 표시는 기존 포맷터 재사용(상대일 클라 재판정 금지).

- 작성=`require_role("booker")` 모듈레벨 싱글톤(`_require_booker`). 소유권: 미존재+비소유=동일 404

### ⚠️ 반복 함정 프리플라이트 (dev-workflow-policy 누적 교훈 — 위반 시 리뷰 차단)

[[dev-workflow-policy-deferred-and-repeat-mistakes]]
1. **service ValueError 500 누출 금지:** 별점 범위·텍스트 길이·naive datetime은 **Pydantic 스키마가 선차단**(422
   `VALIDATION_ERROR`) — service `ValueError`가 미처리 500으로 새지 않게 경계에서 막는다(reservations 반복 함정 #6).
2. **UNIQUE 위반 → 깔끔한 409:** 중복 후기는 `uq_reviews_reservation`가 진실의 원천. INSERT IntegrityError를
   try/except + rollback → `REVIEW_ALREADY_EXISTS`(409)로 변환(favorites `add_favorite` 선례). raw 500 전파 금지.

   reservations.router → reviews.service 방향(라우터=조합 계층). 잘못 service↔service 의존 만들면 import 사이클.
