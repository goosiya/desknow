# Story 5.6: 후기 답글 (제공자)

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 제공자(provider),
I want 내 룸의 후기에 답글을 달길,
so that 손님에게 응대하고 신뢰를 쌓는다 (FR-21, UJ-3).

이 스토리는 **5.5(후기 작성)가 깐 reviews 도메인에 "제공자 답글" 한 겹을 얹는다**. 5.5는 booker가
후기를 쓰고 룸 상세에 익명 노출했고, 5.6은 **그 룸의 provider가 후기에 답글 1건**을 달아 같은 룸 상세
후기 목록에 **후기에 연결되어** 노출되게 한다. 신규 도메인 생성은 없다 — `review_replies` 테이블 1개·
엔드포인트 1개(작성)·기존 공개 조회(`GET /rooms/{id}/reviews`) 응답에 답글 합성 + 웹 룸 상세 표시.

**KTH 확정(create-story 결정 — 반영 완료):**
1. **답글 데이터 모델 = 별도 `review_replies` 테이블**(`review_id` UNIQUE FK·`provider_id`·`text`·
   `created_at`). reviews 컬럼 추가가 아니라 별도 테이블 — 정규화·향후 수정이력/확장 여지. (5.5 forward
   note가 5.6에 위임한 결정.)
2. **작성 표면 = API-only**(provider 인증 `POST` 엔드포인트). 웹 작업은 **booker 룸 상세
   `ReviewSection`에 답글 읽기 표시**만. **provider 답글 작성 웹 UI는 "provider 웹 표면" 버킷으로
   defer** — 웹에 provider 로그인/내비/라우트가 전무하고(Epic 2 공간등록·Epic 6 예약관리 provider
   작업이 전부 API-only), 그 웹 셸을 5.6이 떠안지 않는다([[web-mobile-parity-on-changes]] 정신·5.5 모바일
   버킷 라우팅과 일관).

**파생 설계 결정(create-story 확정):**
3. **소유권 차단 = 403** `REVIEW_REPLY_FORBIDDEN`(epic AC 명시). booker 도메인의 404 비노설 패턴과
   **의도적 분기** — 후기·룸은 공개 데이터라 존재 누설이 무관하고, AC가 명시적으로 403을 요구한다(타
   제공자 룸 후기 답글 시도 → "본인 룸 후기에만 답글" 명료 차단).
4. **후기당 답글 1회 = `uq_review_replies_review` UNIQUE** + **작성 후 불변**(수정/삭제 미제공 — reviews
   불변 정책 일관). 중복 답글 시도 → 409 `REVIEW_REPLY_ALREADY_EXISTS`.
5. **답글도 익명**(제공자 식별 미노출). users엔 email만 있어 표시 이름 부재(5.5 작성자 익명 결정과 동일
   근거). 공개 노출(`ReviewReplyView`)엔 `provider_id` 미포함, FE 라벨은 **"제공자 답글"** 고정.

## Acceptance Criteria

(epics.md Story 5.6 BDD[L984-992] + FR-21[L64] + UJ-3 기반. 5.5 reviews 도메인 선례 미러.)

**AC1 — 제공자가 자기 룸 후기에 답글 작성 → 저장 + 룸 상세에 후기 연결 노출**
**Given** provider 본인이 **소유한 룸**(`room.provider_id == principal.user_id`)에 달린 후기
**When** 텍스트(1~500자)로 답글을 작성하면(`POST /api/v1/reviews/{review_id}/reply`)
**Then** 201 + `ReviewReplyPublic`로 저장되고, 그 룸 상세 후기 목록(`GET /api/v1/rooms/{room_id}/reviews`)
의 **해당 후기에 연결되어**(`ReviewListItem.reply`) 노출된다. room_id는 **후기에서 도출**한다
(`review.room_id` → 경로에 room_id 불요). 답글은 **익명**(제공자 식별 미노출 — KTH 결정 5).

**AC2 — 타 제공자 룸 후기 답글 시도 → 403 소유권 차단**
**Given** **타 제공자가 소유한 룸**의 후기(`room.provider_id != principal.user_id`)
**When** 답글을 시도하면
**Then** **403 `REVIEW_REPLY_FORBIDDEN`**("본인 룸의 후기에만 답글을 달 수 있어요.")로 차단된다(epic AC
명시 403 — booker 404 비노설 패턴과 의도적 분기, KTH 결정 3). 미존재 후기 = **404 `REVIEW_NOT_FOUND`**
("후기를 찾을 수 없습니다."). 입력 형식 위반(텍스트 0자/501자+·공백만) = **422 `VALIDATION_ERROR`**
(Pydantic 선차단 → service `ValueError` 500 누출 금지 — 반복 함정 #1).

**AC3 — 후기당 답글 1회(중복 차단)·불변**
**Given** 이미 답글이 달린 후기
**When** 같은 후기에 또 답글을 시도하면
**Then** **409 `REVIEW_REPLY_ALREADY_EXISTS`**("이미 답글을 작성한 후기예요.")로 차단된다(KTH 결정 4).
중복 차단의 **진실의 원천 = `review_replies.review_id` UNIQUE 제약**(`uq_review_replies_review`) — 서비스
INSERT IntegrityError를 rollback 후 409로 변환(reviews `create_review` UNIQUE 변환 선례 미러). **답글
수정/삭제 미제공**(작성 후 불변 — edit/delete 엔드포인트·UI 0).

**AC4 — RBAC(provider 본인만)·공개 조회는 무인증**
**Given** 답글 작성 엔드포인트
**When** 호출되면
**Then** `Depends(require_role("provider"))`로 **provider만** 통과(booker/admin = 403 `FORBIDDEN_ROLE`,
미인증 = 401 `UNAUTHENTICATED`). 소유권은 그 위에서 룸 `provider_id`로 재검증(AC2). 후기+답글 **조회**
(`GET /rooms/{room_id}/reviews`)는 **공개·무인증 유지**(룸 상세 공개 표면 — 5.5 AC3 동일 근거). 비활성
룸(`is_active=false`)이라도 본인 룸이면 답글 허용(자기 데이터 — `is_active` 게이트 없음).

**AC5 — 웹 룸 상세에 답글 읽기 표시(booker-facing)**
**Given** 룸 상세 후기 섹션(`apps/web/src/features/detail/ReviewSection.tsx`)
**When** 후기에 답글이 있으면
**Then** 그 후기 카드 안에 답글을 **시각적으로 구분되게**(들여쓰기/구분 배경) 중첩 표시한다 — **"제공자
답글" 라벨**(제공자명 미노출 — 익명) + 답글 텍스트 + 답글 작성일(절대 표기·KST). 답글이 없는 후기는
답글 영역 미표시(추가 빈 상태 카피 불요 — 답글은 선택적). 후기 본문(별점·텍스트·작성일·익명·최신순·0건
빈 상태)·기존 동작은 **전부 불변**(답글 한 겹만 append).

**AC6 — 비포함(명시 제외)**
**Given** FR-21 스코프 경계
**When** 구현하면
**Then** **답글 수정/삭제 미제공**(불변) · **provider 답글 작성 웹/모바일 UI 미제공**("provider 웹 표면"
버킷 defer — 작성은 API-only) · **답글 알림 미제공**(예약자에게 답글 통지 배너 0 — FR-21 비요구) · **다중
답글/대댓글 0**(후기당 provider 답글 1건) · **제공자 식별 노출 0**(익명).

**AC7 — 데이터 모델·마이그레이션·도메인 경계·게이트**
**Given** review_replies 신규 테이블 + 라이브 DB
**When** 구현하면
**Then** ① 신규 `review_replies` 테이블 + alembic 마이그레이션(down_revision=`b8f2c4a6d9e3` 현 head=
reviews). 제약명 ≤63자(reviews `test_models.py` 패턴) — `uq_review_replies_review`(후기당 1회)·FK
`fk_review_replies_*`. **CHECK 없음**(별점 없음 — op.f() 이중접두 함정 비해당이나 UNIQUE는 `uq` 규약상
bare명 정상). ② 신규 ErrorCode 3개(`REVIEW_NOT_FOUND` 404·`REVIEW_REPLY_FORBIDDEN` 403·
`REVIEW_REPLY_ALREADY_EXISTS` 409) — core/errors enum + `DEFAULT_STATUS` 양쪽 등록. ③ **도메인 경계:**
reviews.service는 `review_replies`/`reviews` 테이블만 쓴다. 소유권 판정에 필요한 `Room`(provider_id)
조회는 **라우터(조합 계층)가 `session.get(Room, …)`** 으로 한다(reviews.service↔rooms.service 순환 금지 —
list_reservations 룸 메타 합성 선례 동형). 시간=`app.core.time`(`now_utc`) 재사용. ④ **SDK 재생성:** 신규
엔드포인트 + `ReviewListItem.reply`(중첩 `ReviewReplyView`) → openapi.json export → `packages/api-client`
`npm run generate` → 드리프트 0. snake_case 보존. ⑤ **라이브 마이그레이션 의무:** dev 완료 시 라이브
Supabase `alembic upgrade head` **실제 실행**([[dev-workflow-policy-live-db-migration]]) + 실 제약명 대조. ⑥
백엔드 `pytest`·웹 `vitest`/`lint`(직접 fetch 0·set-state-in-effect 0)/`check-types`/`build` + `turbo`
전체 그린.

### 🏗️ 아키텍처 가드레일 (architecture.md + 5.5 reviews 선례 — 반드시 준수)

**도메인 경계·순환 import 금지** [Source: architecture.md L354 · reviews/service.py docstring · reservations/router.py L215-241]
- reviews.service는 `reviews`/`review_replies` 테이블만 쓴다. **소유권 판정에 필요한 `Room`(provider_id)
  조회는 라우터(조합 계층)가 `session.get(Room, …)`** — reviews.service↔rooms.service 순환 금지(`Room`
  타입 import는 허용, service 호출 아님). `list_reservations`가 룸 메타를 라우터에서 합성하는 패턴 동형.
- 프론트는 packages/api-client SDK로만 호출(직접 fetch 금지). 서버 상태=TanStack Query.

**RBAC·소유권 2단** [Source: apps/api/app/core/security.py:203 · rooms/router.py:66 · reviews/router.py:47]
- 1단 = `require_role("provider")` 모듈레벨 싱글톤(`_require_provider` — ruff B008 회피, rooms 선례).
  booker/admin=403 `FORBIDDEN_ROLE`, 미인증=401. 2단 = 룸 `provider_id` 일치(403 `REVIEW_REPLY_FORBIDDEN`).
- **booker 404 비노설과 분기:** reservations/rooms는 비-소유를 404로 합쳐 존재 누설을 막지만(타인
  예약/룸은 사적), 후기·룸은 **공개 데이터**라 존재가 비밀이 아니고 epic AC가 명시 403을 요구한다.

**에러 스키마** [Source: apps/api/app/core/errors.py:34-74]
- 표준 `{detail:{code, message}}`. 신규 코드 3개는 `ErrorCode` enum + `DEFAULT_STATUS` 양쪽 등록(5.5
  RESERVATION_NOT_COMPLETED/REVIEW_ALREADY_EXISTS 추가 패턴). 화면엔 한국어 message, FE 분기는 `detail.code`.

**UNIQUE 위반 → 깔끔한 409** [Source: apps/api/app/reviews/service.py:140-149 · favorites/service.py:49-59]
- 후기당 답글 1건은 `uq_review_replies_review`가 진실의 원천. INSERT IntegrityError → rollback →
  `violated_constraint`로 제약명 식별 → `REVIEW_REPLY_ALREADY_EXISTS`(409) 변환. **무관 위반은 re-raise**
  (과대캐치 금지 — 회고 P2). raw 500 전파 금지.

**시간 규약(NFR·1.5)** [Source: apps/api/app/core/time.py]

### ⚠️ 반복 함정 프리플라이트 (dev-workflow-policy 누적 교훈 — 위반 시 리뷰 차단)

[[dev-workflow-policy-deferred-and-repeat-mistakes]]
1. **service ValueError 500 누출 금지:** 텍스트 길이/공백은 `ReviewReplyCreateRequest`가 선차단(422
   `VALIDATION_ERROR`). service `ValueError`가 미처리 500으로 새지 않게 경계에서 막는다.
2. **UNIQUE 위반 → 깔끔한 409:** 중복 답글은 `uq_review_replies_review`가 진실의 원천. INSERT
   IntegrityError를 try/except + rollback + `violated_constraint` 선별 → `REVIEW_REPLY_ALREADY_EXISTS`.
   **무관 위반 re-raise**(과대캐치 금지 — P2). raw 500 전파 금지.
3. **순환 import 금지:** reviews.service는 rooms.service를 import하지 않는다. 소유권용 `Room` 조회는
   **라우터**가 `session.get(Room, …)`(조합 계층). `Room` 타입/모델 import는 허용(service 호출 아님).
4. **시간 tz-aware:** `created_at`=`now_utc`(aware). 와이어 `...Z`. FE 작성일 절대 표기(KST).
5. **소유권 = 403(이 스토리 한정 분기):** 비-소유 룸 답글은 **403** `REVIEW_REPLY_FORBIDDEN`(epic AC
   명시). booker 도메인의 404 비노설을 **여기 복붙하지 말 것** — 후기·룸은 공개 데이터다.
6. **막다른 화면 금지:** 답글 없는 후기=답글 영역 미표시(에러 아님). 작성 실패(403/404/409/422)=친근한
   안내+`detail.code` 분기(API-only라 본 스토리 웹 작성 UI는 없으나 SDK 소비자 규약).
7. **set-state-in-effect 금지:** 답글 표시=render-time 파생(`review.reply` 렌더). effect 0.
8. **직접 fetch 금지:** 답글 조회는 기존 `useRoomReviews`(SDK)에 합성돼 옴 — 신규 직접 fetch 0.
9. **라이브 마이그레이션 의무:** dev 완료 시 라이브 Supabase `alembic upgrade head` 실제 실행 + **실
   제약명 대조**([[dev-workflow-policy-live-db-migration]] · 5.5 이중접두 함정 교훈 — 본 스토리 CHECK 없어
   op.f() 함정 비해당이나 UNIQUE/FK 실명 확인). 배포=dump→Railway는 별개.
10. **deferred 회수 의무:** 5.5 review가 defer한 **`formatReviewDate` malformed `created_at` 미가드**
    [ReviewSection.tsx:19-26]를 본 스토리가 회수한다 — 5.6이 ReviewSection을 수정하고 답글 날짜 포맷을
    추가해(같은 함수 재사용) **트리거**된다. 가드 추가 후 후기·답글 양쪽 날짜에 적용([[migration-check-constraint-opf-trap]]
    와 무관·별개 회수). 5.5의 다른 defer(손상 슬롯 후기 영구-불가, `is_reservation_completed`)는 본
    스토리가 그 코드를 안 만져 **트리거 아님 — 재-defer 유지**.
