# Story 6.1: 예약현황 조회 (제공자)

Status: done

## Story

As a 제공자,
I want 내 룸에 들어온 예약을 조회하길,
so that 운영 상황을 파악한다 (FR-23, UJ-3, Privacy).

> **이 스토리는 Epic 6(제공자 예약 관리)의 첫 스토리이자, "제공자가 자기 룸의 예약을 보는" 읽기 표면을 처음 만드는 백엔드 수직 슬라이스다.** 4.8이 **예약자 본인** 예약 목록(`GET /reservations`·`list_booker_reservations`·`slot_starts` 스냅샷·라우터 룸이름 합성)을 깔았고, 6.1은 그 **거울상**을 **제공자 소유 룸 단위**로 만든다: **(A)** 제공자 소유 룸 조회(`rooms.service.list_provider_rooms` 신규) + **(B)** 그 룸들의 예약 조회(`reservations.service.list_reservations_for_rooms` 신규 — rooms import 금지·읽기 전용) + **(C)** provider 인증 엔드포인트(`GET /provider/reservations`) + **(D)** 프라이버시 응답 스키마(`ProviderReservationItem` — 예약자 **익명 식별 라벨**·이메일/연락처/raw UUID 비노출) + **(E)** SDK 재생성. **신규 마이그레이션 0**(`idx_reservations_room_id`·`slot_starts` 스냅샷 재사용). **FE/모바일/admin 변경 0**([[web-is-booker-only-provider-api-only]] — provider 웹 UI는 "provider 웹 표면" 버킷 defer).

## 범위 결정 (KTH 확정 2026-06-17 — create-story)

이 스토리가 닿는 경계를 다음과 같이 확정한다:

1. **예약자 표시 = 안정 익명 식별 라벨(`booker_label`), 이메일·연락처·raw UUID 절대 비노출.** AC는 "예약자의 **표시 이름**"을 요구하나, `User` 모델엔 표시명 컬럼이 **없고**(`auth/models.py:40-68` — `email`·`password_hash`·`role`·`is_active`·`created_at`뿐) 유일한 식별 필드는 `email`인데 **FR-23이 이메일 비노출을 명령**한다(reviews가 익명인 것과 동일 제약 — 5.5 deferred 노트: 표시명은 users 스키마 컬럼 추가·가입 변경·마이그레이션·SDK 동반이 선행돼야 하는 별도 미니에픽). 따라서 6.1은 표시명 컬럼을 **추가하지 않고**, `booker_id`에서 **결정적으로 파생한 안정 익명 라벨**(예: `"예약자 #a3f9c1"`)을 노출한다. 같은 예약자는 항상 같은 라벨이라 provider가 동일 예약자의 여러 예약을 **구분·집계**할 수 있고(`"표시 이름"` 문구 충족), 이메일/연락처는 물론 **raw `booker_id` UUID도 응답에 싣지 않는다**(라벨만). 진짜 표시명은 후속(users `display_name` 컬럼) 스토리로 defer. [Source: epics.md#Story 6.1 L1000-1014 / auth/models.py:40-68 / deferred-work.md "5-5 후기-작성 작성자 표시 이름" / KTH 확정]

   - **라벨 파생(raw UUID 미노출):** `booker_id`의 일부를 그대로 자르면 raw UUID 바이트가 새므로, **해시 파생**한다 — `f"예약자 #{hashlib.sha256(str(booker_id).encode()).hexdigest()[:6]}"`. 결정적(같은 booker=같은 라벨)·비가역(이메일/UUID 복원 불가)·스키마 변경 0. 6자 hex 충돌은 MVP 규모(제공자당 소수 예약자)에서 무시 가능(충돌해도 라벨일 뿐 데이터 영향 0) — 정밀 표시명은 deferred 후보.

2. **6.1 표면 = provider 인증 API 엔드포인트 전용(웹/모바일 UI 0).** [[web-is-booker-only-provider-api-only]](KTH 확정): `apps/web`은 완전히 booker 전용이고 provider 로그인/내비/라우트가 전무하다. 6.1의 제공자 예약현황은 **엔드포인트만** 만들고, provider 웹 화면은 "provider 웹 표면" 버킷으로 defer한다. 이 결과는 booker에게 **공개 노출되지 않으므로**(provider 본인만 보는 사설 뷰) booker-facing 웹 작업도 **없다**. SDK는 드리프트 게이트 때문에 **재생성한다**(소비처 UI는 후속). 모바일=별개 "모바일 dev-build 푸시" 버킷(혼동 금지). `apps/web`·`apps/mobile`·`apps/admin` **변경 0**. [Source: 메모 [[web-is-booker-only-provider-api-only]] / 5.6 동일 패턴 확정]

3. **6.1은 읽기 전용 — cancel↔reject race·reader status 필터 부채는 트리거 안 함(6.2 소유).** epic-5 회고 2c(★)는 `cancel↔reject 교차 race`와 `confirmed_slot_starts` reader status 필터 부재를 **6.2(예약 거절)** 의무 회수 대상으로 명시한다 — 거절 엔드포인트(쓰기·종료 전이)가 그 부채의 정면 트리거이기 때문이다. **6.1은 예약을 조회만** 하고 어떤 상태 전이·슬롯 변경도 하지 않으므로 이 부채를 트리거하지 않는다(재-defer 아님 — 책임 스토리는 6.2). 6.1 service 함수는 전부 **읽기 전용**(`commit`/`add`/`delete` 0). ## Acceptance Criteria

**AC1 — 제공자 소유 룸의 예약을 다가오는/지난 단위로 조회 (FR-23, UJ-3)**
**Given** 인증된 제공자가 예약현황을 조회하면(`GET /api/v1/provider/reservations`)
**When** 본인 소유 룸(`Room.provider_id == principal.user_id`)에 예약이 있으면
**Then** 그 룸들의 예약이 **슬롯·예약 단위**로 반환된다 — 각 예약 항목은 **룸 이름 · 예약 슬롯 시작시각(`slot_starts` 스냅샷, `...Z`) · 상태(`confirmed`/`cancelled`/`rejected`) · 예약 생성시각**을 싣는다.
- **상태 무관 flat 목록**(다가오는=`confirmed`+미래 슬롯 / 지난=이용완료·취소·거절)으로 `created_at` 내림차순 반환한다. **다가오는/지난 분류·정렬은 소비처(consumer)가 `slot_starts`·`now` 기준으로** 한다(4.8 `list_booker_reservations` 선례: 서버는 단순 flat, 분류는 소비처 — 서버 시계 의존·캐시 신선도 회피). 취소/거절 예약도 **스냅샷 시간**(`slot_starts`)으로 날짜·시간을 표시한다(슬롯 점유 행은 DELETE됐어도 스냅샷은 잔존 — 범위 결정 #1 of 4.8).
- 본인 소유 룸이 없거나 예약이 없으면 **빈 리스트(정상 200 `[]`)**.

**AC2 — 예약자 식별정보 비노출 (FR-23 Privacy — 표시 이름·슬롯만)**
**Given** 예약 항목이 표시되면
**When** 응답을 직렬화하면
**Then** **예약자의 안정 익명 라벨(`booker_label`, 예: `"예약자 #a3f9c1"`)과 예약 슬롯 정보만** 노출되고, **이메일·연락처·raw `booker_id` UUID는 응답 어디에도 노출되지 않는다**(범위 결정 #1).
- `booker_label`은 `booker_id`에서 **결정적·비가역 해시 파생**(`sha256(str(booker_id))[:6]`)이라 같은 예약자는 항상 같은 라벨(provider가 동일 예약자 구분·집계 가능), 이메일/UUID 복원 불가.
- 응답 스키마(`ProviderReservationItem`)에 `email`·`booker_id`·`password_hash`·좌표 등 **내부/식별 필드를 두지 않는다**(`ReservationPublic`·`ReservationListItem` 정신 — 필요한 것만 노출).

**AC3 — RBAC: 제공자 전용·본인 룸만 (architecture.md §Boundaries L351·L356)**
**Given** `GET /provider/reservations` 요청
**When** 인증된 provider가 호출하면
**Then** **본인 소유 룸의 예약만** 반환한다(타 제공자 룸의 예약 노출 0 — `Room.provider_id` 소유권 필터). 미인증=401 `UNAUTHENTICATED`, booker/admin=403 `FORBIDDEN_ROLE`(`require_role("provider")` — rooms `_require_provider` 패턴 재사용). 본인 룸이 0개여도 빈 리스트(정상 200).

### 예약자 식별정보 비노출 (범위 결정 #1 — KTH 확정)

- **문제:** AC는 "예약자 표시 이름"을 요구하나 `User`엔 표시명 컬럼이 없고(`email`이 유일 식별 필드) FR-23이 이메일 비노출을 명령한다. reviews가 익명인 것과 **동일 제약**(5.5 deferred — 표시명은 users 스키마 변경·가입·마이그레이션·SDK 동반 별도 미니에픽).
- **해법(KTH 확정 — 안정 익명 식별자):** `booker_id`에서 **결정적·비가역 해시 파생**한 `booker_label`(`"예약자 #" + sha256(str(booker_id))[:6]`)만 노출한다. 같은 예약자=같은 라벨(provider가 동일 예약자 구분·집계 — "표시 이름" 문구 충족), **이메일·연락처·raw `booker_id` UUID 절대 비노출**. 스키마 변경 0.
- **왜 prefix가 아니라 hash인가:** `str(booker_id)[:6]`처럼 UUID 일부를 자르면 raw UUID 바이트가 누출된다("raw UUID 미노출" 위반). `sha256` 파생은 UUID 어떤 부분도 노출하지 않으면서 결정적·안정적이다.
- **충돌:** 6자 hex(16.7M 공간)는 MVP 규모(제공자당 소수 예약자)에서 충돌 무시 가능. 충돌해도 표시 라벨일 뿐 데이터 영향 0. 정밀 표시명(users `display_name`)은 deferred 후보.

### 제공자 예약현황 엔드포인트 — `GET /provider/reservations`(top-level) (AC1·AC3)

- **왜 top-level인가:** "내(제공자) 룸들의 예약"은 **여러 룸에 걸친** 조회라 중첩 `/rooms/{room_id}/reservations`(단일 룸 생성·취소)와 별개다. `me_router`(`GET /reservations`)와 동형으로 `provider_router`(`prefix="/provider/reservations"`)를 둔다. 같은 `tags=["reservations"]`라 operationId는 `reservations_*` 규약.
- **소유권 = `Room.provider_id` 필터(AC3):** provider 본인 소유 룸만 조회 대상이다. `list_provider_rooms(session, principal.user_id)`가 소유 룸을 가져오고, 그 룸 id로만 예약을 조회하므로 **타 제공자 룸의 예약은 구조적으로 노출 불가**(룸 소유권이 곧 예약 가시성 경계 — 별도 예약 단위 소유권 검사 불요).
- **도메인 경계(architecture.md L354·L356):** `reservations.service`는 **rooms를 import하지 않는다**(4.9 `rooms.service → reservations.service` 역방향과 순환). 따라서 (a) 룸 소유권 조회는 **rooms.service**(`list_provider_rooms`)가, (b) 룸 단위 예약 조회는 **reservations.service**(`list_reservations_for_rooms` — room_ids만 받음)가, (c) 둘의 합성·룸 이름·익명 라벨은 **라우터(조합 계층)**가 한다. 라우터 → rooms.service/rooms.models import는 단방향(이미 존재 `router.py:59-60`)이라 안전.
- **flat + 소비처 분류:** 서버는 `created_at desc` flat 반환(4.8 선례). 다가오는/지난 분류는 소비처가 `slot_starts`로 계산(서버 시계·캐시 신선도 회피). 6.1은 소비 UI가 없으므로(provider 웹 표면 defer) 분류 로직은 후속 스토리가 소유.

### 6.1 vs 6.2 경계 (혼동 방지 — ★중요)
