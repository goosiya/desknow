# Story 6.2: 예약 거절

Status: done

## Story

As a 제공자,
I want 불가피한 예약을 시작 전까지 거절하길,
so that 받을 수 없는 예약을 정리하고 슬롯을 다시 연다 (FR-24, UJ-3, NFR-7).

> **이 스토리는 Epic 6(제공자 예약 관리)의 두 번째 스토리이자, 제공자 표면 최초의 *쓰기*(상태 전이) 슬라이스다.** 6.1이 제공자 소유 룸 예약 **조회**(읽기 전용)를 깔았고, 6.2는 그 위에 **거절 쓰기**를 얹는다: **(A)** 거절 게이팅 래퍼(`reject_reservation_for_provider` 신규 — 시작 전 only·종료 상태 멱등, `cancel_reservation_for_booker`의 거울상) + **(B)** provider 거절 엔드포인트(`POST /provider/reservations/{reservation_id}/reject`) + **(C)** **통지 생성 배선**(거절 성공 시 예약자에게 `status_change`/reason=`rejected` 통지 1건 — 5.3은 표시만 깔았고 *생성*은 6.2 책임) + **(D)** ★**cancel↔reject 교차 race 의무 회수**(`_transition_to_terminal`을 조건부 원자 UPDATE로 결정화 — epic-5 회고 2c) + **(E)** SDK 재생성. **거절 상태 전이·슬롯 재활성 프리미티브(`reject_reservation`)는 4.1이 이미 만들었으므로 재구현 금지** — 6.2는 게이팅·엔드포인트·통지·race 처리만 더한다. **신규 마이그레이션 0**(`reservations.status`·`reservation_slots`·`notifications` 전부 재사용). **provider 웹/모바일 UI 0**([[web-is-booker-only-provider-api-only]] — 거절 UI는 "provider 웹 표면" 버킷 defer). 예약자 거절 배너 표시는 **5.3이 이미 구축**(6.2는 그 배너에 데이터를 공급만).

## 범위 결정 (create-story 2026-06-17 — 의무 회수 편입 포함)

이 스토리가 닿는 경계와, epic-5 회고 2c가 **6.2에 의무 편입**으로 명시한 부채 처리를 다음과 같이 확정한다:

1. **거절 게이트 = "예약 시작 전까지"(`now < earliest_slot_start`).** FR-24·AC는 "예약 시작 전까지 거절 가능"을 명령한다. 따라서 `confirmed`인 예약은 **가장 이른 점유 슬롯 시작시각보다 현재가 이전일 때만** 거절 가능하고, **시작했거나 지났으면**(`earliest <= now`) 차단한다(409 `REJECT_WINDOW_PASSED` 신규). 종료 상태(`cancelled`/`rejected`)는 시간 판정 *전에* 멱등 반환한다(슬롯 0건 → `earliest` 계산 불가 — `cancel_reservation_for_booker`의 분기 순서와 동형). 6h 윈도우(취소)와 달리 거절은 **고정 윈도우 없이 시작 직전까지** 가능하다(제공자 운영 권한). [Source: epics.md#Story 6.2 L1024-1034 / FR-24 L71 / service.py:225-277 `cancel_reservation_for_booker` 분기 순서 미러]

2. **★cancel↔reject 교차 race 의무 회수 = 조건부 원자 UPDATE(`WHERE status='confirmed'`).** epic-5 회고 2c(★)와 deferred-work.md L43이 **6.2를 이 부채의 책임 스토리**로 명시한다 — 거절 엔드포인트가 두 번째 종료 전이 경로를 도입하므로, booker 취소와 provider 거절이 동시 진입하면 현재 `_transition_to_terminal`(service.py:167-184)이 **Python에서 `reservation.status`를 읽고**(stale `confirmed`) flip하여 **last-write-wins**로 최종 상태(`cancelled`/`rejected`)가 비결정적이 된다. 슬롯 재활성은 양쪽 동일이라 데이터 손상은 없으나, **"누가 종료시켰나" 히스토리와 통지(FR-18a) 의미가 흔들린다** — 6.2가 통지를 *생성*하므로 이 비결정성이 이제 사용자-가시 결함이 된다(거절 안 했는데 거절 통지, 또는 그 역). **해법(이 스토리 확정 — 회수 옵션 중 택1):** `_transition_to_terminal`의 read-then-flip을 **조건부 원자 UPDATE**(`UPDATE reservations SET status=:target WHERE id=:id AND status='confirmed'`)로 바꿔 **DB가 단일 승자를 중재**하게 한다. rowcount=1이면 이 전이가 승자(슬롯 재활성+commit), rowcount=0이면 동시 전이가 선점했거나 이미 종료 상태 → 슬롯/통지 없이 현재 상태로 수렴(멱등). 이로써 cancel↔reject는 **상호 배타·첫-전이-승자**로 결정화되고, **재-defer 금지**(6.2가 책임 스토리). last-write-wins 명시 수용 대신 조건부 UPDATE를 택한 이유: 통지 의미가 6.2에서 비로소 가시화되므로 결정성이 correctness 요건이 된다. [Source: epic-5-retro-2026-06-17.md L59-64·L105 / deferred-work.md L43 "cancel-vs-reject 교차 전이 race" / [[dev-workflow-policy-deferred-and-repeat-mistakes]]]

   - **cancel(4.7) 무회귀 보장:** 이 변경은 `_transition_to_terminal`(공유 프리미티브)을 건드리므로 `cancel_reservation`도 영향받는다. 그러나 (a) cancel-vs-cancel은 여전히 무해(첫째=승자 cancelled, 둘째=rowcount 0 멱등 no-op → 동일 최종 `cancelled`), (b) 단일 cancel은 rowcount=1 → 기존과 동일(slot DELETE+commit), (c) 이미 종료 상태는 fast-path가 DB 왕복 없이 멱등 반환(불변) — 4.7 AC1~AC4·6h 게이트 전부 보존. **기존 cancel 테스트가 전부 그린이어야 한다(무회귀 게이트).**

3. **거절 성공 시 예약자 통지 생성(FR-18a 배선 — 6.2 책임).** 5.3 노트가 명시: status_change 배너 **표시·'확인'은 5.3, 생성 배선은 6.2/8.3**. 따라서 6.2는 거절이 **실제로 confirmed→rejected 전이를 수행했을 때만**(race 패자·멱등 재거절 제외) `notifications.service.create_notification(session, user_id=예약자, reservation_id, type='status_change', reason='rejected')`을 호출해 예약자에게 통지를 남긴다. 통지 생성은 라우터(조합 계층)가 한다 — `reservations.router → notifications.service`는 **단방향**(notifications.service는 reservations를 import하지 않음 = service↔service 순환 아님, 4.8의 `reviews_service`/`rooms_service` import 선례와 동형). `create_notification`은 멱등(`uq_notifications_user_reservation_type`)이라 재시도·동시 중복도 1건으로 수렴. [Source: 5-3...md L20-21·notifications/models.py:55-60·notifications/service.py:32-73]

4. **6.2 표면 = provider 인증 API 엔드포인트 전용(provider 웹/모바일 UI 0).** [[web-is-booker-only-provider-api-only]](KTH 확정): `apps/web`은 booker 전용이고 provider 로그인/내비/라우트가 전무하다. 6.2의 거절은 **엔드포인트만** 만들고 provider 거절 화면은 "provider 웹 표면" 버킷으로 defer한다. **단, 예약자-facing 거절 배너는 신규 작업 0** — 5.3이 status_change 배너 표시를 web에 이미 구축했으므로 6.2가 통지 행을 생성하면 예약자 다음 접속 시 **기존 배너가 자동 표시**한다(공급만). 모바일=별개 "모바일 dev-build 푸시" 버킷. `apps/web`·`apps/mobile`·`apps/admin` **변경 0**. SDK는 드리프트 게이트 때문에 **재생성한다**(provider 거절 소비처 UI는 후속). [Source: 메모 [[web-is-booker-only-provider-api-only]]·[[web-mobile-parity-on-changes]] / 6.1 동일 패턴]

5. **reader status 필터 부채(2c 두 번째 항목)는 6.2에서 *트리거되지 않음* — 명시 평가(재-defer 아님).** epic-5 회고 2c는 `confirmed_slot_starts` reader status 필터 부재를 "**6.2 거절이 *비-DELETE* 종료 경로면** 방어심층 필요"라고 **조건부**로 적었다. 그러나 거절은 `reject_reservation → _transition_to_terminal → _release_slots`로 **점유 슬롯을 DELETE**하는 종료 경로다(취소와 동일 메커니즘) — 즉 `reservation_slots`엔 여전히 **활성 점유만** 잔존하므로 `confirmed_slot_starts` 불변식("DELETE-on-terminal로 status 조인 불요")이 **보존**된다. 따라서 **방어심층(reader status 조인)은 6.2에서 불요**이며, 이는 트리거 조건이 거짓(거절=DELETE 경로)이라는 **명시 평가**이지 침묵 재-defer가 아니다. 이 부채의 실제 트리거는 **비-DELETE 종료 경로**(E8 만료/비활성 캐스케이드·직접 status UPDATE)이고 거기서 회수된다. [Source: epic-5-retro-2026-06-17.md L63 / deferred-work.md L37 "confirmed_slot_starts 계열 reader status 필터 없음" / service.py:153-164 `_release_slots`(DELETE)]

## Acceptance Criteria

**AC1 — 시작 전 거절 → 슬롯 재활성 + 거절 상태 잔존 (FR-24, NFR-7)**
**Given** 제공자 본인 소유 룸의 **확정(`confirmed`)** 예약이 있고 (`POST /api/v1/provider/reservations/{reservation_id}/reject`)
**When** 제공자가 **예약 시작 전**(가장 이른 점유 슬롯 시작시각보다 현재가 이전)에 거절하면
**Then** 예약은 `rejected`로 전이되고, 점유했던 `reservation_slots` 행이 **동일 트랜잭션 내에서 DELETE(슬롯 재활성)**되며, 예약 행은 예약자 히스토리에 **`rejected` 상태로 잔존**(`slot_starts` 스냅샷 보존)한다 → 200 + `ReservationPublic`(status=`rejected`·`slot_starts=[]`). 거절 사유 입력은 **선택**(MVP 미입력 — 통지 reason은 고정 `'rejected'`). 재활성된 슬롯은 같은 `(room_id, slot_start)`를 다른 예약이 재점유할 수 있다.
- **전이·재활성은 4.1 프리미티브(`reject_reservation`)를 그대로 호출**(재구현 금지) — 단 멱등 검사를 **조건부 원자 UPDATE**로 강화(범위 결정 #2).

**AC2 — 거절 성공 시 예약자 상태변경 통지 생성 (FR-18a 배선)**
**Given** 거절이 **실제로 confirmed→rejected 전이를 수행**하면
**When** 거절 트랜잭션 후
**Then** 예약자(`reservation.booker_id`)에게 `status_change` 통지 1건이 생성된다 — `notifications.service.create_notification(user_id=booker_id, reservation_id, type='status_change', reason='rejected')`. 예약자 다음 접속 시 **5.3이 구축한 배너**가 "{룸} {KST 일시} 예약이 거절됐어요."로 표시한다(6.2는 데이터 공급만·web 신규 작업 0). 통지 생성은 **멱등**(`uq_notifications_user_reservation_type` — 재거절·동시 중복도 1건).
- **race 패자·멱등 재거절·종료 상태(이미 cancelled/rejected)에는 통지를 생성하지 않는다**(이 호출이 전이를 수행한 경우에만) — 예: booker가 먼저 취소했으면(`cancelled`) "거절" 통지를 만들지 않는다.

**AC3 — 시작 후/종료 상태 거절 시도 = 차단 또는 멱등 무시 (공통 상태 머신, AC L1032-1034)**
**Given** 예약 시작 후(`earliest <= now`)이거나 이미 종료 상태(`cancelled`·`rejected`)인 예약
**When** 거절을 시도하면
**Then** (a) **시작 후 confirmed** → 409 `REJECT_WINDOW_PASSED`(신규)로 **차단**(상태 전이·슬롯 변경·통지 0). (b) **이미 종료 상태** → **멱등하게 무시**(DB 쓰기 0, 현재 상태 그대로 200 반환, 통지 0 — 에러 아님). 분기 순서: 종료 상태 멱등 검사를 시간 게이트 *전에* 한다(종료 상태는 슬롯 0건이라 `earliest_slot_start`가 `None` → 시간 판정 불가 — `cancel_reservation_for_booker` 선례).

**AC4 — ★cancel↔reject 교차 race 결정화 (epic-5 회고 2c 의무 회수, NFR-7)**
**Given** 같은 `confirmed` 예약에 booker 취소와 provider 거절이 동시 진입
**When** 두 전이가 경합하면
**Then** `_transition_to_terminal`의 **조건부 원자 UPDATE**(`WHERE status='confirmed'`)가 **단일 승자**를 중재한다 — 첫 전이만 status flip + 슬롯 DELETE를 수행(rowcount=1), 둘째는 rowcount=0으로 **슬롯 변경·통지 없이 현재 상태로 수렴**(멱등). 최종 상태와 통지는 **결정적**(첫-전이-승자)이다. last-write-wins 비결정성 제거. **cancel(4.7) 기존 동작은 전부 보존**(cancel-vs-cancel 무해·단일 cancel·종료 상태 멱등 — 무회귀 게이트).

**AC5 — RBAC + 소유권: provider 전용·본인 룸 예약만 (architecture.md §Boundaries L351·L356)**
**Given** `POST /provider/reservations/{reservation_id}/reject` 요청
**When** 인증된 provider가 호출하면
**Then** **본인 소유 룸(`Room.provider_id == principal.user_id`)의 예약만** 거절할 수 있다. 미존재 예약·**타 제공자 룸의 예약**·소유 룸이 아닌 예약은 **404 `RESERVATION_NOT_FOUND`**(403 아님 — 타인 예약 존재 누설 금지, 4.7 cancel·6.1 소유권 패턴 미러). 미인증=401 `UNAUTHENTICATED`, booker/admin=403 `FORBIDDEN_ROLE`(`require_role("provider")` — 6.1 `_require_provider` 재사용).

**AC6 — 결정성·도메인 경계·게이트**
**Given** 도메인 경계·반복 함정·게이트
**When** 구현하면
**Then** ① `reservations.service`는 `reservations`/`reservation_slots`만 만진다(rooms import 0 — 룸 소유권 조회는 라우터가 `session.get(Room)`로, 통지 생성은 라우터가 `notifications.service`로). ② **신규 마이그레이션 0**(스키마 변경 없음)·**신규 ErrorCode 1**(`REJECT_WINDOW_PASSED` 409)·**신규 의존성 0**. ③ **SDK 재생성 1**(`reservationsRejectReservation` — 신규 엔드포인트). 드리프트 게이트(Layer A·B·check-types) 그린. ④ 백엔드 `pytest`/`ruff`/`mypy` 그린(service `ValueError` 500 누출 0). ⑤ **라이브 Supabase head 불변**(`c9d3e5b7f1a4`·`alembic upgrade head` no-op) 확인. [[dev-workflow-policy-live-db-migration]] ⑥ web/mobile/admin 변경 0(예약자 배너=5.3 기존·provider UI defer·모바일 defer). [[web-mobile-parity-on-changes]]

### ★cancel↔reject 교차 race 의무 회수 (epic-5 회고 2c — 재-defer 금지)

- **부채(deferred-work.md L43):** `_transition_to_terminal`은 `if reservation.status in _TERMINAL_STATUSES`를 **Python에서** 검사하고 flip한다. booker 취소(4.7)와 provider 거절(6.2)이 동시 진입하면 둘 다 stale `confirmed`를 읽고 통과해 **last-write-wins** — 최종 status가 `cancelled`/`rejected` 비결정. 슬롯 재활성은 양쪽 동일(데이터 손상 0)이나, **6.2가 통지를 생성**하므로 "거절 통지가 떴는데 실제론 취소" 같은 **사용자-가시 비결정성**이 생긴다.
- **해법(범위 결정 #2 — 조건부 원자 UPDATE):** `UPDATE reservations SET status=:target WHERE id=:id AND status='confirmed'`. DB 행 락이 단일 승자를 중재: rowcount=1=승자(슬롯 DELETE+commit), rowcount=0=패자/멱등(슬롯·통지 0·현재 상태 수렴). 첫-전이-승자로 **결정화**.
- **왜 last-write-wins 명시 수용이 아니라 조건부 UPDATE인가:** 회고는 둘 다 허용하나, 6.2에서 통지가 가시화되며 결정성이 correctness 요건이 됐다. 조건부 UPDATE는 `_transition_to_terminal` 한 곳 수정으로 cancel/reject **공유** 결정화(작은 수술·근본 해결).
- **cancel 무회귀(★):** 공유 프리미티브라 cancel도 바뀌나 동작 불변 — cancel-vs-cancel(첫째 cancelled·둘째 rowcount 0 멱등)·단일 cancel(rowcount 1)·종료 상태 fast-path(DB 왕복 0) 전부 보존. **기존 cancel 테스트 그린이 회수의 안전망.**
- **통지 결정 로직(라우터):** `was_confirmed(전이 전 in-memory) and updated.status == REJECTED`일 때만 통지. race 패자는 `updated.status==cancelled`라 조건 거짓 → 거절 통지 안 만듦(정확). create_notification 멱등이 이중 방어.

### reader status 필터 부채 — 6.2 미트리거 명시 평가 (재-defer 아님)

- **부채(deferred-work.md L37):** `confirmed_slot_starts` 계열 reader가 status 조인 없이 "활성 점유만"을 **DELETE-on-cancel 불변식에만** 의존한다. **비-DELETE** 종료 경로가 추가되면 유령 점유가 가용성 차감에 누출.
- **6.2 평가:** 거절은 `reject_reservation → _transition_to_terminal → _release_slots`로 **점유 슬롯을 DELETE**한다(취소와 동일 메커니즘). 즉 `reservation_slots`엔 여전히 활성 점유만 잔존 → **불변식 보존·방어심층 불요**. 이는 트리거 조건(비-DELETE)이 **거짓**이라는 명시 평가지 침묵 재-defer가 아니다. 실제 트리거=E8 만료/비활성 캐스케이드(비-DELETE) 시 회수.

### 거절 게이트 = "예약 시작 전까지" (AC1·AC3)

- **게이트:** `confirmed` 예약은 `now < earliest_slot_start`(가장 이른 점유 슬롯 시작 *전*)일 때만 거절 가능. `earliest <= now`(시작·경과)면 409 `REJECT_WINDOW_PASSED` 차단. 취소의 6h 윈도우와 달리 **시작 시각 자체가 경계**(제공자 운영 권한 — 시작 직전까지).
- **분기 순서(중요·cancel 선례):** ① 종료 상태(`cancelled`/`rejected`)면 시간 검사 *전* 멱등 반환(종료 상태는 슬롯 0건 → `earliest=None`이라 시간 판정 불가). ② confirmed면 시간 게이트. ③ 통과 → `reject_reservation` 호출.
- **`earliest is None` 방어:** confirmed인데 점유 슬롯 0건은 불변식 위반(정상 경로 불가) → 조용히 거절하지 않고 방어 차단(`cancel_reservation_for_booker`의 동일 fail-safe).

### 소유권 = 룸 소유(`Room.provider_id`) — 라우터 판정 (AC5)

### 반복 함정 프리플라이트 ([[dev-workflow-policy-deferred-and-repeat-mistakes]] · Epic 1~5 회고)

1. **CP949/UTF-8:** 한국어 카피("이미 시작된 예약은 거절할 수 없어요.")·주석·docstring 전부 UTF-8.
