# Story 7.3: 플로팅 챗봇 진입 + 세션 대화 유지

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 사용자,
I want 어느 화면에서든 플로팅 챗봇 "룸메이트"를 열고, 화면을 이동해도 같은 대화 맥락이 이어지며, 로그아웃하면 대화가 초기화되길,
so that 흐름을 끊지 않고 도움을 받고(세션 연속성), 다른 사람·다음 로그인에 내 대화가 새지 않는다 (FR-25, UJ-4, UX-DR6).

## 범위 결정 (KTH 확정 2026-06-17 — 구현 전 반드시 인지)

이 스토리는 4개 surface(BE 챗봇 그래프·세션 / 웹 FAB·대화 UI)를 가로지른다. create-story 단계에서 KTH가 확정한 **4가지 경계 결정**이 ACâ¤Tasks 전체를 지배한다:

1. **7.3↔7.4 경계 = 동작하는 "비스트리밍" 대화 루프.** 7.3은 LangGraph 그래프 + checkpointer + thread 관리 + **비스트리밍** 메시지 엔드포인트(`invoke`)로 **실제 멀티턴 대화가 동작**한다. **SSE 스트리밍 엔드포인트(`/api/v1/chatbot/stream`)·첫토큰≤2초·`aria-live` 토큰 갱신은 7.4 단독 소유** — 본 스토리는 만들지 않는다. 7.4는 **동일 그래프**에 `astream` + SSE 프레이밍만 입힌다(그래프는 재사용·폐기 아님).
2. **Checkpointer = `MemorySaver`(인메모리, MVP).** thread_id로 키잉. 화면 이동 간 맥락 유지(SPA·단일 서버 프로세스 유지)=AC 충족. **신규 DB 테이블 0 → Alembic 충돌 없음·라이브 마이그레이션 트리거 없음.** 영속 PostgresSaver(재시작 내성)는 **배포/스케일 하드닝으로 defer**(자체 테이블 `.setup()`이 1.4 "Alembic 스키마 단독 소유" 규약과 충돌하므로 정책 결정 후행).
3. **디바이스 식별 = 클라이언트 생성 `device_id`.** 클라가 안정 UUID를 로컬 저장(웹 `localStorage`)하고 요청에 전달, 서버는 `thread_id = f"{user_id}:{device_id}"` 도출. **인증 스키마 무변경**(`refresh_tokens`·JWT 클레임 손대지 않음 — "per device" 충족하면서 인증 스택 격리).
4. **웹 전용(web-only).** 모바일은 **인증 스택 부재**(`useSession` 없음·`QueryClientProvider` 없음·Bearer/secure-store 미배선)이고 화면이 **Epic 4 placeholder**라 챗봇 대화 세션의 전제가 없다 → **"모바일 dev-build 푸시" 버킷으로 명시 라우팅**(메모 [[web-mobile-parity-on-changes]]). 모바일 `ChatbotFabSlot` placeholder는 그대로 둔다. ⚠️ **E7=챗봇 에픽이지 모바일 버킷이 아님**(혼동 주의 — 메모 경고).

## Acceptance Criteria

1. **(AC1 — 전역 플로팅 FAB "룸메이트")** 웹 모든 주요 화면(`/`, `/reservations`, `/favorites`, `/rooms/[id]`)에서 플로팅 챗봇 FAB가 전역 노출된다: `{colors.primary}` 원형(`{rounded.full}`) + 그림자 `{elevation.fab}` + `{colors.primary-foreground}` 아이콘, `z-30`(헤더·하단내비 `z-20` 위), `aria-label="룸메이트 챗봇 열기"`. 탭하면 대화 패널이 열린다. (현재 `apps/web/src/components/shell/ChatbotFabSlot.tsx`의 placeholder console.log를 실동작으로 대체 — 스타일·위치·a11y 하한선은 보존.)

2. **(AC2 — 대화 패널 + 첫 진입 제안 칩)** 패널이 열리면 대화 UI(메시지 리스트 + 입력 + 전송)가 표시된다. **첫 진입(빈 대화)** 시 제안 칩 2종("환불 규정?" / "강남 오후 3시 빈 방")이 보이고, 탭하면 그 텍스트가 메시지로 전송된다. 패널은 **포커스 트랩**(열릴 때 포커스 진입·Esc로 닫고 FAB로 복귀), `prefers-reduced-motion` 존중, 새 응답은 `aria-live="polite"` 영역으로 안내된다. 페르소나는 친근한 **해요체** "룸메이트"(스터디룸 도우미). **(스트리밍·타이핑 토큰 단위 갱신은 7.4 — 본 스토리는 비스트리밍 응답 + 전송~응답 사이 로딩(타이핑) 인디케이터만.)**

3. **(AC3 — 비스트리밍 멀티턴 대화 동작)** 메시지를 전송하면 `POST /api/v1/chatbot/messages`(JWT 보호)가 LangGraph 그래프를 `invoke`해 **전체 응답을 한 번에** 반환한다. 같은 `thread_id`(=`user_id:device_id`)로 연속 전송 시 **이전 맥락이 유지**된다(checkpointer가 이전 메시지를 자동 로드·누적). 모델 호출 실패는 7.1 `normalize_llm_error` → `DomainError(LLM_PROVIDER_UNAVAILABLE, 502)`로 정규화되고 UI는 "잠깐 답이 막혔어요. 다시 물어봐 주실래요?" + 재전송으로 graceful degrade한다. **(실제 RAG 근거 검색=7.5, 예약DB 툴=7.6, 범위 밖 거절 골든셋=7.7 — 본 스토리 봇은 툴·거절프롬프트 없는 기본 대화 LLM. 제안 칩 답변의 환각/부정확은 7.3 결함 아님.)**

4. **(AC4 — 화면 이동 간 세션 유지)** 챗봇 패널을 연 채 또는 대화 후 다른 화면으로 라우팅 이동해도 **대화 맥락(표시 transcript + 서버 thread 상태)이 유지**된다. 웹 transcript는 `thread_id`로 키잉된 캐시(TanStack Query)로 네비게이션을 가로질러 보존되고, 패널 재오픈/새로고침 시 `GET /api/v1/chatbot/messages`로 서버 checkpointer 상태에서 재수화(rehydrate)된다. `device_id`는 안정(로그아웃에도 회전 안 함 — 디바이스 식별자).

5. **(AC5 — 로그아웃/미인증 시 대화 초기화, FR-2 연동)** 세션이 종료(로그아웃 또는 refresh 무효화 → `authMe` 401 → `useSession` data=null)되면 챗봇은 **대화를 초기화**한다: 웹은 transcript 캐시 제거 + 패널 닫기, 서버는 `DELETE /api/v1/chatbot/session`로 해당 `user×device` thread의 checkpointer 상태를 폐기(best-effort). 다음 로그인 시 같은 디바이스라도 **빈 대화로 시작**한다(이전 대화 미노출). FR-2 refresh 무효화가 챗봇 세션 종료로 연동됨을 실증.

6. **(AC6 — 게이트 그린 + SDK 재생성, 라이브 마이그레이션 없음)** BE `cd apps/api && uv run ruff check . && uv run mypy && uv run pytest` 전부 통과(네트워크·실키 없이 페이크 모델로 단위 검증). 웹 `pnpm --filter web lint && pnpm --filter web typecheck && pnpm --filter web test` 통과. **신규 라우터가 OpenAPI 계약을 변경하므로 `scripts/export_openapi.py` + `pnpm --filter @desknow/api-client generate`로 SDK 재생성**(Layer A 드리프트 게이트 `test_openapi_export` 그린). **신규 DB 테이블 0 → Alembic 마이그레이션·라이브 `upgrade head` 없음**(MemorySaver·derived thread_id — 라이브 DB 마이그레이션 정책 미트리거).

### ★알려진 함정 (구현 전 반드시 확인 — 프리플라이트)

1. **MemorySaver/그래프 싱글톤 — 최악의 함정:** 컴파일된 그래프(+MemorySaver)를 **요청마다 새로 만들면 매 요청 메모리가 비어 맥락이 절대 유지되지 않는다**(AC3/AC4 전면 실패). 모듈 레벨 lazy 싱글톤(`_get_graph()`)으로 1회 생성·재사용. 라우터 테스트로 "같은 thread 2요청 맥락 유지"를 회귀 고정.
2. **MemorySaver는 프로세스 로컬 — 멀티워커/재시작 시 소실:** MVP 단일 인스턴스(Phase 0) 전제. uvicorn 멀티워커/gunicorn·재배포 시 대화 휘발. AC는 "화면 이동 간 유지"(SPA·동일 프로세스)라 충족하나, **운영 단일 워커 전제를 Completion Notes에 명시**. 영속화=PostgresSaver는 배포/스케일 defer(Alembic 충돌 정책 결정 후행).
3. **device_id 검증 누락 → thread_id 오염:** `thread_id=user_id:device_id`를 외부 입력 device_id로 합성하므로, 검증 없으면 임의 문자열로 타 thread 접근·메모리 남용 가능. 길이 상한·charset(UUID/opaque) 제한·제어문자 거부. user_id는 JWT 도출(신뢰)이라 cross-user 접근은 불가(thread_id 앞에 인증된 user_id 고정).
4. **신규 라우터 = OpenAPI 변경 = SDK 재생성 의무:** 7.2는 라우터 0이라 재생성 불요였으나 7.3은 엔드포인트 3종 추가 → `export_openapi.py` + `generate` 필수. 빠뜨리면 드리프트 게이트(`test_openapi_export`)가 빨강. 웹은 재생성된 `chatbot_*` 함수를 호출.
5. **langgraph pyproject 미선언:** venv에 전이 설치돼 있어 import는 되지만 **명시 의존성 추가 안 하면** 재현성·lock이 깨진다(7.1이 의도적 제외). `uv add langgraph` 후 lock 실버전 기록.
6. **네트워크 단절 카피 표준 오버라이드:** UX 문서는 "오프라인이라 지금은 답하기 어려워요"라 하나, **프로젝트 표준은 "네트워크 연결이 끊겼어요"**(메모 [[terminology-network-disconnect-not-offline]]). UI 카피는 표준을 따른다.
7. **로그아웃 버튼 부재 → AC5 트리거 경로 주의:** 웹에 로그아웃 UI가 아직 없다(deferred-work, 3.7). AC5의 실 트리거는 **`useSession` data→null 전이**(refresh 무효화 시 authMe 401). 이 effect 경로로 FR-2 연동을 충족(전용 로그아웃 버튼이 생기면 그 핸들러도 동일 초기화 호출 — 후속 로그인 스토리가 정렬).
8. **제안 칩 답변 환각은 7.3 결함 아님:** 봇은 툴(RAG/예약)·거절프롬프트 없는 기본 LLM이라 "환불 규정?"에 환각 답할 수 있다. 근거 검색=7.5, 거절=7.7. 리뷰 시 스펙 이탈로 오판 금지(범위 명시).

### ★세션 모델 (AC3, AC4, AC5 — 핵심 설계)

- **thread_id = `f"{user_id}:{device_id}"`** — user_id는 JWT(신뢰), device_id는 클라 생성 UUID(localStorage 영속). "사용자×디바이스" 단위 충족, 인증 스키마 무변경.
- **상태 보관**: 메시지 이력=LangGraph `MemorySaver`(thread_id 키). 웹 표시 transcript=TanStack Query 캐시(`["chatbot","messages",deviceId]`) — 네비게이션 가로질러 보존, 재오픈/새로고침 시 `GET /messages`로 서버 상태 재수화.
- **화면이동 유지(AC4)**: App Router layout(AppShell) 비리마운트 + QueryClient 영속 → 패널 오픈 상태·transcript 모두 보존.
- **로그아웃 초기화(AC5)**: session→null → 웹 캐시 제거+패널 닫기 + `DELETE /session`(서버 thread 폐기). device_id는 유지(디바이스 식별), thread만 초기화 → 재로그인 시 빈 대화.
- **격리**: thread_id가 user_id로 시작 → 타 사용자가 같은 device에 로그인해도 다른 thread(대화 미공유). 이전 사용자 thread는 로그아웃 DELETE로 폐기(미발화 시 메모리 잔존하나 user_id 고정이라 타인 도달 불가 — 경미, Completion Notes 명시).
