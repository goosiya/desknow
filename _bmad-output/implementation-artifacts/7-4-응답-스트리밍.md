# Story 7.4: 응답 스트리밍

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 사용자,
I want 챗봇 응답을 토큰 단위로 빠르게(첫 토큰 ≤2초) 받고, 받는 동안 흘러나오는 걸 보길,
so that 전체 응답을 기다리지 않고 답이 흘러나와 체감 속도가 빠르다 (FR-30, UJ-4, NFR-2·6, SM-6).

## 범위 결정 (KTH 확정 2026-06-18 — 구현 전 반드시 인지)

7.3이 **의도적으로 7.4에 남긴 경계**(비스트리밍 `invoke` → 스트리밍 `astream` + SSE 프레이밍)를 채우는 스토리다. create-story 단계에서 KTH가 확정한 **4가지 경계 결정**이 AC↔Tasks 전체를 지배한다:

1. **SSE 프레이밍 = `sse-starlette`(`EventSourceResponse`).** deferred-work L129 "SSE 서버 프레이밍 견고성"(스파이크의 공백분할 + 리터럴 `[DONE]` → 토큰에 `\n\n`·공백·`[DONE]` 섞이면 프레임/종료 깨짐)을 **의무 회수**한다. EventSourceResponse가 프레이밍·keep-alive ping·클라 disconnect 정리를 라이브러리로 처리하고, 각 토큰은 `data: {json}`(JSON 인코딩 → `\n`·`[DONE]` 오염 원천 차단)로 흘린다. 종료는 명시 `event: done`, 에러는 `event: error`. **신규 의존성 `sse-starlette` 추가**(uv add + lock — 7.3 langgraph 선례와 동형 절차).

2. **동일 그래프 재사용(graph.py 무변경 목표).** 7.3 `get_graph()` 싱글톤을 그대로 쓰고, `g.astream(..., stream_mode="messages")`로 노드 내부 LLM 토큰을 토큰 단위로 소비한다. **그래프 구조·노드 코드 변경 0이 목표** — 단 sync `call_model`(`chat_model.invoke`) 노드가 `astream(stream_mode="messages")`에서 토큰을 실제로 방출하는지는 **착수 전 스파이크로 실증**(7.3이 langgraph 동작을 uv run 스파이크로 확정한 선례). 방출 안 되면 **최소 조정**(노드를 `ainvoke`/스트리밍 활성화)만 허용하고 Completion Notes에 사유 기록.

3. **실패 turn 입력 롤백 — 스트리밍 경로에서 재확인(메모 의무).** [[langgraph-failed-turn-input-rollback]]: LangGraph는 노드 실패 **전에** 입력 HumanMessage를 checkpointer에 영속한다. 7.3 `send_message`는 명시 id + `update_state(RemoveMessage)`로 롤백했다. **스트리밍 `stream_message`도 동일 롤백을 반드시 재현**한다(첫 토큰 전 실패=TTFT 실패, 스트림 중 실패 모두 — 재전송 시 user 메시지 중복 누적 차단, AC4 무결성). 메모가 "7.4~7.7 동일그래프 재사용 시 재확인"을 명시.

4. **웹 전용(web-only) + RN = "모바일 dev-build 푸시" 버킷.** BE `/api/v1/chatbot/stream`은 **웹·모바일 공용**으로 구현하되, 프론트는 **웹 fetch-stream만**. AC2가 `react-native-sse`를 명시하나, 모바일은 **인증 스택 부재**(`useSession`·`QueryClientProvider`·secure-store Bearer 미배선)이고 화면이 **Epic 4 placeholder**라 7.3과 동일하게 전제가 없다 → RN 스트리밍은 deferred-work forward note로 "모바일 dev-build 푸시" 버킷에 라우팅(`apps/mobile` 변경 0). ⚠️ **E7=챗봇 에픽 ≠ 모바일 버킷**(혼동 주의 — 메모 [[web-mobile-parity-on-changes]]).

## Acceptance Criteria

1. **(AC1 — SSE 스트리밍 엔드포인트 + 첫 토큰 ≤2초)** 인증된 `POST /api/v1/chatbot/stream`(JWT 보호)에 질문을 보내면, 응답이 `text/event-stream`으로 **토큰 단위 스트리밍**된다. 기준 프로바이더(OpenAI)·웜 컨테이너·기준 프롬프트에서 **첫 응답 토큰이 ≤2초**에 시작되고(SM-6·NFR-2, p90·콜드스타트/프로바이더 편차 제외 — 자동 강제 단언이 아니라 측정 프로토콜·웜업 설계로 충족하고 Completion Notes에 측정 근거 기록), 첫 토큰 전에는 **타이핑 인디케이터**가 표시된다. 같은 `thread_id`(=`user_id:device_id`)로 연속 전송 시 **이전 맥락이 유지**된다(7.3 checkpointer 누적 — 스트리밍도 동일 그래프).

2. **(AC2 — 견고한 SSE 프레이밍 + 종료/에러 신호)** 스트림은 `sse-starlette` `EventSourceResponse`로 프레이밍된다: 각 토큰은 `data: {"delta": "..."}`(JSON 인코딩 — 토큰에 `\n`·공백·`[DONE]`이 섞여도 프레임 무손상, deferred-work L129 회수), 정상 종료는 `event: done`, LLM 업스트림 실패는 `event: error`(`{"code":"LLM_PROVIDER_UNAVAILABLE","message":...}`)로 **인밴드** 전달된다(스트림 시작 후엔 HTTP 상태코드 불가). 스트림 완료 후 `GET /chatbot/messages` 재수화 시 **전체 응답 turn이 일관되게 보존**된다(checkpointer가 완성 AI 메시지 영속).

3. **(AC3 — 웹 토큰 단위 갱신 + aria-live)** 웹은 표준 fetch-stream(`ReadableStream` reader + SSE 프레임 파서)으로 수신하고(인증=기존 `credentials:"include"` 쿠키), 받는 토큰을 **어시스턴트 버블에 점진적으로 누적 렌더**한다. 스트리밍 갱신은 `aria-live="polite"` 영역으로 안내된다(스크린리더). 첫 토큰 전엔 타이핑 인디케이터, 첫 토큰 도착 시 인디케이터→스트리밍 텍스트 전환. **로컬 UI 피드백(전송 탭) ≤100ms**(옵티미스틱 사용자 버블 즉시 append — 7.3 패턴 유지).

4. **(AC4 — 실패 graceful degrade + 재전송, 무결성 보존)** 스트림 중(또는 첫 토큰 전) LLM 업스트림 실패 시 `event: error`가 전달되고, 웹은 7.3과 동일 카피("잠깐 답이 막혔어요. 다시 물어봐 주실래요?") + **재전송**으로 강등한다(막다른 화면 금지). 실패한 turn의 입력 HumanMessage는 서버 thread에서 **롤백**되어, 재전송 시 user 메시지가 중복 누적되지 않는다([[langgraph-failed-turn-input-rollback]] 회수 — 스트리밍 경로 회귀 테스트로 고정). 부분 수신된 어시스턴트 토큰(있다면)은 정리하고 사용자 버블은 유지(재전송용).

5. **(AC5 — 게이트 그린 + SDK 재생성, 라이브 마이그레이션 없음)** BE `cd apps/api && uv run ruff check . && uv run mypy && uv run pytest` 전부 통과(네트워크·실키 없이 **스트리밍 페이크 모델**로 단위 검증). 웹 `pnpm --filter web lint && pnpm --filter web typecheck && pnpm --filter web test` 통과(스트림 파서·점진 렌더·에러 강등 단위 검증). **신규 라우터(`/stream`)가 OpenAPI 계약을 변경하므로** `uv run python scripts/export_openapi.py` + `pnpm --filter @desknow/api-client generate`로 SDK 재생성(드리프트 게이트 `test_openapi_export` + api-client `check-drift` 그린). 웹 fetch-stream 모듈은 **eslint `no-direct-fetch` allowlist**(preset NOTE가 예고한 "E7 챗봇 SSE만 해당 모듈 allowlist")를 추가해 통과. **신규 DB 테이블 0 → Alembic·라이브 `upgrade head` 없음**(7.3 MemorySaver·derived thread_id 그대로 — 라이브 DB 마이그레이션 정책 미트리거).

### ★알려진 함정 (구현 전 반드시 확인 — 프리플라이트)

1. **sync-invoke 노드의 토큰 방출 — 최우선 스파이크(Task 2):** 7.3 `call_model`은 `chat_model.invoke`(sync)다. `astream(stream_mode="messages")`가 노드 내부 LLM 토큰을 스트리밍하는지 **반드시 uv run 스파이크로 실증**(OpenAI 기준 동작 예상이나 확정 필요). 안 되면 노드 스트리밍 활성화 **최소 조정**만(그래프 재사용 목표 — 경계 결정 2). 7.3이 langgraph 동작을 스파이크로 확정한 선례.
2. **스트림 시작 후 에러는 HTTP 상태 불가 — 인밴드 `event: error`:** EventSourceResponse가 첫 바이트를 보내면 200 헤더가 이미 나간다. LLM 실패를 `raise`로 전파하면 깨진 스트림이 된다 → 반드시 `event: error` 프레임으로 전달하고 클라가 graceful degrade. **단 미인증(401)은 스트림 시작 전 dependency라 정상 HTTP 401**(이 차이 명확히).
3. **실패 turn 롤백 — 스트리밍 경로 누락 시 재전송 중복(메모 의무):** [[langgraph-failed-turn-input-rollback]]. astream도 입력을 노드 실행 전 영속하므로, 실패 시 `aupdate_state(RemoveMessage)` 안 하면 재전송 시 user 중복. **회귀 테스트로 고정**(Task 6). 메모가 "7.4 동일그래프 재사용 시 재확인" 명시 — 이번이 그 회수.
4. **SSE 프레이밍 견고성(deferred L129 회수):** 토큰 content에 `\n`·공백·리터럴 `[DONE]`이 섞이면 순진한 `data: {token}\n\n`/`[DONE]` 프레이밍이 깨진다(스파이크 server.py 결함). **sse-starlette + JSON 인코딩(`data: {"delta":...}`)으로 원천 차단** — 클라 파서도 JSON 디코드. 회귀 단언(Task 6·11). 재-defer 금지.
5. **eslint `no-direct-fetch` allowlist 범위 — 파일 단위로 좁게:** preset(`packages/config/eslint-preset.mjs` L12-14)이 "E7 챗봇 SSE만 해당 모듈 allowlist"를 예고. feature 디렉터리 전체가 아니라 **`streamMessage.ts` 1파일만** 금지 해제(SSE 전송만 raw fetch — 나머지는 SDK 가드 유지). 광역 해제 금지.
6. **baseUrl 이중 출처 드리프트:** `streamMessage.ts`가 baseUrl을 따로 선언하면 `api-client.ts`의 `NEXT_PUBLIC_API_BASE_URL` 로직과 어긋날 수 있다. `getApiBaseUrl()`을 api-client.ts에서 export해 단일 출처화(권장).
7. **첫 토큰 ≤2초는 측정 SLA(자동 강제 단언 아님):** SM-6 측정 프로토콜(웜·기준 프롬프트·기준 프로바이더·p90·콜드/편차 제외). 단위 테스트로 ≤2s를 강제하지 말 것(페이크 모델·CI 환경 무의미). 웜업 설계 + 콜드스타트 회피 + Completion Notes 측정 근거로 충족. 콜드스타트/프로바이더 편차는 명시 제외(PRD §11·review-adversarial H-2).
8. **잦은 setQueryData(델타마다) 렌더 부담:** 토큰마다 transcript 캐시 갱신은 다수 리렌더. MVP는 단순 누적 허용하되, 버벅이면 가벼운 버퍼/rAF throttle(범위 내 하드닝). Zustand 등 신규 상태 라이브러리 도입 금지(7.3 규약 — TanStack Query만).
9. **비스트리밍 경로 보존:** `send_message`/`POST /messages`/`chatbotSendMessage`(SDK)를 제거하지 말 것 — API 완전성·기존 테스트·향후 non-SSE 클라이언트. 웹 send만 streamMessage로 교체(SDK 함수는 웹 미사용으로 잔존 — 무해).

### ★스트리밍 데이터 흐름 (AC1~AC4 — 핵심 설계)

```
[웹] send(text)
  → 옵티미스틱 user 버블 append (≤100ms, AC3)
  → streamMessage(POST /chatbot/stream, credentials:include)   [raw fetch — allowlist]
       body: {message, device_id}
[BE] router.stream (async, get_current_principal → 401 if 미인증)
  → thread_id = derive_thread_id(user_id, device_id)
  → EventSourceResponse(event_gen)
       event_gen: async for delta in service.stream_message(thread_id, message):
                     yield {"data": {"delta": delta}}            [sse-starlette JSON 프레이밍]
                  성공 → {"event":"done"}
                  DomainError → {"event":"error", code/message}   [인밴드, 함정 #2]
[service.stream_message] async
  → g.astream({messages:[Human(id=mid)]}, {thread_id}, stream_mode="messages")
       토큰 chunk(AIMessageChunk, node=call_model) → yield content delta
  → native 예외 시: aupdate_state(RemoveMessage(mid)) 롤백 + normalize_llm_error  [함정 #3]
  → checkpointer가 완성 AI 메시지 자동 영속(스트림 종료 후 GET 재수화 일관, AC2)
[웹] async for ev of streamMessage:
       첫 delta 전: 타이핑 인디케이터
       delta: assistant 버블 누적 갱신 (aria-live polite, AC3)
       done: 최종화
       error: 부분 assistant 제거 + user 버블 유지 + 에러 카피/재전송 (AC4)
```

- **세션 유지·재수화·로그아웃 초기화는 7.3 그대로**(GET /messages 재수화·DELETE /session·session→null effect). 스트리밍은 send 경로만 추가.
- **멀티턴**: 같은 thread_id로 재스트림 시 checkpointer가 이전 메시지 자동 로드(7.3과 동일 — 그래프 재사용).

### ★그래프 확장 경로 (후속 스토리 인지)
