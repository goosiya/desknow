# Story 7.6: 자연어 예약 검색 (예약 DB 툴)

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 사용자,
I want 지역·시간 등 조건을 자연어로 말해 예약 가능한 룸을 추천받길,
so that 화면을 떠나지 않고 빠르게 후보를 찾아 상세로 이동한다 (FR-27, UJ-4).

## Acceptance Criteria

1. **(예약검색 툴 — react 툴콜) 자연어 조건을 해석해 가용 룸 후보를 회수한다.** 7.1 어댑터의 `bind_tools` 능력으로 **예약검색 툴**을 7.5의 문서검색 툴과 **같은 그래프에 추가 바인딩**한다(`_TOOLS`에 append — if 분기·프로바이더 하드코딩 금지). 모델이 "강남 오후 3시 빈 방"처럼 지역·시간 조건을 말한 질문에 대해 이 툴을 호출해 조건(지역명·날짜·시간)을 구조화 인자로 넘기고, 툴이 예약 가능 룸 후보를 회수한다. 잡담·서비스 안내 질문에는 호출되지 않는다(문서검색 툴과 역할 분리).
2. **(아키텍처 경계) 툴은 서비스 계층만 호출하고 SQL/ORM 쿼리에 직접 접근하지 않는다.** 예약검색 툴은 `rooms.service.search_rooms`(및 필요 시 `rooms.service.get_room_slots`)를 호출한다. 이 reader들은 **이미 `reservations.service.confirmed_slot_starts_by_room`로 활성 예약을 차감**(Story 4.9 seam)하므로 "예약 DB를 통해 신선 가용성을 회수하되 raw SQL 미접근"이라는 경계를 충족한다. 툴이 직접 `select(...)`/세션 쿼리를 작성하지 않는다.
3. **(결과 표현) 상위 3개 + 더보기 + 각 항목 상세 링크.** 매칭된 후보 중 **상위 3개**를 이름·가격(원/시간)·룸형태·부대시설·예약 가능 여부와 함께 제시하고, 각 항목은 **해당 룸 상세 페이지(`/rooms/{room_id}`)로 링크**된다. 매칭이 3개를 초과하면 **하단 "더보기"** 링크(탐색 화면 `/`)로 안내한다. 결과 표현은 모델이 툴 반환 컨텍스트를 근거로 생성하며, 프롬프트가 상위 3 + 더보기 형식을 유도한다.
4. **(지역 해석) 자연어 지역명을 법정동 코드로 해석한다.** "강남"·"강남구"·"역삼동" 같은 지역 토큰을 법정동 b_code(시군구/읍면동 레벨)로 변환해 `search_rooms(region_code=...)`에 넘긴다. 해석 불가(미매핑·모호)면 환각 없이 **"그 지역은 못 찾았어요." 신호**를 모델에 돌려 모델이 재질문/안내하게 한다(조용한 빈 결과로 오인 금지). 지역 미지정 질문(예: "지금 빈 방 있어?")은 지역 필터 없이 동작한다.
5. **(시간 조건 — 가용성 정합) 날짜·시간 조건은 신선 슬롯으로 정합 판정한다.** 사용자가 날짜/시각을 말하면(예: "내일 오후 3시") 해당 날짜·시각에 **실제 예약 가능한(`available`) 슬롯이 있는 룸**만 후보로 추린다(`get_room_slots`로 해당 날짜 슬롯 상태 확인 — 활성 예약 차감 반영). 시간 미지정이면 "오늘(ROOM_TZ) 현재시각 이후 잔여 슬롯 ≥ 1"인 룸을 가용으로 본다(`search_rooms`의 `remaining_slots`). [[availability-freshness-policy]]: 상세/예약확정은 신선 조회 책임이므로 챗봇 추천은 "이동을 위한 후보"이며 상세에서 재확인된다.
6. **(상세 링크 클릭 — 웹) 챗봇 답변의 룸 상세 링크가 웹에서 클릭 가능하다.** 웹 `ChatbotPanel`의 어시스턴트 버블이 현재 순수 텍스트만 렌더하므로, 내부 경로(`/rooms/{uuid}`·`/`)를 **클릭 가능한 Next 링크로 렌더**하도록 보강한다(임의 외부 URL 주입 방지 — 화이트리스트 내부 경로만 linkify). 링크 탭 시 해당 화면으로 이동한다. 모바일 동등 적용은 "모바일 dev-build 푸시" 버킷으로 라우팅한다([[web-mobile-parity-on-changes]] — E7=챗봇 에픽, 모바일 챗봇 UI는 별도 버킷).
7. **(기존 대화·스트리밍·세션 보존) 신규 엔드포인트·스키마·SDK·마이그레이션 0.** 답변은 기존 `POST /api/v1/chatbot/messages`(비스트리밍)·`POST /api/v1/chatbot/stream`(SSE)으로 그대로 흐른다. thread_id 세션 유지(7.3)·실패 turn 롤백·재수화·react 빈 tool-call 필터(7.5)가 회귀 없이 동작한다. **백엔드 엔드포인트·스키마·SDK 재생성·alembic 마이그레이션은 0 변경**(스키마 불변).
8. **(검증) 예약검색 react 경로·지역해석·결과형식·경계가 테스트로 단언된다.** 페이크 모델/세션 주입 단위 테스트로: 툴이 `search_rooms`/`get_room_slots`를 호출하고 SQL 직접 접근이 없음, 지역해석 매핑·미해석 신호, 상위 3 + 더보기 직렬화, 가용 0건 신호, react 2-패스 배선(기존 7.5 패턴 미러)을 단언한다. 라이브/통합(실 Postgres+예약 데이터)은 7.5와 동일하게 opt-in 스킵.

### 핵심 설계 결정 (★KTH 확정 2026-06-18)

1. **RAG·검색 = 툴콜 react 패턴(7.5 확정 계승).** 예약검색도 명시 툴이며 한 그래프에 문서검색·(7.7)거절과 공존해야 한다 → `_TOOLS` append가 아키텍처 정합 경로다([[chatbot-rag-tool-calling-react]] — 7.5 KTH 확정·7.6/7.7 공존 전제·그래프 싱글톤이라 툴 자체세션). 매 턴 무조건 검색 아님.
2. **AC "reservations.service 호출"의 정확한 구현 = `rooms.service.search_rooms` 경유.** 후보 회수의 실 reader는 `rooms.service`에 있고, 그것이 내부적으로 `reservations.service.confirmed_slot_starts_by_room`로 활성 예약을 차감한다(4.9). 즉 AC의 의도는 **"raw SQL 미접근·서비스 계층 경유·신선 예약 반영"**이며, search_rooms가 이를 충족한다. 툴이 `reservations.service`를 *직접* 부를 필요는 없다(후보 = 룸이지 예약 레코드가 아니므로 rooms reader가 맞다). **도메인 경계 준수: 툴 → chatbot, chatbot → rooms/reservations service(읽기 전용), service → DB.**
3. **지역 해석 = regions.py 역인덱스 신규(데이터 재사용).** name→code 역방향이 현재 없음(`region_name`은 code→name뿐). `_LEGAL_DONG` 번들로 import 1회 역인덱스 구성. **모호/미매핑은 graceful None → "그 지역은 못 찾았어요." 신호**(조용한 빈 결과 금지·날조 금지).
4. **시간 조건(AC5) 정합 = `get_room_slots` per 후보. ★KTH 확정(포함).** `search_rooms`의 `remaining_slots`는 **오늘 기준 카운트**라 "내일 오후 3시" 같은 특정 시각 가용을 답하지 못한다. 따라서 날짜/시각이 오면 region 후보 각각에 `get_room_slots(room_id, date)`로 그 시각 슬롯 `available` 여부를 확인한다(제공자당 1룸 규모 — N 호출 부담 작음). 시간 미지정이면 `remaining_slots >= 1`로 충분. 시각필터 defer 안 함(이번 스토리 범위).
5. **상위 3 + 더보기 표현 = 모델 생성(툴은 데이터 제공). ★KTH 확정(더보기=`/` 단순).** 툴은 후보 N개를 링크 경로와 함께 직렬화하고 "총 M개 중 3개 표시·더보기=`/`"메타를 넣되, 자연어 문장 구성은 프롬프트가 모델에 위임(7.5 그라운딩 선례). **더보기 타깃은 탐색 화면 `/`로 단순 이동**(현재 `/`는 region 쿼리파라미터 미수용 — region 선필터 전달은 ExploreView/라우팅 추가 스코프라 이번 범위 밖, `/`로만 이동).
6. **상세 링크 클릭(AC6) = 웹 버블 linkify 신규. ★KTH 확정(이번 스토리 포함).** 현 `ChatBubble`은 순수 텍스트 → 링크가 평문으로 보임. 내부 경로 화이트리스트 linkify로 클릭 가능하게(외부 URL·`dangerouslySetInnerHTML` 금지 — LLM 출력 신뢰 경계). 모바일은 "모바일 dev-build 푸시" 버킷([[web-mobile-parity-on-changes]]).

### 그래프 — 7.5 구조 그대로, 툴만 +1

7.5가 이미 `START → call_model →(tools_condition)→ tools → call_model / → END` react 그래프를 구축했다. 7.6은 **`_TOOLS`에 예약검색 툴 1개 append**가 전부다. `ToolNode(_TOOLS)`가 두 툴을 모두 실행하고, `tools_condition`이 tool_calls 유무로 분기한다. **그래프 토폴로지·`build_graph(model)` 시그니처·MemorySaver 싱글톤 변경 없음.**

- **함정 (시간/타임존):** 슬롯은 UTC aware·`target_date`는 ROOM_TZ(KST) 기준. "오후 3시"=KST 15:00 → 해당 slot_start(UTC) 매칭. `datetime.now()` 금지 — `now_utc`/`today_in_tz`(core/time 단일 출처) 주입.
- **함정 (가용성 신선도):** 챗봇 추천은 [[availability-freshness-policy]]상 "이동을 위한 후보"다. 핀/바텀시트가 느슨해도 상세/예약확정에서 신선 재확인·409 차단(FR-15)이 최종 방어선. 챗봇이 "예약 확정 가능"을 보장하지 않음(상세로 유도).
- **함정 (LLM 출력 신뢰 경계 — 웹):** linkify는 **내부 경로 화이트리스트만**. LLM이 생성한 임의 URL/스킴을 클릭 링크로 만들면 오픈리다이렉트·피싱 표면. `dangerouslySetInnerHTML` 사용 금지·정규식/파싱으로 `/rooms/{uuid}`·`/`만 `<Link>`화.
- **deferred 연계:** 실패 turn 롤백은 `native_exceptions`만 커버(7.3 forward note) — 7.6이 이 범위를 넓히지 말 것(7.5와 동형 유지). ToolNode `handle_tool_errors=True` 기본(7.5 defer): 예약검색 툴 예외(DB 단절)도 에러 문자열이 ToolMessage로 모델에 갈 수 있음 — 저확률·E7 하버닝 후보(이번에 신규 정책 도입 불요).
