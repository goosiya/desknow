## Story

As a 시드 관리자,
I want 분리된 관리자 웹(`apps/admin`)에 이메일·비밀번호로 로그인해 운영 화면(/accounts·/reservations·/ingest)에 접근하길,
so that 공격 표면이 분리된 별도 환경에서 운영 작업을 수행하고, 관리자 권한이 백엔드에서 최종 강제된다 (FR-31~33, NFR-6).

## Acceptance Criteria

1. **(AC1 — 관리자 로그인 & 셸 접근)** 시드 관리자가 분리된 관리자 웹(`apps/admin`, Next.js, **포트 3001**)에서 이메일·비밀번호로 로그인하면, `/accounts`·`/reservations`·`/ingest` 운영 화면에 접근할 수 있고 **데이터테이블·폼 중심 레이아웃**으로 표시된다. 로그인은 기존 백엔드 `POST /api/v1/auth/login`(1.8)을 재사용하며(관리자 전용 로그인 엔드포인트 없음), 인증은 httpOnly 쿠키(`desknow_access`)로 유지된다.

2. **(AC2 — 비-admin 백엔드 403 최종 차단)** admin 역할이 아닌 사용자(booker·provider)가 관리자 백엔드 엔드포인트(`/api/v1/admin/*`)에 접근하면 **백엔드 RBAC 의존성이 403 `FORBIDDEN_ROLE`로 최종 차단**한다. 인증 토큰 없이 접근하면 **401 `UNAUTHENTICATED`**. 프론트의 역할 게이트(비-admin·미인증 → 로그인 화면 유도)는 **보조일 뿐 최종 강제가 아니다**(아키텍처 L167: 백엔드 최종, Next 미들웨어/클라이언트 보조).

3. **(AC3 — 시드 관리자만 존재, 가입·생성 미제공)** 관리자 **가입·생성 기능은 어떤 표면에도 제공하지 않는다**(시드 관리자만 존재). 기존 `POST /api/v1/auth/register`는 1.8에서 이미 `role="admin"`을 422로 거부한다(불변 — 회귀 금지). 시드 관리자 계정은 **독립 시드 스크립트**(`scripts/seed_admin.py`)가 환경변수(`SEED_ADMIN_EMAIL`/`SEED_ADMIN_PASSWORD`)로 **멱등 생성/갱신**하며, 관리자 웹에는 회원가입·계정생성 UI가 없다.

4. **(AC4 — 운영 계정목록 수직 슬라이스)** 관리자가 `/accounts` 화면에 진입하면, **admin 가드된** `GET /api/v1/admin/accounts`(읽기전용·페이지네이션)가 booker·provider 계정 목록(이메일·역할·활성여부·가입일)을 반환하고 데이터테이블로 표시된다. 이로써 로그인→RBAC 403 게이트→실데이터 조회가 **end-to-end로 실증**된다. *계정 비활성(8.2)·예약 임의취소(8.3)·문서 인제스트(8.4)의 실제 쓰기 동작은 본 스토리 범위가 아니다 — `/reservations`·`/ingest`는 데이터테이블·폼 골격 + "준비 중" 빈 상태로 렌더한다.*

> **이 스토리의 범위 결정(명시 — 매우 중요, 1.8과 동일 원칙):**
> - **8.1은 "셸 + 접근 + RBAC 증명"이다:** 관리자 웹 인증 배선(Providers·세션·로그인/로그아웃·역할 게이트) + 3개 운영 라우트(셸) + 백엔드 `admin` 도메인 부트스트랩(읽기전용 계정목록 1개 엔드포인트) + 시드 스크립트. 
> - **실 운영 쓰기 기능은 후속 스토리:** 계정 **비활성+캐스케이드 = 8.2**, 예약 **임의취소 = 8.3**, 문서 **인제스트 트리거 = 8.4**. 본 스토리는 이들 화면을 **빈 골격(데이터테이블/폼 + "준비 중")**으로만 둔다.
> - **스키마 마이그레이션 0:** `users.role`(admin 허용 CHECK `ck_users_role`)·`users.is_active`는 이미 라이브에 적용됨(191d9c7dab2d·b2c4f1a9d3e7). 본 스토리는 **새 테이블/컬럼/제약을 만들지 않는다.** 라이브 반영 = `python scripts/seed_admin.py` **1회 실행**(데이터 insert, 마이그레이션 아님).
> - **production CORS·도메인·HTTPS 쿠키(Secure)·레이트리밋은 배포 스토리.** 로컬은 `localhost:3001`이 이미 CORS allowlist에 있음(main.py). 본 스토리는 로컬 dev 기준.
> - **웹(사용자) 로그인 UI는 여전히 별도 스토리(1.8/3.7 defer).** 본 스토리의 로그인 폼은 **admin 앱 전용**이다(`apps/web`·`apps/mobile` 무변경 — 관리자는 모바일 표면 없음, web-mobile parity 비적용).

### 정확한 인증/인가 규약 (AC1·AC2·AC3 — 그대로 적용, 1.8 재사용)

- **관리자 로그인 = 기존 `POST /api/v1/auth/login`(1.8).** 관리자 전용 로그인 엔드포인트를 새로 만들지 않는다. 로그인은 role을 검사하지 않고(어떤 role이든 동일 엔드포인트로 인증), `is_active` 거부·enumeration 단일 401만 한다. 시드 admin은 발급된 access 토큰에 `"role":"admin"` 클레임을 갖는다.
- **쿠키(1.8 확정):** `desknow_access`(httpOnly·Secure·SameSite=lax·path `/`·15분), `desknow_refresh`(path `/api/v1/auth`·14일). 웹은 쿠키로 인증 유지 → admin SDK 호출에 `credentials:"include"` **필수**(cross-origin 3001→8000).
- **RBAC(1.8 `core/security` 재사용):** `require_role("admin")` → 비-admin 403 `FORBIDDEN_ROLE`, 무토큰 401 `UNAUTHENTICATED`. `get_current_principal`이 헤더 Bearer 우선·쿠키(`desknow_access`) 폴백으로 토큰 추출(DB 미접근, 클레임 신뢰). **새 RBAC 코드 작성 금지 — 기존 의존성 소비만.**
- **admin 생성 차단(AC3):** `RegisterRequest.role: Literal["booker","provider"]`이 가입으로 admin을 422 거부(1.8, `test_register_admin_role_returns_422` 존재). 시드 스크립트가 `User(role="admin")` 직접 구성(DB `ck_users_role`이 'admin' 허용)이 유일 경로.

### 익명화 예외 — 관리자는 실 이메일을 본다 (중요)

- 메모 `anonymous-booker-label-no-display-name`: provider/타인-facing 표면은 sha256 파생 익명 라벨, 이메일·raw UUID 비노출. **그러나 admin은 운영자**(타인-facing 아님)이고 계정 관리(8.2 비활성 대상 식별)를 위해 **실 이메일·실 역할·실 활성여부**가 필요하다. `AdminAccountItem`은 실값을 노출한다(`password_hash`만 절대 금지). 이 예외를 명확히 인지하고 익명화하지 말 것.

### deferred 교훈 선반영 (반복 함정 프리플라이트 — 메모 dev-workflow-policy)

- **정렬 결정성:** `list_booker_reservations`(4.8)·`list_reservations_for_rooms`(6.1)가 `created_at` 단일 정렬키로 동률 시 비결정 순서 → deferred. **본 계정목록은 처음부터 `.order_by(created_at.desc(), id)`** 보조키 포함(재-defer 금지).
