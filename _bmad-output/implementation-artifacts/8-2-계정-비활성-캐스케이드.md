## Story

As a 시드 관리자,
I want 관리자 웹에서 예약자·제공자 계정을 비활성하고(제공자는 룸 노출 중단·신규 예약 차단까지 캐스케이드),
so that 운영상 필요한 계정을 차단하되 기존 확정 예약은 보존한다 (FR-31).

## Acceptance Criteria

1. **(AC1 — 계정 비활성 → 로그인·서비스 이용 차단)** 시드 관리자가 활성 계정(booker 또는 provider)을 비활성하면(`POST /api/v1/admin/accounts/{account_id}/deactivate`, admin 가드), 해당 계정의 `User.is_active`가 `false`가 되고 **로그인이 차단**된다(기존 1.8 — 로그인은 `is_active=false`를 단일 401로 거부, 불변). **서비스 이용 차단은 최종 강제(≤15분)**: 발급된 access 토큰(15분)은 만료까지 유효하나, refresh 갱신이 거부되어(기존 `auth/service.py:131` — 비활성 refresh 거부, 불변) 만료 후 세션 연장 불가 → 결국 차단된다. *인증 핫패스(`get_current_principal`)는 stateless 유지(매 요청 DB 미조회) — 변경하지 않는다.*

2. **(AC2 — 제공자 비활성 캐스케이드: 룸 노출 중단·신규 예약 차단·기존 예약 유지)** **provider** 계정 비활성 시, **동일 트랜잭션**에서 그의 모든 룸(`Room.provider_id == account_id`)의 `is_active`가 `false`가 된다. 이로써:
   - **지도/목록/반경/행정동 노출 중단**: 기존 노출 쿼리(`aggregate_availability` 3.1·`list_active_rooms` 3.2·`list_regions` 3.4·`search_rooms` 3.5)가 모두 `where(Room.is_active)`를 적용하므로 비활성 룸은 자동으로 제외된다(신규 쿼리 수정 없음).
   - **신규 예약 차단**: 예약 생성 경로(`router.py:120` → `get_room_slots` → `_get_active_room_or_404`)가 `Room.is_active`를 보고 404(`ROOM_NOT_FOUND`)로 차단한다(자동).
   - **기존 확정 예약 유지(★)**: `reservations`/슬롯은 **건드리지 않는다**. 예약자의 기존 예약 조회(4.8)·취소(4.7)·provider 예약 조회(6.1)는 `room.is_active` 가드를 경유하지 않으므로(검증 완료) 룸 비활성화 후에도 정상 동작한다. (예약 임의취소는 FR-32/8.3 별도.)

3. **(AC3 — 비활성 룸 즐겨찾기 '비활성' 라벨)** 비활성된 룸을 즐겨찾기한 예약자에게는 즐겨찾기 목록에서 '비활성' 라벨이 표시된다(FR-10 연동). 이는 캐스케이드로 `room.is_active=false`가 되면 **기존 즐겨찾기 표면이 이미 처리**한다: `FavoriteRoomItem.is_active`(`favorites/schemas.py:46`)·`build_favorite_item`(`favorites/service.py:107`)·웹 `FavoriteRow` 비활성 배지/상세 진입 차단(`FavoriteRow.tsx:36-42,54,82-95`)이 모두 존재. **즐겨찾기 코드 수정은 없으며**, 본 스토리는 캐스케이드가 이 경로를 올바르게 트리거함을 **회귀 테스트로 실증**한다.

4. **(AC4 — 멱등·정책 가드)** 비활성 엔드포인트는:
   - **멱등**: 이미 비활성인 계정에 재호출하면 추가 쓰기 없이 현재 상태를 200으로 반환한다(조건부 원자 UPDATE, rowcount 0 = no-op).
   - **admin 대상 금지**: 대상 계정이 admin(자기 자신 포함)이면 비활성하지 않고 **404 `ACCOUNT_NOT_FOUND`**로 응답한다(관리 대상 집합=booker·provider만 — `list_accounts`의 `_EXCLUDED_ROLE` 미러, admin 존재 누설 방지).
   - **미존재 404**: 존재하지 않는 `account_id` → 404 `ACCOUNT_NOT_FOUND`.
   - **RBAC 최종(8.1 미러)**: 비-admin 403 `FORBIDDEN_ROLE`, 무토큰 401 `UNAUTHENTICATED`.

> **이 스토리의 범위 결정(명시 — 매우 중요):**
> - **비활성 단방향(KTH 2026-06-18 선택):** 재활성(reactivate)·토글 UI/엔드포인트는 만들지 않는다(PRD:516 "계정 비활성/재활성 정책 고도화"=범위밖, AC도 비활성 방향만 기술). 되돌리기는 후속 스토리/수동 DB. → AccountsTable은 **비활성 버튼만**(활성 계정에만 노출), 재활성 버튼 없음.
> - **최종 강제(≤15분, KTH 2026-06-18 선택):** 인증 핫패스를 바꾸지 않는다(stateless JWT 유지). 즉시 취소(per-request DB 조회/토큰 거부목록)는 본 스토리 범위밖.
> - **캐스케이드 = `is_active` 플래그 전파, 하드 삭제 아님:** `users`/`rooms` 행을 삭제하지 않는다(FK `ondelete` 미지정·RESTRICT 전제 — `rooms/models.py:91-99`). 룸/예약 보존.
> - **reservations·slots 무변경:** 기존 확정 예약은 유지(AC2). 예약 취소·슬롯 재활성은 **8.3**(FR-32).
> - **스키마 마이그레이션 0건:** `users.is_active`·`rooms.is_active` 기존. 새 테이블/컬럼/제약 없음. 신규 코드는 ErrorCode enum 1개 추가뿐(DB 무관).
> - **즐겨찾기/노출 쿼리/예약 도메인 코드 무변경:** 캐스케이드가 기존 `Room.is_active` 경로를 트리거할 뿐. 본 스토리는 admin 도메인 쓰기 + 회귀 검증.
> - **production CORS·도메인·Secure 쿠키·레이트리밋은 배포 스토리.** 본 스토리는 로컬 dev 기준(admin 포트 3001 기존 allowlist).

### 캐스케이드 설계 — 플래그 전파 (왜 노출/예약/즐겨찾기 코드를 안 건드리나)

- AC2가 `(is_active=false)`를 명시 → 캐스케이드는 **provider 룸의 `is_active`를 false로 전파**하는 설계다(조인 추가 아님).
- 기존 코드가 이미 `Room.is_active`를 단일 기준으로 삼는다:
  - **노출(공개 탐색):** `aggregate_availability`(`rooms/service.py:189`)·`list_active_rooms`(`:266`)·`list_regions`(`:554`)·`search_rooms`(`:642`) 모두 `where(col(Room.is_active).is_(True))`.
  - **단일 룸 가드(요약·슬롯·예약 생성):** `_get_active_room_or_404`(`rooms/service.py:283-288`)가 `not room.is_active`면 404. 예약 생성은 라우터(`reservations/router.py:120`)가 `get_room_slots`를 거쳐 이 가드를 탄다.
  - **즐겨찾기 라벨:** `build_favorite_item`(`favorites/service.py:99-107`)이 `room.is_active`로 `is_active`/`remaining_slots=0` 합성, 웹 `FavoriteRow`가 비활성 배지+상세 진입 차단.
- 따라서 룸 `is_active=false` **데이터 변경만으로** AC2/AC3가 충족된다. 대안(노출 쿼리 6곳에 provider `User.is_active` 조인 추가)은 더 침습적·회귀 위험·AC의 `(is_active=false)`와 불일치 → 채택 안 함.

### 무회귀 검증 결과 (정독 — 룸 비활성화가 기존 예약을 깨지 않음)

- **예약자 기존 예약 조회(4.8):** `list_booker_reservations`(`reservations/service.py:397-402`)는 `booker_id`로만 필터, 룸 JOIN/`is_active` 가드 없음. 라우터(`:252-263`)가 룸 메타를 PK로 합성하되 활성 가드로 행 제외 안 함(`is_active=room.is_active` 전달). → 비활성 룸 예약도 조회됨. **무회귀.**
- **예약 취소(4.7):** 취소 라우터(`:188-194`)는 소유권만 검사, `room.is_active` 미검사. service도 룸 미터치. → 비활성 룸의 기존 예약 취소 정상. **무회귀.**
- **provider 예약 조회(6.1):** `list_provider_rooms`(`rooms/service.py:814`)는 `provider_id`만 필터(상태 무관·is_active 미적용 — 소유자 뷰). → 비활성 룸 예약도 provider에게 노출. **무회귀.**
- **로그인/refresh:** 로그인은 `is_active=false` 거부(1.8). refresh는 `rotate_token_pair`(`auth/service.py:130-132`)가 DB에서 `is_active` 재확인 후 거부. → 비활성 계정 세션 연장 불가. **AC1 최종 강제 충족(신규 코드 0).**

### 정확한 멱등·원자성 규약 (AC4 — 취소/거절 4.7/6.2 패턴 미러)

- **조건부 원자 UPDATE로 race 결정화:** `UPDATE users SET is_active=false WHERE id=:id AND is_active=true` — `_transition_to_terminal`(`reservations/service.py:198-205`)의 단일 승자 중재 정신. `rowcount==0`=이미 비활성(멱등 패자) → no-op 200(예외 아님).
- **캐스케이드는 같은 트랜잭션:** 계정 UPDATE + 룸 UPDATE를 한 트랜잭션에서 묶고 끝에 `commit` 1회(부분 적용 방지 — `_release_slots`의 "commit은 호출처가" 정신).
- **멱등 룸 캐스케이드:** `WHERE provider_id=:id AND is_active=true` — 이미 비활성 룸은 재변경 안 함(provider가 2.3에서 자체 비활성한 룸 포함, 중복 쓰기 회피).
