## Story

As a 시드 관리자,
I want 관리자 웹에서 확정 예약 목록을 보고 예외 상황의 예약을 임의 취소하길(점유 슬롯 재활성 + 예약자 통지),
so that 운영 문제를 해소하고 슬롯을 다시 연다 (FR-32, NFR-7).

## Acceptance Criteria

1. **(AC1 — 확정 예약 임의 취소 → 점유 슬롯 동일 트랜잭션 재활성)** 시드 관리자가 확정(`confirmed`) 예약을 임의 취소하면(`POST /api/v1/admin/reservations/{reservation_id}/cancel`, admin 가드), **동일 트랜잭션 내에서** ① 예약 `status`가 `cancelled`로 전이되고 ② 그 예약이 점유한 `reservation_slots` 행이 **DELETE되어 슬롯이 재활성**(다시 빔)된다. 슬롯 재활성은 기존 `_release_slots`(DELETE) 경로를 쓴다 — bare status UPDATE로 슬롯을 남기면 가용성 reader의 DELETE-on-cancel 불변식이 깨져 유령 점유가 누출된다(deferred L78). 취소 후 가용성 집계(3.1·4.3·4.9)에서 그 슬롯이 즉시 다시 예약 가능해진다. **시간 윈도우 없음(admin 권한):** booker 6h(4.7)·provider 시작전(6.2) 게이트와 달리, 관리자는 시작된/지난 예약도 임의 취소할 수 있다(예외 운영 도구 — 게이팅 래퍼 우회·프리미티브 직접 호출).

2. **(AC2 — 예약자 status_change 통지·전이와 원자적)** 임의 취소가 실제로 `confirmed→cancelled` 전이를 수행하면, 예약자(`booker_id`)에게 **`status_change`/`reason="cancelled"` 통지 1건**이 생성된다(FR-18a 배선 — 5.3 배너가 표시, 생성은 8.3 책임). **★통지는 전이·슬롯 재활성과 동일 트랜잭션에 편입(원자):** 통지 INSERT가 status flip + 슬롯 DELETE와 한 번의 commit으로 묶인다. 따라서 통지 생성 실패 시 전체가 롤백되어 예약은 `confirmed`로 남고(부분 영속 없음), 재시도가 전부를 다시 수행한다 → **통지 영구 손실 불가**(deferred L42 회수). 예약자 web 배너 표시는 5.3이 이미 구축(`NotificationBanner`가 `reason="cancelled"` 분기 보유) — web 신규 0. 통지는 멱등(`uq_notifications_user_reservation_type`)이라 같은 예약·종류 통지는 1건이다.

3. **(AC3 — 종료 상태 멱등: 추가 효과 0)** 이미 종료 상태(`cancelled`/`rejected`)인 예약에 임의 취소를 시도하면, **멱등하게 무시되고 현재 상태가 200으로 반환**된다(공통 상태머신 — `_transition_to_terminal` fast-path). 멱등 경로엔 **status 변경·슬롯 DELETE·통지 생성이 전부 0**이다(전이를 수행한 winner만 통지 — 종료 상태 재호출은 winner 아님). 미존재 `reservation_id`는 404 `RESERVATION_NOT_FOUND`(누설 방지 — 4.7/6.2 미러).

4. **(AC4 — 확정 예약 목록 조회 + 행별 취소 UI)** 관리자 웹 `/reservations` 화면(8.1이 만든 `ComingSoon` 골격을 교체)이 **확정 예약 목록**(`GET /api/v1/admin/reservations` — `status='confirmed'`만·페이지네이션·안정 정렬)을 표시하고, 각 행에 **취소 버튼**(파괴적 작업 확인 단계 포함)을 둔다. 취소 성공 시 목록이 갱신되어(invalidate) 취소된 예약이 confirmed 목록에서 사라진다. 목록은 무제한 결과셋을 막고(LIMIT≤100·offset 페이지네이션) `created_at desc, id` 보조키로 안정 정렬한다(8.2 선반영 — 4.8/6.1 비결정 정렬·무페이지네이션 함정 회피).

5. **(AC5 — RBAC 최종 강제·8.1/8.2 미러)** 두 엔드포인트 모두 `_require_admin` 가드: 비-admin(booker/provider) 403 `FORBIDDEN_ROLE`, 무토큰 401 `UNAUTHENTICATED`. 관리자 화면은 에러코드·숫자 상태를 노출하지 않고 고정 한국어 카피만 보인다(UX-DR10 — FE 분기는 `detail.code`). 네트워크 단절은 "네트워크 연결이 끊겼어요" 카피([[terminology-network-disconnect-not-offline]]).

> **이 스토리의 범위 결정(명시 — 매우 중요):**
> - **통지 원자화로 deferred L42 의무 회수(KTH 2026-06-18 선택):** 8.3은 6.2와 동일한 status_change 통지 배선을 추가하므로, 알려진 "비원자 통지 재시도 영구손실"(deferred L42, retro·sprint-status L105가 8.3을 회수 스토리로 지정)을 **동일 트랜잭션 편입**으로 고친다. **6.2 reject 라우터도 함께 리팩터**(통지블록을 router→service로 이동·원자화) — 공유 프리미티브 변경이라 6.2가 함께 retry-safe가 된다(반복함정 회피: 같은 코드를 쓰며 알려진 함정을 재도입하지 않음).
> - **예약 식별 = 확정 목록 + 행별 취소(KTH 2026-06-18 선택):** UUID 입력식(최소안)이 아니라 `GET /admin/reservations`(confirmed·페이지네이션) 목록 + 행별 취소를 만든다(accounts 8.2 패턴 미러·운영 사용성).
> - **force-cancel은 시간 게이트 없음:** booker 6h(`cancel_reservation_for_booker`)·provider 시작전(`reject_reservation_for_provider`) **게이팅 래퍼를 호출하지 않는다** — 프리미티브(`_transition_to_terminal`)를 직접 호출해 admin이 시작된/지난 예약도 취소 가능(예외 운영 도구).
> - **슬롯 재활성 = `_release_slots`(DELETE) 경로 유지:** bare status UPDATE 금지(deferred L78 — DELETE-on-cancel 불변식 보존). 재구현 금지 — `_transition_to_terminal`이 이미 status flip + 슬롯 DELETE를 한 트랜잭션에 수행.
> - **스키마 마이그레이션 0건:** Notification·Reservation·ReservationSlot 전부 기존. 새 테이블/컬럼/제약 없음 → 라이브 `alembic upgrade head` 신규 실행분 없음([[dev-workflow-policy-live-db-migration]]).
> - **웹/모바일 작업 0:** 예약자 web `NotificationBanner`가 `reason="cancelled"`를 **이미 처리**(5.3) → 통지 데이터만 공급. 모바일은 챗봇/예약 화면 placeholder + 배너 미구현이라 "모바일 dev-build 푸시" 버킷(5.3/6.2 동일) — admin은 모바일 표면 없음([[web-mobile-parity-on-changes]] 비적용).
> - **비활성 캐스케이드 미통지(deferred L8)는 별도 스코프:** 8.2 provider 비활성으로 룸이 닫힌 booker에게 자동 통지/자동취소는 8.3이 **하지 않는다**(8.3=특정 예약 수동 강제취소). 운영자가 필요 시 해당 예약을 8.3으로 개별 취소(→ 통지)한다. 비활성→자동 강제취소+통지는 별도 product 결정으로 deferred 유지(아래 Dev Notes).
> - **즉시 토큰 무효화·재활성·production CORS/쿠키/레이트리밋은 배포/후속 스토리.**

### ★통지 원자화 — deferred L42 회수의 정확한 메커니즘 (반복함정 방지)

- **기존 6.2 버그(deferred-work L42):** reject 라우터가 `reject_reservation_for_provider`로 `confirmed→rejected`를 **commit한 *뒤*** `create_notification`(자체 commit)을 **별도 트랜잭션**으로 호출. 통지 INSERT가 raise하면(희박 동시삽입 `existing is None`→re-raise 500, 또는 무관 DB 오류) **거절·슬롯 재활성은 이미 영속인데 응답 500** → 재시도 시 예약이 이미 `rejected`라 `was_confirmed` 거짓 → 통지 분기 skip → **예약자 status_change 통지 영구 미발송**(자가치유 아님).
- **왜 rowcount 반환만으론 불충분:** "`_transition_to_terminal` rowcount를 라우터로 반환"(deferred L42 후보안)은 **race 정확성**(전이 수행자만 통지)은 고치나 **재시도 손실은 못 고친다** — 재시도 시 이미 종료라 rowcount=0(=전이 안 함)이라 여전히 통지 skip. **진짜 해법은 원자성**: 통지 INSERT를 전이 commit과 **한 트랜잭션**에 묶어야 "통지 실패 ⇒ 전이도 롤백 ⇒ 재시도가 전부 재수행"이 성립.
- **본 스토리 해법(원자):** `stage_status_change_notification`(add-only) → `_transition_to_terminal`이 winner일 때 `_release_slots` 직후·commit 전에 호출 → status flip + 슬롯 DELETE + 통지 INSERT가 **단일 commit**. 멱등 no-op(이미 종료)은 winner 아니라 통지 staged 안 함(AC3 효과 0). 6.2·8.3 둘 다 이 경로를 타 retry-safe.
- **반복함정 경고:** 통지를 다시 별도 `create_notification`(자체 commit)으로 두면 deferred L42를 **재도입**한다([[dev-workflow-policy-deferred-and-repeat-mistakes]] — 같은 코드를 쓰며 알려진 함정 재발 금지). 반드시 add-only + 호출처 commit.

### 재사용 자산 — 재구현 절대 금지 (슬롯/멱등/동시성은 이미 해결됨)

- **`_transition_to_terminal`(`reservations/service.py:167-219`):** 조건부 원자 UPDATE(`WHERE id AND status='confirmed'`)로 cancel↔reject race를 단일 winner로 결정화(epic-6 회고 회수) + winner면 `_release_slots`(슬롯 DELETE) + commit + refresh. fast-path(종료 상태)=멱등 no-op. **8.3 force-cancel은 이 프리미티브에 `notify_reason="cancelled"`만 더해 호출** — 슬롯 재활성·멱등·동시성을 공짜로 얻는다.
- **`_release_slots`(`:153-164`):** `DELETE reservation_slots WHERE reservation_id` — commit은 호출처(전이)가. **DELETE 경로 유지가 deferred L78 안전의 핵심**(가용성 reader가 status 조인 없이 "DELETE-on-cancel 불변식"에 의존 — bare status UPDATE로 슬롯 남기면 유령 점유 누출).
- **`ReservationStatus`/`_TERMINAL_STATUSES`(`models.py:58-79`):** `confirmed`/`cancelled`/`rejected` + 종료셋. force-cancel 타깃=`CANCELLED`.
- **`create_notification`(`notifications/service.py:32-73`):** reminder(5.2) 경로는 **그대로**(시그니처 불변). 8.3은 reminder가 아닌 원자 편입이라 add-only 신규 프리미티브를 쓴다(혼동 금지).
- **admin 도메인(8.1/8.2):** `_require_admin` 싱글톤·`AdminAccountListResponse` 페이지네이션 포맷·`deactivate_account` 조건부 원자 UPDATE + 안정정렬·`list_accounts` offset/limit — **모두 미러 대상**. admin.service는 `User`(auth.models)·`Room`(rooms.models) 이미 import → reservations 추가는 단방향.
- **admin 웹(8.1/8.2):** `AccountsTable.tsx`(테이블·2단계 파괴 확인·행별 작업)·`useDeactivateAccount.ts`(SDK 뮤테이션+invalidate)·`useAdminAccounts.ts`(목록 쿼리)·`AdminGate`·`ComingSoon` — reservations 피처가 그대로 미러.
- **예약자 web 배너(5.3):** `NotificationBanner.tsx:71-76`이 `reason==="cancelled"` 분기·카피 **기보유** → 8.3 통지 데이터만 공급(web 신규 0).
