# Story 9.1: 모바일 인증 + 탐색·검색

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 예약자,
I want 모바일 앱에서 로그인/가입(역할 선택)하고 지도·목록으로 스터디룸을 탐색·검색·즐겨찾기 하길,
so that 웹과 동일한 첫 진입 경험을 모바일에서도 누린다 (FR-18, FR-1·2 / Epic 1·3 표면).

> **이 스토리는 Epic 9(모바일 패리티)의 첫 스토리이자 파운데이션이다.** 현 `apps/mobile`은 셸/슬롯 골격(탭 네비·온보딩 오버레이·챗봇 FAB no-op·인앱배너 빈 슬롯·SDK baseUrl 배선)만 있고 **실제 기능 화면이 0개**다(`apps/mobile/src/app/index.tsx·favorites.tsx·reservations.tsx` 전부 placeholder 텍스트). 9.1은 9.2/9.3가 재사용할 **3대 파운데이션** — ① secure-store+Bearer 인증 배선(세션 프로바이더·토큰 저장·SDK `client` 헤더 주입·앱 재시작 세션 복원) ② WebView 카카오맵 지도 컴포넌트(지도 캔버스만 WebView·핀↔바텀시트 postMessage 브릿지) ③ E2E 세션주입 하니스(`EXPO_PUBLIC_E2E_*`+`__DEV__` 게이팅, 프로덕션 빌드 제외) — 을 세우고, 그 위에 **로그인/가입·홈 지도탐색·지역/반경 검색·즐겨찾기**를 웹과 기능 동등(parity)으로 구현한다. 백엔드·SDK·디자인 토큰은 **이미 모바일을 지원**하므로(Bearer 헤더 추출 완료·SDK `workspace:*` 의존·토큰 본문 발급) 백엔드 변경은 없다.

## 범위 결정 (KTH 확정 2026-06-19 — create-story)

이 스토리가 닿는 네 경계(provider 룸등록=9.3 · 룸상세·예약=9.2 · 챗봇=9.3 · E2E 전화면 통합=9.3)를 다음과 같이 확정한다. 침범 금지.

1. **인증 = secure-store Bearer + 세션 스택 전체.** 웹은 httpOnly 쿠키(`credentials:"include"`)지만 모바일은 **응답 본문 토큰을 `expo-secure-store`에 저장 + `Authorization: Bearer` 헤더 주입**이다(아키텍처 L165 계획·1.8 AC1 명시·백엔드 `_extract_access_token` 헤더 우선 이미 지원). 9.1이 토큰 저장소·SDK 인터셉터·`QueryClientProvider`·세션 훅(`useSession`/`useAuth` RN 포팅)·부팅 시 세션 복원·401→refresh 1회 재시도를 **전부 신설**한다. 백엔드/SDK는 무변경.

2. **provider 가입 = 역할 선택 UI + booker 완결 / provider는 룸폼(9.3)까지 지연.** 웹과 동일하게 가입 화면에 **역할 토글(예약자/제공자)**을 둔다. **booker**는 9.1이 원자 가입(register→자동 login→토큰 저장)을 완결한다. **provider**는 웹처럼 register를 **룸 등록 시점까지 지연**(클라 1차 검증 통과 → `pendingSignup` 메모리 보관 → 룸등록 라우트로 이동)하는데, **룸 등록 폼(`RoomForm`)·지오코딩은 Story 9.3 소유**다. 따라서 9.1은 provider 선택 시 `pendingSignup`을 보관하고 **9.3이 채울 룸등록 라우트(`/provider/room` 스텁)**로 네비게이트하는 **seam까지만** 만든다(4.2가 예약 패널을 placeholder로 둔 패턴과 동형). 9.1에서 provider 가입을 끝까지 완료시키지 않는다(룸 없는 떠도는 계정 방지 = 웹과 동일 원칙).

3. **지도 = 캔버스만 WebView 카카오맵(앱 전체 WebView 아님).** 핀 렌더 캔버스만 `react-native-webview`에 웹 카카오맵 JS를 띄우고, **데이터는 RN이 SDK로 조회해 WebView에 주입**(postMessage), **핀 탭은 WebView→RN 브릿지로 네이티브 바텀시트**를 연다. `react-native-maps`(한국 구글지도 규제로 패리티 깨짐)·비공식 카카오 RN 래퍼는 **둘 다 기각**(아키텍처 결정 ②). 착수 즉시 **브릿지 왕복 1건(핀 탭→RN 수신)을 먼저 검증**한다(KTH 지시).

4. **바텀시트 = 요약 + "상세 보기" 스텁 링크.** 핀/목록행 탭 시 네이티브 바텀시트로 룸 요약을 보여주되, **"상세 보기"는 9.2 소유 룸상세 라우트(`/rooms/[room_id]` 스텁)로 네비게이트**한다. 룸 상세 화면 본체·날짜·슬롯·예약은 **Story 9.2**가 이 자리를 채운다. 9.1은 탐색→요약까지만.

5. **즐겨찾기·온보딩 = 9.1 완결.** 즐겨찾기 추가/조회/제거 + 미로그인 안내는 9.1이 완결한다. 온보딩 오버레이는 **3.9에서 이미 RN 구현 완료**(`apps/mobile/src/components/OnboardingOverlay.tsx`·`useOnboarding` AsyncStorage)이므로 **재사용**하고 홈 화면 마운트만 확인한다(AC: "다시 보지 않기"만 영속).

6. **9.1이 만드는 화면 = index(홈 탐색)·favorites·login·signup.** `reservations` 탭은 **9.2 placeholder 그대로** 둔다. provider 화면·챗봇·E2E 전화면 통합테스트는 9.3.

## Acceptance Criteria

**AC1 — secure-store Bearer 인증이 배선되고 세션이 재시작에도 복원된다 (FR-18, FR-3 / 아키텍처 L165 · 1.8 AC1)**
**Given** 모바일 앱
**When** 이메일·비밀번호로 로그인(또는 booker 가입)하면
**Then** 응답 본문 `TokenResponse`의 `access_token`·`refresh_token`이 `expo-secure-store`에 저장되고, 이후 모든 SDK 호출에 `Authorization: Bearer <access>` 헤더가 주입되며(인증 필요 엔드포인트 200), **앱을 재시작해도** 저장 토큰을 읽어 헤더 주입 → `authMe`로 세션이 복원된다. 로그아웃 시 secure-store가 비워지고 `authLogout`에 refresh 토큰을 **본문**으로 전송한다(쿠키 없음).

**AC2 — 로그인/가입 화면이 웹과 동일 흐름으로 동작한다 (FR-1·2 / 1.7 가입 · 1.8 로그인)**
**Given** 로그인/가입 화면
**When** 입력·제출하면
**Then** 웹 `LoginView`/`SignupView`와 동일하게 — 로그인 401="이메일 또는 비밀번호가 올바르지 않아요.", 가입 409="이미 가입된 이메일이에요.", 422=서버 message(폴백=비밀번호 정책 안내), 네트워크 단절="네트워크 연결이 끊겼어요. 연결되면 다시 시도해주세요." — 인라인 안내로 분기되고(막다른 화면 금지), 가입 화면 상단에 역할 토글(예약자/제공자)이 있다. **booker** 가입은 register→자동 login→토큰 저장→홈으로 완결되고, **provider** 선택은 클라 1차 검증 통과 후 `pendingSignup` 보관 + 룸등록 라우트(9.3 스텁)로 이동한다(register 미호출 — §범위 2).

**AC3 — 홈 지도/목록 전환이 웹과 동일한 핀·가용성 색으로 표시된다 (FR-1·8·9 / Epic 3 표면)**
**Given** 첫 진입 화면(`index` 탭)
**When** 지도/목록을 전환하면
**Then** WebView 카카오맵이 웹과 동일한 핀(예약가능 `pinAvailable #157F45`+✓ / 마감 `pinFull #7E7466`+✕, 색 단독 금지)으로 표시되고, **핀 데이터는 RN이 `roomsListRooms`+`roomsAggregateAvailability`를 SDK로 조회해 `room_id`로 조인 후 WebView에 주입**하며(WebView는 캔버스만·직접 fetch 안 함), 핀 탭 시 postMessage 브릿지로 **네이티브 바텀시트 요약**이 올라온다(앱 전체 WebView 아님). 목록 모드는 `FlatList`로 무한스크롤(`onEndReached` → `next_cursor`)된다.

**AC4 — 지역(2단)·반경 검색과 즐겨찾기가 웹과 동일 구성으로 동작한다 (FR-8·9·10 / 3.4·3.5·3.7)**
**Given** 검색·즐겨찾기
**When** 지역(시군구→지역 2단)·반경(프리셋 1/2/3/5/10km) 검색하거나 즐겨찾기를 토글하면
**Then** `roomsListRegions`·`roomsSearchRooms`(region_code 또는 lat/lng/radius_km)·`favoritesAddFavorite`/`favoritesListFavorites`/`favoritesRemoveFavorite`가 호출되어 웹과 동일 결과가 표시되고, **미로그인 시 즐겨찾기 토글은 "로그인하면 저장돼요." 안내**(옵티미스틱 호출 안 함)로 graceful 처리된다. 용어는 **"지역"으로 통일**(UI/주석에 "행정동/법정동" 금지).

**AC5 — 위치 권한·막다른 화면 5상태가 일관 처리된다 (FR-9 / 3.6·3.8)**
**Given** 위치 권한과 데이터 상태
**When** 권한 확인·조회하면
**Then** 마운트 즉시 깜짝 프롬프트 없이 권한을 **먼저 확인**(`expo-location` `getForegroundPermissionsAsync`) → granted면 좌표 확보까지 지도 생성 보류 후 **내 위치로 생성**, prompt/denied면 **즉시 서울 기본값**(`37.5665,126.978`)으로 생성(서울 렌더→좌표 점프 금지)하고, **로딩/에러/빈/네트워크단절/미로그인 5상태**가 친절한 인라인 카피로 일관 처리된다(네트워크 단절="네트워크 연결이 끊겼어요…"·"오프라인" 단어 금지).

**AC6 — 온보딩 오버레이가 첫 방문 시 표시되고 "다시 보지 않기"만 영속된다 (FR / 3.9 재사용)**
**Given** 온보딩 오버레이(기존 3.9 구현 `OnboardingOverlay`+`useOnboarding`)
**When** 첫 방문이면
**Then** 홈에서 표시되고, "다시 보지 않기"만 `AsyncStorage`(`desknow:onboarding:seen`)로 영속(이번만 닫기는 비영속).

**AC7 — E2E 세션주입 하니스가 환경변수 게이팅으로 동작하고 프로덕션 빌드엔 미포함된다 (NFR-4 / 신규 아키텍처 결정 ③)**
**Given** Expo Web 빌드 + Playwright(MCP)
**When** `__DEV__` && `EXPO_PUBLIC_E2E_ENABLED==="1"` 게이팅이 켜진 상태에서 세션 주입 훅으로 토큰을 주입하면
**Then** 로그인 UI를 거치지 않고 인증 화면을 구동할 수 있고, **프로덕션 빌드(`!__DEV__`)에는 주입 코드가 정적 제거(dead-code elimination)되어 번들에 포함되지 않는다**(번들 grep으로 주입 심볼 부재 확인). 9.2/9.3가 이 하니스를 재사용한다.

## Dev Notes

### 이 스토리의 본질

9.1은 **세 파운데이션을 세우는 스토리**다 — 이후 9.2(예약)·9.3(provider·챗봇·통합검증)이 전부 ① 인증 스택 ② WebView 지도 ③ E2E 하니스 위에 얹힌다. 그래서 "탐색·검색 화면 1개"가 아니라 **모바일 앱을 처음으로 '인증된·데이터 있는' 앱으로 만드는** 작업이다. 가장 큰 리스크는 **WebView 카카오맵 origin/브릿지**와 **secure-store Bearer 배선**이므로 둘을 착수 초기에 1건씩 먼저 검증한다.

핵심 통찰: **백엔드·SDK·디자인 토큰은 이미 모바일을 지원한다.** 백엔드는 `Authorization: Bearer` 헤더를 쿠키보다 먼저 추출하고(`apps/api/app/core/security.py:167`), 로그인/refresh가 토큰을 **응답 본문**으로도 내려준다(`router.py:102-105`). SDK(`@desknow/api-client`)는 web·admin·mobile 공용이고 모바일이 이미 `workspace:*`로 의존 중이다. 따라서 **백엔드 변경 0** — 9.1은 순수 **클라이언트(모바일) 작업**이다. 웹 `features/auth`·`features/list`·`features/map`·`features/favorites`의 **로직은 거의 그대로 재사용**하되, 단 하나의 차이 — **인증 저장이 쿠키→secure-store 토큰** — 만 반영한다.

### 신규 아키텍처 결정 (Epic 9 동반, 경량 ADR — sprint-change-proposal §4.3)

> **ADR-9.1-A · 모바일 인증 = expo-secure-store + Bearer 헤더.**
> 웹=httpOnly 쿠키(`credentials:"include"`), 모바일=응답 본문 토큰을 `expo-secure-store`에 저장 + `Authorization: Bearer` 헤더 주입. 백엔드는 이미 헤더 우선 추출(쿠키 폴백)을 지원(`security.py:167`)하므로 **백엔드 무변경**. 주입점=SDK `client.interceptors.request`(매 요청 secure-store 토큰 읽어 헤더 세팅·`client.gen.ts:91-95`에서 매 요청 실행 확인). 401→refresh 1회 재시도도 같은 자리(응답/에러 인터셉터). `configureApiClient`에 `credentials` **추가 금지**(웹 전용). [근거: architecture.md:165 · 1-8 AC1 · packages/api-client/src/index.ts:15-16,24-25]

> **ADR-9.1-B · 지도 = WebView + 카카오맵 JS SDK 재사용(지도 캔버스만).**
> 웹은 카카오맵 JS SDK(`<script>` 주입)라 RN 직접 이식 불가. `react-native-webview`에 웹 카카오맵을 띄워 동일 시각·핀 재현, **데이터는 RN이 SDK로 조회해 WebView에 주입**(WebView는 직접 fetch 안 함 — eslint 가드·단일 데이터 경로 유지), 핀 탭↔바텀시트는 `postMessage` 브릿지. 카카오 JS 키 재사용(WebView origin 화이트리스트 추가 필요). `react-native-maps`(한국 구글지도 규제로 패리티 깨짐)·비공식 카카오 RN 래퍼 **둘 다 기각**. **착수 시 브릿지 왕복 1건 먼저 검증**. [근거: sprint-change-proposal:53,125 · epics.md:243]

> **ADR-9.1-C · E2E 세션주입 = 환경변수+`__DEV__` 게이팅.**
> 프로젝트에 자동화 E2E 계층이 **없음**("E2E"=Playwright MCP 수동 검증). `EXPO_PUBLIC_E2E_*`+`__DEV__` 게이팅으로 Expo Web 빌드에 세션 주입 훅을 노출(프로덕션 빌드 dead-code 제거), Playwright로 구동. 기존 `api-client.ts:16`의 `__DEV__` fail-fast 게이팅 패턴 재사용. 9.2/9.3가 재사용. [근거: sprint-change-proposal:54,104,126,138]

### secure-store Bearer 인증 — 정확한 배선

웹 `useAuth.ts`는 로그인 성공 시 **응답 본문 토큰을 의도적으로 무시**한다(`useAuth.ts:80` 주석 "웹은 쿠키 인증 — 응답 본문 토큰은 무시"). **모바일은 정확히 이 지점에서 갈린다** — `authLogin`이 돌려준 `{ data }`(=`TokenResponse{access_token, refresh_token}`)를 `setTokens(data)`로 secure-store에 저장해야 한다. `authRegister`는 토큰을 안 주므로(`UserPublic` 201) 웹과 동일하게 **register→자동 login 연쇄**(`useAuth.ts:115`)를 유지하고 그 login 응답 토큰을 저장한다.

Bearer 주입은 SDK `client`의 **request 인터셉터**가 정석이다(토큰이 로그인/로그아웃/refresh로 바뀌어도 매 요청 시점 최신값을 읽음): `client.gen.ts:91-95`가 `for (const fn of interceptors.request.fns) request = await fn(request, opts)`로 **매 요청** 실행함을 확인했다. `client.setConfig({ headers })` 정적 방식도 가능하나 토큰 변경 시 재호출이 필요해 인터셉터가 우월. **주의:** OpenAPI 스펙에 `securitySchemes`가 비어 SDK per-endpoint `security`/`auth` 콜백은 **발화하지 않으므로**(생성 `sdk.gen.ts` `security:` 선언 0건) 자동 Bearer 주입에 의존하지 말 것 — 인터셉터로 직접 주입.

세션 복원: 웹은 httpOnly 쿠키가 자동 복원하지만 **모바일은 명시 부트스트랩** — `_layout.tsx`에서 부팅 시 secure-store 토큰 로드 → 인터셉터가 헤더 주입 → `authMe`로 복원. 로그아웃/refresh는 쿠키가 없으므로 refresh 토큰을 **본문 `{refresh_token}`**으로 전송(`LogoutRequest`/`RefreshRequest` 타입이 이미 지원·`schemas.py:98-107`).

### WebView 카카오맵 — 데이터 흐름·브릿지·origin 리스크

**데이터 흐름(반드시 준수):** WebView는 **지도 렌더 캔버스만** 담당한다. ① RN이 SDK로 핀 조회(`roomsListRooms`+`roomsAggregateAvailability` → `joinAvailability`) ② RN이 핀 배열을 WebView에 **주입**(`webViewRef.injectJavaScript` 또는 postMessage) ③ WebView가 카카오 마커 렌더(색·아이콘은 `pinVisual`) ④ 핀 탭 → `window.ReactNativeWebView.postMessage(JSON.stringify({type:"pinTap",room_id}))` ⑤ RN `onMessage`가 수신 → 네이티브 바텀시트 오픈. **WebView가 백엔드를 직접 호출하지 않는다**(단일 SDK 데이터 경로·eslint 가드 정신 유지).

**지도 HTML:** 웹 `kakao-map.ts`의 `<script src="//dapi.kakao.com/v2/maps/sdk.js?appkey=KEY&autoload=false">` + `kakao.maps.load(cb)` 명시 초기화 패턴 재현. 키=**`EXPO_PUBLIC_KAKAO_JS_KEY`**(JS 키만·REST 키 절대 금지). 9.1은 저장된 `lat`/`lng`를 쓰므로 지오코딩 불요(provider 지오코딩=9.3).

**⚠️ origin 화이트리스트 리스크(가장 큰 함정):** 카카오 JS SDK는 요청 origin을 콘솔 등록 도메인과 대조한다. WebView를 `source={{ html }}`로 띄우면 origin이 `null`/`about:blank`이라 **카카오가 거부**할 수 있다. 해결: `source={{ html, baseUrl }}`의 **`baseUrl`을 카카오 콘솔에 등록된 화이트리스트 origin**(예: 개발 `http://localhost:3000` 또는 등록 도메인)으로 설정하고, 필요 시 카카오 콘솔 [플랫폼→Web]에 WebView origin을 추가 등록(1-1 인적 작업). **착수 시 브릿지 왕복 1건과 함께 origin 수용 여부를 먼저 검증**한다(KTH 지시). 실패는 AC5 "지도를 못 불러왔어요" graceful degrade로 떨어지게(웹 동형).
