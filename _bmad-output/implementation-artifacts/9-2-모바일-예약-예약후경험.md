# Story 9.2: 모바일 예약 + 예약 후 경험

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 예약자,
I want 모바일 앱에서 룸 상세를 보고 슬롯을 골라 즉시 예약하고, 예약현황 확인·취소·공유·후기까지 하길,
so that 예약 전 과정을 모바일에서 완결한다 (FR-18 / Epic 4·5 표면).

> **이 스토리는 9.1 파운데이션 위에 예약 도메인을 얹는다.** 9.1이 ① secure-store Bearer 인증 스택 ② WebView 카카오맵·`RoomSheet` 요약 ③ E2E 세션주입 하니스를 세웠고, 현재 `apps/mobile`의 **룸 상세(`rooms/[room_id].tsx`)·예약현황(`(tabs)/reservations.tsx`)은 둘 다 placeholder 스텁**이다. 9.2는 이 두 스텁을 웹과 기능 동등(parity)으로 채운다 — **룸 상세 본체(3단 위계·미니지도·예약 패널 전개·후기 표시)·날짜/슬롯/연속선택·즉시 예약(409 차단)·예약현황(다가오는/지난 분할)·취소(6h 리드타임)·후기 작성·인앱 배너·카카오 공유**. 핵심: 웹 `features/reservation`·`features/detail`·`features/notifications`의 **비즈니스 로직은 거의 그대로 재사용**된다 — 날짜·슬롯·연속선택·취소판정·에러판별·분류·공유텍스트가 전부 **프레임워크 무관 순수 함수**(`slots.ts`/`reservations.ts`/`errors.ts`/`share.ts`)이고, React Query 훅도 SDK+키만 쓴다. 포팅이 필요한 건 JSX(웹 `<button>`/Tailwind/카카오 JS SDK 공유)뿐이다. **백엔드·SDK·디자인 토큰은 이미 모바일을 지원**하므로(SDK `@desknow/api-client` 전량 re-export·Bearer 인터셉터 완비) **백엔드/SDK 변경은 0**. 순수 클라이언트(모바일) 작업이다.

## 범위 결정 (KTH 확정 2026-06-19 — create-story)

이 스토리가 닿는 경계(9.1 탐색·9.3 provider·챗봇·E2E 통합)를 다음과 같이 확정한다. 침범 금지.

1. **9.2 = 예약자(booker) 예약 전 과정 완결.** 룸 상세·예약·예약현황·취소·후기 작성·인앱 배너·카카오 공유까지 booker 관점을 9.2가 완결한다. **provider 화면(예약자현황·거절·후기 답글 작성·룸 등록)·챗봇·9.1~9.3 전화면 통합 Playwright는 9.3 소유** — 침범 금지.

2. **룸 상세 미니지도 = WebView 단일 핀(9.1 `mapHtml` 카카오 로드 패턴 재사용).** 웹 `RoomLocationMap`(카카오 JS SDK 단일 핀)을 모바일은 9.1이 세운 WebView 카카오맵 패턴(`features/map/mapHtml.ts` `buildMapHtml`·origin baseUrl 화이트리스트·graceful degrade)으로 재현한다. 전체화면 다중 핀(`MapWebView`)이 아니라 **상세용 단일 핀 미니지도 래퍼**를 새로 만든다(인터랙션 최소·줌/팬 비활성 가능). **Expo Web에서는 react-native-webview 미지원**이라 9.1과 동일하게 graceful degrade("지도를 못 불러왔어요")로 떨어진다(웹 `RoomLocationMap`도 동형 degrade) — 정상. 네이티브 렌더는 dev build에서만.

3. **카카오 공유 = RN 네이티브 공유(OS 공유 시트) + `buildReservationShareText` 재사용. 네이티브 카카오 전용 모듈은 EAS dev-build 버킷 보류.** 웹 `kakao-share.ts`(`window.Kakao` JS SDK `<script>` 주입)는 RN에 `window`/`document`가 없어 **통째로 폐기**한다. 모바일은 **RN 내장 `Share.share()`**(추가 네이티브 의존·키 불필요)로 OS 공유 시트를 열고, 공유 텍스트는 **순수 함수 `buildReservationShareText`를 그대로 복사 재사용**, 링크는 등록 웹 origin `${EXPO_PUBLIC_WEB_ORIGIN}/rooms/{room_id}`. **카카오톡 정확 템플릿 공유(`@react-native-kakao/share` 네이티브 모듈)는 EAS dev build + 네이티브 카카오 키 설정이 필요**(epics.md L136 명시·5.4 forward note)하고 **Expo Web/Playwright로 검증 불가**하므로, 진입점·콘텐츠·OS 공유는 9.2가 회수하되 **카카오 전용 모듈만 "모바일 dev-build 푸시" 버킷에 명시 보류**한다(§deferred 회수 참조). → **이 분기는 착수 전 KTH 확정 대기**(Change Log·하단 결정).

4. **후기 답글 작성 = 9.3(provider).** 9.2는 후기를 **표시(read)**하고(룸 상세 `ReviewSection`·예약현황 행에 내 후기·제공자 답글 read-only), **후기 작성(`useCreateReview`)**까지만 한다. 제공자 답글 **작성**(`useReplyToReview`)은 9.3 소유 — 침범 금지.

5. **단위 테스트 러너 미도입(9.1 패리티).** 모바일은 `test` 스크립트·러너 부재. 9.2 검증도 **typecheck(`tsc --noEmit`)+lint(`expo lint`)+Playwright(MCP) Expo Web 실검증**을 1차로 한다(자동 단위테스트 추가는 dev 판단·선택). 신규 인프라 도입 금지(웹은 3.2에서 vitest 최초 도입한 선례 — 모바일 러너 도입은 별도 결정).

## Acceptance Criteria

**AC1 — 룸 상세 화면이 웹과 동일한 3단 정보 위계·미니지도·후기로 표시된다 (FR-11 / 4.2 · 5.5 표시)**
**Given** 바텀시트 "상세 보기" 또는 `/rooms/[room_id]` 진입
**When** 룸 상세가 마운트되면
**Then** 웹 `RoomDetail`과 동일하게 — **1차 위계**(`StatusBadge`+`formatPrice`+오늘 영업시간+"예약 가능 시간 보기" 토글)·**2차 위계**(부대시설 칩+수용/룸형태+미니지도 `RoomLocationMap`)·**3차 위계**(후기 `ReviewSection` 읽기) — 가 네이티브로 표시되고, `useRoomSummary`(`roomsGetRoom`·`["rooms", roomId]`)로 **신선 조회**(`refetchOnMount:"always"`, [[availability-freshness-policy]] 상세=신선)하며, **404=`isRoomNotFound`("그 방은 더 이상 없어요")**·로딩/에러/네트워크단절이 인라인 카피로 처리된다(막다른 화면 금지). 룸 상세 파라미터명은 **`room_id`(snake_case)** 유지(`useLocalSearchParams<{ room_id: string }>`).

**AC2 — 날짜·슬롯 가용성과 연속 슬롯 선택이 웹과 동일하게 동작한다 (FR-12·13 / 4.3 · 4.4)**
**Given** 룸 상세에서 "예약 가능 시간 보기" 토글을 펼치면
**When** 날짜(달력)·슬롯을 선택하면
**Then** `ReservationPanel`이 전개되어 **전개·날짜 변경 시에만** `useRoomSlots(roomId, date)`(`roomsGetRoomSlots`·`["room", roomId, "slots", date]`)로 슬롯을 신선 조회하고, **커스텀 달력**(`Calendar` — 라이브러리 0·`RESERVATION_HORIZON_DAYS=30`·`isSelectableDate`)에서 today≤date≤today+29만 선택 가능하며, **슬롯 그리드**(`SlotGrid`)에서 **연속 가용 구간만**(`clampContiguousAvailable`·점유/지난 슬롯 가로지르기 구조적 차단) 선택되고 선택 라벨("14:00–17:00")·합계 금액(`selectionTotalPrice`)·"연속된 시간만 고를 수 있어요" 안내가 표시된다. **클라가 슬롯을 재계산하지 않는다**(서버 `slot.status`만 신뢰 — `available`/`past`/`reserved`, architecture.md L298·367). 빈 날="이 날은 다 찼어요. 다른 날을 골라보세요."+다음 빈 날짜 점프. `safeSelection`(bounds+content 이중 가드)로 stale 선택 무효화.

**AC3 — 즉시 예약이 결제 없이 확정되고 중복(409)이 차단된다 (FR-14·15 / 4.5 · 4.6)**

**When** "예약하기"를 누르면
**Then** `useCreateReservation`이 `selectionSlotStarts`로 `slot_starts[]`를 추출해 `reservationsCreateReservation`(POST `/rooms/{room_id}/reservations`)을 호출하고(**옵티미스틱 없음**·이중제출 방지 `disabled={isPending}`), 성공 시 "예약이 완료됐어요!" 배너+**카카오 공유 진입점**이 노출되며 정확 키(`["room",id,"slots",date]`+`["rooms","availability"]`)만 invalidate된다. **중복은 409=`isSlotConflict`("앗, 방금 다른 분이 먼저 잡았어요. 가까운 빈 시간을 다시 보여드릴게요.")로 graceful 처리**(선택 해제·그날 슬롯 재조회·에러코드 화면 노출 금지). 409 최종 강제는 서버(UNIQUE 제약). generic 실패="예약을 완료하지 못했어요. 다시 시도해 주세요."

9.2는 **9.1 파운데이션 위에 예약 도메인 전체를 얹는 스토리**다. 9.1이 인증·SDK 인터셉터·WebView 지도·세션·테마·E2E 하니스를 세웠으므로, 9.2는 **순수 클라이언트 작업**이고 **백엔드/SDK 변경 0**이다. 가장 큰 통찰: **웹 예약 비즈니스 로직은 거의 전부 프레임워크 무관 순수 함수**(`slots.ts`·`reservations.ts`·`errors.ts`·`share.ts`)이고 React Query 훅도 SDK+키만 쓴다 → **복사·미러로 재사용**하고, 포팅이 필요한 건 JSX(달력·슬롯 그리드·폼·카카오 공유)뿐이다. 유일한 진짜 갭 = **카카오 공유**(웹 `window.Kakao` JS SDK → RN 네이티브 공유). 두 번째 리스크 = **WebView 미니지도 origin/브릿지**(9.1과 동일 함정·Expo Web degrade).

### 신선도·상태·동시성 계약 (반드시 준수)

- **신선도**([[availability-freshness-policy]]·FR-15): 지도 핀=느슨 스냅샷, **룸 상세=신선 조회**(`useRoomSummary` `refetchOnMount:"always"`·슬롯도 전개/날짜 변경 시 신선), **예약 확정=서버 409 차단**(클라 사전판정 아님). 슬롯 키 `["room",id,"slots",date]`.
- **클라가 슬롯 재계산 금지**(architecture.md L298·367): 슬롯 리스트·상태(`available`/`past`/`reserved`)·`remaining_slots`는 **전부 서버**가 준다. 순수함수는 표시 포맷·인덱스 산술·금액 곱셈만.
- **옵티미스틱 경계**(architecture.md L277): 즐겨찾기·**슬롯 선택(UI 선택만)**만 옵티미스틱. **예약 확정·취소·후기 작성은 서버 확인 후**(옵티미스틱 없음·서버가 진실).
- **동시성 409**(architecture.md L180): 중복 예약·취소창 경과는 서버 409→**UI 우아한 안내+인접 슬롯**(에러코드 화면 노출 금지·UX-DR10). 6h 취소 경계 최종 강제=서버 timedelta, FE 계산은 버튼 활성/비활성 표시용.
- **invalidate 정확 키만**(architecture.md L275-276): `["rooms"]` 광역 금지. 통지 키 `["notifications"]`는 **최상위 독립**(`["rooms"]`/`["reservations"]` 프리픽스로 오염 시 지도/예약현황 캐시 깨짐). 취소는 `["reservations"]`+`["room",id,"slots"]` prefix+`["rooms","availability"]`+`["notifications"]`.

### 카카오 공유 — 갭과 결정 (§범위 3·deferred 회수)

웹은 `window.Kakao.Share.sendDefault`(JS SDK `<script>` 동적 주입)로 카카오톡 정확 템플릿 공유를 한다. **RN엔 `window`/`document`가 없어 통째로 폐기**한다. 5.4 forward note(deferred-work.md:67)가 **"모바일 예약 화면 자체가 미존재라 공유 전제 부재 → 모바일 dev-build 버킷 보류"**로 라우팅했는데, **9.2가 바로 그 예약 화면을 만들어 전제를 채우므로 회수 트리거가 발동**한다([[dev-workflow-policy-deferred-and-repeat-mistakes]] 의무 회수). 단 native 카카오 전용 모듈(`@react-native-kakao/share`)은 **EAS dev build+네이티브 키 설정 필요·Expo Web/Playwright 검증 불가**(epics.md L136). 그래서 **회수 가능한 부분(진입점·공유 텍스트·OS 공유)은 9.2가 RN 내장 `Share.share()`로 즉시 구현·검증**하고, **카카오 전용 정확 템플릿 모듈만 EAS dev-build 버킷에 명시 보류**한다(재-defer 아님·검증 불가 인프라 의존이라 예외 인정·E7/E8/배포 예외 정신과 동형). `buildReservationShareText`는 순수 함수라 그대로 재사용. → **착수 전 KTH 확정**(Change Log 결정 항목).

### 반복 함정 프리플라이트 ([[dev-workflow-policy-deferred-and-repeat-mistakes]])

1. **재발명 금지** — 예약 로직은 웹 순수함수·훅을 **복사/미러**. 9.1이 이미 둔 `roomSummary`/`pin`/`StatusBadge`/`useRoomSummary`/`pagination`/`ListStates`/`NetworkNotice`/`FavoriteButton`/`mapHtml`은 **재사용**(중복 복사 금지).
2. **직접 fetch 금지** — 백엔드 호출 전부 `@/lib/api-client` SDK(`reservations*`/`reviews*`/`notifications*`/`roomsGet*`). eslint `no-restricted-globals`/`no-restricted-syntax` 위반 0. WebView 미니지도는 RN 주입 데이터만(직접 호출 안 함).
3. **클라 슬롯 재계산 금지** — 서버 `slot.status`/`remaining_slots`만. 핀·슬롯 색은 서버 집계.
4. **옵티미스틱 경계** — 예약 확정·취소·후기는 서버 확인 후(옵티미스틱 없음). 슬롯 선택·즐겨찾기만 옵티미스틱.
5. **invalidate 정확 키·`["notifications"]` 독립** — 광역 `["rooms"]` 금지. 통지 키 프리픽스 오염 금지.
6. **에러코드 화면 노출 금지** — 409/취소창/후기 에러는 코드를 분기에만, 화면엔 친절 카피.
7. **네트워크 단절 카피** — "오프라인" 금지·"네트워크 연결이 끊겼어요. 연결되면 다시 보여드릴게요."([[terminology-network-disconnect-not-offline]]). 단절>에러 우선.
8. **용어 "지역"** — 해당 시 UI/주석 "행정동/법정동" 금지([[region-code-legal-not-admin-dong]]). 9.2는 예약 도메인이라 빈도 낮음.
9. **snake_case 와이어 보존** — `room_id`/`slot_start`/`slot_starts`/`reservation_id`/`next_cursor`/`remaining_slots`/`notification_id`/`reservation_reminder` 그대로(camelCase 변환 금지).
10. **카카오 공유 갭** — `window.Kakao`/`document`/`<script>` 사용 금지(RN 무존재). RN `Share.share()`+`buildReservationShareText`. 카카오 전용 모듈=EAS dev-build 보류.
11. **WebView origin/브릿지**(9.1 함정 상속) — 미니지도 카카오 키=`EXPO_PUBLIC_KAKAO_JS_KEY`(JS 키만)·baseUrl origin 화이트리스트. Expo Web degrade 정상.
12. **library 실측** — 신규 의존성 추가 시 실제 패키지/버전 확인(환각 금지). **9.2는 신규 의존 0 권장**(RN `Share` 내장·`buildMapHtml` 재사용). `@gorhom/bottom-sheet`·`react-native-webview`·`react-query`·netinfo·secure-store·expo-location은 **이미 설치됨**(9.1·재설치 금지).
13. **python(=python3 아님)·uv --no-sync**(API 재기동 필요 시) — 9.2는 백엔드 무변경이라 대개 무관([[python-command-use-python-not-python3]]·[[uv-run-no-sync-when-api-running]]).

### deferred 후보·회수 (회수 트리거)

- **[회수 트리거] 모바일 카카오 예약 공유** — 5.4 forward note(deferred-work.md:67)가 "모바일 예약 화면 미존재"로 보류한 항목. 9.2가 예약 화면을 만들어 전제 충족 → **진입점·공유텍스트·OS 공유는 9.2가 회수**(RN `Share`). **카카오 전용 모듈(`@react-native-kakao/share`)+EAS dev build만 "모바일 dev-build 푸시" 버킷 보류**(검증 불가 인프라 의존). KTH 확정 대기.
- **[9.1 상속] WebView 카카오 origin 하드코딩·SDK 로드 워치독 부재·RoomSheet 닫힘 중 탭** — 9.2 미니지도가 `buildMapHtml`/WebView 패턴을 재사용하면 동일 갭 상속(deferred-work.md:5-7). 네이티브 하드닝 시 origin env화(`EXPO_PUBLIC_KAKAO_WEBVIEW_ORIGIN`)+로드 타임아웃 워치독 회수.
- **[배포 의존] 카카오 콘솔 프로드 도메인 등록·후기 공개조회 페이지네이션/레이트리밋** — 배포 스토리 소유(deferred-work.md:62,68·[[deployment-first-time-checklist]]). 9.2 무관.
- **[5.4 동형] 자정 넘는 예약 시간범위 표기** — `reservationTimeRangeLabel`/`buildReservationShareText` 공통(deferred-work.md:72). 그대로 복사(웹 패리티)·시간표기 UX 하드닝 시 "+1일" 마커 공통 승격.
