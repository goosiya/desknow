## 1. ★의무 회수 검증 (focus ②의 핵심 — epic-5 회고 2c)

epic-5 회고가 **★중대·6.2 의무 회수·재-defer 금지**로 못박은 2c 두 항목을 **실제 소스에서 검증**했다.

### 1a. cancel↔reject 교차 race → ✅ 진짜 회수됨 (조건부 원자 UPDATE)
- **부채(deferred-work L43·4.7 defer):** `_transition_to_terminal`이 Python에서 `reservation.status`를 read-then-flip → booker 취소(4.7)와 provider 거절(6.2)이 동시 진입 시 둘 다 stale `confirmed`를 읽고 last-write-wins(최종 status·통지 비결정).
- **회수 확인 [service.py:194-219]:** read-then-flip을 **조건부 원자 UPDATE**(`UPDATE reservations SET status=:target WHERE id=:id AND status='confirmed'`)로 교체. `result.rowcount == 1`=승자(`_release_slots` DELETE + commit + refresh), `rowcount == 0`=패자(slot·통지 0·`rollback`+`refresh`로 현재 상태 수렴). DB 행 락이 단일 승자를 중재 → **첫-전이-승자로 결정화**. 종료 상태 fast-path(DB 왕복 0 멱등)는 게이트 *전*에 보존.
- **회수 품질:** 문서 주장이 아니라 코드가 실제로 그렇게 동작(rowcount 분기·refresh 필수 처리·rollback 정리까지 정확). 통합 테스트가 SQLite/PG 실 rowcount를 핀(`test_reservations_migration.py`).
- **공유 프리미티브 무회귀:** cancel/reject 공유라 cancel도 결정화되나 동작 불변 — cancel-vs-cancel(둘째 rowcount 0 멱등)·단일 cancel(rowcount 1)·종료 fast-path 전부 보존. 기존 cancel 테스트 전부 그린이 안전망(563 passed·무회귀 0).
- **통지 정확성 [router.py:380-397]:** `was_confirmed and updated.status == REJECTED`일 때만 통지 → race 패자(post=`cancelled`)·멱등 재거절(was_confirmed 거짓)엔 통지 0. "거절 안 했는데 거절 통지" 비결정성 제거. `create_notification` 멱등이 이중 방어.

### 1b. `confirmed_slot_starts` reader status 필터 → ✅ 미트리거 명시 평가 (재-defer 아님)
- **부채(deferred-work L37):** reader가 status 조인 없이 "활성 점유만"을 **DELETE-on-cancel 불변식에만** 의존. **비-DELETE** 종료 경로가 추가되면 유령 점유 누출.
- **6.2 평가 확인 [service.py:153-164·371]:** 거절은 `reject_reservation → _transition_to_terminal → _release_slots`로 점유 슬롯을 **DELETE**(취소와 동일 메커니즘). 즉 `reservation_slots`엔 활성 점유만 잔존 → 불변식 보존·방어심층 불요. 트리거 조건(비-DELETE)이 **거짓**이라는 명시 평가지 침묵 재-defer가 아니다.
- **잔존:** 실제 트리거 = E8 비-DELETE 종료 경로(만료/비활성 캐스케이드·직접 status UPDATE). deferred-work에 그대로 유지(책임 = E8).

**관찰:** Epic 1 회고 A3("트리거 시 의무 회수, 재-defer 금지")가 **3연속 동작**(Epic 5 = L103·L6/L7·maps 누수 회수 / Epic 6 = 2c race 회수 + reader 필터 명시 평가). 정책이 형식이 아니라 실효 — **의무 회수가 코드로 검증됨이 이번 회고의 핵심 성과.**

## 3. deferred 트리아지 (focus ②)

### 3a. 신규 deferred — 정상 이관 (유지 OK)
- **★거절 commit 후 통지 비원자 — 통지 실패 시 500 + 재시도 영구 손실** [router.py:384-398·notifications/service.py:57-71] (6-2 신규) — 거절을 commit한 *뒤* 라우터가 `create_notification`(자체 commit)을 호출(2 트랜잭션). 통지가 raise하면 거절·슬롯 재활성은 영속인데 응답 500이고, 재시도 시 이미 `rejected`라 `was_confirmed` 거짓 → status_change 통지 **영구 미발송**(자가치유 아님). 극희박(거절 commit 직후 통지 INSERT 실패+동시경합)이라 defer 유지. **회수: outbox 또는 동일 트랜잭션 통지, 혹은 rowcount 기반 "전이 수행" 신호로 `was_confirmed` 대체 — 8.3(임의취소)이 동일 통지 배선 추가 시 함께 정리.**
- **제공자 예약현황 페이지네이션 부재** [service.py:344-349] (6-1) — favorites·4.8과 동형 미페이지네이션. MVP 제공자당 룸 1개지만 룸당 예약 수는 무상한. 회수: 4.8·favorites와 일관 페이지네이션/cursor.
- **`created_at desc` 동률 비결정 정렬** [service.py:347] (6-1) — 보조 정렬키(`id`) 부재. `list_booker_reservations`(4.8) 동일 갭 미러. 회수: `.order_by(created_at.desc(), id)`를 **6.1+4.8 동시 적용**(둘 중 하나만 고치면 비대칭).

### 3b. 누적 추세 패턴 (관찰)
- **미페이지네이션 + 비결정 보조정렬**이 이제 **favorites·4.8·6.1 3회 반복**. 매번 "MVP 규모 무리없음·동형 선례"로 defer. 진짜 결함은 아니나 **반복 신호** → 공개/목록 조회 일괄 페이지네이션 + `(created_at desc, id)` 보조키를 한 번에 정리할 후보. **배포 스토리에 레이트리밋과 함께 묶기 권장.** [[deployment-first-time-checklist]]
- **비원자 통지/멱등 경합 클래스**(create_notification None→500·favorites add IntegrityError None→500)도 동근 — 통지 *생성* 배선이 6.2에서 처음 가시화. 8.3이 동일 배선 확장 시 outbox/트랜잭션 통합으로 근본 정리 후보.

### 3c. Epic 5 회고 2d(옵티미스틱 dismiss/toggle stale clobber) 상태
- Epic 6은 **백엔드 전용**(FE 변경 0)이라 재발도·해소도 없음. 여전히 pending(favorites 3.7·5.1·5.2 3회) — 임계 도달 시 전용 정리 스토리 권장. provider 웹 표면/모바일 UI 착수 시 동일 키 뮤테이션 직렬화 검토.

## 4. 코드리뷰 반복 지적 (focus ③)

### 4a. 프리플라이트 정착 ✅
Epic 1 A1("반복 함정 프리플라이트")이 6-1·6-2 Dev Notes에 고정 12항 섹션으로 존재(rooms import 금지·UTF-8·조건부 UPDATE refresh 필수·cancel 무회귀·reason 정확 'rejected' 등). **차단성 리뷰 지적 0**(decision-needed 0·patch 0) — 프로세스가 실효.

### 4b. "신규 결함 아님" defer가 다수 (패턴)
- 6-1 defer 2건·6-2 defer 1건 **전부 pre-existing**(스토리가 이미 deferred 후보로 명시했거나 4.8/favorites 미러). 적대적 리뷰가 잡은 것은 새 결함이 아니라 **선재 부채의 재확인** → 프리플라이트가 새 결함 유입을 막고 있다는 방증.
- **dismiss 패턴 반복(noise):** 6-1 5건·6-2 5건 dismiss — 도달 불가 false positive(`slot_starts` coerce·소유 룸 id로만 조회·해시 역산 비현실)·문서화된 결정(충돌 수용)·이식성(통합 테스트가 rowcount 핀). Epic 5 3b("컨텍스트 0 Blind Hunter 고심각도 오탐")와 동형 — **리뷰 결과를 그대로 받지 말고 런타임/구조로 확인하는 규율이 매번 필요**(A1 ④와 동형).

### 4c. reason 자유문자열 하드코딩 분기 (5.3→6.2 인지 계승)
- 6.2가 `reason="rejected"`를 **정확히** 넘김(오타 시 FE가 조용히 generic 폴백 = 거짓 정상). BE CHECK/Literal 없는 자유문자열 + FE 하드코딩 분기는 deferred-work L281에 인지됨 → 8.3 합류 시 근본 방어(CHECK·상수 공유) 후보.
