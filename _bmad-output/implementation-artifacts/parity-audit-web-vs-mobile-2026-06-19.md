# 웹↔모바일 시각·인터랙션 패리티 audit (2026-06-19)

> 목적: 모바일(RN/Expo) 화면이 웹과 구조·기능은 차용했으나 **렌더 결과(버튼 모양·정렬·메시지 위치·셸)** 가 어긋나는 드리프트를 화면쌍 스크린샷으로 전수 적출한다. 이 문서가 Correct Course(Epic 9 정정·시각 패리티 정정 스토리)의 입력이다.

## 환경(검증 하니스)

- **웹 = `http://localhost:3002`** (Next.js `apps/web`, `next dev`, 모바일 뷰포트 390×844). ※기본 :3000은 Expo가 점유해 임시로 :3002 기동 + API CORS에 :3002 추가(main.py, dev 전용).
- **모바일 = `http://localhost:3001`** (Expo Web `apps/mobile`, 최신 번들).
- API = `http://localhost:8000`. 시드 계정 `booker@test.desknow` / `_pweb0@test.desknow` (Test1234!).
- ⚠️ **중대 전제 정정**: 직전까지 ":3000=웹"으로 알고 비교했으나 **:3000도 :3001도 모두 Expo 모바일**이었다(같은 `expo-router/entry.bundle` 서빙). 즉 과거 "픽셀 일치"는 모바일↔모바일 비교였고 무의미. :3000은 9.3 이전 stale 번들이라 provider/room이 옛 stub("곧 제공돼요")으로 보였던 것. **진짜 웹앱은 어느 포트에도 안 떠 있었다.** 9.x 모바일 스토리의 "리뷰/테스트 통과"가 실제 웹과 대조 없이 이뤄졌음을 시사 — 드리프트 누락의 근본 원인.

- **스크린샷 근거**: `_bmad-output/implementation-artifacts/parity-audit-2026-06-19-shots/` — 웹 `w2-*.png` / 모바일 `m1-*.png` 쌍(login·signup·home·roomform(+hours)·chatbot·prov-res·prov-rev·favorites·reservations·detail).

## ⚠️ Expo Web ≠ 네이티브 (반드시 인지)

이 audit은 **웹 vs Expo Web(react-native-web)** 비교다. 실제 iOS/Android 네이티브 빌드는 폰트 렌더링·그림자·터치 하이라이트·세이프에어리어·키보드 회피 등에서 **Expo Web보다 더 벌어질 수 있다.** 여기서 "일치"로 적힌 항목도 실기기에서 재검증 필요. 카카오 지도/지오코더는 :3001 origin 미등록이라 캔버스가 degrade("지도를 못 불러왔어요") — 이는 환경 artifact이지 드리프트 아님.
