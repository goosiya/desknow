## 2026-06-14
- 브리프 작성 착수(intent: create). 입력 자료: `docs/idea.md`, 기술조사 리포트(`_bmad-output/planning-artifacts/research/technical-desknow-mvp-tech-feasibility-research-2026-06-13.md`).
- **용도/이해관계:** 개인 학습 프로젝트. → 시장·투자 증명 강도는 낮게, 비전·범위 명료성 위주로 캘리브레이션.
- **예외:** 디자인 및 UI/UX 완성도는 **높은 기준**을 적용하기로 함. → 브리프에 UX/디자인 품질 섹션을 별도로 강하게 둘 것.
- **작업 방식:** 코칭 경로(섹션별 동행, 얇은 가정은 밀어붙임).
- **동기 프레이밍(확정):** 기술 연습이 본질(주), 현실 문제는 종이되 "진짜로 믿는 빈틈"으로 다룸. → 브리프는 이 이중성을 솔직하게 드러낸다(가짜 시장 해자 만들지 않음).
- **문제 정의 핵심:** 흩어진 탐색 창구 통합 + 실시간 예약가능 여부·가격을 "한눈에" → 이 "한눈에"가 높은 UI/UX 기준과 직결되는 핵심 가치.
- **경쟁 인식:** 스페이스클라우드/네이버 예약 등 기존재. DeskNow는 "예약처 부재"가 아니라 "경험의 빈틈(실시간 빈슬롯 가시성·가격 비교 편의)"을 노린다(가설, B 본질 + A 빈틈).
- **포지셔닝 재조정(중요):** 사용자가 "포트폴리오지만 나중에 진짜 제품이 될 수 있다는 가정으로 만들고 싶다"고 명시. → 브리프 본문에서 "학습 프로젝트" 명시 문구는 **제거**, 진짜 제품 브리프로 작성(톤만 정직). 동기의 이중성은 결정 로그/내부 맥락으로만 보존.
- **주 사용자(확정):** 시험준비에 국한 X → **다양한 형태의 스터디 그룹 + 소규모 모임**. 제공자(스터디룸 사장)는 공급 측.
- **문제 섹션:** 승인됨(마지막 "솔직한 맥락" 문단은 제거).
- **디자인 북극성(UX 원칙):** 단일 영웅 화면이 아니라 **"예약에 이르는 흐름 전체에 마찰이 없어야 한다"**. 사용자는 어느 한 장면을 고르지 못함 = 예약 과정 전체가 핵심. → UX 섹션에서 "전체"를 측정 가능한 마찰 지점들로 분해 예정.
- **챗봇 역할(확정):** (다) 둘 다 동등 — ① 서비스 안내 도우미(문서/FAQ 기반) + ② 자연어 예약 검색 파트너(DB 기반).
- **노쇼/신뢰(확정):** MVP는 **의도적 단순화**(결제 없는 즉시 예약 + 6시간 전 취소 + 제공자 거절). 노쇼·환불·신뢰장치는 향후 고도화로 명시.
- **찾기(확정·수정):** 지도 중심이되 **시/군/구·동/읍/면 콤보 목록 조회는 1급 기능으로 필수 유지**(드롭 금지).
- **UX 기준(확정):** 4개 차원 **모두** 높은 기준 — ①시각적 완성도·일관성 ②인터랙션·마이크로애니메이션 ③정보 명료성·한눈에 ④반응성·접근성·성능.
- **UX 레퍼런스(확정):** 토스(단순함·마이크로인터랙션) + 에어비앤비(지도 예약) + 카카오맵(핀·바텀시트). "장점만 골라 좋은 기준으로."
- **부록 분리:** PRD/아키텍처 인계용 `addendum.md` 생성(연속 다중슬롯 미해결 질문, 기술결정 3종, 챗봇 멀티LLM 유의, 동시성 DB모델, 아키텍처 방향, UX 수치 승계).
- **Finalize 폴리시:** 구조·프로즈 에디토리얼 리뷰 적용 — Executive Summary↔Problem/Solution/Vision 중복 3건 정리, 도메인 용어 일관화(가용→예약 가능, 칸/시간대→슬롯, 시군구·동→시/군/구·동/읍/면, 별점(5)→별점(5점 만점) 등). 의미 변경 없음.
- **Scope(확정):** idea.md 기반 In/Out 경계 승인. 카카오맵·비밀번호 정책(차기 완화)·연속 다중 슬롯 반영.
- **Vision(확정):** 방향 승인하되 **소박한 톤**으로 — "국내 기준점" 같은 거창한 표현 제거, "내가 직접 쓰고 싶은 수준의 경험을 완성" 결의 중심.
- **성공 기준(확정):** 제품/경험 신호 + 포트폴리오/역량을 **동등 비중**, 모두 측정 가능 형태. 실사용 트래픽 부재 → 경험 지표는 소규모 사용성 테스트로 측정한다고 정직하게 명시.
- **UX 섹션 구조(확정):** "흐름 전체 무마찰" 원칙을 7개 마찰 지점으로 분해 + 4개 품질 차원. 핵심 몇 개에 **수치 목표**를 명시(첫 지도 표시 2초·단독 예약 3스텝·챗봇 첫 토큰 2초·상호작용 응답 100ms 등), 나머지는 정성 기준. 수치는 PRD 검증 기준으로 승계.
- **사용자(확정):** 예약자는 **"모임 총무/주최자"와 "혼자 쓸 개인"을 동등 비중**으로. → UX는 공유·일정조율 흐름 + 빠른 단독 예약 흐름 둘 다 1급. 관리자는 페르소나 X, **운영 주체**로만 가볍게 다룸.
- **예약 슬롯(확정·중요 수정):** 1시간 단위 + **연속된 다중 슬롯 선택 가능**(한 칸만 X). → PRD/아키텍처 영향: 슬롯당 1행 모델(`UNIQUE(room_id, slot_start)`)은 그대로 유효하되 1회 예약이 N개 연속 슬롯행에 매핑. 비연속 다중선택 허용 여부는 PRD에서 확정(기본: 연속 구간).
