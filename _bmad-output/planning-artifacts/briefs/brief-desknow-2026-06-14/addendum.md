# DeskNow Brief — Addendum (다운스트림 인계용)

> 브리프 본문엔 넣지 않았지만 **PRD·아키텍처 단계로 반드시 넘겨야 할** 결정·제약·미해결 항목. 출처: 코칭 세션(2026-06-14) + 기술조사 리포트(`_bmad-output/planning-artifacts/research/technical-desknow-mvp-tech-feasibility-research-2026-06-13.md`).

## 예약 모델 (PRD에서 확정 필요)
- **연속 다중 슬롯 예약:** 1시간 단위, 한 번의 예약이 **N개의 연속 슬롯**에 매핑. 기본 정책은 *연속 구간만* 허용.
  - ❓ **비연속 다중 선택**(예: 10–11시 + 15–16시를 한 번에) 허용 여부는 PRD에서 결정. 기본값=불허(연속만).
- **동시성/중복 방지:** 애플리케이션 락이 아니라 **DB 제약**으로 해결 — 고정 1시간 슬롯이므로 `UNIQUE(room_id, slot_start)` 한 줄로 원자적 차단. (가변/겹침 시 `EXCLUDE USING gist`나 PG18 `WITHOUT OVERLAPS`는 현재 불필요.)
- pgvector(챗봇)와 예약 테이블은 **동일 PostgreSQL 공존** 무리 없음.

## 기술 결정 3종 (기술조사 → 본문 Scope에 반영, 구현 시 유의)
1. **지도/주소 = 카카오맵 우선**(네이버 무료정책 재편 리스크). 최종 채택 전 **카카오 약관의 서비스 유형 적합성 1차 확인** 필요. 좌표 + 행정동 코드 **둘 다 저장**해 반경/행정동 두 검색 UX 지원.
2. **Railway 상시 컨테이너 과금** — 트래픽 0이어도 과금, 서비스 수 = 비용. 비용 모니터링/알림 필수. 초기엔 챗봇을 API에 통합했다가 부하 시 분리하는 단계적 접근 고려.
3. **비밀번호 강제 복잡도(8자+대/특/숫) ↔ NIST SP 800-63B Rev.4 충돌** — MVP 수용, **차기 완화**(길이 우선 + 유출목록 대조)를 향후 항목으로 명시.

## 챗봇 (구현 유의)
- 역할 2종 동등: ① 서비스 안내 도우미(문서/FAQ RAG) + ② 자연어 예약 검색 파트너(예약 DB 툴).
- **멀티 LLM 파라미터 비대칭** — 예: Anthropic 최신 모델은 `temperature`/`top_p`/`top_k` 미지원 → 공통 파라미터를 전 프로바이더에 그대로 전달하면 깨질 수 있음. 래핑 레이어에서 **프로바이더별 분기** 필요. MVP는 채팅+툴콜 공통기능에 한정.
- 스트리밍 = SSE 권장. **RN은 네이티브 EventSource 없음** → `react-native-sse`/fetch-event-source. 임베딩 모델은 **단일 고정**(교체 시 전체 재임베딩).

## UX 수치 목표 → PRD 검증 기준으로 승계
- 첫 지도 표시 ≤2초 · 단독 예약 ≤3스텝 · 챗봇 첫 토큰 ≤2초 · 주요 상호작용 응답 ≤100ms.
- 경험 지표(첫 예약 ≤3분, 완주율 ≥80%, 제공자 등록 ≤5분)는 **소규모 사용성 테스트(5명 내외)**로 측정.

## 아키텍처 방향 (기술조사 권고, 아키텍처 단계에서 확정)
- FastAPI **모듈러 모놀리스** + 챗봇은 같은 레포·별도 서비스 배포(스트리밍 부하 격리).
- Turborepo + pnpm 모노레포(사용자 웹 / 관리자 웹 / RN) + FastAPI 디렉터리, OpenAPI → 타입 안전 TS SDK 자동생성.
- 관리자 웹은 별도 Next.js 앱으로 분리(공격 표면 축소). RBAC 최종 강제는 FastAPI.
- 인증: PyJWT + pwdlib(Argon2), refresh 무효화 저장소(Redis/DB), 웹=쿠키·앱=헤더 이원 추출.
