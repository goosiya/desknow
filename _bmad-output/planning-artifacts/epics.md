# desknow - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for desknow, decomposing the requirements from the PRD, UX Design if it exists, and Architecture requirements into implementable stories.

> **포스트-MVP 동기화 주석 (Correct Course 2026-06-19):** Epic 1~8 done 이후의 보강(provider 웹 표면·"지역" 용어 통일·목록 커서 페이징 등)은 BMad 밖에서 구현되었으며, **구현 소스를 정본으로 수용**한다(기존 스토리 본문 재작성 안 함). 또한 모바일 앱은 각 에픽에 "웹/앱 동일"로 암묵 포함되었으나 매번 deferral되어 미구현이었고, 이를 **Epic 9(모바일 화면 패리티)**로 묶어 회수한다. 상세 근거=`sprint-change-proposal-2026-06-19.md`.

## Requirements Inventory

### Functional Requirements

*PRD §4 기준 — 총 34개 (FR-1~33 + FR-18a). UJ realize 관계는 PRD 인라인 참조.*

**계정 & 인증 (§4.1)**
- **FR-1: 이메일 회원가입** — 예약자/제공자가 이메일·비밀번호·역할로 가입. 이메일 형식 검증, 역할 무관 중복 차단, 가입 즉시 로그인 가능. Realizes UJ-1, UJ-2, UJ-3.
- **FR-2: 로그인 / 로그아웃 / 세션 유지** — 이메일·비밀번호 로그인, 로그아웃 전까지 인증 유지. 역할은 가입 시 확정·전환 불가. 로그아웃 시 세션·챗봇 대화 종료. 웹/앱 동일 백엔드 인증. Realizes UJ-1~UJ-4.
- **FR-3: 비밀번호 정책** *(횡단 정책 — UJ 외)* — 최소 8자 + 대문자1 + 특수문자1 + 숫자1. 백엔드 신뢰경계 검증, 평문 저장 금지(해싱).

**스터디룸 탐색 — 지도 (§4.2)**
- **FR-4: 첫 진입 지도 + 주변 핀** — 첫 화면에서 추가 조작 없이 주변 룸을 핀으로. 위치 권한 시 현위치 주변 핀, 첫 표시 ≤2초, 핀 탭→바텀시트. Realizes UJ-2.
- **FR-5: 핀 색상 — 예약 가능 구분** — 오늘(룸 타임존) 현재시각 이후 남은 빈 슬롯 ≥1이면 초록, 없으면 회색. 서버측 집계값으로 결정, 색+보조 표식 병행. Realizes UJ-2.
- **FR-6: 바텀시트 요약** — 1차 정보(가격·영업시간·예약 가능)는 스크롤 없이, 2차(부대시설)는 하위 위계. 상세 이동·즐겨찾기 토글. Realizes UJ-1, UJ-2.
- **FR-7: 위치 권한 거부 우회** — 권한 거부 시 목록 조회(행정동 콤보)로 즉시 우회, 반경 검색 비활성+이유 안내, 막다른 화면 금지. Realizes UJ-2.

**스터디룸 탐색 — 목록 (§4.3)**
- **FR-8: 행정동 콤보 목록 조회** — 시/군/구 선택→하위 동/읍/면 갱신, 지역 룸 목록 나열, 상세 이동·즐겨찾기, 결과 없음 빈 화면 디자인. Realizes UJ-1.
- **FR-9: 반경 검색** — 현위치 기준 반경 내 룸, 기본 3km·사용자 조정, 반경↔행정동 전환, 위치 없으면 비활성. Realizes UJ-2.

**즐겨찾기 (§4.4)**
- **FR-10: 즐겨찾기 추가 / 조회** — 어디서든 토글, 즐겨찾기 페이지 모아보기, 사용자별 영속, 비활성 룸은 '비활성' 라벨+진입 차단. Realizes UJ-1.

**스터디룸 상세 & 예약 (§4.5)**
- **FR-11: 상세 정보 화면** — 가격·부대시설·영업시간·주소·후기 위계 표시, 예약 UI 같은 페이지 내 전개. Realizes UJ-1, UJ-2.
- **FR-12: 날짜·슬롯 가용성 표시** — 달력 날짜 선택, 과거·당일 현재시각 이전·예약된 슬롯 비활성, 영업시간 도출 1시간 단위. Realizes UJ-1, UJ-2.
- **FR-13: 연속 슬롯 선택** — 연속 구간 드래그/탭 선택, 비연속 불허(D1), 구간 중 하나라도 점유면 막힘. Realizes UJ-1.
- **FR-14: 즉시 예약 (결제 없음)** — 결제 단계 없이 즉시 확정, 상세 진입~3스텝 이내, 성공 피드백, all-or-nothing 점유. Realizes UJ-1, UJ-2.
- **FR-15: 중복 예약 방지 (동시성)** — 동시 확정 시 정확히 하나만 성공(중복 0), 겹치는 연속 예약 부분 점유 0, 단일 트랜잭션 원자 성립/실패, 실패 측 우아한 안내. Realizes UJ-1(edge).
- **FR-16: 예약 취소** — 예약 시작 6시간 전까지 취소(이후 비활성), 점유 슬롯 동일 트랜잭션 재활성, 히스토리 취소 상태, 상태 전이 원자·멱등. Realizes UJ-2.

**예약 관리 (예약자) (§4.6)**
- **FR-17: 예약현황 / 히스토리** — 다가오는/지난(이용완료·취소·거절) 구분, 상세 이동, 취소 동작 연결. Realizes UJ-1, UJ-2.

**알림 (§4.7)**
- **FR-18: 예약 도래 인앱 배너** — 시작 24시간 이내 확정 예약 접속 시 배너, '다시 보지 않기' 사용자별 영속, 웹/앱 동일. Realizes UJ-1.
- **FR-18a: 예약 상태변경 통지 배너** — 거절(FR-24)·임의취소(FR-32) 발생 시 다음 접속 1회 통지(24h 규칙 비종속), 확인 시 소멸, 도래 리마인드와 독립 트리거. Realizes UJ-3(거절 결말).

**공유 (§4.8)**
- **FR-19: 카카오톡 예약 공유** — 확정 예약 한 번의 동작으로 카카오톡 공유, 룸·일시 핵심 정보 포함, 카톡 외 SNS 비포함. Realizes UJ-1.

**후기 & 평점 (§4.9)**
- **FR-20: 후기 작성** — 이용 완료(확정 유지+마지막 슬롯 종료 경과) 예약만, 예약당 1회, 별점 1~5 정수+텍스트 500자, 상세 노출. 취소·거절은 작성 불가. Realizes UJ-1.
- **FR-21: 후기 답글 (제공자)** — 제공자가 자기 룸 후기에 답글, 후기에 연결 노출. Realizes UJ-3.

**제공자 — 예약 관리 (§4.11)**

### NonFunctional Requirements

*PRD §10 Cross-Cutting NFRs — 시스템 전체 적용 1급 요건.*

- **NFR-1 (시간·타임존):** 모든 slot_start는 절대시각(UTC) 저장, 모든 "오늘/현재시각/N시간 전/24시간 이내" 판정은 룸 소재지 타임존(MVP=Asia/Seoul) 기준. FR-5·12·16·18·20의 시간 경계 결정성 전제.
- **NFR-2 (성능):** 첫 지도 표시 ≤2초, 챗봇 첫 응답 토큰 ≤2초(양호 네트워크·웜·기준 프롬프트·기준 프로바이더·p90, 콜드스타트/프로바이더 편차 제외). 로컬 옵티미스틱 UI 피드백(탭·토글·전환 시작) ≤100ms.
- **NFR-3 (예약 단순성):** 단독 예약은 상세 진입~3스텝 이내 완료(스텝=별개 탭/입력 동작, 스크롤·드래그는 1스텝).
- **NFR-4 (시각적 완성도·일관성):** 디자인 토큰(간격·색·타이포) 기반으로 웹/앱·화면 간 이질감 제로. 인터랙션·마이크로애니메이션이 로딩·전환·선택에 부드러운 피드백.
- **NFR-5 (접근성·상태 설계):** 키보드·스크린리더 접근성(WCAG 2.2 AA), 색 비의존(색+보조 표식 병행). 빈/에러/오프라인/로딩 상태까지 설계(막다른 빈 화면 금지) — 주요 화면 × {빈/에러/오프라인/로딩} 매트릭스 각 셀에 디자인 존재.
- **NFR-6 (보안):** 외부 API 키(카카오·LLM·지도)는 백엔드에서만 관리(지도 JS SDK 도메인 키는 도메인 화이트리스트). 단일 백엔드 발급 토큰 웹/앱 공용(웹=쿠키·앱=헤더). 역할 권한은 백엔드 최종 강제. 챗봇 스트리밍 엔드포인트 인증 보호. 비밀번호 해싱 저장.
- **NFR-7 (신뢰성·데이터 정합성):** 동시 예약(겹치는 연속 구간 포함)에서 중복·부분 점유 0건(DB 제약+단일 트랜잭션 원자 보장). 예약 상태 전이(확정→취소/거절)는 원자·멱등, 슬롯 재활성은 전이와 동일 트랜잭션 내에서만.

### Additional Requirements

*Architecture.md 기준 — 에픽/스토리 구성에 직접 영향을 주는 기술 요건.*

**🚩 스타터 템플릿 (Epic 1, Story 1 — 최우선):**
- **하이브리드 스캐폴드** = create-turbo 골격 + 앱별 공식 스캐폴드. 초기화 명령(아키텍처 §Starter):
  - `pnpm dlx create-turbo@latest desknow --package-manager pnpm` (모노레포 골격)

  - `create-expo-app` → `apps/mobile` (Expo SDK 56, RN 0.85)

  - `packages/api-client` @hey-api/openapi-ts 설정 (openapi.json 소비)
- 버전(2026-06 기준): Turborepo 2.9.x / Next.js 16.2.x / Expo SDK 56. init 시점 최신 재확인.

**Phase 0 검증 스파이크 (구현 전 권장 — 별도 스토리화 가능):**
- ① 카카오맵 지도표시 + 주소변환 e2e (+ 약관 서비스유형 적합성 1차 확인)
- ② 챗봇 SSE 스트리밍 (FastAPI ↔ 웹 ↔ RN)
- ③ Supabase → Railway pgvector 덤프/복원 1회

**데이터·인프라:**
- DB: PostgreSQL 단일 + pgvector(HNSW), 예약 테이블과 벡터 테이블 공존. ORM=SQLModel, 마이그레이션=Alembic.
- 동시성: `UNIQUE(room_id, slot_start)` + 단일 트랜잭션 multi-row INSERT(`ON CONFLICT` 후 affected=N 검증, 아니면 ROLLBACK).
- 영업시간/휴무: room별 요일별(weekday 0~6) 영업시간 행 + 휴무일 예외(date) 테이블. 슬롯 = (영업시간 − 휴무일 − 예약) 도출.
- 지도/주소: 좌표(lat/lng) + 행정동 코드 둘 다 저장. 반경=PostGIS ST_DWithin 또는 Haversine.
- 인증: 단일 JWT(access+refresh), refresh 무효화=PostgreSQL 테이블, 토큰 보관 이원화(웹 쿠키/RN SecureStore). PyJWT + pwdlib(Argon2). RBAC=JWT role 클레임(booker/provider/admin) + FastAPI 의존성.
- API: REST + OpenAPI, `/api/v1` 버저닝. 와이어 전 구간 snake_case. 에러 표준 스키마(`{detail:{code,message}}`), 동시성 충돌 409. CORS 웹 origin 등록.
- 타입 안전: OpenAPI → @hey-api/openapi-ts → packages/api-client 공유 SDK. 프론트는 SDK로만 호출(직접 fetch 금지).

**챗봇:**
- FastAPI 모듈러 모놀리스(도메인: auth, rooms, reservations, reviews, notifications, chatbot, admin). 챗봇은 API 통합 시작 → 부하 시 분리.

- 임베딩: OpenAI text-embedding-3-small 단일 고정(교체 시 전체 재임베딩).
- 인제스트: 디렉터리 배치(docs_corpus/) → 멱등 인제스트, 부분 실패 식별.
- 챗봇 세션: LangGraph checkpointer + thread_id(사용자×디바이스), 로그아웃 시 초기화.

**배포·운영:**

### Story 1.1: 외부 서비스 가입 & 사전 설정 (선행 — 사람이 직접 수행)

As a 프로젝트 소유자(KTH),
I want 구현 착수 전에 필요한 모든 외부 서비스에 가입하고 API 키·과금·플랫폼 등록을 직접 완료하고 그 값을 백엔드가 안전하게 로드하도록 정리하길,
So that 이후 모든 개발 스토리가 외부 의존성 차단 없이 진행되고, 키는 프론트에 노출되지 않고 백엔드에서만 관리된다(NFR-6).

> ⚠️ **이 스토리는 "사람만" 수행 가능한 선행 작업이다.** dev 에이전트는 (1) 체크리스트·`.env.example`·키 로딩 코드(`core/config`)를 준비하고, (2) 각 키가 실제로 로드/검증되는지 확인하는 역할만 한다. 가입·결제·플랫폼 등록 자체는 소유자가 직접 한다. **이 스토리가 완료되기 전 Phase 0 스파이크(1.3)·챗봇(E7)·배포는 시작할 수 없다.**

**필요 서비스 체크리스트 (소유자 수행):**
- **카카오 디벨로퍼스** — 계정 가입 → 애플리케이션 생성 → ① **JavaScript 키**(웹 지도용, 도메인 화이트리스트 등록), ② **REST API 키**(지오코딩/주소변환), ③ 카카오 공유용 **네이티브 앱 키** + iOS `LSApplicationQueriesSchemes`·URL 스킴 / Android **키 해시** 등록. **카카오맵 약관의 서비스 유형 적합성 1차 확인**(PRD §8 Q1).
- **OpenAI** — 가입 → **결제 수단 등록(API 과금 발생)** → API 키 발급. 용도: 기준 LLM 프로바이더(GPT, ≤2초 SLA 대상) + **임베딩 `text-embedding-3-small`**(단일 고정).
- **Anthropic** — 가입 → API 키 발급(과금). 용도: 멀티 LLM best-effort 프로바이더.
- **Google AI (Gemini)** — 가입 → API 키 발급(과금). 용도: 멀티 LLM best-effort 프로바이더.
- **Railway** — 가입 → **결제 등록(상시 컨테이너 과금)** → 프로젝트 생성. 비용 모니터링/알림 설정.
- **Apple Developer Program** ($99/년) — RN 앱 EAS 빌드·App Store 제출용. (앱 배포 시점 필요 — 가입은 선행 권장.)
- **Google Play Developer** ($25 1회) — RN 앱 Play 제출용. (동상.)
- **(스파이크 ③ 대상)** Supabase 등 pgvector 소스 인스턴스 — Railway 이관 검증용(해당 시).

**Acceptance Criteria:**

**Given** 위 체크리스트의 각 서비스
**When** 소유자가 가입·키 발급·과금 설정을 완료하면
**Then** 발급된 모든 키·시크릿이 백엔드 `apps/api/.env`에 채워지고, `apps/api/.env.example`에 키 이름·용도·획득 출처가 주석으로 문서화된다
**And** 어떤 키도 프론트 코드(`apps/web`·`apps/admin`·`apps/mobile`)나 저장소(.gitignore 적용)에 커밋되지 않는다.

**Given** `core/config`(pydantic-settings)
**When** 백엔드를 기동하면
**Then** 필수 키(카카오 REST·OpenAI·임베딩 모델명 등)가 누락된 경우 명확한 오류 메시지와 함께 기동이 실패한다(fail-fast)
**And** 모든 필수 키가 존재하면 정상 기동되고, 키 존재 여부를 확인하는 헬스/진단 출력이 제공된다.

**Given** 카카오맵 약관 적합성 확인
**When** 서비스 유형이 약관에 부적합으로 판명되면
**Then** 그 사실과 대안(지도 제공자 대체 검토) 필요성이 문서에 기록되어 후속 스토리(E3) 진행 판단에 반영된다.

**Given** 지도 JS SDK 도메인 키
**When** 키를 발급·설정하면
**Then** 허용 도메인이 화이트리스트로 제한되어, 프론트에 노출되는 유일한 키(도메인 키)도 타 도메인에서 무단 사용되지 않는다(NFR-6).

**Then** 단일 백엔드 발급 JWT(access 단기 + refresh 장기)가 발급되고, refresh 해시가 `refresh_tokens` 테이블에 저장되며, 토큰은 웹=httpOnly+Secure+SameSite 쿠키 / RN=SecureStore+Bearer 헤더로 보관된다(백엔드는 양쪽 추출 지원).

**Given** 잘못된 자격 증명
**When** 로그인을 시도하면
**Then** 401로 거부되고 안내가 표시된다.

**Given** access 토큰 만료
**When** 인증 요청이 401을 받으면
**Then** refresh로 1회 자동 재발급되고, 실패 시 로그인으로 유도된다.

**Given** 로그아웃
**When** 사용자가 로그아웃하면
**Then** 해당 refresh 토큰이 `refresh_tokens`에서 즉시 무효화되고 세션이 종료된다(챗봇 대화 종료 연동은 E7).

**Given** 역할(booker/provider/admin) 권한
**When** 권한이 없는 역할이 보호된 엔드포인트에 접근하면
**Then** 백엔드 의존성이 403 FORBIDDEN_ROLE로 최종 강제한다(프론트 라우트 보호는 보조). 역할은 가입 시 확정되어 전환할 수 없다.

### Story 1.9: OpenAPI→TS SDK 자동생성 파이프라인

As a 개발자,
I want 백엔드 OpenAPI 스키마에서 타입 안전 TS SDK를 자동 생성해 공유 패키지로 두길,
So that 세 프론트가 단일 계약으로 백엔드를 호출하고 계약 변경이 컴파일 타임에 드러난다.

**Acceptance Criteria:**

**Given** FastAPI가 노출하는 `openapi.json`(현재 auth 엔드포인트 포함)

**Then** `packages/api-client`에 타입 안전 클라이언트가 생성되고, 와이어 전 구간 snake_case가 변환 없이 유지된다.

**Given** 프론트 코드

**Then** 오직 `packages/api-client` SDK를 통해서만 호출하고 직접 fetch는 사용하지 않는다.

**Given** CI
**When** 백엔드 계약이 변경되었는데 SDK가 재생성되지 않으면
**Then** drift 검사가 실패해 계약-SDK 불일치를 차단한다.

## Epic 2: 제공자 공간 등록

제공자가 공간 1개를 5분 내 등록·수정해 지도/목록에 노출한다. rooms 도메인(요일별 영업시간 + 휴무 예외, 좌표 + 행정동 코드, is_active)이 확립되어 이후 탐색·예약의 데이터 전제가 된다. (FR-22 / UJ-3 전반부)

### Story 2.1: rooms 데이터 모델 & 영업시간/휴무 모델

As a 개발자,
I want 스터디룸·영업시간·휴무 예외 테이블과 슬롯 도출 규칙을 정의하길,
So that 등록된 공간으로부터 1시간 슬롯이 결정적으로 도출되고 탐색·예약이 이를 사용할 수 있다.

**Acceptance Criteria:**

**Given** rooms 도메인 마이그레이션
**When** `alembic upgrade head`를 실행하면
**Then** `rooms`(이름·시간당 금액·수용 인원·룸 형태·부대시설·lat/lng·행정동 코드·`is_active`), `business_hours`(room별 weekday 0~6, 같은 날 내 시작/종료), `holiday_exceptions`(room별 date) 테이블이 생성된다.

**Given** 한 룸의 영업시간·휴무·날짜
**When** 슬롯을 도출하면
**Then** 슬롯 = (요일별 영업시간 − 휴무일 − 이미 예약) 으로 계산되고, 고정 1시간 단위이며 영업시간은 같은 날 내(자정 넘김 없음)로 슬롯 소속 날짜가 모호하지 않다. (이미 예약 차감은 reservations가 존재하는 E4에서 연결되며, 그 전까지 예약 집합은 공집합으로 취급한다.)

**And** 모든 slot_start는 UTC로 저장되고 판정은 Asia/Seoul 기준이다(NFR-1, core/time 사용).

### Story 2.2: 공간 등록 & 주소 검색

As a 제공자,
I want 공간 정보를 입력하고 주소 검색으로 위치를 확정해 저장하길,
So that 등록 한 번으로 내 공간이 지도/목록에 노출된다 (FR-22, UJ-3, SM-5).

**Acceptance Criteria:**

**Given** 제공자 계정으로 로그인한 상태
**When** 이름·시간당 금액·영업시간·주소·수용 인원·룸 형태(개방형/독립룸)·부대시설(주차·빔프로젝터/TV·커피머신·화이트보드·wifi·기타 다중 선택)을 입력하고 저장하면
**Then** 룸이 생성되어 지도/목록에 노출되고, `is_active=true`로 시작한다.

**Given** 주소 검색(카카오 지오코딩)
**When** 주소를 선택하면
**Then** 좌표(lat/lng)와 행정동 코드가 함께 저장되어 반경 검색(FR-9)과 행정동 검색(FR-8)을 모두 지원한다.

**Given** 영업시간 입력
**When** 자정을 넘기는 시간(예: 22:00~02:00)을 입력하면
**Then** 같은 날 내(0~24시) 제약으로 거부된다.

**Given** 이미 룸을 1개 보유한 제공자
**When** 추가 등록을 시도하면
**Then** MVP 제약(제공자당 1개)으로 차단된다.

**And** 등록 흐름은 제공자가 5분 이내 마칠 수 있도록 설계되고, 저장 후 지도에서 자기 핀을 확인할 수 있다.

### Story 2.3: 공간 수정 & 영업시간 변경 독립성

As a 제공자,
I want 등록한 공간 정보와 영업시간을 수정하길,
So that 운영 상황에 맞춰 정보를 갱신하되 기존 확정 예약은 영향받지 않는다 (FR-22).

**Acceptance Criteria:**

**Given** 등록된 공간
**When** 정보·영업시간을 수정하면
**Then** 변경이 이후 도출되는 슬롯에 반영된다.

**Given** 영업시간을 축소해 기존 확정 예약의 슬롯이 도출 집합 밖으로 나가는 경우
**When** 변경을 저장하면
**Then** 이미 확정된 예약은 자신의 점유 슬롯(slot_start)을 자체 보유하므로 정상 표시·취소·후기 대상이 되고 영향받지 않는다.

**And** 스터디룸 삭제는 제공하지 않으며, 운영 중단은 계정 비활성(FR-31, E8)으로 갈음한다.

## Epic 3: 스터디룸 탐색 (지도·목록·즐겨찾기)

예약자가 첫 화면 지도에서 주변 룸을 핀(서버 집계 가용성 색)으로 보고, 바텀시트로 요약을 확인하며, 행정동 콤보·반경으로 목록을 훑고, 마음에 든 룸을 즐겨찾기한다. 위치 권한 거부 시 행정동 목록으로 우회한다. (FR-4~10 / UX-DR2·3·7·8·9)

### Story 3.1: 슬롯 도출 & 룸 가용성 집계 엔드포인트

As a 개발자,
I want 룸별 잔여 빈 슬롯 수를 서버에서 집계하는 엔드포인트를 두길,
So that 핀 색이 서버 집계값으로 결정되고 클라이언트가 핀마다 슬롯을 N회 계산하지 않는다 (FR-5 근거, NFR-2).

**Acceptance Criteria:**

**Given** 룸 집합
**When** 가용성 집계를 요청하면
**Then** 각 룸의 "오늘(룸 타임존) 현재시각 이후 남은 빈 슬롯 수"가 서버측에서 1회 계산되어 반환된다.

**Given** 슬롯 도출
**When** 집계를 계산하면
**Then** (영업시간 − 휴무일 − 예약) 규칙을 사용하며, reservations가 존재하지 않는 시점에는 예약 집합을 공집합으로 취급한다(예약 차감 연결은 Story 4.9에서 수행).

**And** 응답은 핀 색 결정에 충분한 집계값(잔여 빈 슬롯 수)만 제공하고, 클라이언트 N회 계산을 유발하지 않는다.

### Story 3.2: 첫 진입 지도 + 주변 핀 + 가용성 색/아이콘

As a 예약자,
I want 첫 화면에서 추가 조작 없이 주변 룸을 핀으로 보고 예약 가능 여부를 색·아이콘으로 즉시 인지하길,
So that 지금 내 근처 어디가 비는지 한눈에 안다 (FR-4, FR-5, UJ-2, UX-DR2, NFR-2).

**Acceptance Criteria:**

**Given** 위치 권한 허용
**When** 앱/웹을 열면
**Then** 현재 위치 주변 룸이 카카오맵 핀으로 표시되고, 첫 지도 표시는 양호 네트워크에서 ≤2초(p90) 시작된다.

**Given** Story 3.1의 가용성 집계
**When** 핀을 렌더하면
**Then** 오늘 남은 빈 슬롯 ≥1이면 예약 가능 색(`pin-available` 초록 + 체크 아이콘), 없으면 마감 색(`pin-full` 회색 + X 아이콘)으로 표시된다 — 색은 절대 단독 신호가 아니며 항상 아이콘을 병행한다.

**Given** 핀
**When** 핀을 탭하면
**Then** 해당 룸 바텀시트가 열린다.

**Given** 스크린리더
**When** 핀에 도달하면
**Then** "○○ 스터디룸, 예약 가능/마감"으로 읽힌다(NFR-5).

### Story 3.3: 바텀시트 요약

As a 예약자,
I want 핀/목록 항목에서 가격·영업시간·예약 가능 여부를 한눈에 보길,

**Acceptance Criteria:**

**Given** 핀/목록 항목 탭
**When** 바텀시트가 열리면
**Then** 1차 정보(가격·영업시간·예약 가능 여부)가 스크롤 없이 한 화면에 표시되고, 2차 정보(부대시설 등)가 그 아래 위계로 배치된다.

**Given** 바텀시트
**When** 사용자가 조작하면
**Then** 상세 페이지 이동과 즐겨찾기 토글이 가능하고, 그래버/드래그로 펼침·접힘이 동작한다.

**And** 바텀시트는 지도 위로 떠오르는 유일 면으로 그림자(`elevation.sheet`)·상단만 라운드(`rounded.xl`)를 가지며, 열릴 때 포커스가 진입하고 Esc로 닫히며 포커스 트랩이 적용된다(NFR-5).

### Story 3.4: 행정동 콤보 목록 조회

As a 예약자,
I want 시/군/구·동/읍/면을 선택해 해당 지역 룸을 목록으로 보길,
So that 특정 동네 전체를 훑어 모임 장소를 비교한다 (FR-8, UJ-1, UX-DR8).

**Acceptance Criteria:**

**Given** 행정동 콤보
**When** 시/군/구를 선택하면
**Then** 하위 동/읍/면 콤보가 갱신되고, 선택 지역의 룸이 목록(룸 카드/리스트 로우)으로 나열된다.

**Given** 목록 항목
**When** 항목을 조작하면
**Then** 상세 이동과 즐겨찾기 토글이 가능하다.

**Given** 선택 지역에 룸이 없는 경우
**When** 조회하면
**Then** "이 지역엔 등록된 곳이 없어요." 빈 상태가 인접 동/반경 확대 제안과 함께 표시된다(막다른 화면 금지).

### Story 3.5: 반경 검색

As a 예약자,
I want 현재 위치 기준 반경 내 룸을 찾고 반경을 조정하길,

### Story 3.6: 위치 권한 거부 우회

As a 위치 권한을 거부한 예약자,
I want 지도 대신 행정동 목록으로 즉시 우회하길,

**Given** 즐겨찾기 페이지
**When** 진입하면
**Then** 저장한 룸을 모아보고 상세로 이동할 수 있으며, 빈 상태는 "마음에 든 곳을 즐겨찾기해두면 여기 모여요."로 안내된다.

**Given** 참조 룸이 비활성(`is_active=false`)인 경우
**When** 즐겨찾기 목록에 표시되면
**Then** '비활성' 라벨이 표시되고 신규 예약 진입이 차단된다(막다른 화면 금지). (관리자 비활성 동작은 FR-31/E8이며, 본 스토리는 `room.is_active` 필드만 읽는다 — 순방향 의존 아님.)

### Story 3.8: 탐색 상태 매트릭스 마감

As a 예약자,
I want 탐색 화면의 빈/에러/오프라인/로딩 상태가 모두 설계되어 있길,
So that 어떤 상황에서도 막히지 않고 다음 행동을 안내받는다 (UX-DR9, NFR-5).

**Acceptance Criteria:**

**Given** 찾기(지도) 화면
**When** 주변 룸 없음/지도 로드 실패/오프라인/로딩 상황이면
**Then** 각각 "이 근처엔 아직 없어요"(반경 확대·행정동 전환) / "지도를 못 불러왔어요"(재시도·목록 우회) / 오프라인 안내(마지막 핀 캐시) / 지도 스켈레톤(≤2초) 상태가 표시된다.

**Given** 찾기(목록) 화면
**When** 결과 없음/조회 실패/오프라인/로딩 상황이면
**Then** 각각 빈 상태 안내 / 재시도 / 캐시 표시+오프라인 토스트 / Skeleton 행이 표시된다.

**And** 모든 마이크로카피는 친근한 해요체이며 막힘 뒤 항상 다음 행동을 제시한다(UX-DR10).

### Story 3.9: 첫 방문 온보딩 소개 오버레이

As a 처음 방문한 사용자,
I want 첫 진입에서 서비스를 간략히 소개받고 "지금 시작"을 누르면 다시는 그 안내가 뜨지 않길,
So that 처음엔 무엇을 하는 서비스인지 알고, 재방문부터는 마찰 없이 바로 탐색한다 (PRD "마찰 없는 흐름" 보강 — FR 외 신규, KTH 2026-06-15).

> 프론트엔드 전용(백엔드 API/FR 없음). 웹 + 앱 표면(관리자 웹 제외 — 운영자용). **실행 의존: Story 3.2(첫 진입 지도)** 이후 — 오버레이는 진입 화면(지도) 위에 뜬다. 번호는 비충돌 위해 3.9이나 실행은 3.2 직후 가능.

**Acceptance Criteria:**

**Given** 처음 방문(온보딩 플래그 없음)
**When** 탐색 진입 화면을 열면
**Then** 간략한 서비스 소개와 하단 행동 버튼("지금 시작" 등 — 문구는 카피 단계 확정)이 진입 화면 위 오버레이로 표시된다.

**Given** 온보딩 오버레이
**When** 사용자가 "지금 시작" 버튼을 누르면
**Then** 오버레이가 사라지고 바로 탐색(지도)으로 진입하며, **본 적 있음 플래그가 클라이언트 로컬에 영속**된다(웹=`localStorage`, 앱=AsyncStorage 등가물 — 쿠키 대신 서버 미전송 경량 플래그).

**Given** 플래그가 있는 재방문(같은 기기·캐시 유지)
**When** 진입하면
**Then** 오버레이는 **표시되지 않고** 곧장 탐색으로 진입한다. 캐시/스토리지 삭제·새 기기·시크릿 모드면 플래그가 없어 다시 1회 노출된다(로그인 무관·기기 로컬 — 서버에 "본 적 있음" 추적 없음).

**And** 오버레이는 접근성(포커스 진입·Esc/버튼으로 닫힘·포커스 트랩)과 reduced-motion을 준수하고(NFR-5·UX-DR12), 막다른 화면을 만들지 않는다(닫으면 항상 탐색으로).

## Epic 4: 예약 핵심 (상세·슬롯·즉시예약·취소·현황)

예약자가 상세 화면을 떠나지 않고 빈 연속 슬롯을 선택해 결제 없이 즉시 예약한다. 동시 예약은 정확히 하나만 성립하고(중복·부분 점유 0건), 시작 6시간 전까지 취소하며, 예약현황/히스토리에서 조회한다. (FR-11~17 / UX-DR4 / SM-1·2·3·4)

### Story 4.1: reservations 데이터 모델 & 동시성 제약·상태머신

As a 개발자,
I want 예약 테이블과 동시성 제약·상태 전이 머신을 정의하길,
So that 중복·부분 점유 0건과 원자·멱등 상태 전이가 데이터 계층에서 보장된다 (FR-15·16 기반, NFR-7).

**Acceptance Criteria:**

**Given** reservations 도메인 마이그레이션
**When** `alembic upgrade head`를 실행하면
**Then** `reservations`(예약 단위)와 슬롯 점유 행(`room_id`, `slot_start`, status)이 생성되고, `UNIQUE(room_id, slot_start)`(`uq_reservations_room_slot`) 제약이 적용된다.

**Given** 예약 상태(확정/취소/거절)
**When** 상태 전이를 수행하면
**Then** 전이는 원자적 단일 연산이고, 종료 상태(취소·거절)에 대한 추가 전이는 멱등하게 무시되고 현재 상태를 반환한다.

**And** 슬롯 점유/재활성은 상태 전이와 동일 트랜잭션 내에서만 일어난다.

### Story 4.2: 상세 정보 화면

As a 예약자,
I want 룸의 가격·부대시설·영업시간·주소·후기를 상세에서 보고 같은 페이지에서 예약을 전개하길,
So that 화면 이동 없이 정보 확인부터 예약까지 이어간다 (FR-11, UJ-1·2).

**Acceptance Criteria:**

**Given** 상세 화면
**When** 진입하면
**Then** 가격·부대시설·영업시간·위치·후기/평점이 위계를 갖고 표시된다(1차=가격·영업시간·예약가능, 2차=시설·주소, 3차=후기).

**Given** 상세 화면
**When** 예약을 시작하면
**Then** 별도 페이지 이동 없이 같은 페이지 내에서 예약 UI(달력+슬롯)가 전개된다(여백 우선 레이아웃).

**And** 상세 빈 상태는 N/A(룸이 있어야 진입)이며, 정보 로드 실패/오프라인/로딩 상태가 각각 설계된다(재시도·뒤로 / 읽기 캐시 / 스켈레톤).

### Story 4.3: 날짜·슬롯 가용성 표시

As a 예약자,
I want 달력에서 날짜를 고르고 그날의 빈 슬롯만 선택 가능한 형태로 보길,
So that 예약 가능한 시간만 명확히 구분해 고른다 (FR-12, UJ-1·2, NFR-1).

**Acceptance Criteria:**

**Given** 상세의 달력
**When** 날짜를 선택하면
**Then** 그날의 1시간 슬롯이 표시되고, 과거 날짜·당일 현재시각 이전 슬롯(룸 타임존 기준)·이미 예약된 슬롯은 비활성으로 표시된다.

**Given** 그날 빈 슬롯이 0개인 경우
**When** 날짜를 선택하면
**Then** "이 날은 다 찼어요. 다른 날을 골라보세요." + 다음 빈 날짜 제안이 표시된다.

### Story 4.4: 연속 슬롯 선택

As a 예약자,
I want 연속된 여러 슬롯을 한 번에 선택하길,
So that 실제 사용 길이에 맞춰 예약한다 (FR-13, UJ-1, UX-DR4, NFR-5).

**Acceptance Criteria:**

**Given** 슬롯 피커
**When** 연속 구간(예: 14~17시)을 드래그 또는 탭 연속으로 선택하면
**Then** 선택 범위가 `primary`로 강조되고, 비연속 선택은 시각적으로 불가능하게 막힌다(D1).

**Given** 선택 구간 중 하나라도 빈 슬롯이 아닌 경우
**When** 선택하면
**Then** 해당 구간 선택이 막힌다.

**Given** 키보드 사용자
**When** 슬롯을 선택하면
**Then** 방향키 이동 + Enter로 시작/끝을 지정하는 키보드 대안이 제공되고, 스크린리더에 "14시부터 17시까지 선택됨"이 피드백된다(드래그가 유일 경로가 아님).

### Story 4.5: 즉시 예약 (결제 없음)

As a 예약자,
I want 선택한 연속 슬롯을 결제 없이 즉시 확정하길,
So that 최소 단계로 예약을 끝낸다 (FR-14, UJ-1·2, SM-1·2).

**Acceptance Criteria:**

**Given** 빈 연속 슬롯이 선택된 상태
**When** 예약을 확정하면
**Then** 결제 단계 없이 즉시 확정되고 "예약이 완료됐어요!" 성공 피드백이 표시된다.

**Given** 단독 예약 흐름
**When** 상세 진입부터 예약을 완료하면
**Then** 3스텝 이내(스텝=별개 탭/입력, 스크롤·드래그는 1스텝)로 완료된다.

**And** 확정 예약은 선택한 모든 슬롯을 all-or-nothing으로 점유한다(일부만 잡힌 채 확정되는 상태 없음).

### Story 4.6: 중복 예약 방지 (동시성)

As a 예약자,
I want 같은 시간에 다른 사람과 동시에 눌러도 한쪽만 성공하고 실패해도 우아하게 안내받길,
So that 잘못 잡힐 걱정 없이 안심하고 예약한다 (FR-15, UJ-1 edge, NFR-7, SM-4).

**Acceptance Criteria:**

**Given** 동일 슬롯에 대한 동시 확정 시도
**When** 두 예약이 경합하면
**Then** 정확히 하나만 성공한다(슬롯 점유 중복 0건). 예약은 선택 슬롯 전체에 대해 단일 트랜잭션 multi-row INSERT(`ON CONFLICT` 후 affected=N 검증, 아니면 ROLLBACK)로 원자적으로 성립/실패한다.

**Given** 겹치되 시작 슬롯이 다른 두 연속 예약(14~17시 vs 16~18시)의 동시 확정
**When** 경합하면
**Then** 한쪽 예약 전체가 성립하고 다른 쪽은 단 하나의 슬롯도 점유하지 않는다(부분 점유 0건).

**Given** 실패한 쪽
**When** 충돌이 반환되면
**Then** 409 SLOT_CONFLICT가 "앗, 방금 다른 분이 먼저 잡았어요. 가까운 빈 시간을 다시 보여드릴게요."로 안내되고 인접 빈 슬롯이 즉시 재표시된다(에러코드 노출 금지).

**And** 동시성 불변식은 자동화 테스트(동일 슬롯 + 겹치는 연속 구간)로 검증된다(SM-4).

### Story 4.7: 예약 취소

As a 예약자,
I want 예약 시작 6시간 전까지 예약을 취소하길,
So that 계획이 바뀌어도 슬롯이 다른 사람에게 다시 열린다 (FR-16, UJ-2, NFR-7).

**Acceptance Criteria:**

**Given** 확정 예약
**When** 예약 시작 6시간 전(룸 타임존 기준) 이내가 되면
**Then** 취소 버튼이 비활성되고 "이제 6시간이 안 남아서 취소가 어려워요." 안내가 표시된다.

**Given** 시작 6시간 전 이전
**When** 취소하면
**Then** 점유했던 모든 슬롯이 동일 트랜잭션 내에서 즉시 재활성되고, 예약은 히스토리에 취소 상태로 남는다.

**Given** 이미 종료 상태(취소·거절)인 예약
**When** 추가 취소를 시도하면
**Then** 멱등하게 무시되고 현재 상태가 반환된다.

### Story 4.8: 예약현황 / 히스토리 (예약자)

As a 예약자,
I want 다가오는 예약과 지난 예약을 한 곳에서 보길,
So that 내 예약을 관리하고 취소·상세로 이어간다 (FR-17, UJ-1·2).

**Acceptance Criteria:**

**Given** 예약현황 화면
**When** 진입하면
**Then** 다가오는 예약과 지난(이용 완료/취소/거절) 예약이 구분되어 표시되고, 각 예약에서 룸 상세로 이동할 수 있다.

**Given** 취소 가능한 예약
**When** 목록에서 조작하면
**Then** 취소 동작(FR-16)으로 연결된다.

**And** 빈 상태("아직 예약이 없어요. 마음에 드는 곳을 찾아볼까요?" → 찾기 링크)·에러·오프라인·로딩 상태가 설계된다.

### Story 4.9: 가용성 집계에 예약 차감 연결

As a 개발자,
I want Story 3.1의 가용성 집계와 Story 4.3의 슬롯 표시가 실제 예약을 차감하도록 연결하길,
So that 핀 색·슬롯 가용성이 실제 점유를 정확히 반영하고 예약 직후 갱신된다 (FR-5·12 완성, NFR-2).

**Acceptance Criteria:**

**Given** reservations 테이블에 점유 행이 존재
**When** 가용성 집계(3.1)와 슬롯 도출을 계산하면
**Then** (영업시간 − 휴무일 − 예약) 규칙에서 예약된 슬롯이 차감되어 핀 색·슬롯 비활성에 정확히 반영된다.

**Given** 예약 확정·취소·거절 뮤테이션
**When** 성공하면
**Then** 관련 TanStack Query 키(`['room', roomId, 'slots', date]`·핀 집계)가 invalidate되어 가용성이 즉시 갱신된다.

## Epic 5: 예약 후 경험 (알림·공유·후기)

예약자가 접속 시 도래 리마인드(24h)·상태변경 통지 배너를 받고, 확정 예약을 카카오톡으로 한 번에 공유하며, 이용 완료 예약에 별점·후기를 남기고 제공자가 답글한다. (FR-18, 18a, 19, 20, 21 / UX-DR5)

### Story 5.1: notifications 모델 & 인앱 배너 인프라

As a 개발자,
I want 인앱 배너의 데이터 모델과 전역 배너 컴포넌트를 두길,
So that 도래 리마인드·상태변경 통지가 접속 시점에 표시되고 사용자별로 영속 관리된다 (UX-DR5, NFR-5).

**Acceptance Criteria:**

**Given** `notifications` 테이블과 전역 배너 컴포넌트
**When** 접속 시 표시할 배너가 있으면
**Then** 면 위 플랫(`secondary`/`muted` 배경 + 아이콘 + 텍스트)으로 표시되고, 출현이 aria-live로 안내된다(푸시 아님 — 접속 시점에만).

**Given** 배너 상태
**When** '다시 보지 않기' 또는 '확인'을 하면
**Then** 사용자별 상태가 영속되어 해당 배너 규칙대로 더는/소멸 처리된다.

### Story 5.2: 예약 도래 리마인드 배너

As a 예약자,
I want 예약이 임박하면 접속 시 배너로 안내받길,
So that 약속을 잊지 않는다 (FR-18, UJ-1).

**Acceptance Criteria:**

**Given** 시작 24시간 이내(정확히 24시간, 달력일 아님)의 확정 예약
**When** 접속하면
**Then** 도래 리마인드 배너가 표시된다(웹·앱 동일).

## Epic 7: 챗봇 "룸메이트" (RAG·예약 검색·멀티 LLM·스트리밍)

### Story 7.1: 멀티 LLM 어댑터 레이어

As a 개발자,
I want OpenAI/Google/Anthropic의 요청/응답/스트리밍/에러 스키마를 공통 인터페이스로 정규화하는 어댑터를 두길,
So that 프로바이더를 전환해도 공통 5종 기능이 동일하게 동작한다 (FR-29, NFR-2).

**Acceptance Criteria:**

**Given** `chatbot/llm/adapters/{openai,anthropic,google}.py` + base
**When** 설정으로 프로바이더/모델을 전환하면
**Then** 공통 5종(① 단일·멀티턴 채팅, ② 시스템 프롬프트, ③ 2개 툴콜[문서검색·예약DB], ④ 토큰 스트리밍, ⑤ 범위 밖 거절)이 동일하게 동작한다.

**Given** 프로바이더별 고유 샘플링 파라미터(temperature 등)
**When** 정규화하면
**Then** 공통 표면에서 정규화하지 않고(Out of Scope), 요청/응답/스트리밍/에러 차이만 어댑터가 흡수한다(if 분기 금지).

**When** 인제스트를 실행하면

**Given** 재인제스트
**When** 동일 문서를 다시 처리하면

**Then** 7.2의 파이프라인으로 문서가 인제스트되어 챗봇 검색에 반영된다(UI 업로드 아님 — 디렉터리 배치 후 트리거).

**Given** 재인제스트
**When** 동일 문서를 다시 처리하면
**Then** 멱등하게 동작해 중복 벡터를 생성하지 않는다.

**Given** 인제스트 처리
**When** 완료되면
**Then** 성공/실패 상태를 확인할 수 있고, 부분 실패 시 어떤 문서가 실패했는지 식별된다.

## Epic 9: 모바일 앱 화면 패리티

웹에 구현된 전 화면을 Expo/React Native 모바일 앱에 기능 동등(parity)으로 구현한다. 디자인·구조·인터랙션은 웹과 동일 구성으로 재현하되, RN 특성(네이티브 네비·제스처·세이프에어리어·secure-store Bearer 인증·WebView 지도·react-native-sse)을 반영한다. 화면 기준 통합검증(Playwright, 환경변수 게이팅 세션주입)으로 마감한다. (FR-18·NFR-4 / UX 디자인토큰 `@desknow/ui` 공유 / 모바일 일괄 개발 결정 2026-06-19 / 신규 결정: secure-store 인증·WebView 카카오맵·E2E 세션주입)

### Story 9.1: 모바일 인증 + 탐색·검색

As a 예약자,
I want 모바일 앱에서 로그인/가입하고 지도·목록으로 스터디룸을 탐색·검색·즐겨찾기 하길,
So that 웹과 동일한 첫 진입 경험을 모바일에서도 누린다 (FR-18, FR-1·2 / Epic 3 표면).

> 이 스토리는 파운데이션을 포함한다: ① secure-store+Bearer 인증 배선(세션 프로바이더·토큰 저장·SDK `client` 헤더 주입) ② WebView 카카오맵 지도 컴포넌트(postMessage 핀↔시트 브릿지) ③ E2E 세션주입 하니스(`EXPO_PUBLIC_E2E_*`/`__DEV__` 게이팅, 프로덕션 빌드 제외) — 9.2/9.3가 재사용.

**Acceptance Criteria:**

**Given** 모바일 앱
**When** 이메일·비밀번호로 로그인/가입(역할 선택)하면
**Then** secure-store에 토큰이 저장되고 이후 SDK 호출에 Bearer 헤더가 주입되며, 앱 재시작 시 세션이 복원된다.

**Given** 첫 진입 화면
**When** 지도/목록을 전환하면
**Then** WebView 카카오맵이 웹과 동일한 핀·가용성 색으로 표시되고, 핀 탭 시 네이티브 바텀시트 요약이 올라온다(앱 전체 WebView 아님).

**Given** 지역(2단)·반경 검색과 즐겨찾기
**When** 조회·토글하면
**Then** 웹과 동일 구성으로 결과가 표시되고(FlatList 무한스크롤), 막다른 화면 5상태(로딩/에러/빈/단절/미로그인) 매트릭스가 일관 처리된다.

**Given** 온보딩 오버레이(기존 3.9 구현)
**When** 첫 방문이면
**Then** 표시되고 "다시 보지 않기"만 AsyncStorage로 영속된다.

### Story 9.2: 모바일 예약 + 예약 후 경험

As a 예약자,
I want 룸 상세에서 슬롯을 골라 즉시 예약하고 예약현황 확인·취소·공유·후기까지 하길,
So that 예약 전 과정을 모바일에서 완결한다 (FR-18 / Epic 4·5 표면).

**Acceptance Criteria:**

**Given** 룸 상세 화면
**When** 진입하면
**Then** 웹과 동일한 3단 정보 위계·미니지도(WebView)·예약 전개가 네이티브로 표시되고, 날짜·슬롯 가용성과 연속 슬롯 선택이 동작한다.

**Given** 슬롯 선택 후
**When** 즉시 예약(결제 없음)하면
**Then** 중복 예약이 차단되고(409) 예약이 확정되며, 카카오 공유 진입점이 노출된다.

**Given** 예약현황 화면
**When** 조회하면
**Then** 다가오는/지난으로 분할 표시되고, 취소(리드타임 게이팅)·후기 작성이 동작하며, 인앱 배너가 접속 시점 GET으로 표시된다.

### Story 9.3: 모바일 provider + 챗봇 + E2E 통합검증

As a 제공자/예약자,
I want 제공자 화면(예약자현황·거절·후기답글·룸등록)과 플로팅 챗봇을 모바일에서 쓰고 전 화면이 실검증되길,
So that 제공자 운영과 챗봇까지 포함해 모바일 MVP가 완성된다 (FR-18 / Epic 6·7 표면).

**Acceptance Criteria:**

**Given** 제공자 계정
**When** provider 화면에 진입하면
**Then** 예약자현황(익명 라벨)·예약 거절·후기 답글·룸 등록(지오코딩·영업시간)이 웹과 동일 구성으로 네이티브 동작하고, 역할 가드가 적용된다.

**Given** 플로팅 챗봇 FAB
**When** 열고 메시지를 보내면
**Then** `react-native-sse`로 응답이 스트리밍되고 세션 대화가 유지되며, RAG 안내·자연어 예약 검색·범위 밖 질문 거절이 동작한다.

**Given** 9.1의 E2E 세션주입 하니스
**When** Playwright로 Expo Web 빌드를 구동해 9.1~9.3 전 화면을 검증하면
**Then** 핵심 플로우가 통과하고, **프로덕션 빌드에는 주입 코드가 포함되지 않는다**(환경변수 게이팅 확인).

### Story 9.4: 모바일 시각·인터랙션 패리티 정정

As a 예약자/제공자,
I want 모바일 전 화면이 진짜 웹앱과 시각·인터랙션까지 동등해지길,
So that 9.1~9.3이 가렸던 셸·컴포넌트·온보딩 드리프트가 제거되어 웹·앱 이질감이 사라진다 (FR-18·NFR-4 / audit 2026-06-19 정정).

> 배경: 9.1~9.3은 기능을 포팅했으나 `:3000`·`:3001`이 둘 다 Expo라 **진짜 웹과 대조된 적이 없었다.** 본 스토리는 웹(`:3002`)을 정본으로 전 화면 드리프트를 정정한다. 입력 = `parity-audit-web-vs-mobile-2026-06-19.md`(증거 스크린샷 `parity-audit-2026-06-19-shots/`).

**Acceptance Criteria:**

**AC1 — 시스템성 셸 패리티 (S1/S2)**
**Given** 모바일 전 화면
**When** 렌더하면
**Then** ① 글로벌 헤더(좌 DeskNow 로고 + 우 인증/화면 컨트롤)가 웹과 동일하게 모든 화면 상단에 노출되고(SYS-1), ② **인증화면(로그인·가입) 포함** booker 하단 탭 바(아바타+스터디룸찾기/예약현황/즐겨찾기)가 웹과 동일하게 노출되며(SYS-2, 웹=정본 확정), ③ provider 하단 탭 순서·라벨이 웹과 동일 `[스터디룸, 예약자 현황, 후기]`이고(SYS-4), ④ 챗봇 FAB 아이콘이 웹과 동일한 💬 말풍선(원형)이다(SYS-3).

**AC2 — 온보딩 4슬라이드 캐러셀 (S1)**
**Given** 첫 방문 홈 진입
**When** 온보딩 오버레이가 뜨면
**Then** 웹 `OnboardingOverlay`의 `SLIDES[4]`와 동일한 4장 캐러셀(① 환영 ② 지도와 목록으로 찾기 ③ 원하는 시간에 바로 예약 ④ 궁금하면 룸메이트 챗봇)이 점 인디케이터·이전/다음·X와 함께 표시되고(ONB-1·2), "다음"→마지막 "시작하기" CTA·**"다시 보지 않기" 영속 dismiss**(AsyncStorage)가 웹과 동일 동작하며(ONB-3·4), 환영 본문 문구도 웹과 일치한다(ONB-5).

**AC3 — provider 룸 폼 입력 컨트롤 (S1/S2)**
**Given** provider 룸 등록/수정 폼
**When** 영업시간·룸 형태를 입력하면
**Then** 영업시간 시간 입력이 웹과 동일한 **12시간제 표기(오전/오후)** 로 표시되고(ROOM-1, 웹 `<input type=time>` 정본), 룸 형태 세그먼트가 웹과 동일한 **반폭 테두리 박스 2개(개방형|독립룸)** 구성이며(ROOM-2), 하단 탭바가 폼 하단 콘텐츠(부대시설 칩/저장)를 가리지 않는다(ROOM-5).

**AC4 — 화면별 S2/S3 시각 정렬**
**Given** 각 화면(로그인·가입·홈·챗봇·provider후기·룸상세 등)
**When** 렌더하면
**Then** 로그인 폼 상단정렬·필드 라벨 주황색(LOGIN-1·2), 가입 콘텐츠 상단정렬(SIGNUP-2), 홈 제목·부제 "내 주변 스터디룸"+부제(HOME-2), 챗봇 입력바 시트 최하단 고정·시트 높이(CHAT-1·2·4), 후기 "사장님 답글" 박스 복숭아빛 틴트(PREV-2), 룸상세 "‹ 뒤로" 인앱버튼 처리(DET-1) 등 audit의 **S2·S3 항목 전수**가 웹 스크린샷과 일치한다.

**AC5 — 미검증 인터랙션 전수 쌍비교 (audit TODO 9건)**
**Given** 9.1의 E2E 세션주입 하니스 + 웹(`:3002`)·모바일 동시 기동(**포트 정체 확인 선행**)
**When** 다음을 웹↔모바일 쌍으로 실행하면 — 가입 역할 토글 강조 · 홈 목록탭/지역(2단)콤보/반경/핀탭 요약 바텀시트 · **룸상세 예약 플로우**(캘린더·슬롯 그리드·연속 선택·예약 요약) · 예약 취소 · provider 거부 2단 확인 · 후기 답글 작성폼 · 챗봇 전송→스트리밍 버블 · 즐겨찾기 해제 토글
**Then** 각 결과 화면이 웹과 시각·동작 동등하다.

**Given** audit의 S1·S2·S3 + 미검증 인터랙션 전 항목
**When** fix 후 재촬영하면
**Then** 화면별 웹↔모바일 스크린샷 쌍을 스토리에 첨부하고 KTH 승인을 받는다(audit S1·S2·S3 전부 클리어).
