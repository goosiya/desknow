## 예약/슬롯 모델 (아키텍처에서 확정)
- 고정 1시간 슬롯. **단일 슬롯 중복은 `UNIQUE(room_id, slot_start)`로 원자적 차단.**
- **연속 N슬롯 예약의 원자성(리뷰어 C-1 반영):** 한 예약 = 연속 N개 슬롯. N개 행 삽입을 **단일 트랜잭션**으로 묶어 all-or-nothing 보장 — 예: 단일 multi-row INSERT(필요 시 `ON CONFLICT DO NOTHING` 후 affected rows = N 검증, 아니면 ROLLBACK). 한 슬롯이라도 충돌하면 예약 전체 실패, **부분 점유 0건**. ⚠️ "FOR UPDATE·SERIALIZABLE은 과설계" 단정은 **단일 슬롯에 한함** — 멀티슬롯 원자성은 트랜잭션 경계 설계가 결정한다.
- 비연속 불허(D1).
- **시간/타임존(리뷰어 C-2 반영):** slot_start는 UTC 저장, "오늘/현재/N시간 전" 판정은 룸 소재지 타임존(MVP=Asia/Seoul). 영업시간은 같은 날 내(자정 넘김 미지원, PRD §5)로 제한해 슬롯 소속 날짜 모호성 제거.
- **상태 전이 머신(리뷰어 C-3 반영):** 예약 상태(확정→취소/거절)는 원자적 단일 연산. 종료 상태(취소·거절)에 대한 추가 전이는 멱등하게 무시하고 현재 상태 반환. 슬롯 재활성은 상태 전이와 **동일 트랜잭션** 내에서만. 6시간 경계·재활성 직후 점유 레이스를 이로써 결정적으로 닫는다.
- **확정 예약의 영업시간 독립성(리뷰어 H-6 반영):** 확정 예약은 점유 슬롯의 slot_start를 **자체 보유**하여 영업시간 도출과 독립적으로 존재. 영업시간 축소로 도출 집합 밖으로 나간 기존 예약도 정상 표시·취소·후기 대상.
- 가변/겹침 슬롯이 도입되면(향후) `EXCLUDE USING gist` 또는 PG18 `WITHOUT OVERLAPS`로 확장.
- pgvector(챗봇)와 예약 테이블은 동일 PostgreSQL 공존 가능.
- 영업시간/휴무 모델은 슬롯 도출의 입력 — 요일별 영업시간·임시 휴무 반영 방식 미확정(PRD §8 Q7).

## 계정/권한 (구현 유의)
- **역할 고정(리뷰어 H-5 반영):** 한 이메일=한 역할, 가입 시 확정·전환 불가. 한 사람이 양쪽을 시연하려면 2계정. 향후 겸용 허용은 별도 결정.
- **비활성 캐스케이드(리뷰어 H-9 반영):** 제공자 비활성 → 룸 노출 중단·신규 예약 차단·기존 확정 예약 유지. 룸 삭제는 미지원(비활성 갈음). 즐겨찾기/히스토리의 비활성 참조 항목은 '비활성' 라벨 + 진입 차단.

## 지도/주소 (구현 유의)
- **카카오맵 우선**(네이버 무료정책 재편 리스크). 최종 채택 전 카카오 약관의 서비스 유형 적합성 1차 확인(PRD §8 Q1).
- **좌표 + 행정동 코드 둘 다 저장** → 반경/행정동 두 검색 UX 지원(FR-9, FR-8).

## 인증/세션 (구현 유의)
- 단일 백엔드 발급 JWT를 웹/앱 공용. access(단기)+refresh(장기). 즉시 무효화 위해 **서버측 무효화 저장소(Redis 또는 DB)** 사실상 필수 → 선택은 아키텍처 단계(PRD §8 Q2).
- 추출 이원화: **웹=쿠키 / 앱=헤더** 양쪽 지원 의존성(흔히 빠뜨림).
- 권장 라이브러리: PyJWT + pwdlib(Argon2). RBAC는 토큰 role 클레임 + 백엔드 의존성, **최종 강제는 백엔드**.

## 챗봇 (구현 유의)
- 역할 2종 동등: 문서/FAQ RAG 안내 + 자연어 예약 검색(예약 DB 툴). 결과는 상위 3 + 더보기(FR-27). 응답은 3분기(근거있음=답 / 근거없음=모름 / 범위밖=거절, FR-26).
- **멀티 LLM = 어댑터 레이어, '파라미터 분기' 아님(리뷰어 H-1 반영):** 프로바이더별로 sampling 파라미터(`temperature`/`top_p`/`top_k`)·assistant prefill·thinking 설정의 지원이 다르며, 일부 현행 모델은 이를 전달하면 요청을 거부할 수 있다. 따라서 단순 if 분기가 아니라 **요청/응답/스트리밍/에러 스키마를 공통 인터페이스로 정규화하는 어댑터 레이어**가 필요. 정규화 범위 = FR-29의 공통 5종(채팅·시스템프롬프트·2툴콜·스트리밍·범위밖 거절). 모델 고유 파라미터는 정규화 대상 아님. (정확한 모델별 파라미터 지원은 구현 시 각 프로바이더 최신 문서로 실측 — claude-api 등.)
- 스트리밍 = SSE 권장. **RN은 네이티브 EventSource 없음** → `react-native-sse`/fetch-event-source(인증 헤더·POST 바디 필요 시 fetch 기반). 프로바이더별 TTFT 편차로 가장 느린 프로바이더가 ≤2초(FR-30) 못 맞출 수 있음 → 기준 프로바이더 정책(PRD §8 Q5).
- **임베딩 모델 단일 고정**(교체 시 전체 재임베딩 — PRD §8 Q3). 인제스트는 디렉터리 배치 기반(UI 업로드 아님).
- **인제스트 멱등성(리뷰어 M-5 반영):** 재인제스트 시 동일 문서 중복 벡터 미생성, 부분 실패 시 실패 문서 식별, 디렉터리 제거 문서의 벡터 동기화 정책 확정 필요.

## 웹/앱 ↔ FastAPI REST (구현 유의 — 리뷰어 GAP 반영)
- OpenAPI → 타입 안전 TS SDK 자동생성(`@hey-api/openapi-ts` 등), 모노레포 공유 패키지화.
- 흔한 함정: ① **CORS** — 웹은 origin 등록 필수, RN은 미적용("웹에서만 깨짐"), ② **API 버저닝**(`/api/v1`) — 모바일 구버전 장기 잔존 대비 하위 호환, ③ 토큰 저장 차이(웹 쿠키 vs RN SecureStore), ④ RN의 `localhost` 접근 불가(에뮬레이터 별도 IP).

## 카카오톡 공유 (구현 유의 — 리뷰어 GAP 반영)
- 웹(Next.js): Kakao JS SDK `Kakao.Share.sendDefault()` + Feed/Text 템플릿(무료·검수 불필요).
- **RN(Expo): `@react-native-kakao/share` — 네이티브 모듈이라 Expo Go 불가 → EAS development build/prebuild 필요.** iOS Info.plist(URL 스킴·LSApplicationQueriesSchemes)·Android 키 해시 사전 등록 선행. 카카오 디벨로퍼스 앱 등록 필요.

## 아키텍처 방향 (기술조사 권고, 아키텍처 단계 확정)
- FastAPI **모듈러 모놀리스** + 챗봇은 같은 레포·별도 서비스 배포(장시간 스트리밍 부하 격리).
- Turborepo + pnpm 모노레포(사용자 웹 / 관리자 웹 / RN) + FastAPI 디렉터리, OpenAPI → 타입 안전 TS SDK 자동생성.
- 관리자 웹은 별도 Next.js 앱으로 분리(공격 표면 축소).

## 비용/운영 (PRD §11 Constraints 상세)
- Railway 상시 컨테이너 과금 — 서비스 수 = 비용. 비용 모니터링/알림 필수. 초기 챗봇을 API에 통합 후 부하 시 분리하는 단계적 접근 고려.
- 외부 연동 비용 0원(카카오 공유·EAS Free). 별도: Apple $99/년, Google Play $25(1회).
