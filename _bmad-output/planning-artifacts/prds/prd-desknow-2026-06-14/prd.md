# PRD: DeskNow
*Working title — 확정됨.*

## 0. Document Purpose

이 PRD는 DeskNow MVP를 만드는 사람(개발자 본인)과 다운스트림 워크플로우(UX 설계 `bmad-ux`, 아키텍처 `bmad-create-architecture`, 에픽/스토리 `bmad-create-epics-and-stories`)를 위한 기능 계약서다. 어휘는 §3 Glossary에 고정하고, 기능은 묶음(§4 Features)으로 그루핑하되 그 아래 전역 번호 FR로 중첩한다. 추론은 `[ASSUMPTION]` 태그로 인라인 표기하고 §9에 색인한다. 구현 방식(기술 스택·라이브러리·전송 방식)은 PRD가 아니라 **제품 브리프의 애드덤**(`_bmad-output/planning-artifacts/briefs/brief-desknow-2026-06-14/addendum.md`)과 **기술 타당성 조사**(`_bmad-output/planning-artifacts/research/technical-desknow-mvp-tech-feasibility-research-2026-06-13.md`), 그리고 본 PRD의 동반 `addendum.md`에 둔다. 이 PRD는 그 자료들 위에 세워지며 중복하지 않는다.

## 1. Vision

DeskNow는 네이버·인스타그램·개별 업체 사이트로 흩어진 스터디룸 탐색을 **지도 한 장과 목록 한 곳에 모으고, 빈 시간을 결제 단계 없이 바로 잡게** 해주는 양면 예약 서비스다. 예약자는 *위치·예약 가능 여부·가격*을 한눈에 비교하고, 제공자는 등록 한 번으로 빈 시간을 손님 앞에 내놓는다.

**문제와 통찰.** 스페이스클라우드·네이버 예약처럼 인접 서비스가 이미 있으므로 DeskNow가 푸는 문제는 *"예약할 곳이 없다"*가 아니다. 사람들은 지금도 네이버·인스타·개별 사이트·오프라인 간판을 오가며, 정작 알고 싶은 세 가지(① 근처에 어디가, ② 그 시간에 비는지, ③ 얼마인지)를 한자리에서 비교하기 번거로워 "그냥 늘 가던 곳"으로 타협한다. DeskNow가 메우는 건 **흩어진 탐색을 한 곳에 모으고 "한눈에" 비교하게 하는 경험적 빈틈**이다. 그래서 승부처는 새로운 기능이 아니라 **완성도**다. 디자인 원칙은 하나의 영웅 화면이 아니라 "예약에 이르는 흐름 전체에 마찰이 없을 것"이며, 같은 예약을 *더 매끄럽게* 해내는 데 집중한다. 어느 화면에서든 부르는 RAG 챗봇이 서비스 안내와 자연어 예약 검색을 함께 돕는다.

DeskNow는 포트폴리오로 출발하되 *언젠가 진짜 제품이 될 수 있다*는 가정 위에서 만든다. 성공의 잣대도 **좋은 제품 경험**과 **만든 사람의 풀스택 역량 증명**을 동등하게 둔다. 첫 버전은 결제·푸시·OAuth 같은 무거운 장치를 의도적으로 미루고, 찾기→예약→공유→문의의 핵심 흐름을 흠 없이 완성하는 데 집중한다. 합격선은 거창하지 않다 — **"내가 직접 쓰고 싶은 수준의 예약 경험을 끝까지 완성하는 것."**

## 2. Target User

DeskNow는 **예약자**와 **제공자**를 동등한 비중으로 위한다. 예약자는 다시 총무형·개인형 두 유형으로 나뉘며, 둘을 동등하게 다룬다.

### 2.1 Jobs To Be Done

- **(예약자·총무)** "이번 주 우리 모임, 어디서 볼까?"를 빠르게 정하고 일행에게 **자신 있게 공유·확정**하고 싶다 — 잘못 잡으면 본인이 곤란하므로 *예약 가능의 확실함*과 *과정의 단순함*이 핵심.
- **(예약자·개인)** 지금 또는 가까운 시일, 근처의 조용한 빈 방을 **최소 탭으로 빠르게** 혼자 잡고 싶다.
- **(제공자)** 여러 창구에 수동 홍보하는 대신, **등록 한 번으로 지도에 노출되고** 들어온 예약을 가볍게 확인·관리하고 싶다.
- **(막힐 때)** 규정이 궁금하거나 조건에 맞는 방을 찾고 싶을 때, 화면을 떠나지 않고 **바로 물어** 답을 얻고 싶다.
- **(만든 사람)** 사용자 웹·관리자 웹·모바일 앱이 단일 백엔드로 end-to-end 동작하는 **풀스택 역량을 증명**하고 싶다.

### 2.2 Non-Users (v1)

- 결제·정산·세금계산서가 필요한 정식 거래 사업자 운영(MVP는 결제 없음).
- 다중 공간·좌석 유형을 운영하는 대형 제공자(MVP는 제공자당 공간 1개).
- 미접속 중에도 푸시 알림을 받아야 하는 사용자(MVP는 접속 시 인앱 배너만).
- **관리자는 사용자 페르소나가 아니다** — 시드 관리자가 운영 역할로만 존재한다. 따라서 관리자·횡단 정책 성격의 FR(FR-3, FR-29, FR-31~33)은 어떤 UJ도 realize하지 않으며, 이는 누락이 아니라 의도다.

### 2.3 Key User Journeys

*명명된 페르소나가 실제로 겪는 시나리오. UJ-1 ~ UJ-4. FR이 인라인으로 참조한다("realizes UJ-X").*

- **UJ-1. 총무 "지우"가 다음 주 강남 모임 장소를 잡아 단톡방에 공유한다.**
  - **Persona + context:** 사이드 프로젝트 모임의 총무. 단톡방에 "다음 주 강남 어디서 볼까?"가 떴고, 잘못 잡으면 본인이 곤란하다.
  - **Entry state:** 모바일 앱, 로그인 상태. 약속은 6일 뒤.
  - **Path:** 목록 탐색에서 **시/군/구·동/읍/면 콤보**로 "강남구 역삼동" 선택 → 후보 목록에서 가격·시설 비교 → 한 곳을 골라 상세에서 달력으로 그날 날짜 선택 → **14~17시 연속 슬롯** 드래그 선택 → 즉시 예약.
  - **Climax:** 즉각적 성공 피드백 후, **카카오톡 공유** 한 번으로 일행에게 장소·시간이 한 번에 전달된다.
  - **Resolution:** 예약현황에 확정 예약이 남고, 약속 24시간 전 접속 시 인앱 배너로 리마인드된다.
  - **Edge case:** 슬롯 선택 직후 같은 시간을 다른 사람이 먼저 확정하면, 당황 없는 안내와 함께 인접 빈 슬롯을 다시 보여준다.

- **UJ-2. 개인 "민재"가 지금 당장 근처 빈 방을 혼자 잡는다.**
  - **Persona + context:** 카페는 시끄럽고 집은 산만한 개인. 조율할 일행이 없으니 속도가 곧 만족.
  - **Entry state:** 앱을 막 열었고 위치 권한 허용. 지금부터 2시간 집중하고 싶다.
  - **Path:** 첫 화면 지도에 내 주변 룸이 핀으로 뜸 → **초록 핀(오늘 빈 슬롯 있음)** 하나를 탭 → 바텀시트에서 가격·영업시간 확인 → 상세로 들어가 오늘 날짜의 빈 2개 연속 슬롯 선택 → 즉시 예약.
  - **Climax:** **상세 진입부터 3스텝 이내**로 예약이 끝나고 성공 피드백을 받는다.
  - **Resolution:** 예약현황에 오늘 예약이 보이고, 6시간 전까지 취소 가능.
  - **Edge case:** 위치 권한을 거부하면 지도 대신 행정동 목록 조회로 즉시 우회한다.

- **UJ-3. 제공자 "현우"가 공간을 등록해 지도에 올리고 예약을 받는다.**
  - **Persona + context:** 스터디룸 한 곳을 운영하며 빈 시간을 채우고 싶다. 가벼운 운영만 기대한다.
  - **Entry state:** 제공자 계정으로 로그인, 첫 등록.
  - **Path:** 공간 정보(이름·시간당 금액·영업시간·주소·수용 인원·룸 형태·부대시설) 입력 → 주소 검색으로 지도 위치 확정 → 저장하면 지도에 노출 → 예약현황 화면에서 들어온 예약을 확인.
  - **Climax:** **5분 이내**에 등록을 마치고 지도에서 자기 핀을 확인한다.
  - **Resolution:** 들어온 예약을 보고, 불가피하면 거절(해당 슬롯 재활성), 후기에는 답글로 응대.

- **UJ-4. 누구든 막히면 챗봇에게 묻는다.**
  - **Persona + context:** 규정이 궁금하거나 조건에 맞는 방을 빠르게 찾고 싶은 사용자.
  - **Entry state:** 어느 화면이든. 플로팅 챗봇 아이콘이 늘 떠 있다.
  - **Path:** 아이콘 탭 → *"환불 규정이 어떻게 돼?"* (문서 기반 안내) 또는 *"강남 근처 오후 3시 빈 방 추천해줘"* (예약 DB 기반 검색) 질문 → **첫 응답 토큰 2초 이내** 스트리밍 시작.
  - **Climax:** 검색이면 상위 3개 + 더보기로 결과가 뜨고, 링크로 상세 페이지로 바로 이동.
  - **Resolution:** 대화는 로그아웃 전까지 유지된다. 서비스 범위 밖 잡담은 정중히 거절한다.

## 3. Glossary

*다운스트림과 독자는 이 용어를 정확히 사용한다. FR·UJ·SM은 Glossary 용어를 그대로 쓰며, 동의어 도입은 규율 위반이다. (시간 표기 규약은 §10 "시간·타임존" 참조.)*

- **예약자(Booker)** — 스터디룸을 찾아 예약하는 사용자. 총무형·개인형으로 나뉘나 동일 역할.
- **제공자(Provider)** — 스터디룸을 등록·운영하는 사용자. MVP에서 제공자당 **스터디룸 1개**.
- **관리자(Admin)** — 서비스 운영 역할. MVP는 **시드 관리자**만 존재(가입·생성 기능 없음).
- **스터디룸(Room)** — 제공자가 등록한 공간. 이름·시간당 금액·영업시간·주소(좌표+행정동 코드)·수용 인원·룸 형태(개방형/독립룸)·부대시설을 가진다.
- **슬롯(Slot)** — 예약 단위. **고정 1시간**. 스터디룸의 영업시간에서 자동 도출된다(영업시간은 같은 날 내, 자정 넘김 없음 — §5). 한 슬롯은 한 예약에만 귀속된다. 각 슬롯의 시작 시각(slot_start)은 절대시각으로 저장된다(§10).
- **예약(Reservation)** — 한 예약자가 한 스터디룸의 **연속된 N개 슬롯**(N≥1)에 대해 성립시킨 건. 상태: **확정 / 취소 / 거절**. 상태 전이는 원자적이며 종료 상태(취소·거절)는 되돌릴 수 없다.
- **이용 완료** — 예약이 **확정 상태를 유지한 채** 마지막 슬롯의 종료 시각이 지난 것. 취소·거절된 예약은 이용 완료가 아니다.
- **즐겨찾기(Favorite)** — 예약자가 스터디룸을 저장해둔 것.
- **핀(Pin)** — 지도 위 스터디룸 표시. 색으로 예약 가능 여부를 구분한다.
- **바텀시트(Bottom Sheet)** — 핀·목록 항목 선택 시 하단에서 펼쳐지는 요약 카드.
- **후기(Review)** — 예약자가 이용 완료 예약에 대해 남기는 별점(1~5 정수) + 텍스트(500자).
- **답글(Reply)** — 제공자가 후기에 다는 응답.
- **인앱 배너(In-App Banner)** — 접속 시점에 화면에 표시되는 알림. 푸시가 아니다. 두 종류: **도래 리마인드**(FR-18)와 **상태변경 통지**(FR-18a).
- **챗봇(Chatbot)** — 플로팅 RAG 어시스턴트. 문서 안내 + 자연어 예약 검색 두 역할.
- **인제스트(Ingest)** — 챗봇 RAG 문서를 벡터 저장소에 적재하는 처리.
- **행정동 코드** — 시/군/구·동/읍/면 표준코드. 목록/행정동 검색에 사용.

## 4. Features

### 4.1 계정 & 인증

**Description:** 예약자와 제공자는 이메일로 가입·로그인한다. 한 이메일은 **하나의 역할**만 가지며, 역할은 가입 시 확정되어 전환할 수 없다 `[ASSUMPTION: 한 이메일로 예약자·제공자 겸용 불가; 중복가입은 역할 무관 이메일 단위로 차단; 한 사람이 양쪽을 시연하려면 2개 이메일을 쓴다]`. OAuth·약관 동의·이메일 인증 메일 발송·비밀번호 변경은 MVP 비포함. 관리자는 시드 계정만 존재한다. 세션은 로그인 후 유지되며 로그아웃으로 종료된다.

**Functional Requirements:**

#### FR-1: 이메일 회원가입

예약자 또는 제공자는 이메일·비밀번호와 역할을 지정해 가입할 수 있다. Realizes UJ-1, UJ-2, UJ-3.

**Consequences (testable):**
- 이메일 형식이 유효하지 않으면 가입이 거부된다.
- 이미 가입된 이메일로는 역할과 무관하게 가입이 거부된다.
- 가입 성공 시 해당 역할로 즉시 로그인 가능한 상태가 된다.

#### FR-2: 로그인 / 로그아웃 / 세션 유지

가입한 사용자는 이메일·비밀번호로 로그인하고, 로그아웃 전까지 인증 상태가 유지된다. Realizes UJ-1~UJ-4.

**Consequences (testable):**
- 잘못된 자격 증명은 로그인이 거부되고 안내가 표시된다.
- 인증이 필요한 기능에 비로그인 상태로 접근하면 로그인으로 유도된다.
- **로그인 사용자의 역할(예약자/제공자)은 가입 시 확정되며 로그인 후 전환할 수 없다.**
- 로그아웃하면 세션과 챗봇 대화가 종료된다.
- 웹·앱 모두 동일 백엔드 인증으로 동작한다(웹/앱 토큰 보관 방식 차이는 addendum 참조).

#### FR-3: 비밀번호 정책 *(횡단 정책 — UJ 외, §2.2)*

가입 시 비밀번호는 강제 복잡도를 만족해야 한다.

**Consequences (testable):**
- **최소 8자 + 대문자 1 + 특수문자 1 + 숫자 1** 이상을 만족하지 않으면 거부된다.
- 검증은 백엔드가 신뢰 경계로 수행한다(프론트는 UX 보조).
- 비밀번호는 평문으로 저장되지 않는다(해싱 저장).

**Notes:** `[NOTE FOR PM]` 이 정책은 최신 NIST SP 800-63B Rev.4 권고(강제 복잡도 제거·길이 우선·유출목록 대조)와 어긋남을 알고도 MVP 단순성을 위해 수용한다. *차기 완화*를 §8/§6.2에 명시. 비밀번호 변경/재설정 흐름은 MVP 비포함(§6.2).

### 4.2 스터디룸 탐색 — 지도

**Description:** 앱/웹을 열면 지도 위에 주변 스터디룸이 핀으로 뜨고, **예약 가능 여부가 핀 색으로** 구분된다. 핀을 누르면 바텀시트로 가격·시설·영업시간 요약이 펼쳐지고, 상세로 이동하거나 즐겨찾기할 수 있다. 지도·주소는 카카오맵 기반. Realizes UJ-2.

**Functional Requirements:**

#### FR-4: 첫 진입 지도 + 주변 핀

예약자는 첫 화면에서 추가 조작 없이 자기 주변 스터디룸을 핀으로 본다. Realizes UJ-2.

**Consequences (testable):**
- 위치 권한 허용 시 현재 위치 주변의 핀이 표시된다.
- 첫 지도 표시는 양호 네트워크에서 **2초 이내** 시작된다(측정 기준 §10·SM-6).
- 핀을 탭하면 해당 스터디룸 바텀시트가 열린다.

#### FR-5: 핀 색상 — 예약 가능 구분

예약자는 핀 색만으로 오늘 예약 가능 여부를 즉시 인지한다. Realizes UJ-2.

**Consequences (testable):**
- **오늘(스터디룸 소재지 타임존 기준, §10) 현재시각 이후 남은 빈 슬롯이 1개 이상**이면 '예약 가능' 색(초록)으로 표시된다.
- 남은 빈 슬롯이 없으면 '마감' 색(회색)으로 표시된다.
- 핀 색은 **서버측 집계값**(룸별 잔여 빈 슬롯 수)으로 결정되며, 클라이언트가 핀마다 실시간으로 슬롯을 N회 계산하지 않는다(FR-4 성능과의 충돌 방지).
- 색 구분은 텍스트를 읽기 전에 인지된다(색 + 보조 표식 병행, §10 접근성).

#### FR-6: 바텀시트 요약

예약자는 핀/목록 항목에서 가격·시설·영업시간을 한눈에 본다. Realizes UJ-1, UJ-2.

**Consequences (testable):**
- **1차 정보(가격·영업시간·예약 가능 여부)**가 스크롤 없이 한 화면에 표시되고, 2차 정보(부대시설 등)가 그 아래 위계로 배치된다(시각적 위계의 구체 디자인은 UX 단계).
- 바텀시트에서 상세 페이지 이동과 즐겨찾기 토글이 가능하다.

#### FR-7: 위치 권한 거부 우회

위치 권한이 없거나 거부된 예약자도 탐색을 계속할 수 있다. Realizes UJ-2.

**Consequences (testable):**
- 위치 권한 거부 시 지도 대신 **목록 조회(행정동 콤보)**로 즉시 우회된다.
- 위치가 없으면 반경 검색(FR-9)은 비활성되고 그 이유가 안내된다.
- 막다른 빈 화면 없이 다음 행동이 안내된다.

---

### 4.3 스터디룸 탐색 — 목록

**Description:** 지도가 중심이지만 **목록 조회도 1급 기능**이다. 시/군/구·동/읍/면 콤보로 특정 동네 전체를 훑거나, 현재 위치 기준 반경으로 찾는다. Realizes UJ-1.

**Functional Requirements:**

#### FR-8: 행정동 콤보 목록 조회

예약자는 시/군/구·동/읍/면을 선택해 해당 지역 스터디룸을 목록으로 본다. Realizes UJ-1.

**Consequences (testable):**
- 시/군/구 선택 → 하위 동/읍/면 콤보가 갱신된다.
- 선택 지역의 스터디룸이 목록으로 나열되고, 각 항목에서 상세 이동·즐겨찾기가 가능하다.
- 결과 없음 상태가 빈 화면 디자인으로 안내된다.

#### FR-9: 반경 검색

예약자는 현재 위치 기준 일정 반경 내 스터디룸을 찾을 수 있다. Realizes UJ-2.

**Consequences (testable):**
- 위치 좌표 기준 반경 내 스터디룸만 결과에 포함된다.
- 기본 반경은 **3km**이며 사용자가 조정할 수 있다 `[ASSUMPTION: 기본 반경 3km·사용자 조정 가능]`.
- 반경/행정동 두 검색이 동일한 탐색 도구 안에서 전환 가능하다.
- 위치 권한이 없으면 반경 검색은 비활성되고 행정동 검색으로 안내된다(FR-7 교차).

**Feature-specific NFRs:**
- 스터디룸은 **좌표 + 행정동 코드를 둘 다** 보유해 두 검색 UX를 지원한다(구현 근거 addendum 참조).

---

### 4.4 즐겨찾기

**Description:** 예약자는 마음에 든 스터디룸을 즐겨찾기하고 모아본다.

#### FR-10: 즐겨찾기 추가 / 조회

예약자는 스터디룸을 즐겨찾기에 추가·제거하고 목록으로 모아본다. Realizes UJ-1.

**Consequences (testable):**
- 바텀시트·상세·목록 어디서든 즐겨찾기 토글이 가능하다.
- 즐겨찾기 페이지에서 저장한 스터디룸을 모아보고 상세로 이동할 수 있다.
- 즐겨찾기 상태는 로그인 사용자별로 영속된다.
- **참조 대상 스터디룸이 비활성(제공자 비활성, FR-31)된 경우** 해당 항목은 '비활성' 라벨로 표시되고 신규 예약 진입이 차단된다(막다른 화면 금지, §10).

---

### 4.5 스터디룸 상세 & 예약

**Description:** 상세 화면을 떠나지 않고(같은 페이지 내 UI 전개) 달력에서 날짜를 고르면 **빈 1시간 슬롯만 활성**된다. **연속된 여러 슬롯**을 함께 선택해 실제 사용 길이에 맞춰 예약하며, 결제가 없으니 **바로 예약**으로 끝난다. 같은 시간에 두 사람이 동시에 눌러도 한쪽만 성공한다. Realizes UJ-1, UJ-2.

**Functional Requirements:**

#### FR-11: 상세 정보 화면

예약자는 스터디룸의 가격·부대시설·영업시간·주소·후기를 상세 화면에서 확인한다. Realizes UJ-1, UJ-2.

**Consequences (testable):**
- 가격·부대시설·영업시간·위치·후기/평점이 위계를 갖고 표시된다(1차=가격·영업시간·예약가능, 2차=시설·주소, 3차=후기; 시각 위계 디자인은 UX 단계).
- 상세에서 예약 UI가 같은 페이지 내에서 전개된다(별도 페이지 이동 없음).

#### FR-12: 날짜·슬롯 가용성 표시

예약자는 달력에서 날짜를 고르고, 그날의 빈 슬롯만 선택 가능한 형태로 본다. Realizes UJ-1, UJ-2.

**Consequences (testable):**
- 과거 날짜와 **당일 현재시각 이전 슬롯**(스터디룸 소재지 타임존 기준, §10)은 비활성된다.
- 이미 예약된 슬롯은 비활성으로 표시된다.
- 슬롯은 스터디룸 영업시간에서 도출된 1시간 단위다(영업시간은 같은 날 내, §5).

#### FR-13: 연속 슬롯 선택

예약자는 연속된 여러 슬롯을 한 번에 선택한다. Realizes UJ-1.

**Consequences (testable):**
- 연속 구간(예: 14~17시)을 드래그/탭으로 선택할 수 있다.
- **비연속 선택은 허용되지 않는다**(D1 — 떨어진 슬롯 동시 선택 불가).
- 선택 구간 중 하나라도 빈 슬롯이 아니면 해당 구간 선택이 막힌다.

**Out of Scope:** 비연속 다중 슬롯(향후 고도화 가능, §8).

#### FR-14: 즉시 예약 (결제 없음)

예약자는 선택한 연속 슬롯을 결제 없이 즉시 확정한다. Realizes UJ-1, UJ-2.

**Consequences (testable):**
- 예약 확정에 결제 단계가 없다.
- 단독 예약이 **상세 진입부터 3스텝 이내**로 완료된다(스텝 정의 §7·SM-2).
- 확정 즉시 명확한 성공 피드백이 표시된다.
- 확정 예약은 선택한 모든 슬롯을 **전부(all-or-nothing)** 점유한다 — 일부 슬롯만 잡힌 채 확정되는 상태는 없다(FR-15).

#### FR-15: 중복 예약 방지 (동시성)

같은 슬롯에 동시 예약이 들어와도 한 예약만 성립한다. Realizes UJ-1(edge).

**Consequences (testable):**
- 동일 슬롯에 대한 동시 확정 시도 중 **정확히 하나만 성공**한다(슬롯 점유 중복 0건).
- **겹치되 시작 슬롯이 다른 두 연속 예약**(예: 14~17시 vs 16~18시)의 동시 확정 시, 한쪽 예약 전체가 성립하고 다른 쪽은 **단 하나의 슬롯도 점유하지 않는다**(부분 점유 0건).
- 예약은 선택 슬롯 전체에 대해 단일 트랜잭션으로 원자적으로 성립/실패한다(전략 addendum: 단일 트랜잭션 multi-row INSERT + `UNIQUE(room_id, slot_start)`).
- 실패한 쪽은 당황 없는 안내와 함께 갱신된 가용성을 본다.

#### FR-16: 예약 취소

예약자는 예약 시작 시각 기준 일정 시간 전까지 예약을 취소한다. Realizes UJ-2.

**Consequences (testable):**
- **예약 시작 6시간 전까지**(스터디룸 소재지 타임존 기준, §10) 취소 가능, 그 이후 취소 버튼 비활성.
- 취소 시 점유했던 모든 슬롯이 **동일 트랜잭션 내에서** 즉시 재활성된다.
- 취소된 예약은 히스토리에 취소 상태로 남는다.
- 예약 상태 전이는 원자적이며, 이미 종료 상태(취소·거절)인 예약에 대한 추가 전이 시도는 멱등하게 무시되고 현재 상태를 반환한다(FR-24·FR-32 공통).

### 4.6 예약 관리 (예약자)

**Description:** 예약자는 다가오는 예약과 지난 예약을 한 곳에서 본다.

#### FR-17: 예약현황 / 히스토리

예약자는 자신의 예약을 현황과 히스토리로 조회한다. Realizes UJ-1, UJ-2.

**Consequences (testable):**
- 다가오는 예약과 지난(이용 완료/취소/거절) 예약이 구분되어 표시된다.
- 각 예약에서 스터디룸 상세로 이동할 수 있다.
- 취소 가능한 예약은 취소 동작으로 연결된다.

### 4.7 알림

**Description:** 미접속 중 도달하는 푸시는 MVP 비포함. 대신 **접속 시점 인앱 배너**로 알린다. 배너는 두 종류이며 — 다가오는 예약 리마인드(FR-18)와 예약 상태변경 통지(FR-18a) — 트리거가 다르다.

#### FR-18: 예약 도래 인앱 배너

예약자는 예약이 임박하면 접속 시 인앱 배너로 안내받는다. Realizes UJ-1.

**Consequences (testable):**
- **예약 시작 24시간 이내(=정확히 24시간, 달력일 아님)** 도래한 확정 예약이 있으면 접속 시 배너가 표시된다(D3).
- **'다시 보지 않기'** 선택 시 해당 예약 배너는 사용자별로 더 이상 표시되지 않는다(상태 영속).
- 웹·앱 모두 동일하게 동작한다.

#### FR-18a: 예약 상태변경 통지 배너

예약자는 자신의 예약이 거절·임의취소되면 다음 접속 시 통지 배너로 안내받는다. Realizes UJ-3(거절 결말).

**Consequences (testable):**
- 제공자 거절(FR-24) 또는 관리자 임의취소(FR-32)가 발생하면, **거절/취소 시점과 무관하게**(24h 규칙에 종속되지 않음) 다음 접속 시 1회 통지 배너가 표시된다.
- 통지를 확인하면 해당 배너는 소멸한다.
- 도래 리마인드(FR-18)와 상태변경 통지는 서로 독립된 트리거를 가진다.

### 4.8 공유

**Description:** 확정 예약을 일행에게 카카오톡으로 한 번에 전달한다(총무 흐름의 핵심).

#### FR-19: 카카오톡 예약 공유

예약자는 확정 예약을 카카오톡으로 공유한다. Realizes UJ-1.

**Consequences (testable):**
- 확정 예약에서 카카오톡 공유가 한 번의 동작으로 실행된다.
- 공유 내용에 스터디룸·일시 핵심 정보가 포함된다.
- 카카오톡 외 SNS는 MVP 비포함.

---

### 4.9 후기 & 평점

**Description:** 예약자는 **이용 완료 예약에 한해** 별점·후기를 남기고, 제공자는 답글로 응대한다.

#### FR-20: 후기 작성

예약자는 이용 완료 예약에 대해 별점과 후기를 작성한다. Realizes UJ-1.

**Consequences (testable):**
- **확정 상태를 유지한 채 마지막 슬롯 종료 시각이 지난(이용 완료) 예약**에 한해 작성 가능하다(D4). **취소·거절된 예약은 작성 불가.**
- 예약당 **1회**만 작성 가능하다.
- 별점은 **1~5 정수**, 후기 텍스트는 **500자 이내**.
- 작성된 후기·평점은 해당 스터디룸 상세에 노출된다.

**Out of Scope:** 후기 수정/삭제, "내가 쓴 후기 모아보기"는 MVP 비포함 `[ASSUMPTION: 작성 후 수정·삭제 미지원]`. 노쇼(예약 후 미이용) 구분은 MVP 밖 — 이용 완료는 시각·상태로만 판정한다(§6.2).

#### FR-21: 후기 답글 (제공자)

제공자는 자기 스터디룸의 후기에 답글을 단다. Realizes UJ-3.

**Consequences (testable):**
- 제공자는 자기 스터디룸 후기에 한해 답글을 작성할 수 있다.
- 답글은 해당 후기에 연결되어 노출된다.

---

### 4.10 제공자 — 공간 관리

**Description:** 제공자는 공간 1개를 등록·수정하면 지도에 노출된다.

#### FR-22: 스터디룸 등록 / 수정

제공자는 스터디룸 정보를 등록·수정한다. Realizes UJ-3.

**Consequences (testable):**
- 다음 항목을 입력해 저장하면 지도/목록에 노출된다: **이름, 시간당 금액, 영업시간, 주소, 수용 인원, 룸 형태(개방형/독립룸), 부대시설**(주차·빔프로젝터/TV·커피머신·화이트보드·wifi·기타 중 다중 선택).
- 주소 검색으로 좌표가 확정되고 행정동 코드가 함께 저장된다.
- 영업시간은 **같은 날 내(0~24시, 자정 넘김 없음 — §5)**로 입력된다.
- **MVP는 제공자당 스터디룸 1개**만 등록 가능하다.
- 영업시간 변경은 이후 도출되는 슬롯에 반영된다. **이미 확정된 예약은 자신의 점유 슬롯(slot_start)을 자체 보유하므로 영업시간 변경·축소에 영향받지 않고 정상 표시·취소·후기 대상이 된다**(addendum 데이터 모델).
- 제공자는 **5분 이내** 등록을 마칠 수 있는 흐름을 갖는다.

**Notes:** 스터디룸 삭제는 MVP 비포함 — 운영 중단은 제공자 계정 비활성(FR-31)으로 갈음한다(§6.2).

### 4.11 제공자 — 예약 관리

**Description:** 제공자는 들어온 예약을 확인하고 필요 시 거절한다.

#### FR-23: 예약현황 조회 (제공자)

제공자는 자기 스터디룸에 들어온 예약을 조회한다. Realizes UJ-3.

**Consequences (testable):**
- 다가오는/지난 예약이 슬롯·예약 단위로 표시된다.
- 제공자에게는 예약자의 **표시 이름과 예약 슬롯 정보**만 노출되며, **이메일 등 식별 연락처는 노출되지 않는다** `[ASSUMPTION: 제공자에 예약자 이메일 비노출, 표시 이름만]`(§11 Privacy).

#### FR-24: 예약 거절

제공자는 들어온 예약을 거절할 수 있다. Realizes UJ-3.

**Consequences (testable):**
- 거절 시 해당 예약이 점유했던 슬롯이 **동일 트랜잭션 내에서** 즉시 재활성된다.
- 거절된 예약은 예약자의 히스토리에 거절 상태로 남는다.
- 거절은 **예약 시작 전까지** 가능하다 `[ASSUMPTION: 거절 시한=예약 시작 전; 거절 사유 입력은 선택]`.
- 거절 사실은 예약자에게 **상태변경 통지 배너(FR-18a)**로 안내된다.
- 상태 전이는 원자적이며, 이미 종료 상태(취소·이용 완료)인 예약에 대한 거절은 멱등하게 무시된다(FR-16 공통 상태 머신).

### 4.12 챗봇

**Description:** 어느 화면에서든 떠 있는 플로팅 챗봇이 두 가지를 한다 — 문서 기반 **서비스 안내**와 예약 DB 기반 **자연어 예약 검색**. 멀티 LLM(OpenAI/Google/Anthropic) 스위칭으로 동작하고, 서비스 범위 밖 잡담은 정중히 거절한다. 대화는 로그아웃 전까지 유지된다. Realizes UJ-4.

**Functional Requirements:**

#### FR-25: 플로팅 진입 + 세션 대화 유지

사용자는 어느 화면에서든 챗봇을 열고, 로그아웃 전까지 대화 맥락을 이어간다. Realizes UJ-4.

**Consequences (testable):**
- 챗봇 진입점이 모든 주요 화면에 플로팅으로 노출된다.
- 화면을 이동해도 같은 세션 대화 맥락이 유지된다.
- 대화 맥락의 범위는 **사용자×디바이스(클라이언트) 단위**다 `[ASSUMPTION: MVP는 단일 디바이스 가정 — 웹/앱 동시 접속 시 대화는 디바이스별로 분리]`.
- 로그아웃 시 대화가 종료·초기화된다.

#### FR-26: 서비스 안내 (문서 RAG)

사용자는 서비스 이용방법·FAQ·규정을 자연어로 물어 문서 기반 답을 받는다. Realizes UJ-4.

**Consequences (testable):**
- 챗봇 응답은 세 갈래로 분기한다: **(a) 범위 내 + 근거 있음 → 답변, (b) 범위 내 + 근거 없음 → 환각 대신 "모름/확인 불가" 안내, (c) 범위 밖 → 정중한 거절(FR-28)**.
- (a) 답변은 인제스트된 문서 범위 내 근거에 기반한다.
- 정확성은 **골든셋**(근거 질문 N개 + 근거없음 질문 M개 + 범위밖 질문 K개)에 대한 기대 분기 일치율로 검증한다(acceptance는 UX/구현 단계에서 셋 확정 — SM-7 연계).

#### FR-27: 자연어 예약 검색 (예약 DB 툴)

사용자는 조건을 자연어로 말해 예약 가능한 스터디룸을 추천받는다. Realizes UJ-4.

**Consequences (testable):**
- 지역·시간 등 조건을 해석해 예약 DB에서 후보를 찾는다.
- 결과는 **상위 3개 + 하단 더보기**로 표현되고, 각 항목은 상세 페이지로 링크된다.

#### FR-28: 범위 밖 질문 거절

챗봇은 서비스 범위를 벗어난 요청을 정중히 거절한다. Realizes UJ-4.

**Consequences (testable):**
- 일반 잡담·서비스 무관 질문은 정중한 거절로 안내된다(FR-26 분기 c).
- 거절 시나리오가 골든셋 테스트로 검증된다.

#### FR-29: 멀티 LLM 스위칭 *(횡단 기능 — UJ 외)*

챗봇은 OpenAI/Google/Anthropic 모델을 전환해 동작한다.

**Consequences (testable):**
- 설정으로 프로바이더/모델을 전환해도 아래 **공통 기능**이 동일하게 동작한다: ① 단일·멀티턴 채팅, ② 시스템 프롬프트 적용, ③ 2개 툴콜(문서검색·예약DB), ④ 토큰 스트리밍(FR-30), ⑤ 범위 밖 거절(FR-28).
- 프로바이더별 고유 샘플링 파라미터(temperature 등)는 공통 표면에서 **정규화하지 않는다**(Out of Scope) — 요청/응답/스트리밍/에러 스키마 차이를 흡수하는 어댑터 레이어로 처리(addendum).

**Out of Scope:** 프로바이더별 고유 파라미터 노출, 구조화출력 등 비대칭 고급 기능(MVP는 위 공통 5종 한정).

#### FR-30: 응답 스트리밍

챗봇 응답은 토큰 단위로 스트리밍된다. Realizes UJ-4.

**Consequences (testable):**
- **첫 응답 토큰이 2초 이내** 스트리밍 시작된다(측정 기준 §10·SM-6 — 기본 프로바이더 1종·웜 컨테이너·기준 프롬프트).
- 웹·앱 모두 스트리밍이 동작한다(RN 전송 제약 대응은 addendum).

### 4.13 운영 (시드 관리자)

**Description:** 시드 관리자는 관리자 웹에서 계정·예약 예외와 챗봇 문서를 관리한다. 관리자 가입·생성 기능은 없다. *(이 절의 FR은 운영 FR — UJ 외, §2.2.)*

#### FR-31: 계정 관리 (비활성)

관리자는 예약자·제공자 계정을 비활성할 수 있다.

**Consequences (testable):**
- 비활성된 계정은 로그인·서비스 이용이 차단된다.
- **제공자 계정 비활성 시 그의 스터디룸은 지도/목록 노출이 중단되고 신규 예약이 차단된다. 기존 확정 예약은 유지된다**(취소는 FR-32로 별도 처리).
- 비활성된 스터디룸을 즐겨찾기한 예약자에게는 '비활성' 라벨이 표시된다(FR-10).

#### FR-32: 예약 임의 취소

관리자는 예외 상황에서 예약을 임의 취소할 수 있다.

**Consequences (testable):**
- 임의 취소 시 점유 슬롯이 동일 트랜잭션 내에서 재활성되고 예약은 취소 상태가 된다.
- 예약자에게 상태변경 통지 배너(FR-18a)로 안내된다.
- 상태 전이는 원자적이며 종료 상태 예약에 대해 멱등하다(FR-16 공통).

#### FR-33: 챗봇 문서 인제스트 관리

관리자는 지정 디렉터리에 배치된 문서의 인제스트 처리를 관리한다.

**Consequences (testable):**
- 지정 디렉터리의 문서를 인제스트해 챗봇 검색에 반영한다(MVP는 UI 업로드 아님 — 디렉터리 배치 후 인제스트 트리거).
- **재인제스트는 멱등**하다(동일 문서 재처리 시 중복 벡터를 생성하지 않음).
- 인제스트 처리 상태(성공/실패)를 확인할 수 있고, 부분 실패 시 어떤 문서가 실패했는지 식별된다.

**Notes:** `[ASSUMPTION: 챗봇 임베딩 모델은 단일 고정 — 교체 시 전체 재임베딩 필요(§8). 디렉터리에서 제거된 문서의 벡터 동기화 정책은 구현 단계에서 확정]`.

## 5. Non-Goals (Explicit)

- DeskNow는 **결제·정산 플랫폼이 아니다** — 실결제(PG)·환불·노쇼 정책을 v1에서 만들지 않는다.
- **계정 시스템을 고도화하지 않는다** — OAuth, 약관 동의/버전관리, 이메일 인증 메일, 비밀번호 변경/재설정, 계정 비활성/중복가입 정책 고도화, 역할 전환 제외.
- **알림 인프라를 만들지 않는다** — 푸시·메일 알림 제외(접속 시 인앱 배너만).
- **파일/이미지 스토리지를 다루지 않는다** — 업로드·스토리지 정책 제외.
- **대형 공급자 운영 도구가 아니다** — 다중 공간·좌석 유형 등록 제외(제공자당 1개), 스터디룸 삭제 제외(비활성으로 갈음).
- **소셜/마케팅 채널 허브가 아니다** — 카카오톡 외 SNS 확장 제외.
- **개인정보 분리보관/삭제 체계를 v1에서 구축하지 않는다**(향후 고도화).
- **영업시간은 같은 날 내(0~24시)로 제한**한다 — 자정을 넘기는 영업시간(예: 22:00~02:00)은 MVP 비지원.

## 6. MVP Scope

### 6.1 In Scope

- **계정:** 이메일 가입·로그인(예약자/제공자), 역할 가입 시 확정, 중복가입 차단, 비밀번호 강제 복잡도(8자+대/특/숫). OAuth 없음.
- **찾기:** 지도(핀·예약 가능 색구분·바텀시트) + 목록(시/군/구·동/읍/면 콤보) + 반경 검색(기본 3km) + 즐겨찾기. 카카오맵 기반.
- **예약:** 1시간 단위 **연속 다중 슬롯**, 빈 슬롯만 선택, **즉시 예약(결제 없음)**, 원자적 중복 예약 방지(DB 제약+트랜잭션), 예약 6시간 전까지 취소, 예약현황·히스토리.
- **알림·공유·후기:** 접속 시 인앱 배너 2종(도래 리마인드 24h + 상태변경 통지), 카카오톡 공유, 이용 완료 예약당 별점(1~5)·후기(500자)·제공자 답글.
- **제공자:** 공간 1개 등록/수정, 예약현황·거절, 후기 답글.
- **운영(시드 관리자):** 챗봇 문서 인제스트 관리, 계정 비활성, 예약 임의 취소.
- **챗봇:** RAG 안내(3분기) + 자연어 예약 검색(상위 3+더보기), 플로팅, 세션 유지, 멀티 LLM 스위칭(공통 5종), 스트리밍, 범위 밖 거절.
- **플랫폼:** 사용자 웹 · 관리자 웹 · 모바일 앱, 단일 백엔드.

### 6.2 Out of Scope for MVP

- 실결제(PG)·환불·노쇼 정책 — *신뢰·거래 완성은 다음 단계*. `[NOTE FOR PM]` 비전의 핵심 다음 수순 — 타임라인 여유 시 재검토.
- OAuth / 약관 동의·버전관리 / 이메일 인증.
- **비밀번호 변경/재설정** — MVP 미지원.
- 푸시·메일 알림 — *미접속 중 도달 필요 시 차기*.
- 이미지/파일 업로드·스토리지.
- 다중 공간·좌석 유형 등록 — *제공자 확장은 차기*.
- **스터디룸 삭제** — 비활성으로 갈음(향후 삭제·이관 정책).
- 카카오톡 외 SNS 확장.
- 개인정보 분리보관/삭제.
- 비밀번호 정책 완화(길이 우선 + 유출목록 대조) — *NIST Rev.4 정렬, 차기*. `[NOTE FOR PM]`
- 비연속 다중 슬롯 예약.
- **자정 넘김 영업시간**(같은 날 내로 제한).
- **노쇼 구분** — 이용 완료는 시각·상태로만 판정.
- 후기 수정/삭제, 내가 쓴 후기 모아보기.
- 관리자 계정 생성/가입 기능 / 역할 전환.

## 7. Success Metrics

두 잣대를 동등하게 둔다 — **사용자에게 좋은 제품인가**, **만든 사람의 역량을 증명하는가**. 경험 지표는 소규모 사용성 테스트(5명 내외)로 측정하며, 표본이 작으므로 **정량 KPI가 아니라 막힘 지점을 드러내는 정성 신호**로 해석한다(합격은 "전원 완주 + 막힘 지점 소수"로 판정, 시간은 중앙값 기준).

**스텝 정의(SM-2용):** "스텝" = 사용자가 의도적으로 하는 **별개의 탭/입력 동작**. 스크롤·드래그 내 연속 이동은 1스텝. 기준 시나리오 = 단독·당일·웜 상태.

**Primary**
- **SM-1 (첫 예약 속도):** 처음 온 사용자가 안내 없이 **첫 예약을 3분 이내(중앙값)** 완료. 지도 경로와 목록 경로 모두 측정. Validates FR-4~FR-6, FR-8, FR-11~FR-14.
- **SM-2 (예약 단순성):** 단독 예약이 상세 진입부터 **3스텝 이내** 완료(위 스텝 정의). Validates FR-11, FR-14.
- **SM-3 (완주율):** 사용성 테스트 참가자(5명 내외)가 막힘 없이 예약 흐름을 완주 — 합격 = **전원 완주 + 치명적 막힘 0건**(표본이 작아 % 대신 절대 기준). Validates FR-4~FR-16.
- **SM-4 (데이터 정합성):** 동시 예약 시도(동일 슬롯 + **겹치는 연속 구간** 포함)에서 **중복·부분 점유 0건**. Validates FR-15.

**Secondary**
- **SM-5 (제공자 등록 속도):** 제공자가 **5분 이내** 공간 등록을 마치고 지도에 노출. Validates FR-22.
- **SM-6 (성능 체감):** 측정 프로토콜 — *양호 네트워크·웜 컨테이너·기준 프롬프트·기본 프로바이더 1종, p90 기준*. 첫 지도 표시 **≤2초**, 챗봇 첫 토큰 **≤2초**(콜드스타트·프로바이더 편차 제외), 로컬 UI 피드백(탭/토글/전환 시작) **≤100ms**. Validates FR-4, FR-30 + §10.
- **SM-7 (챗봇 경계):** 골든셋에서 3분기(근거있음=답 / 근거없음=모름 / 범위밖=거절)가 기대대로 동작. Validates FR-26~FR-28.
- **SM-8 (역량 증명):** 구체 산출물 체크리스트 — ① 사용자 웹·관리자 웹·모바일 앱이 단일 백엔드로 E2E 동작, ② 멀티 LLM 스위칭 + 2개 툴(문서검색·예약DB) 챗봇이 실제 배포 도달, ③ 핵심 플로우(찾기→예약→공유→챗봇) 데모 영상/스크린샷. Validates FR-25~FR-30 + 플랫폼(§14).

**Counter-metrics (do not optimize)**
- **SM-C1 (기능 수 ↛ 목표):** 기능 개수를 늘리지 않는다 — DeskNow의 가치는 폭이 아니라 **마찰 없는 완성도**다. SM-1·SM-3을 상쇄.
- **SM-C2 (챗봇 답변률 ↛ 목표):** 챗봇 응답률을 높이려 **범위 밖 답변이나 근거 없는 답변을 허용하지 않는다** — 정중한 거절·"모름"이 정답인 경우가 있다. SM-7을 상쇄.
- **SM-C3 (속도 ↛ 안전 희생):** 예약 3스텝/3분이나 첫 토큰 ≤2초를 위해 동시성 안전장치(FR-15)·빈 슬롯 검증(FR-12·13)·챗봇 근거성(FR-26)을 약화하지 않는다. SM-1·SM-2·SM-6을 상쇄.

## 8. Open Questions

1. **카카오맵 약관 적합성** — 최종 채택 전 카카오 약관의 서비스 유형 적합성 1차 확인 필요(기술조사 R1).
2. **refresh 토큰 무효화 저장소** — 즉시 무효화를 위해 서버측 저장소(Redis 또는 DB)가 사실상 필수. 선택은 아키텍처 단계에서 확정(기술조사 §인증).
3. **임베딩 모델 고정 선택** — 단일 고정 원칙 하 어떤 모델로 고정할지(교체 시 전체 재임베딩). 디렉터리 제거 문서의 벡터 동기화 정책 포함.
4. **챗봇 배포 토폴로지** — "초기 API 통합 후 부하 시 분리"(비용)와 "같은 레포 별도 서비스"(부하 격리) 중 MVP 기본값 결정(아키텍처 단계).
5. **프로바이더별 TTFT 편차** — 가장 느린 프로바이더가 첫 토큰 ≤2초(FR-30)를 못 맞출 경우의 기준 프로바이더 정책.
6. **Supabase→Railway 벡터 이관 검증** — 1회 실덤프/복원 스파이크로 검증(기술조사 R6).
7. **요일별 영업시간·임시 휴무 모델** — 슬롯 도출에 반영하는 방식(아키텍처/스토리 단계 상세화). *자정 넘김은 §5에서 범위 제외로 확정됨.*
8. **비밀번호 정책 완화 시점** — NIST Rev.4 정렬(길이 우선 + 유출목록 대조)을 차기 어느 시점에 반영할지.

## 9. Assumptions Index

- **§4.1 / FR-1, FR-2** — 한 이메일은 하나의 역할만(예약자·제공자 겸용 불가), 역할은 가입 시 확정·전환 불가, 중복가입은 이메일 단위 차단; 한 사람의 양쪽 시연은 2계정.
- **§4.3 / FR-9** — 반경 검색 기본값 3km, 사용자 조정 가능.
- **§4.9 / FR-20** — 후기 작성 후 수정·삭제 미지원, "내가 쓴 후기 모아보기" 없음.
- **§4.10 / FR-22** — 확정 예약은 점유 슬롯(slot_start)을 자체 보유해 영업시간 변경에 영향받지 않음.
- **§4.11 / FR-23** — 제공자에게 예약자 이메일 비노출(표시 이름·슬롯만).
- **§4.11 / FR-24** — 거절 시한 = 예약 시작 전; 거절 사유 입력은 선택.
- **§4.12 / FR-25** — 챗봇 대화 범위 = 사용자×디바이스(MVP 단일 디바이스 가정).
- **§4.13 / FR-33** — 챗봇 임베딩 모델은 단일 고정(교체 시 전체 재임베딩); 디렉터리 제거 문서의 동기화 정책은 구현 단계.

## 10. Cross-Cutting NFRs

DeskNow의 차별점이 "마찰 없는 완성도"인 만큼, 아래 비기능 요건은 단일 기능이 아니라 **시스템 전체**에 적용되는 1급 요건이다.

**시간·타임존 (시간 판정의 신뢰 경계)**
- 모든 slot_start는 **절대시각(UTC)으로 저장**하고, 모든 "오늘 / 현재시각 / N시간 전 / 24시간 이내" 판정은 **스터디룸 소재지 타임존(MVP = Asia/Seoul)** 기준으로 한다.
- 영업시간은 같은 날 내(자정 넘김 없음, §5)이므로 슬롯의 소속 날짜는 모호하지 않다.
- 이 규약은 FR-5(핀 색)·FR-12(슬롯 활성)·FR-16(6시간 취소)·FR-18(24h 배너)·FR-20(이용 완료)의 시간 경계 consequence가 결정적으로 테스트되기 위한 전제다.

**성능 (체감 속도 — 측정 프로토콜 SM-6)**
- 첫 지도 표시 **≤2초**, 챗봇 첫 응답 토큰 **≤2초**(양호 네트워크·웜·기준 프롬프트·p90, 콜드스타트/프로바이더 편차 제외).
- 로컬 옵티미스틱 UI 피드백(탭·토글·전환 애니메이션 시작) **≤100ms** — 네트워크 의존 동작은 제외하며 별도 지표로 관리.
- 단독 예약 **상세 진입~3스텝 이내**(스텝 정의 §7).

**시각적 완성도·일관성**
- 디자인 토큰(간격·색·타이포) 기반으로 **웹/앱·화면 간 이질감 제로**.
- 인터랙션·마이크로애니메이션이 로딩·전환·선택에 "손에 감기는" 부드러운 피드백을 준다.

**접근성·상태 설계**
- 키보드·스크린리더 접근성 확보. 핀 색 구분은 색에만 의존하지 않고 보조 표식 병행(FR-5).
- **빈/에러/오프라인/로딩** 상태까지 설계(막다른 빈 화면 금지). 검증 대상 = 주요 화면 목록 × {빈/에러/오프라인/로딩} 매트릭스를 UX 단계에서 산출하고 각 셀에 디자인이 존재함을 합격 기준으로 한다.

**보안**
- 외부 API 키(카카오·LLM·지도)는 **백엔드에서만** 관리, 프론트 노출 금지(지도 JS SDK 도메인 키는 도메인 화이트리스트 제한).
- 인증은 단일 백엔드 발급 토큰을 웹/앱이 공용(웹=쿠키·앱=헤더 추출 이원화 — addendum).
- 역할(예약자/제공자/관리자) 권한은 **백엔드에서 최종 강제**(프론트 라우트 보호는 보조).
- 챗봇 스트리밍 엔드포인트도 인증으로 보호.
- 비밀번호는 해싱 저장(평문 금지 — FR-3).

**신뢰성·데이터 정합성**
- 동시 예약(겹치는 연속 구간 포함)에서도 중복·부분 점유 0건(FR-15) — DB 제약 + 단일 트랜잭션으로 원자적 보장.
- 예약 상태 전이(확정→취소/거절)는 원자적·멱등이며, 슬롯 재활성은 전이와 동일 트랜잭션 내에서만 일어난다(FR-16·24·32).

## 11. Constraints & Guardrails

**Cost.** 배포 플랫폼(Railway)은 트래픽이 0이어도 **상시 컨테이너 과금**이며 서비스 수 = 비용이다. 운영 비용 모니터링/알림이 필요하고, 초기엔 챗봇을 API에 통합했다가 부하 시 분리하는 단계적 접근을 고려한다(기술조사 R2, §8 Q4). LLM API 호출 비용도 주요 변수.

**Privacy.** MVP는 최소 개인정보(이메일)만 수집한다. 제공자에게 예약자 이메일은 노출하지 않는다(FR-23). 개인정보 분리보관·삭제 체계는 v1 비포함이며 향후 고도화 대상(§6.2). 외부 공유는 사용자가 명시적으로 실행하는 카카오톡 공유에 한한다.

**Safety (챗봇).** 챗봇은 **서비스 범위 안에서만**, **근거에 기반해서만** 답한다 — 범위 내라도 근거가 없으면 "모름"을 안내하고(환각 금지), 범위 밖은 정중히 거절한다(FR-26 3분기, FR-28).

## 12. Aesthetic & Tone

**미감의 닻 (장점만 취함):** 토스의 *단순함·단계별 진행감*, 에어비앤비의 *지도+카드 예약 경험*, 카카오맵의 *핀·바텀시트 패턴*.

**디자인 원칙:** 하나의 영웅 화면이 아니라 **예약에 이르는 흐름 전체에 마찰이 없을 것**. 7개 마찰 지점(첫 진입→주변 룸 / 핀에서 가용·가격 읽기 / 바텀시트 상세 / 날짜·연속 슬롯 / 예약 확정 / 공유·조율 / 챗봇 호출) 각각이 통과 기준을 만족해야 한다(브리프 §Experience & Design Bar). 인터랙션은 "손에 감기는" 단계별 진행감을 준다.

**챗봇 어조:** 정중하고 간결. 범위 밖 요청·근거 없는 질문은 정중한 거절/"모름"으로. 사용자의 흐름을 끊지 않는다.

## 13. Information Architecture

**사용자 웹/앱 (반응형 일관):**
- 1급 진입: **스터디룸 찾기**(지도/목록 전환), **예약현황**, **즐겨찾기**.
- 전역: **플로팅 챗봇**(모든 주요 화면), 로그인/로그아웃·계정.
- 탐색 → 상세(같은 페이지 내 예약 전개) → 확정 → 공유의 선형 흐름.

**제공자 (사용자 앱/웹 내 제공자 영역):** 공간 등록/수정, 예약현황·거절, 후기 답글. (역할은 가입 시 확정되므로 한 계정은 예약자 영역 또는 제공자 영역 중 하나를 본다.)

**관리자 웹 (분리된 앱):** 계정 관리(비활성), 예약 임의 취소, 챗봇 문서 인제스트 관리. 시드 관리자 전용.

## 14. Platform

- **사용자 웹** — 반응형 웹(예약자·제공자 공용).
- **관리자 웹** — 반응형 웹, 보안상 별도 앱으로 분리.
- **모바일 앱** — React Native(예약자·제공자 공용).
- **백엔드** — 단일 FastAPI(OpenAPI 계약)로 세 클라이언트를 end-to-end 지원.
- 세 클라이언트는 동일 핵심 경험을 제공하되 surface 특성(위치 권한·카카오 공유·SSE 등)에 맞춘 구현 차이는 addendum에 둔다.
