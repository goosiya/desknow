# DeskNow PRD — 적대적 리뷰 (Adversarial Review)

> 작성: 2026-06-14 · 대상: `prd.md`, `addendum.md`
> 관점: 냉소적·적대적 검토자. 저자가 스스로를 속이고 있다고 가정하고, 손에 잡히는 곳마다 그것을 입증한다.
> 심각도: **critical** (출시·신뢰·데이터 정합성 깨짐) / **high** (테스트 불가·자기모순·큰 작업 은폐) / **medium** (모호·근거 부족) / **low** (정리·일관성)

---

## 한 줄 총평

이 PRD는 "마찰 없는 완성도"를 무기로 내세우면서, 정작 **시간(slot_start의 타임존/DST/자정 넘김), 경쟁 상태(취소·거절·동시예약의 레이스), 챗봇 멀티-LLM 추상화**라는 세 개의 진짜 어려운 문제를 모두 한 줄짜리 가정이나 `[ASSUMPTION]` 태그로 덮어버렸다. "DB 제약 한 줄로 동시성 끝"이라는 자신감은 단일 슬롯에는 맞지만 **연속 N슬롯 예약의 원자성**에는 거짓이다. 성능 수치(≤2초, ≤100ms, 3스텝)는 측정 방법도 환경 정의도 없이 선언되어 있어 합격/불합격을 가를 수 없다.

## Critical findings

### C-1. 연속 N슬롯 예약의 원자성을 `UNIQUE(room_id, slot_start)` 한 줄이 보장하지 못한다 — 동시성 안전 주장(FR-15/SM-4)이 거짓일 수 있다

- **위치:** FR-15 (prd.md L263-270), FR-14 "확정 예약은 선택한 모든 슬롯을 점유"(L261), addendum 예약/슬롯 모델 L17-18, SM-4(L524), SM-C3(L535)
- **문제:** 예약 = 연속 N개 슬롯(FR-13/FR-14). `UNIQUE(room_id, slot_start)` 제약은 **개별 슬롯 행 한 줄**의 중복만 막는다. 사용자 A가 14~17시(3슬롯), 사용자 B가 16~18시(2슬롯)를 동시에 확정하면:
  - 두 트랜잭션이 각자 자기 슬롯들을 INSERT하다가 16시 슬롯에서 한쪽이 UNIQUE 위반으로 실패한다. 여기까진 의도대로다.
  - 그러나 **부분 삽입 후 롤백 처리**, **N개 INSERT를 하나의 트랜잭션으로 원자적으로 묶는 것**, **충돌 시 "정확히 하나만 성공"의 판정**은 제약 한 줄이 아니라 트랜잭션 경계·격리수준·INSERT 전략(예: 단일 multi-row INSERT + ON CONFLICT) 설계가 결정한다. addendum이 "애플리케이션 락·FOR UPDATE·SERIALIZABLE은 과설계"라고 단언한 것은 **단일 슬롯 시나리오에만 참**이고, 연속 N슬롯에서는 검증되지 않은 주장이다.
  - "중복 예약 0건"(SM-4)은 슬롯 점유 0중복은 보장할 수 있어도, **예약(Reservation) 단위의 부분 성공/유령 예약**(일부 슬롯만 잡힌 채 확정으로 표시) 가능성을 닫지 못한다.
- **테스트 가능성:** SM-4는 "동시 예약 시도에서 중복 예약 0건"이라고만 한다. **겹치되 동일하지 않은 연속 구간**의 동시 시도(가장 위험한 케이스)가 테스트 시나리오에 명시돼 있지 않아, 통과해도 안전을 증명하지 못한다.
- **수정:**
  1. FR-15의 testable consequence에 "**겹치되 시작 슬롯이 다른** 두 연속 예약의 동시 확정 시, 한쪽 예약 전체가 성립하고 다른 쪽은 **단 하나의 슬롯도 점유하지 않는다**(부분 점유 0건)"를 추가.
  2. addendum에 "N슬롯 예약은 단일 트랜잭션 내 multi-row INSERT(... ON CONFLICT DO NOTHING 후 affected-rows = N 검증, 아니면 ROLLBACK) 또는 동등한 원자 전략으로 보장한다"고 명시. "FOR UPDATE는 과설계"라는 단정은 삭제하거나 "단일 슬롯에 한함"으로 한정.

### C-2. slot_start의 타임존/DST/자정 넘김(영업시간이 0시를 가로지름) 모델이 전무하다

- **위치:** Glossary 슬롯(L82), FR-12(L233-240), FR-22 영업시간(L363-366), addendum 영업시간/휴무 L21, §8 Q6(L544)
- **문제:** 모든 시간 판정(핀 색 "오늘 현재시각 이후"=FR-5/D2, 슬롯 비활성 "당일 현재시각 이전"=FR-12, 취소 "6시간 전"=FR-16, 후기 "예약 시작 시각이 지난"=FR-20/D4, 배너 "24시간 이내"=FR-18/D3)이 **slot_start라는 절대시각**과 **"오늘/현재시각"이라는 상대 개념**에 의존하는데, 어느 타임존 기준인지 PRD 어디에도 없다.
  - 한국(KST, DST 없음) 단일 시장이라 "DST는 무관"이라고 넘길 수 있다고 착각할 수 있으나, **저장은 UTC, 판정은 KST**라는 결정이 빠지면 "오늘 빈 슬롯"·"6시간 전" 경계에서 off-by-one이 난다. 포트폴리오로 "언젠가 진짜 제품"(L21)을 표방하면 멀티 타임존은 시한폭탄.
  - **영업시간이 자정을 넘는 경우**(예: 22:00~02:00) 슬롯 도출이 정의되지 않았다. "고정 1시간 슬롯을 영업시간에서 자동 도출"(L82)인데, 02:00은 어느 날짜의 슬롯인가? 핀 색 "오늘 빈 슬롯"의 "오늘"은 자정 경계에서 어떻게 끊기나? §8 Q6은 "요일별 영업시간·임시 휴무"만 미루고 **자정 넘김은 언급조차 없다.**
- **테스트 가능성:** FR-5/FR-12/FR-16/FR-18/FR-20의 모든 시간 경계 consequence가 타임존 정의 없이는 결정적으로 테스트 불가. "오늘 현재시각 이후 빈 슬롯 1개 이상"(L154)을 23:30에 평가하면 결과가 타임존·자정 처리에 따라 달라진다.
- **수정:** §3 또는 §10에 "모든 slot_start는 UTC 저장, 모든 '오늘/현재시각/N시간 전' 판정은 **스터디룸 소재지 타임존(MVP=Asia/Seoul)** 기준" 명시. §8에 "영업시간 자정 넘김 허용 여부 — 허용 시 슬롯의 소속 날짜 규칙" Q를 신설하거나, Non-Goal에 "영업시간은 같은 날 내(0~24시 비-자정넘김)로 제한"을 명시해 범위에서 제거.

### C-3. 거절·관리자 임의취소·동시예약 실패의 "in-progress 레이스" 처리가 정의되지 않았다

- **위치:** FR-16(취소), FR-24(거절), FR-32(임의취소), UJ-1 edge(L50), §8 Q4(L542)
- **문제:** 여러 행위자가 같은 예약/슬롯에 동시에 작용할 때의 결말이 없다.
  - **예약 시작 직전 6시간 경계**에서 예약자 취소(FR-16, 6시간 전까지)와 제공자 거절(FR-24, 시작 전까지)이 동시에 들어오면? 둘 다 "슬롯 재활성"을 트리거하는데 상태 전이(확정→취소 vs 확정→거절)가 충돌한다. 최종 상태가 비결정적.
  - **거절·임의취소로 슬롯이 재활성된 직후** 다른 예약자가 그 슬롯을 잡는 레이스. 재활성과 신규 점유 사이에 윈도우가 있다.
  - **예약이 이미 "이용 완료"(시작 시각 경과) 상태인데 제공자가 거절을 시도**하면? FR-24는 "예약 시작 전까지"라 막히지만, "시작 직후"의 경계 정의가 없다.
- **테스트 가능성:** "취소 시 점유 슬롯 즉시 재활성"(L278), "거절 시 즉시 재활성"(L387)은 단일 행위자 가정에서만 테스트된다. 동시 전이 결말이 명시되지 않아 회귀 테스트로 못 박을 수 없다.
- **수정:** FR-16/FR-24/FR-32에 공통 상태 머신을 추가 — "예약 상태 전이는 단일 원자 연산이며, 이미 종료 상태(취소/거절)인 예약에 대한 추가 전이 시도는 멱등하게 무시되고 현재 상태를 반환한다." 슬롯 재활성은 상태 전이와 **동일 트랜잭션** 내에서만 일어난다고 명시.

## High findings

### H-1. 멀티 LLM 스위칭(FR-29)은 존재하지 않는 공통 파라미터 표면을 가정한다 — addendum의 서술이 문제를 축소한다

- **위치:** FR-29(L433-439), addendum 챗봇 L34, §8 없음
- **문제 (사실 확인 완료):** addendum은 "**일부** Anthropic 최신 모델은 temperature/top_p/top_k **미지원** → 공통 파라미터를 그대로 전달하면 깨질 수 있음 → 래핑 레이어에서 분기"라고 적었다. 실제는 더 강하다:
  - 현행 Claude 모델(Opus 4.8/4.7, Fable 5)에서 `temperature`/`top_p`/`top_k`는 **"미지원"이 아니라 전달 시 HTTP 400으로 거부**된다(제거됨). `thinking`도 adaptive 전용이며 `budget_tokens`/명시적 `disabled`도 400. assistant prefill도 400. 즉 **파라미터뿐 아니라 요청 스키마 자체가 프로바이더마다 갈린다.**
  - 이는 "if 분기 한 번"이 아니라, **OpenAI/Google/Anthropic의 채팅·툴콜·스트리밍·에러 표면을 공통 인터페이스로 정규화하는 어댑터 레이어**라는 본격 작업이다. FR-29의 testable consequence "설정으로 전환해도 채팅·툴콜 공통 기능이 동작"은 이 어댑터가 세 프로바이더 각각에서 동일 동작을 내야 한다는 뜻인데, 그 정규화 범위가 어디까지인지(온도? 시스템 프롬프트 위치? 툴콜 스키마? 스트리밍 이벤트 모양? 에러 코드 매핑?) PRD에 없다.
- **테스트 가능성:** "공통 기능이 동작"은 측정 불가 — 공통 기능의 목록이 없다. SM-8의 "멀티 LLM 스위칭"도 "전환되면 OK" 수준이라 정규화 정확성을 검증하지 못한다.
- **수정:**
  1. addendum 문구를 "현행 Anthropic 모델은 sampling 파라미터(temperature/top_p/top_k)·assistant prefill·일부 thinking 설정을 **400으로 거부**한다. 멀티 LLM은 파라미터 분기가 아니라 **요청/응답/스트리밍/에러 스키마를 정규화하는 어댑터 레이어**가 필요하다"로 강화.
  2. FR-29에 "공통 기능"의 명시적 목록(채팅 1턴, 시스템 프롬프트, 2개 툴콜, 토큰 스트리밍, 범위밖 거절)을 testable로 고정. 정규화하지 **않는** 것(온도 등 모델별 고유 파라미터)도 Out of Scope로 명시.
  3. §8에 "스트리밍 시작 지연(첫 토큰 ≤2초, FR-30)이 프로바이더별로 다른데 가장 느린 프로바이더가 기준을 못 맞추면?"을 Open Question으로 추가.

### H-2. "첫 토큰 ≤2초"(FR-30/SM-6)는 측정 불가능하고, 단일 컨테이너·콜드스타트 비용 제약과 충돌한다

- **위치:** FR-30(L446), SM-6(L528), §10 성능(L560), §11 Cost(L584), addendum 아키텍처 L39
- **문제:**
  - "첫 응답 토큰 2초 이내"는 **누구의 네트워크에서, 어느 프로바이더 모델로, 어떤 프롬프트 길이/RAG 컨텍스트 크기로, 콜드/웜 중 무엇으로** 측정하는지 전부 빠졌다. RAG는 임베딩 검색 + 컨텍스트 주입 + LLM TTFT의 합인데, 인제스트 문서량이 늘면 검색 지연이 변한다.
  - §11은 "Railway 상시 컨테이너 과금, 초기엔 챗봇을 API에 통합했다가 부하 시 분리"라고 단계적 접근을 권한다. 통합 배포면 **요청 폭주 시 큐잉/콜드 영향**으로 TTFT가 튄다. 즉 비용 절감 결정(통합)과 성능 약속(≤2초)이 같은 문서 안에서 상충한다.
  - SM-C3은 "속도를 위해 안전을 희생하지 않는다"는데, 정작 **속도 자체가 비용 결정에 종속**된다는 역방향 충돌은 다루지 않는다.
- **테스트 가능성:** 측정 환경·표본·통계(평균? p95? 첫 호출 포함?)가 없어 합격선을 못 긋는다.
- **수정:** SM-6/§10에 측정 프로토콜 명시 — "양호 네트워크(정의: 다운 ≥X Mbps, RTT ≤Y ms), 웜 컨테이너, 기준 프롬프트(토큰 N±), 기본 프로바이더 1종, **p90** 기준 ≤2초". 콜드스타트·프로바이더 편차는 측정에서 제외함을 명시하거나 별도 목표로 분리.

### H-3. "주요 상호작용 응답 ≤100ms"는 RN+단일 FastAPI+카카오맵 아키텍처에서 비현실적이거나, 정의가 빠져 무의미하다

- **위치:** SM-6(L528), §10(L561)
- **문제:** "로딩·전환·선택 ≤100ms"가 **클라이언트 렌더 프레임 예산**(로컬 UI 반응)을 말하는 건지 **네트워크 왕복 포함 서버 응답**을 말하는 건지 불분명하다. 후자라면 단일 백엔드·외부 카카오맵 호출·RN 브릿지를 고려할 때 100ms는 비현실적(네트워크 RTT만으로 종종 초과). 전자라면 굳이 "응답"이라 부를 게 아니라 "입력 피드백"이라 해야 하고, 그 경우 RN 리스트/지도 핀 렌더에서 100ms 지키는 것도 자명하지 않다.
- **테스트 가능성:** 무엇을 재는지가 모호해 통과/불통과 불가.
- **수정:** "≤100ms"는 **로컬 옵티미스틱 UI 피드백(탭/토글/전환 애니메이션 시작)에 한함, 네트워크 의존 동작 제외**로 한정 정의. 네트워크 동작은 별도 지표(예: 목록 조회 p90 ≤Xms)로 분리.

### H-4. "상세 진입부터 3스텝 이내"(FR-14/SM-2)는 '스텝'의 정의가 없어 자기충족적이다

- **위치:** FR-14(L259), SM-2(L522), UJ-2 climax(L56), §10(L562)
- **문제:** "스텝"이 무엇인가 — 탭? 화면 전환? 사용자 의사결정? UJ-1의 총무 흐름은 "날짜 선택 → 슬롯 드래그 → 즉시 예약"인데 여기에 "달력 열기", "슬롯 확정 다이얼로그"를 스텝으로 셀지 말지에 따라 3이 되기도 5가 되기도 한다. 정의가 없으면 어떤 구현도 사후에 "3스텝"이라 주장 가능 → 측정 불가.
- **테스트 가능성:** SM-2 "3스텝 이내 완료"는 스텝 카운팅 규칙 없이는 검증 불가.
- **수정:** §3 또는 SM-2에 "스텝 = **사용자가 의도적으로 하는 별개의 탭/입력 동작**(스크롤·드래그 내 연속 이동은 1스텝). 기준 시나리오: 단독·당일·웜 상태"라고 카운팅 규칙과 기준 시나리오를 못 박는다.

### H-5. 더블롤(한 이메일 예약자+제공자) 차단 가정이 핵심 흐름과 충돌하고, 차단의 testable consequence가 모순적이다

- **위치:** FR-1(L102-109), §4.1 Description(L98), §9 Assumptions(L548), UJ-3(제공자), §13 "제공자=사용자 앱/웹 내 제공자 영역"(L605)
- **문제:**
  - `[ASSUMPTION]`은 "한 이메일=한 역할, 중복가입은 역할 무관 이메일 단위 차단". 그런데 현실의 제공자(현우)는 자기 룸을 **예약자로서 둘러보거나 경쟁 룸을 즐겨찾기**하고 싶을 수 있다. 더 큰 문제: 제공자가 **다른 사람의 룸을 예약**하려면 예약자 계정이 필요한데 이메일 1개로는 불가능 → 별도 이메일 강요. 포트폴리오 데모에서 "한 사람이 양쪽을 보여주는" 시연이 어색해진다.
  - FR-1 consequence "이미 가입된 이메일로는 **역할과 무관하게** 가입 거부"는 명확하지만, §13이 제공자를 "사용자 앱 내 제공자 영역"으로 두는 것과 합치면 "같은 앱·다른 역할"의 UX 기대와 "역할 고정"이 미묘하게 어긋난다.
- **테스트 가능성:** "역할과 무관하게 거부"는 테스트 가능하나, **로그인 후 역할이 어떻게 결정/표시되는지**(가입 시 고정? 전환 불가?)가 빠져 권한 경계 테스트가 불완전.
- **수정:** 더블롤 차단의 **제품적 정당성**을 Vision/Non-Goal에 명시하거나(예: "MVP는 역할 분리를 단순화 위해 강제, 시연은 2계정으로"), 아니면 "한 계정이 두 역할을 동시 보유"를 허용으로 재검토. 최소한 FR-2에 "로그인 사용자의 역할은 가입 시 확정되며 전환 불가"를 추가.

### H-6. "이미 확정된 예약은 영업시간 변경에 영향받지 않음" 가정이 슬롯 도출 모델과 충돌한다

- **위치:** FR-22(L366), §9(L550), Glossary 슬롯 "영업시간에서 자동 도출"(L82)
- **문제:** 슬롯이 영업시간에서 **자동 도출**되는데, 제공자가 영업시간을 14~18시→16~20시로 줄이면, 이미 14~15시에 잡힌 확정 예약은 "더 이상 도출되지 않는 슬롯"에 속하게 된다. 가정은 "영향 안 받음"이라지만, 그러면 **확정 예약은 영업시간 스냅샷을 별도로 보유**해야 한다(도출에만 의존하면 불가능). 이건 데이터 모델 결정이지 한 줄 가정으로 끝날 일이 아니다. 동시에 핀 색·가용성 표시는 새 영업시간을 따라야 하므로 "도출된 슬롯 집합"과 "기존 점유 슬롯"이 불일치하는 상태를 UI가 어떻게 보여줄지 미정.
- **테스트 가능성:** "이후 도출되는 슬롯에 반영"(L366)은 테스트 가능하나, "기존 예약이 새 영업시간 밖으로 나간" 케이스의 표시·취소·후기 자격이 정의 안 됨.
- **수정:** addendum 예약/슬롯 모델에 "확정 예약은 점유 슬롯의 slot_start를 **자체 보유**하며 영업시간 도출과 독립적으로 존재한다. 영업시간 밖으로 벗어난 기존 예약도 정상 표시·취소·후기 가능"을 명시.

### H-7. 거절·취소 통지가 "인앱 배너"인데, 배너 노출 조건(24h 도래)과 통지 목적(즉시 알림)이 양립 불가

- **위치:** FR-18(L302-309), FR-24 "거절 사실 인앱 배너"(L390), §8 Q4(L542), §9(L551)
- **문제:** FR-18 배너는 "예약 시작 24시간 이내 도래 예약"을 접속 시 보여주는 **리마인드** 메커니즘이다. 그런데 FR-24는 제공자 거절 사실을 **인앱 배너로** 알린다고 가정한다. 거절은 예약 시작 한참 전(예약 시작 전까지 언제든)에 일어날 수 있는데:
  - 거절 시점이 24h 밖이면 FR-18의 24h 규칙에 안 걸려 배너가 안 뜬다.
  - 거절된 예약은 "도래 예약"이 아니라 **취소된 예약**이므로 FR-18 트리거 대상도 아니다.
  - 즉 "거절 통지=배너"라는 가정은 FR-18 배너의 동작 정의와 충돌하며, **별도의 이벤트성 배너 시스템**이 필요하다는 큰 작업을 한 줄 가정으로 숨겼다. 관리자 임의취소(FR-32) 통지도 동일.
- **테스트 가능성:** "거절 사실이 예약자에게 인앱 배너로 안내"(L390)는 트리거·노출 시점·소멸 규칙·'다시 보지 않기' 적용 여부가 없어 테스트 불가.
- **수정:** 알림을 두 종류로 분리 명시 — (a) **도래 리마인드**(24h, FR-18), (b) **상태 변경 이벤트 통지**(거절/임의취소, 다음 접속 시 1회, 확인하면 소멸). §8 Q4를 FR로 승격하거나 FR-18을 "도래 + 상태변경" 두 트리거로 확장.

### H-8. 후기 자격(D4) "예약 시작 시각이 지난(이용 완료)"이 거절·취소·노쇼를 구분하지 못한다

- **위치:** FR-20(L337), D4(addendum L14), Glossary 후기 "이용 완료 예약"(L87)
- **문제:** D4는 "**예약 시작 시각이 지난** 예약 = 이용 완료"로 정의한다. 그러나:
  - **거절된** 예약도 시작 시각은 지나간다(거절 후 시간이 흐르면). 거절된 예약에 후기를 쓸 수 있나? Glossary는 "이용 완료"라 하나 D4는 "시작 시각 경과"라 해서 **정의가 두 갈래**다. 거절·취소된 예약은 이용한 적이 없는데 "시작 시각 경과"만 보면 자격이 생긴다.
  - **노쇼**(예약했지만 안 감)도 "시작 시각 경과"면 후기 가능 — 가보지도 않고 별점 1점 어뷰징 경로.
  - FR-17은 지난 예약을 "완료/취소/거절"로 구분(L292)하는데, FR-20/D4는 "시작 시각 경과"만 본다. **상태(확정 유지)와 시각(경과)을 함께 봐야** 하는데 D4는 시각만 본다.
- **테스트 가능성:** "예약 시작 시각이 지난 예약에 한해 작성 가능"(L337)을 그대로 구현하면 거절·취소 예약에도 후기가 열린다 — Glossary 의도와 모순. 어느 정의로 테스트할지 불명.
- **수정:** D4와 FR-20을 "**확정 상태를 유지한 채** 예약 시작 시각이 지난 예약에 한함(취소·거절 제외)"으로 통일. 노쇼 구분이 MVP 밖이면 그 한계를 Non-Goal에 명시.

### H-9. 즐겨찾기·후기·예약현황에서 삭제/비활성된 룸·계정의 처리가 정의되지 않았다

- **위치:** FR-10 즐겨찾기(L208-215), FR-17 히스토리(L287-294), FR-31 계정 비활성(L455-460), FR-22 룸(룸 삭제 FR 자체가 없음)
- **문제:**
  - 제공자 계정이 **비활성**(FR-31)되면 그의 룸은 지도에서 사라져야 하나? 그 룸을 **즐겨찾기**한 예약자 목록(FR-10)에는 무엇이 보이나? 그 룸의 **기존 확정 예약**(FR-17)은? 비활성이 룸·슬롯·예약·즐겨찾기에 미치는 캐스케이드가 전무하다.
  - **룸 삭제** FR이 아예 없다(등록/수정만, FR-22). 제공자가 그만두면 룸은 영원히 남나? 삭제가 없으면 비활성으로 갈음하는 건가? 미정.
  - 즐겨찾기한 룸이 사라지면 즐겨찾기 페이지는 깨진 항목을 보여주나, 막다른 화면 금지(§10 L570)와 충돌하나?
- **테스트 가능성:** FR-10 "즐겨찾기 상태 영속"(L215), FR-31 "비활성 계정 로그인 차단"(L460)은 테스트 가능하나, 캐스케이드가 없어 "비활성 후 즐겨찾기/예약/지도 상태"는 검증 불가.
- **수정:** FR-31에 비활성 캐스케이드 명시(룸 지도 노출 중단·신규 예약 차단·기존 확정 예약 유지 여부). 즐겨찾기/히스토리에 "참조 대상이 사라진 항목"의 표시 규칙(비활성/삭제 라벨, 상세 진입 차단) 추가. 룸 삭제 정책(미지원이면 Non-Goal 명시)을 결정.

### M-1. 핀 색 기준(D2) "오늘 현재시각 이후 빈 슬롯"이 반경/행정동 목록의 대량 룸에서 성능·정합성 문제를 숨긴다

- **위치:** FR-5(L153-156), D2(addendum L12), FR-9 반경(L195)
- **문제:** 핀 하나당 "오늘 현재시각 이후 남은 빈 슬롯 ≥1" 계산이 필요하다. 지도에 수십~수백 핀이면 핀마다 슬롯 가용성 집계를 실시간 계산해야 하고, 이는 "첫 지도 표시 ≤2초"(FR-4)와 충돌할 수 있다. 캐싱·집계 전략이 없다. 또 "오늘"의 경계(C-2 타임존)와 맞물려 23시대에 핀 색이 어떻게 변하는지 미정.
- **수정:** FR-5에 "핀 색은 서버측 집계값(룸별 잔여 빈 슬롯 수)을 반환하며 실시간 N회 계산을 클라이언트가 하지 않는다"를 명시. 성능 영향과 갱신 주기를 addendum에 남김.

### M-2. 비밀번호 정책(FR-3)이 NIST와 어긋남을 "알고도 수용"하면서도, 정책 자체의 testable가 백엔드 강제만 다룬다

- **위치:** FR-3(L121-130), §6.2(L511)
- **문제:** "8자+대/특/숫"을 강제하면서 NIST Rev.4와 어긋남을 인지(NOTE FOR PM). 적대적으로 보면, 이건 합리적 트레이드오프지만 **유출목록 대조 없음**은 포트폴리오의 "보안" 어필(§10 보안)과 톤이 안 맞는다. 또 "비밀번호 변경" 흐름이 FR로 없다(FR-3은 "가입·변경 시"라 하지만 변경 UI/엔드포인트 FR 부재). 변경이 없으면 강제 복잡도의 "변경 시" 절반은 죽은 조항.
- **수정:** 비밀번호 변경 FR을 추가하거나, 없으면 "MVP는 비밀번호 변경 미지원"을 Non-Goal에 명시(그러면 FR-3 문구에서 "변경 시" 삭제).

### M-3. 챗봇 세션 "로그아웃 전까지 유지"가 멀티 디바이스·웹/앱 동시 접속에서 정의되지 않았다

- **위치:** FR-25(L400-407), FR-2 세션(L111-119), UJ-4 resolution(L72)
- **문제:** "대화는 로그아웃 전까지 유지"인데, 같은 사용자가 웹과 앱에 동시 로그인하면 대화가 공유되나 분리되나? "화면 이동해도 같은 세션 대화 유지"(L406)는 단일 클라이언트 내 이야기로 읽히지만, 멀티 디바이스가 명시 안 됨. 대화 저장 위치(서버? 클라이언트?)도 불명 — 서버면 디바이스 간 공유, 클라이언트면 분리.
- **수정:** FR-25에 "대화 맥락의 범위 = 사용자×디바이스 단위인지 사용자 단위인지" 명시. MVP가 단일 디바이스 가정이면 그 한계를 적는다.

### M-4. "RAG 근거 없으면 모름/범위 밖 안내"(FR-26)와 "범위 밖 거절"(FR-28)이 같은 동작을 두 FR로 쪼개 테스트가 중복·모호

- **위치:** FR-26(L414-415), FR-28(L429-431), §11 Safety(L588)
- **문제:** FR-26 "근거 없는 질문엔 환각 대신 모름/범위 밖 안내" vs FR-28 "서비스 무관 질문 정중 거절"은 경계가 겹친다. "환불 규정"은 인제스트 안 된 경우 FR-26인가 FR-28인가? "근거 없음"(문서에 없음)과 "범위 밖"(서비스 무관)은 다른 개념인데 둘 다 "거절/안내"로 수렴해 테스트 시나리오가 어느 FR을 검증하는지 모호. SM-7도 "범위 내=답, 범위 밖=거절" 이분법이라 "범위 내 질문인데 문서에 답이 없는" 제3 케이스를 빠뜨린다.
- **수정:** 3분류 명시 — (a) 범위 내+근거 있음=답, (b) 범위 내+근거 없음=모름 안내(환각 금지), (c) 범위 밖=정중 거절. SM-7을 3케이스로 확장.

### M-5. 인제스트(FR-33)가 "디렉터리 배치 후 트리거"인데 멱등성·중복·삭제·실패 처리가 없다

- **위치:** FR-33(L469-477), addendum 챗봇 L36
- **문제:** "지정 디렉터리 문서를 인제스트"인데, 같은 문서를 다시 인제스트하면 벡터가 중복되나? 디렉터리에서 파일을 빼면 벡터가 삭제되나? 인제스트 실패(임베딩 API 오류) 시 부분 적재 상태는? "처리 상태 확인 가능"(L474)만 있고 멱등성·삭제 동기화·부분 실패 복구가 없다. "임베딩 모델 교체 시 전체 재임베딩"(L477)은 인지했으나 일상 운영의 증분 갱신이 비어 있다.
- **수정:** FR-33에 "재인제스트는 멱등(동일 문서 재처리 시 중복 벡터 미생성), 디렉터리에서 제거된 문서의 벡터 처리 정책, 부분 실패 시 동작"을 testable로 추가.

### M-6. SM-1/SM-3의 사용성 테스트(5명)는 통계적으로 무의미한 표본인데 "80% 완주"를 정량 목표로 쓴다

- **위치:** SM-1(L521), SM-3(L523), §7 머리말(L518)
- **문제:** "5명 내외"에서 "80% 이상 완주"는 4/5다. 1명 더 막히면 60%로 떨어진다 — 표본 노이즈가 합격선을 좌우한다. 포트폴리오 맥락에서 정성 신호로는 쓸 만하나, **정량 KPI처럼 제시**하면 자기기만. "3분 이내 첫 예약"도 5명 평균/중앙값/최댓값 중 무엇인지 없다.
- **수정:** SM-1/SM-3을 "정성 신호(막힘 지점 식별)"로 격하하거나, 합격 판정을 "전원 완주 + 막힘 지점 3개 이하" 같은 노이즈에 둔감한 기준으로 바꾸고 측정 통계(중앙값 등)를 명시.

### M-7. 반경 검색(FR-9) "일정 반경"의 값·조정 가능성이 미정, 위치 거부 시(FR-7)와 충돌

- **위치:** FR-9(L195-197), FR-7 위치거부 우회(L170-172)
- **문제:** "일정 반경 내"의 반경값(1km? 3km? 사용자 조절?)이 없다. 또 FR-7은 위치 거부 시 "목록(행정동)으로 우회"인데, FR-9 반경 검색은 위치 좌표가 필수다. 위치 거부 사용자에게 반경 검색 탭은 어떻게 동작하나(비활성? 숨김?) — "반경/행정동 전환 가능"(L197)과 "위치 없음" 상태의 상호작용이 미정.
- **수정:** FR-9에 기본 반경값과 조정 여부 명시. 위치 권한 없는 상태에서 반경 검색 탭의 동작(비활성+안내)을 FR-7/FR-9 교차로 정의.

### M-8. "막다른 빈 화면 금지"(§10)가 1급 NFR인데 검증 기준이 없어 사실상 선언

- **위치:** §10 접근성·상태(L568-570), FR-7/FR-8 결과없음
- **문제:** "빈/에러/오프라인/로딩 상태까지 설계", "막다른 빈 화면 금지"는 좋은 원칙이나 **어느 화면 목록에 대해 검증하는지** 없다. 모든 화면×4상태(빈/에러/오프라인/로딩) 매트릭스가 없으면 통과 판정 불가 — "마찰 없는 완성도"의 핵심 약속이 측정 불가 상태로 남는다.
- **수정:** §10에 "상태 설계 검증 대상 화면 목록 × {빈/에러/오프라인/로딩} 매트릭스를 UX 단계에서 산출하고 각 셀에 디자인이 존재함을 합격 기준으로 한다"를 명시.

가장 위험한 것은 **저자가 어려운 줄 모르는 어려운 문제들**이다: (1) 연속 N슬롯 동시성을 "한 줄 제약"으로 끝났다고 믿는 것(C-1), (2) 시간 모델 전체가 타임존·자정 경계 정의 없이 떠 있는 것(C-2), (3) 멀티-LLM을 "파라미터 분기"로 축소한 것(H-1, 사실은 어댑터 레이어). 성능 약속들은 거의 전부 측정 프로토콜이 없어 "출시 품질"이라는 자기 평가를 검증 불가능하게 만든다. 다음 단계(UX·아키텍처)로 넘기기 전에 최소한 critical 3건과 H-1·H-7·H-8은 FR/Glossary 수준에서 못 박아야 한다.
