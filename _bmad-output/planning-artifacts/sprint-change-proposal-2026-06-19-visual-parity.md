## Section 1 — 이슈 요약

**문제 진술:** Epic 9(모바일 화면 패리티)의 스토리 9.1~9.3이 전부 `done`으로 닫혔으나, 모바일(RN/Expo) 화면의 **렌더 결과(앱 셸·버튼 모양·정렬·입력 컨트롤·온보딩)** 가 웹과 어긋난다. 콘텐츠(카드·폼·뱃지)는 대체로 잘 포팅됐지만 화면 골격이 다르다.

**발견 경위 (근본 원인):** 직전까지 `:3000`을 "웹"으로 알고 비교해 왔으나, **`:3000`·`:3001`이 모두 Expo 모바일**(같은 `expo-router/entry.bundle`)이었다. 진짜 Next 웹앱은 어느 포트에도 떠 있지 않았다. 즉 9.x 모바일 스토리들의 "리뷰/패리티 통과"가 **모바일↔모바일 비교**였고 무의미했다 → 셸·입력 컨트롤·온보딩 드리프트가 적출되지 않았다. 이번 세션에 진짜 웹을 `:3002`로 세워 전수 대조했다.

**증거:** `parity-audit-web-vs-mobile-2026-06-19.md` + 스크린샷 쌍(`parity-audit-2026-06-19-shots/`). 심각도 롤업:
- **S1(배포 차단) 4건:** SYS-1 글로벌 헤더 유무 · SYS-2 인증화면 하단탭 노출 여부 · ONB-1·2·3 온보딩(4슬라이드 중 1개만 구현, 캐러셀·"다시 보지 않기" 누락) · ROOM-1 영업시간 시간 입력(웹 12h `<input type=time>` vs 모바일 24h ComboSelect).
- **S2(출시 전 수정) 14건:** SYS-3 FAB 아이콘, SYS-4 provider 탭 순서·라벨, LOGIN-1·2, SIGNUP-2, HOME-1·2, ROOM-2·3·5, CHAT-1, PRES-1, PREV-1, FAV-1, RES-1, DET-2.
- **S3(일괄 정리) 4건:** CHAT-2·4, PREV-2, DET-1.
- **미검증 인터랙션 9건(=fix 스토리 필수):** 가입 역할 토글 · 홈 목록탭/지역콤보/반경/핀탭 요약 · 룸상세 예약 플로우(진짜 웹) · 예약 취소 · provider 거부 2단 · 후기 답글 작성폼 · 챗봇 전송→스트리밍 · 즐겨찾기 해제.

## Section 2 — 영향 분석

### 2.1 에픽 영향
- **Epic 9:** `done`으로 닫지 않고 `in-progress` **유지**. 9.1~9.3은 "기능 포팅 완료 + 시각 패리티 미달"로 재해석한다(스토리 본문·기능 AC는 유효, 재작성 안 함).
- **신규 Story 9.4 추가:** 시스템성 셸 정렬 + 화면별 S2/S3 + 미검증 인터랙션 전수 + 실기기 재검증을 단일 스토리로 흡수(KTH "단일 9.4" 확정).
- **Epic 1~8:** 영향 없음(웹·백엔드 기준 완료 유효).

### 2.2 아티팩트 충돌
- **PRD:** 충돌 없음. FR-18·NFR-4가 정정의 근거. MVP 범위 변동 없음(누락 품질 회수).
- **아키텍처(`architecture.md`):** **신규 결정 없음.** 9.x에서 확정한 WebView+카카오맵 JS 지도, secure-store+Bearer 인증, E2E 환경변수 게이팅 주입은 전부 유효하며 그대로 사용. 드리프트는 아키텍처가 아니라 **구현 충실도(셸·컴포넌트·온보딩 완성도)** 문제다.
- **UX:** 신규 화면 설계 아님. 웹을 정본으로 시각 동등 재현. 디자인 토큰 `@desknow/ui` 공유 유지.
- **기타 산출물:** 실기기(iOS/Android) 재검증은 EAS dev build·시뮬레이터/실기기가 필요해 본 루프에서 자동검증 불가 → `⚠️사람-확인-필수`로 분리(2.4·Section 5).

### 2.3 ⚠️ 인증화면 앱 셸 방향 (KTH 확정)
웹은 루트 레이아웃(`apps/web/src/app/layout.tsx`)이 **모든 페이지를 `AppShell`로 감싸** `/login`·`/signup`에도 글로벌 헤더(SYS-1) + booker 하단 탭 바(SYS-2)가 노출되는 **의도된 구조**다(웹 버그 아님).
→ **결정: 웹=정본. 모바일 인증화면에도 동일 셸을 추가**한다. done인 웹 코드는 무수정. (대안 "양쪽 인증화면 셸 제거"는 기각 — done 웹 AppShell 수정 회피.)

### 2.4 ⚠️ 실기기 재검증 (KTH 확정 — 사람-확인-필수)
본 audit·9.4 검증은 **웹 vs Expo Web(react-native-web)** 한정이다. 실제 iOS/Android 네이티브 빌드는 폰트 렌더링·그림자·터치 하이라이트·세이프에어리어·키보드 회피에서 더 벌어질 수 있다.
→ 9.4의 Expo Web 패리티 검증을 done 기준으로 하되, **네이티브 실기기 재검증은 `⚠️사람-확인-필수` 항목**으로 분리해 배포 체크리스트에 의무 라인화한다(자동 빌드/푸시로 안 잡히는 항목).

> ### Story 9.4: 모바일 시각·인터랙션 패리티 정정
>
> As a 예약자/제공자,
> I want 모바일 전 화면이 진짜 웹앱과 시각·인터랙션까지 동등해지길,
> So that 9.1~9.3이 가렸던 셸·컴포넌트·온보딩 드리프트가 제거되어 웹·앱 이질감이 사라진다 (FR-18·NFR-4 / audit 2026-06-19 정정).
>
> > 배경: 9.1~9.3은 기능을 포팅했으나 `:3000`·`:3001`이 둘 다 Expo라 **진짜 웹과 대조된 적이 없었다.** 본 스토리는 웹(`:3002`)을 정본으로 전 화면 드리프트를 정정한다. 입력 = `parity-audit-web-vs-mobile-2026-06-19.md`.
>
> **Acceptance Criteria:**
>
> **AC1 — 시스템성 셸 패리티 (S1/S2)**
> **Given** 모바일 전 화면
> **When** 렌더하면
> **Then** ① 글로벌 헤더(좌 DeskNow 로고 + 우 인증/화면 컨트롤)가 웹과 동일하게 모든 화면 상단에 노출되고(SYS-1), ② **인증화면(로그인·가입) 포함** booker 하단 탭 바(아바타+스터디룸찾기/예약현황/즐겨찾기)가 웹과 동일하게 노출되며(SYS-2, 웹=정본 확정), ③ provider 하단 탭 순서·라벨이 웹과 동일 `[스터디룸, 예약자 현황, 후기]`이고(SYS-4), ④ 챗봇 FAB 아이콘이 웹과 동일한 💬 말풍선(원형)이다(SYS-3).
>
> **AC2 — 온보딩 4슬라이드 캐러셀 (S1)**
> **Given** 첫 방문 홈 진입
> **When** 온보딩 오버레이가 뜨면
> **Then** 웹 `OnboardingOverlay`의 `SLIDES[4]`와 동일한 4장 캐러셀(① 환영 ② 지도와 목록으로 찾기 ③ 원하는 시간에 바로 예약 ④ 궁금하면 룸메이트 챗봇)이 점 인디케이터·이전/다음·X와 함께 표시되고(ONB-1·2), "다음"→마지막 "시작하기" CTA·**"다시 보지 않기" 영속 dismiss**(AsyncStorage)가 웹과 동일 동작하며(ONB-3·4), 환영 본문 문구도 웹과 일치한다(ONB-5).
>
> **AC3 — provider 룸 폼 입력 컨트롤 (S1/S2)**
> **Given** provider 룸 등록/수정 폼
> **When** 영업시간·룸 형태를 입력하면
> **Then** 영업시간 시간 입력이 웹과 동일한 **12시간제 표기(오전/오후)** 로 표시되고(ROOM-1, 웹 `<input type=time>` 정본), 룸 형태 세그먼트가 웹과 동일한 **반폭 테두리 박스 2개(개방형|독립룸)** 구성이며(ROOM-2), 하단 탭바가 폼 하단 콘텐츠(부대시설 칩/저장)를 가리지 않는다(ROOM-5).
>
> **AC4 — 화면별 S2/S3 시각 정렬**
> **Given** 각 화면(로그인·가입·홈·챗봇·provider후기·룸상세 등)
> **When** 렌더하면
