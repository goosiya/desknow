# DeskNow — 경험 스파인 (Experience Spine)

> 이 문서는 **무엇이 어떻게 동작하는가(HOW IT WORKS)** 를 정의한다. 색·간격·타이포·그림자 같은 **시각 명세(HOW IT LOOKS)** 는 `DESIGN.md`가 소유하며, 본 문서는 토큰 이름을 `{경로.토큰}` 구문으로 참조만 한다(예: `{colors.primary}`, `{colors.pin-available}`). 충돌 시 시각은 DESIGN.md, 행동은 본 문서가 정본이다.

---

## Foundation (토대)

DeskNow는 **멀티 서피스** 제품이다. 같은 핵심 경험을 세 표면이 공유하되, 각 표면의 입력·제약에 맞춰 구현이 갈라진다.

| 표면 | 기술 | 사용자 | 입력 모델 |
|---|---|---|---|
| 사용자 웹 | 반응형 웹 (Next.js) | 예약자 · 제공자 공용 | 키보드 + 마우스 + 터치 |
| 모바일 앱 | React Native (Expo) | 예약자 · 제공자 공용 | 터치 |
| 관리자 웹 | 별도 반응형 웹 (Next.js) | 시드 관리자 전용 | 키보드 + 마우스 |

- **UI 시스템 = shadcn.** 두 스파인 모두 shadcn에서 상속하며, 브랜드는 "shadcn 기본값을 존중하되 브랜드 레이어로 덮는 것"이다. `DESIGN.md`가 시각 정체성과 오버라이드 표면을 명명한다.
- **단일 백엔드(FastAPI, OpenAPI 계약).** 세 클라이언트는 자동생성 TS SDK로 동일 백엔드를 호출한다. 라이트 모드 전용(토큰은 다크 확장 가능한 시맨틱 구조로 설계됨).
- **세션/토큰 보관 이원화:** 웹 = 쿠키 토큰, RN 앱 = SecureStore. 인증 자체는 단일 백엔드 발급 토큰을 공용한다.

### 플랫폼 델타 (구현이 갈라지는 지점)
- **카카오톡 공유 (RN):** 네이티브 모듈이 필요해 Expo Go가 아닌 **EAS dev build**에서만 동작한다. 웹은 카카오 JS SDK. → Foundation 단계에서 빌드 경로를 분기해야 함.
- **챗봇 스트리밍 (RN):** 브라우저 `EventSource`가 없으므로 **`react-native-sse`** 로 SSE를 받는다. 웹은 표준 SSE/스트리밍.
- **localhost 미접근 (RN):** 앱은 개발 머신의 `localhost`에 직접 닿지 못한다 — LAN IP/터널 기반 API 베이스 URL이 필요.
- **위치 권한 (앱):** OS 권한 다이얼로그를 거치며, 거부 시 행정동 콤보로 우회(아래 IA·State 참조).

## Information Architecture (정보구조)

### 사용자 웹/앱 — 표면 지도

| 표면 | 진입 경로 | 목적 |
|---|---|---|
| 스터디룸 찾기 (지도) | 앱/웹 첫 진입 | 주변 룸을 핀으로 탐색, 핀 탭 → 바텀시트 |
| 스터디룸 찾기 (목록) | 찾기 화면 내 지도/목록 토글 | 행정동 콤보·반경으로 동네 전체 훑기 |
| 스터디룸 상세 | 바텀시트·목록·즐겨찾기·챗봇 결과 | 정보 확인 + **같은 페이지 내 예약 전개** |
| 예약현황 | 1급 진입 (탭/네비) | 다가오는/지난 예약, 취소·후기·공유로 연결 |
| 즐겨찾기 | 1급 진입 (탭/네비) | 저장한 룸 모아보기 → 상세 이동 |
| 로그인 / 계정 | 전역 (헤더/메뉴) | 가입·로그인·로그아웃 |
| 플로팅 챗봇 "룸메이트" | 전역 (모든 주요 화면) | 문서 안내 + 자연어 예약 검색 |

**1급 진입 3개:** 스터디룸 찾기(지도/목록 전환) · 예약현황 · 즐겨찾기. 전역으로 플로팅 챗봇 + 로그인/계정.
**선형 흐름:** 탐색 → 상세(같은 페이지 내 예약 전개) → 확정 → 공유.

### 제공자 영역 (사용자 웹/앱 내, 역할 가입 시 확정)

| 표면 | 목적 |
|---|---|
| 공간 등록/수정 | 룸 1개 등록(이름·금액·영업시간·주소·수용·형태·부대시설) → 지도 노출 |
| 예약현황 (제공자) | 들어온 예약 조회(표시 이름·슬롯만, 이메일 비노출) |
| 예약 거절 | 예약 시작 전까지 거절 → 슬롯 재활성 + 상태변경 통지 |
| 후기 답글 | 자기 룸 후기에 답글 |

역할은 가입 시 확정되므로 한 계정은 **예약자 영역 또는 제공자 영역 중 하나**만 본다.

### 관리자 웹 (분리된 앱, 시드 관리자 전용)

| 표면 | 목적 |
|---|---|
| 계정 관리 | 예약자·제공자 계정 비활성 |
| 예약 임의취소 | 예외 상황 예약 취소 → 슬롯 재활성 + 통지 배너 |
| 챗봇 문서 인제스트 | 디렉터리 배치 문서 인제스트 트리거·상태 확인 |

**커버리지:** 모든 명시된 사용자 니즈가 표면을 가진다 — 탐색(지도/목록), 예약(상세 전개), 관리(예약현황), 저장(즐겨찾기), 문의(챗봇), 공유(상세→카톡), 후기(상세/예약현황), 제공자 운영(제공자 영역), 운영 예외(관리자 웹). 막다른 진입 없음.

→ 시각 구성 참조 (`mockups/` 내 핵심 화면 HTML):
- [지도+바텀시트](mockups/key-map-bottomsheet.html) — 지도 핀 탐색 + 핀 탭 시 떠오르는 바텀시트 3단 정보 위계.
- [상세+슬롯 피커](mockups/key-detail-slotpicker.html) — 같은 페이지 내 예약 전개, 달력 + 연속 슬롯 선택.
- [챗봇 룸메이트](mockups/key-chatbot-roommate.html) — 플로팅 챗봇 대화 + 검색 결과 상위 3개.
- [예약현황](mockups/key-reservation-list.html) — 다가오는/지난 예약 목록과 도래 리마인드 배너.

충돌 시 스파인이 정본이다(spines win on conflict).

## Voice and Tone (보이스 & 톤 — 마이크로카피)

**친근한 해요체.** 토스/당근 결의 따뜻하고 말 걸기 쉬운 톤. 브랜드 보이스 자체는 `DESIGN.md.Brand & Style` 참조. 아래는 화면별 마이크로카피.

| 상황 | DeskNow는 이렇게 | 이렇게는 안 함 |
|---|---|---|
| 예약 성공 | "예약이 완료됐어요!" | "예약 요청이 정상 처리되었습니다." |
| 동시성 충돌 실패 | "앗, 방금 다른 분이 먼저 잡았어요. 가까운 빈 시간을 다시 보여드릴게요." | "Error: slot conflict (409)" |
| 빈 상태 (룸 없음) | "아직 등록된 스터디룸이 없어요." | "검색 결과가 존재하지 않습니다." |
| 빈 상태 (즐겨찾기 없음) | "마음에 든 곳을 즐겨찾기해두면 여기 모여요." | "즐겨찾기 목록이 비어 있습니다." |
| 일반 에러 | "잠깐 문제가 생겼어요. 다시 한번 해볼까요?" | "Unknown error occurred." |
| 위치 권한 거부 | "현재 위치를 못 받았어요. 동네를 골라서 찾아볼게요." | "위치 권한이 거부되었습니다." |
| 취소 불가 (6시간 경과) | "이제 6시간이 안 남아서 취소가 어려워요." | "취소 가능 시간이 만료되었습니다." |
| 후기 작성 유도 | "이용은 어떠셨어요? 짧게 후기 남겨주실래요?" | "후기를 작성하십시오." |
| 챗봇 범위 밖 거절 | "그건 제가 도와드리기 어려운 주제예요. 스터디룸 찾기나 예약은 얼마든지 도와드릴게요!" | "I cannot answer that." |

규칙: 짧고 완결된 문장, 책임을 사용자에게 떠넘기지 않음("당신이 ~했습니다" 금지), 막힘 뒤에는 항상 **다음 행동**을 제시.

## Component Patterns (컴포넌트 패턴 — 행동만)

시각 명세는 `DESIGN.md.Components` 및 shadcn 기본값. 여기서는 행동 규칙만.

| 컴포넌트 | 사용처 | 행동 규칙 |
|---|---|---|
| 지도 + 핀 | 찾기(지도) | 핀 색으로 가용성 구분(`{colors.pin-available}` 초록 / `{colors.pin-full}` 회색). 색 + 보조 아이콘 병행(색만 의존 금지). 핀 탭 → 바텀시트 열림. 핀 색은 **서버측 집계값**으로 결정(클라이언트가 핀마다 슬롯 N회 계산 금지). |
| 바텀시트 | 핀/목록 탭 시 | **3단 정보 위계** — 1차(가격·영업시간·예약 가능 여부)는 스크롤 없이 한 화면, 2차(부대시설 등) 아래, 3차는 상세로. 그래버/드래그로 펼침·접힘. 떠야 하므로 그림자(`{elevation.sheet}`). 상세 이동 + 즐겨찾기 토글 제공. |
| 상세 내 예약 전개 | 상세 | 별도 페이지 이동 없이 **같은 페이지 내**에서 예약 UI(달력+슬롯)가 전개된다. 상세 진입~확정 ≤3스텝. |
| 달력 + 슬롯 피커 | 상세 | 날짜 선택 → 그날 슬롯 표시. **드래그/탭으로 연속 선택**, 비연속 불허(D1). 과거 날짜·당일 현재시각 이전·이미 예약된 슬롯은 **비활성**. 선택 구간 중 하나라도 안 비면 그 구간 선택이 막힘. |
| 즐겨찾기 토글 | 바텀시트·상세·목록 어디서나 | 옵티미스틱 토글(≤100ms 로컬 피드백). 로그인 사용자별 영속. 비활성 룸은 '비활성' 라벨 + 신규 예약 진입 차단. |
| 인앱 배너 | 접속 시 전역 | 두 종류 — **도래 리마인드**(예약 24h 이내) / **상태변경 통지**(거절·임의취소). '다시 보지 않기' 선택 시 해당 배너 **사용자별 영속 숨김**. 푸시 아님(접속 시점에만). |
| 플로팅 챗봇 "룸메이트" | 전역(모든 주요 화면) | 아이콘 늘 떠 있음(그림자 `{elevation.fab}`). 화면 이동해도 **같은 세션 맥락 유지**. 응답은 토큰 스트리밍. 검색 결과는 상위 3개 + 더보기. 로그아웃 시 대화 초기화. |
| 캘린더 토글 / 지도-목록 전환 | 찾기 | 같은 탐색 도구 안에서 지도↔목록, 반경↔행정동 전환. 상태(선택 지역·반경) 보존. |

## State Patterns (상태 매트릭스)

**필수 매트릭스: 주요 화면 × {빈 상태 / 에러 / 오프라인 / 로딩}. 모든 셀에 설계된 상태가 존재한다(막다른 빈 화면 금지).**

| 화면 | 빈 상태 | 에러 | 오프라인 | 로딩 |
|---|---|---|---|---|
| 찾기 (지도) | 주변에 룸 없음 → "이 근처엔 아직 없어요. 동네를 넓혀볼까요?" + 반경 확대/행정동 전환 버튼 | 지도 로드 실패 → "지도를 못 불러왔어요. 다시 시도" + 목록 보기로 우회 | "오프라인이에요. 연결되면 다시 보여드릴게요." 마지막 핀 캐시 유지 | 지도 스켈레톤 + 핀 placeholder (첫 표시 ≤2초) |
| 찾기 (목록) | 결과 없음 → "이 지역엔 등록된 곳이 없어요." 인접 동/반경 확대 제안 | 조회 실패 → 재시도 액션 | 캐시된 목록 표시 + 오프라인 토스트 | shadcn Skeleton 행 4–6개 |
| 상세 | (해당 없음 — 룸이 있어야 진입) | 정보 로드 실패 → 재시도 / 뒤로 | "오프라인이에요. 예약은 연결 후 가능해요." 읽기는 캐시 | 상세 스켈레톤 |
| 슬롯 피커 | 그날 빈 슬롯 0개 → "이 날은 다 찼어요. 다른 날을 골라보세요." + 다음 빈 날짜 제안 | **동시성 충돌** → "앗, 방금 다른 분이 먼저 잡았어요." + **인접 빈 슬롯 재표시** | 예약 버튼 비활성 + "연결 후 예약할 수 있어요." | 가용성 조회 스피너(비파괴적) |
| 예약현황 | "아직 예약이 없어요. 마음에 드는 곳을 찾아볼까요?" → 찾기로 링크 | 조회 실패 → 재시도 | 캐시된 예약 표시 + 오프라인 토스트 | Skeleton 카드 |
| 즐겨찾기 | "마음에 든 곳을 즐겨찾기해두면 여기 모여요." → 찾기로 링크 | 조회 실패 → 재시도 | 캐시 표시 | Skeleton |
| 챗봇 "룸메이트" | 첫 진입 → 제안 칩("환불 규정?" / "강남 오후 3시 빈 방") | 모델 호출 실패 → "잠깐 답이 막혔어요. 다시 물어봐 주실래요?" + 재전송 | "오프라인이라 지금은 답하기 어려워요." | 첫 토큰 ≤2초, 그 전엔 타이핑 인디케이터 |
| 챗봇 — 범위밖/모름 | 범위 밖 → 정중한 거절 / 범위 내·근거 없음 → "그건 확인이 안 돼요." (환각 금지) | (위 에러와 동일) | (위와 동일) | (위와 동일) |
| 위치 권한 거부 | 지도 대신 **행정동 콤보 목록**으로 즉시 우회(막다른 화면 없음) | — | — | — |
| 제공자 등록 | (입력 폼 — 빈 상태 = 첫 등록 안내) | 저장 실패 → 입력값 보존 + 재시도 | "연결 후 저장돼요." 입력 보존 | 저장 스피너 |

핵심 셀(요구 명시): 지도 로드 실패, 목록 결과없음, 슬롯 동시성 충돌, 챗봇 범위밖/모름, 위치 권한거부 우회 — 모두 위에 설계됨.

## Interaction Primitives (인터랙션 기본기)

- **탭/토글/전환:** 로컬 피드백 **≤100ms**(옵티미스틱). 네트워크 결과는 별도로 도착.
- **드래그:** 연속 슬롯 선택의 1급 제스처(탭 연속도 허용, 비연속 불허).
- **풀/그래버:** 바텀시트를 끌어올리고 내림(카카오맵식). 지도 위로 부드럽게 상승.
- **스트리밍 토큰:** 챗봇 응답은 토큰 단위. **첫 토큰 ≤2초**(웜·기준 프롬프트·기본 프로바이더).
- **모션:** `[ASSUMPTION]` 활기·경쾌 감성에 맞춘 **부드럽되 약간 생기 있는 마이크로모션**(짧은 스프링감, 과하지 않게). 구체 곡선·시간은 `DESIGN.md.Motion` 참조.
- **금지:** 색만으로 상태 표현, 콜드 진입 시 히어로 애니메이션, 흐름을 끊는 모달 2단 중첩, 비연속 슬롯 선택.

## Accessibility Floor (접근성 하한선 — 행동)

시각 대비는 `DESIGN.md`(shadcn WCAG AA 기본값 상속, 브랜드 오버라이드는 비율 유지 검증).

- **키보드 완전 지원(웹):** Tab으로 모든 인터랙티브 요소 도달, Enter로 실행. 슬롯 선택은 **키보드 대안**(방향키 이동 + Enter로 시작/끝 지정) 제공 — 드래그가 유일 경로가 되지 않음.
- **스크린리더:** 모든 인터랙티브 요소에 역할 + 상태 ARIA 라벨. 핀은 "○○ 스터디룸, 예약 가능/마감"으로 읽힘. 챗봇 스트리밍은 `aria-live`로 갱신 안내.
- **색상 의존 금지:** 색 단독은 어디서도 상태 신호가 아니다 — 핀뿐 아니라 **예약 가능 배지·슬롯 상태·인앱 배너**까지 모두 색 + 아이콘/텍스트를 병행한다. (핀 가용성 = `{colors.pin-available}` 초록 + 가능 아이콘 / `{colors.pin-full}` 회색 + 마감 아이콘.)
- **터치 타겟:** 모든 탭 대상(핀·슬롯 셀·토글·FAB·배너 닫기) ≥44×44px(iOS) / 48×48dp(Android).
- **aria-live:** 챗봇 스트리밍뿐 아니라 **인앱 배너(도래/상태변경) 출현**도 aria-live로 안내.
- **아이콘 단독 컨트롤 접근 이름:** 즐겨찾기 하트, 챗봇 FAB, 지도/목록 토글 등 아이콘만 있는 컨트롤에 접근 이름(aria-label) 필수.
- **슬롯 선택 피드백:** 슬롯 선택 시 스크린리더에 선택 구간(예: "14시부터 17시까지 선택됨")을 피드백.
- **(RN) 모달 포커스 격리:** 모달·바텀시트 열릴 때 접근성 포커스 격리(`accessibilityViewIsModal`).
- **prefers-reduced-motion 존중:** 비필수 모션 축소/제거(시각 명세는 `DESIGN.md.Motion`).
- **포커스 관리:** 바텀시트·모달 열릴 때 포커스 진입, Esc로 닫고 트리거로 복귀. 포커스 트랩 적용.
- **탭 순서:** 모든 화면에서 읽기 순서와 일치.

## 시간·타임존 규칙 (제품 고유)

모든 "오늘 / 현재시각 / N시간 전 / 24시간 이내" 판정은 **룸 소재지 타임존 기준**(MVP = Asia/Seoul). slot_start는 절대시각(UTC) 저장.

- 핀 색(오늘 빈 슬롯), 슬롯 활성(당일 현재시각 이전 비활성), 6시간 취소 마감, 24h 도래 배너, 이용 완료 판정 — 전부 이 규약으로 결정적으로 동작한다.
- 영업시간은 **같은 날 내(0~24시, 자정 넘김 없음)** 이라 슬롯의 소속 날짜가 모호하지 않다.
- UI는 사용자에게 항상 룸 기준 시각을 보여준다(타임존 혼동 방지).

## 동시성 & 예약 무결성 UX (제품 고유)

`UNIQUE(room_id, slot_start)` + 단일 트랜잭션 multi-row INSERT 위에서:

- 동일 슬롯 동시 확정 시 **정확히 한쪽만 성공**, 슬롯 점유 중복 0건.
- 겹치되 시작 슬롯이 다른 두 연속 예약(14~17시 vs 16~18시)도 한쪽 전체 성립 / 다른 쪽 **단 하나도 점유 안 함**(부분 점유 0건, all-or-nothing).
- **UI는 실패를 우아하게 안내** — 비난·에러코드 없이 "앗, 방금 다른 분이 먼저 잡았어요." + **인접 빈 슬롯 즉시 재표시**로 흐름 회복.
- 취소·거절·임의취소 시 슬롯은 **동일 트랜잭션 내**에서 즉시 재활성되어 다른 사람에게 다시 보인다.

## 챗봇 "룸메이트" (제품 고유)

스터디룸 컨셉 + "함께 찾아주는 친구" 결의 친근한 해요체 도우미.

- **두 역할:** ① 문서 RAG 서비스 안내, ② 예약 DB 자연어 검색.
- **3분기 응답:** (a) 범위 내 + 근거 있음 → 답변 / (b) 범위 내 + 근거 없음 → "확인 불가/모름"(환각 금지) / (c) 범위 밖 → 정중한 거절.
- **검색 결과:** 상위 3개 + 더보기, 각 항목은 상세 페이지로 링크.
- **멀티 LLM 스위칭(OpenAI/Google/Anthropic)은 사용자에게 투명** — 사용자는 프로바이더를 의식하지 않고, 공통 5종 기능(채팅·시스템프롬프트·2개 툴콜·스트리밍·범위밖 거절)만 동일하게 경험한다.
- **세션:** 사용자×디바이스 단위로 화면 이동 간 맥락 유지, 로그아웃 시 초기화.

## Key Flows (핵심 플로우)

PRD의 UJ 이름을 그대로 따른다. 각 플로우는 번호 스텝 + 명확한 **climax beat**.

### UJ-1 — 총무 "지우"가 강남 모임 장소를 잡아 단톡방에 공유한다 (모바일 앱)

1. 지우가 앱을 연다(로그인 상태). 단톡방에 "다음 주 강남 어디?"가 떴고, 잘못 잡으면 본인이 곤란하다.
2. 찾기에서 목록으로 전환, **시/군/구·동/읍/면 콤보**로 "강남구 역삼동" 선택.
3. 후보 목록에서 가격·시설 비교 → 한 곳 탭 → 상세로 진입.
4. 달력에서 약속 날짜 선택 → **14~17시 연속 슬롯**을 드래그로 선택.
5. 즉시 예약 → "예약이 완료됐어요!" 성공 피드백.
6. **Climax:** 확정 화면에서 **카카오톡 공유 한 번** — 장소·시간이 일행에게 한 번에 전달된다. 총무의 불안이 확신으로 바뀌는 순간.
7. Resolution: 예약현황에 확정 예약이 남고, 24시간 전 접속 시 도래 리마인드 배너.
- **Edge:** 슬롯 선택 직후 다른 사람이 같은 시간을 먼저 확정 → "앗, 방금 다른 분이 먼저 잡았어요." + 인접 빈 슬롯 재표시. 당황 없이 옆 시간으로 회복.

### UJ-2 — 개인 "민재"가 지금 당장 근처 빈 방을 혼자 잡는다 (지도)

1. 민재가 앱을 막 열었다(위치 권한 허용). 지금부터 2시간 집중하고 싶다.
2. 첫 화면 지도에 주변 룸이 핀으로 뜸 → **초록 핀(오늘 빈 슬롯 있음)** 하나를 탭.
3. 바텀시트에서 가격·영업시간·가용성 확인 → 상세로 진입.
4. 오늘 날짜의 빈 2개 연속 슬롯 선택 → 즉시 예약.
5. **Climax:** **상세 진입부터 3스텝 이내**로 예약 완료 + 성공 피드백. 조율할 일행이 없으니 속도가 곧 만족.
6. Resolution: 예약현황에 오늘 예약 표시, 6시간 전까지 취소 가능.
- **Edge:** 위치 권한 거부 → 지도 대신 **행정동 콤보 목록**으로 즉시 우회. "위치를 못 받았어요. 동네를 골라서 찾아볼게요." 막다른 화면 없음.

### UJ-3 — 제공자 "현우"가 공간을 등록해 지도에 올리고 예약을 받는다

1. 현우가 제공자 계정으로 로그인(첫 등록).
2. 공간 정보 입력 — 이름·시간당 금액·영업시간·주소·수용 인원·룸 형태(개방형/독립룸)·부대시설(다중 선택).
3. 주소 검색으로 지도 위치 확정(좌표 + 행정동 코드 동시 저장).
4. 저장 → 지도/목록에 노출.
5. **Climax:** **5분 이내** 등록을 마치고 지도에서 자기 핀을 확인한다. 여러 창구 수동 홍보 대신 등록 한 번으로 손님 앞에 선다.
6. Resolution: 예약현황에서 들어온 예약 확인(표시 이름·슬롯만). 불가피하면 예약 시작 전까지 거절(슬롯 재활성 + 예약자에게 상태변경 통지 배너). 후기에는 답글로 응대.

### UJ-4 — 누구든 막히면 챗봇 "룸메이트"에게 묻는다

1. 어느 화면이든 플로팅 챗봇 아이콘이 떠 있다 → 탭.
2. 질문 — *"환불 규정이 어떻게 돼?"*(문서 RAG) 또는 *"강남 근처 오후 3시 빈 방 추천해줘"*(예약 DB 검색).
3. **첫 응답 토큰 2초 이내** 스트리밍 시작.
4. **Climax:** 검색이면 **상위 3개 + 더보기**로 결과가 뜨고, 링크로 상세 페이지로 바로 이동 — 화면을 떠나지 않고 답에서 행동으로 이어진다.
5. Resolution: 대화는 로그아웃 전까지 유지(화면 이동해도 맥락 유지). 범위 밖 잡담은 정중히 거절, 범위 내라도 근거 없으면 "확인 불가"로 안내(환각 금지).
